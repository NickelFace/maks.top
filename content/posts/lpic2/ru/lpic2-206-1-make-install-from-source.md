> **Тема экзамена 206.1** — Сборка и установка программ из исходного кода (вес: 2). Охватывает полный цикл: скачивание, конфигурирование, компиляция и установка программы из исходных текстов.

---

## Зачем собирать из исходников

Иногда нужная программа отсутствует в репозитории дистрибутива. Или требуется проверить код перед установкой, включить нестандартные опции, собрать под конкретную архитектуру. Пакетный менеджер вроде `apt` или `yum` автоматически решает зависимости, но при ручной сборке всё приходится делать самостоятельно.

> **Внимание:** При установке из исходников нет менеджера зависимостей. Все необходимые библиотеки и компиляторы нужно установить вручную до начала сборки.

Сначала установите базовые инструменты разработки:

```bash
# Red Hat / CentOS / Fedora
yum groupinstall "Development Tools"

# Debian / Ubuntu
apt-get install build-essential
```

## Где хранить исходный код

По стандарту GNU исходный код хранится в `/usr/src`. Это принятое место для архивов перед распаковкой. В процессе сборки можно работать в любом каталоге (например, `/tmp` или домашнем), но `/usr/src` — расположение по умолчанию.

```
/usr/src/
    curl-7.46.0/       <- распакованное дерево исходников
    curl-7.46.0.tar.gz <- оригинальный архив
```

> **Примечание:** Для экзамена — `/usr/src` является каталогом хранения исходного кода по умолчанию.

## Распаковка архивов

Исходный код чаще всего распространяется в виде сжатого tar-архива. Три основных формата сжатия: `gzip` (.tar.gz / .tgz), `bzip2` (.tar.bz2) и `xz` (.tar.xz).

### Распаковка с помощью GNU tar (рекомендуется)

```bash
# gzip (.tar.gz или .tgz)
tar zxvf archive.tar.gz

# bzip2 (.tar.bz2)
tar jxvf archive.tar.bz2

# xz (.tar.xz)
tar Jxvf archive.tar.xz
```

Флаги: `z` = gzip, `j` = bzip2, `J` = xz, `x` = extract (распаковать), `v` = verbose (подробный вывод), `f` = filename (имя файла).

### Двухэтапная распаковка (для старого tar без поддержки сжатия)

```bash
# Шаг 1: распаковать сжатие
gunzip archive.tar.gz      # или: gzip -d archive.tar.gz
bunzip2 archive.tar.bz2    # или: bzip2 -d archive.tar.bz2
unxz archive.tar.xz        # или: xz -d archive.tar.xz

# Шаг 2: извлечь tar
tar xvf archive.tar
```

### Создание архива

```bash
tar czf archive.tar.gz ./mydir/
```

Флаг `c` = create (создать), в отличие от `x` = extract (распаковать).

> **Совет:** Сохраняйте вывод `configure` и `make` в журнал с помощью `tee`:
> ```bash
> ./configure | tee ../configure.log
> make | tee ../make.log
> ```

## Файлы документации перед сборкой

Перед запуском `configure` прочитайте файлы в каталоге с исходниками. Автор программы может использовать нестандартный процесс установки.

| Файл | Назначение |
|---|---|
| `README` | Общая документация и инструкции по установке |
| `INSTALL` | Подробные инструкции по установке |
| `COPYING` | Лицензия программы |
| `RELEASE-NOTES` | Изменения в текущей версии |
| `NEWS` | Новые функции и исправления |
| `AUTHORS` | Авторы и контакты |

> **Примечание:** Не все файлы присутствуют в каждом проекте. Имена также могут отличаться.

## configure: что это и зачем нужен

`configure` — это shell-скрипт, проверяющий окружение перед компиляцией. Он проверяет наличие компилятора, необходимых библиотек и опциональных зависимостей, затем генерирует `Makefile`.

Скрипт создаётся инструментами GNU autoconf/automake. `automake` создаёт `Makefile.in`, а `autoconf` превращает его в итоговый скрипт `configure`.

```bash
./configure           # запуск с настройками по умолчанию
./configure --help    # просмотр всех доступных параметров
```

Если `configure` не может найти обязательную зависимость, он выводит ошибку и останавливается. Отсутствие опциональных зависимостей просто отключает соответствующие функции.

### Что делает configure

1. Проверяет наличие компилятора (обычно `gcc`)
2. Ищет обязательные и опциональные библиотеки
3. Определяет архитектуру через `uname` (запускает скрипт `config.guess`)
4. Генерирует `Makefile` из `Makefile.in`
5. Адаптирует код под целевую систему

---

## Параметры configure

```bash
./configure                      # установка в /usr/local (по умолчанию)
./configure --prefix=/opt        # установка в /opt
./configure --prefix=/opt/sue    # установка в /opt/sue

# Сборка для развёртывания на другой машине (DESTDIR)
./configure --prefix=/opt/sue
make
make install DESTDIR=/tmp/sue_tmp
# Все файлы программы теперь в /tmp/sue_tmp/opt/sue/
```

> **Примечание:** `--prefix` задаёт путь установки в итоговой системе. `DESTDIR` при `make install` указывает make, куда класть файлы прямо сейчас. Это разные вещи: `DESTDIR` используется для сборки пакетов или развёртывания на нескольких машинах.

Полный цикл передачи на другую машину:

```bash
# На сборочной машине
tar zxvf sue-2.0.3.tar.gz
cd sue-2.0.3
./configure --prefix=/opt/sue
make
make install DESTDIR=/tmp/sue_tmp
cd /tmp
tar czf sue-installed.tar.gz sue_tmp/

# Скопировать архив на целевые машины, затем:
tar zxvf sue-installed.tar.gz -C /
```

## make и цели сборки

`make` читает `Makefile` и компилирует программу. Без аргументов запускает цель `all` (собрать всё).

```bash
make           # компиляция (цель: all)
make install   # установка скомпилированной программы
make clean     # удаление объектных файлов после сборки
make distclean # удаление всего, включая Makefile (для полной пересборки)
```

| Цель make | Что делает |
|---|---|
| `all` | Компиляция (цель по умолчанию) |
| `install` | Копирование бинарных файлов в места назначения |
| `clean` | Удаление объектных файлов (.o) из дерева исходников |
| `distclean` | Удаление всего сгенерированного, включая Makefile |
| `uninstall` | Удаление установленной программы (если реализовано) |

> **Совет:** `make install` требует прав root — бинарные файлы помещаются в системные каталоги.

## Утилита install

`install` копирует файлы в нужное место с правильными правами доступа и владельцем. В отличие от `cp`, устанавливает права и владельца/группу за один шаг.

```bash
# Установить бинарный файл с правами 755 в /usr/local/bin
install -m 755 mybinary /usr/local/bin/

# Установить с явным указанием владельца и группы
install -m 755 -o root -g root mybinary /usr/local/bin/

# Создать резервную копию существующего файла перед перезаписью
install -b -m 755 mybinary /usr/local/bin/

# Создать каталог, если он не существует
install -d /usr/local/myapp/
```

| Флаг | Описание |
|---|---|
| `-m mode` | Установить права (например, 755) |
| `-o owner` | Установить владельца |
| `-g group` | Установить группу |
| `-b` | Создать резервную копию существующего файла |
| `-d` | Создать каталог |

> **Примечание:** Поведение резервного копирования с `-b` управляется переменной окружения `VERSION_CONTROL`. Значения: `numbered`, `existing`, `simple`, `none`.

## uname

`uname` выводит информацию о системе. `configure` использует его через скрипт `config.guess` для определения архитектуры при кросс-компиляции.

```bash
uname
# Linux

uname -a
# Linux myhost 5.15.0-91-generic #101-Ubuntu SMP x86_64 GNU/Linux
```

| Флаг | Вывод |
|---|---|
| `-a` | Вся информация |
| `-s` | Имя ядра (Linux) |
| `-n` | Сетевое имя хоста |
| `-r` | Версия выпуска ядра |
| `-v` | Версия (дата сборки) |
| `-m` | Аппаратная архитектура (x86_64) |
| `-p` | Тип процессора |
| `-o` | Операционная система (GNU/Linux) |

## patch

`patch` применяет diff-файл к оригинальному файлу. Файл патча создаётся командой `diff` между старой и новой версиями файла.

```bash
patch -p1 < patchfile.patch       # применить патч
patch -R -p1 < patchfile.patch    # откатить патч
patch -s -p1 < patchfile.patch    # тихий режим, только ошибки
```

Флаг `-p` указывает, сколько начальных компонентов пути нужно удалить из имён файлов в патче. `-p0` оставляет путь как есть, `-p1` удаляет первый компонент.

```bash
# Пример: путь в файле патча /home/user/src/prog/main.c
patch -p0 < fix.patch   # ищет /home/user/src/prog/main.c
patch -p1 < fix.patch   # ищет home/user/src/prog/main.c
patch -p4 < fix.patch   # ищет main.c
```

Если `patch` не может применить часть патча, отклонённые фрагменты сохраняются в файл с расширением `.rej`.

> **Внимание:** `patch -R` предполагает, что старый и новый файлы поменяны местами. Используйте это для возврата к исходному состоянию.

## Полный цикл сборки

Стандартный процесс установки программы из исходников:

```bash
# 1. Перейти в /usr/src и распаковать архив
cd /usr/src
tar zxvf sue-2.0.3.tar.gz

# 2. Перейти в каталог с исходниками
cd sue-2.0.3

# 3. Прочитать README и INSTALL

# 4. Запустить configure
./configure

# 5. Скомпилировать
make

# 6. Установить (от root)
sudo make install
```

Установка в нестандартный путь:

```bash
./configure --prefix=/opt
make
sudo make install
```

Или с использованием `install` вручную:

```bash
make
sudo install -m 755 -o root -g root ./sue /usr/local/bin/sue
```

## Шпаргалка для экзамена

### Пути

| Путь | Назначение |
|---|---|
| `/usr/src` | Хранение исходного кода по умолчанию |
| `/usr/local/bin` | Бинарные файлы, установленные из исходников (по умолчанию с `--prefix=/usr/local`) |
| `Makefile` | Генерируется configure, используется make |
| `Makefile.in` | Шаблон для генерации Makefile |
| `config.guess` | Скрипт для определения архитектуры системы |

### Команды и утилиты

| Команда | Назначение |
|---|---|
| `gzip` / `gunzip` | Сжатие / распаковка .gz |
| `bzip2` / `bunzip2` | Сжатие / распаковка .bz2 |
| `xz` / `unxz` | Сжатие / распаковка .xz |
| `tar zxvf` | Распаковать .tar.gz |
| `tar jxvf` | Распаковать .tar.bz2 |
| `tar Jxvf` | Распаковать .tar.xz |
| `./configure` | Проверка окружения и генерация Makefile |
| `./configure --prefix=PATH` | Установить путь установки |
| `make` | Компиляция (цель: all) |
| `make install` | Установка скомпилированной программы |
| `make clean` | Удаление объектных файлов |
| `install -m 755` | Копирование файла с правами |
| `uname -a` | Вся системная информация |
| `uname -r` | Версия ядра |
| `patch -p1 < file` | Применить патч |
| `patch -R -p1 < file` | Откатить патч |

### Типичные ошибки

- Запуск `make install` без прав root: ошибка доступа.
- Забыть об зависимостях: `configure` завершается с ошибкой.
- Перепутать `-p0` и `-p1` в `patch`: файл не найден.
- Запустить `make` до `./configure`: нет Makefile, команда завершится с ошибкой.
- Установка без `--prefix`: программа попадёт в `/usr/local`, возможен конфликт с пакетным менеджером.

## Практические вопросы

**В1.** Какой флаг `tar` используется для распаковки архива `.tar.bz2`?

**Ответ:** `-j`. Полная команда: `tar jxvf archive.tar.bz2`.

---

**В2.** Где по умолчанию хранится исходный код в системе Linux?

**Ответ:** `/usr/src`.

---

**В3.** Что делает `./configure --prefix=/opt`?

**Ответ:** Устанавливает `/opt` как корневой путь установки. Все файлы программы попадут в `/opt/bin`, `/opt/lib` и т.д.

---

**В4.** Какой файл создаёт `configure` и зачем он нужен?

**Ответ:** `Makefile`. Он читается командой `make` для компиляции и установки программы.

---

**В5.** В чём разница между `make clean` и `make distclean`?

**Ответ:** `make clean` удаляет только объектные файлы (.o). `make distclean` удаляет всё, сгенерированное configure и make, включая сам `Makefile`, возвращая дерево исходников в исходное состояние.

---

**В6.** Как применить патч и удалить первый компонент пути из файла патча?

**Ответ:** `patch -p1 < fix.patch`.

---

**В7.** Почему утилита `install` лучше `cp` для установки бинарных файлов?

**Ответ:** `install` устанавливает права доступа и владельца/группу за один шаг. С флагом `-b` создаёт резервную копию существующего файла перед перезаписью.

---

**В8.** Зачем `configure` вызывает `uname`?

**Ответ:** Для определения архитектуры системы через скрипт `config.guess`. На основе этой информации configure адаптирует код и флаги компилятора под целевую платформу.
