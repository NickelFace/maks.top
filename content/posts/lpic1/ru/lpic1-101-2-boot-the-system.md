## Последовательность загрузки

Загрузка Linux проходит через несколько последовательных этапов. Знайте их порядок — экзамен часто проверяет именно это.

```
Включение питания
      |
      v
BIOS / UEFI  (POST, инициализация оборудования)
      |
      v
Загрузчик (GRUB 2)
      |
      v
Ядро Linux (vmlinuz)
      |
      v
initramfs (временная корневая файловая система)
      |
      v
init / systemd (PID 1)
      |
      v
Уровни запуска / цели
```

## BIOS и UEFI

### BIOS (Basic Input/Output System)

BIOS хранится в энергонезависимом чипе на материнской плате и запускается при каждом включении машины. Это прошивка — физически отдельная от обычных устройств хранения.

Точная последовательность загрузки BIOS:

1. POST (Power-On Self Test) проверяет оборудование сразу при включении питания
2. BIOS активирует основные компоненты: видеовыход, клавиатуру, хранилище
3. BIOS считывает **первые 440 байт** первого устройства хранения (согласно настройкам порядка загрузки) — это загрузчик первой ступени
4. Загрузчик первой ступени вызывает вторую ступень, которая показывает меню и загружает ядро

О MBR: первые **512 байт** диска, использующего схему разделов DOS, называются MBR (Master Boot Record). MBR содержит загрузчик первой ступени (первые 440 байт) и таблицу разделов. Повреждённый MBR означает, что система не загрузится.

Примечание для экзамена: BIOS **инициирует** процесс загрузки после включения питания, и некоторые параметры загрузки настраиваются через него. BIOS **не** загружается с жёсткого диска — он встроен в чип на материнской плате.

### UEFI (Unified Extensible Firmware Interface)

UEFI также является прошивкой, но работает принципиально иначе. Ключевые отличия:

- Может читать таблицы разделов и работать с файловыми системами напрямую
- **Не использует MBR** — читает настройки из NVRAM на материнской плате
- NVRAM хранит пути к EFI-приложениям, которые запускаются автоматически или из меню загрузки
- Поддерживает Secure Boot: запускает только подписанные EFI-приложения, авторизованные производителем

EFI-приложения (загрузчик, диагностика, выбор ОС) хранятся на **EFI System Partition (ESP)**. ESP не разделяется с корневой файловой системой или пользовательскими данными. Совместимые файловые системы для ESP: FAT12, FAT16, FAT32 для блочных устройств; ISO-9660 для оптических носителей.

Точная последовательность загрузки UEFI:

1. POST проверяет оборудование
2. UEFI активирует основные компоненты: видеовыход, клавиатуру, хранилище
3. UEFI читает NVRAM и запускает указанное EFI-приложение с ESP — обычно загрузчик
4. Загрузчик загружает ядро и запускает операционную систему

ESP обычно монтируется в `/boot/efi`. На ней хранится `.efi`-файл, например `grubx64.efi`. Каталог `/EFI` на ESP содержит приложения, на которые ссылаются записи NVRAM.

Примечание для экзамена: на системе с UEFI загрузчик хранится **на ESP**, а не в MBR. UEFI **может** читать таблицы разделов и работать с файловыми системами FAT. Конфигурация UEFI хранится в NVRAM, а не в `/boot`.

## GRUB 2

GRUB (Grand Unified Bootloader) — наиболее распространённый загрузчик для Linux на x86. BIOS или UEFI передаёт управление GRUB, который затем показывает список доступных операционных систем.

Если меню не появляется автоматически: на системах с BIOS удерживайте `Shift` во время загрузки GRUB; на системах с UEFI нажмите `Esc`.

### Ключевые файлы

```
/etc/default/grub            — пользовательские настройки (редактировать здесь)
/etc/grub.d/                 — скрипты для генерации grub.cfg
/boot/grub/grub.cfg          — итоговая конфигурация (не редактировать напрямую)
/boot/grub/i386-pc/          — модули GRUB для систем с BIOS (например, lvm.mod)
```

`grub.cfg` нужно регенерировать каждый раз при изменении `/etc/default/grub`:

```bash
grub-mkconfig -o /boot/grub/grub.cfg
```

На Debian/Ubuntu есть удобный псевдоним:

```bash
update-grub
```

### Установка GRUB

```bash
# Установить GRUB в MBR первого диска
grub-install /dev/sda

# Установить в MBR третьего диска
grub-install /dev/sdc
```

### Передача параметров ядра через GRUB

В меню GRUB нажмите `e` для редактирования записи. Найдите строку, начинающуюся с `linux`, и добавьте параметры в конец. Например:

```
linux /boot/vmlinuz-5.15.0 root=/dev/sda1 ro quiet splash
```

Нажмите `Ctrl+X` или `F10` для загрузки с изменёнными параметрами. Изменения **временные** — они не сохраняются после перезагрузки.

Чтобы сделать параметры постоянными, добавьте их в `/etc/default/grub` в строке `GRUB_CMDLINE_LINUX`, затем регенерируйте конфигурацию с помощью `grub-mkconfig`.

### Параметры ядра

Большинство параметров используют формат `option=value`. Полный список из целей экзамена LPI:

| Параметр | Описание |
|---|---|
| `acpi=off` | Отключить поддержку ACPI |
| `init=/bin/bash` | Установить альтернативный инициализатор (например, запустить bash вместо init) |
| `systemd.unit=graphical.target` | Установить цель systemd при загрузке |
| `1` или `S` | Загрузиться в уровень запуска 1 (однопользовательский режим) |
| `mem=512M` | Ограничить доступную оперативную память (полезно в виртуальных машинах) |
| `maxcpus=2` | Ограничить видимые ядра CPU |
| `maxcpus=0` | Отключить поддержку SMP (аналогично `nosmp`) |
| `quiet` | Подавить большинство сообщений загрузки |
| `vga=ask` | Показать список доступных видеорежимов для выбора |
| `root=/dev/sda3` | Задать корневой раздел, отличный от указанного в конфигурации загрузчика |
| `rootflags=` | Параметры монтирования корневой файловой системы |
| `ro` | Смонтировать корень только для чтения при начальной загрузке |
| `rw` | Разрешить запись в корневую файловую систему при начальной загрузке |

Параметр `init=/bin/bash` полезен для восстановления: bash запускается сразу после ядра, минуя init и все службы. Обратите внимание, что корневая файловая система монтируется только для чтения, поэтому сразу выполните `mount -o remount,rw /`.

## Ядро и initramfs

### Ядро Linux

Ядро загружается как сжатый файл, обычно `vmlinuz` или `vmlinuz-<версия>`, расположенный в `/boot/`. После распаковки ядро инициализирует оборудование, монтирует initramfs и запускает первый процесс.

Параметры, переданные ядру при загрузке, доступны в работающей системе:

```bash
cat /proc/cmdline
```

### initramfs

initramfs (initial RAM filesystem) — временная корневая файловая система в оперативной памяти. Ядро монтирует её до переключения на реальный корневой раздел.

Почему она нужна: реальный корневой раздел может находиться на LVM, RAID, зашифрованном устройстве или в сети. Для монтирования всего этого нужны драйверы, которых у ядра ещё нет. initramfs содержит минимальный набор инструментов и модулей для поиска и монтирования реального корня.

Важные факты для экзамена:
- initramfs — это **сжатый архив cpio** — его можно распаковать и изучить
- ядро использует initramfs **временно**, затем переключается на реальный корень
- после успешной загрузки содержимое initramfs доступно в `/run/initramfs/`
- initramfs **привязан к конкретной версии ядра**

Просмотр содержимого initramfs:

```bash
# Для формата cpio
lsinitramfs /boot/initrd.img-$(uname -r)

# Вручную
mkdir /tmp/initramfs_test
cd /tmp/initramfs_test
zcat /boot/initrd.img-$(uname -r) | cpio -idmv
```

## Системы инициализации

После завершения работы ядро распаковывает initramfs, монтирует файловые системы из `/etc/fstab`, затем запускает первую программу — `init`. Этот процесс `init` (PID 1) запускает все скрипты инициализации и системные демоны. Как только init запущен, initramfs освобождается из памяти.

Скрипты инициализации выполняют короткие задачи и завершаются. Демоны (службы) работают непрерывно. Любая система инициализации как минимум умеет запускать, останавливать и перезапускать службу.

Исторически в Linux использовались три различные системы инициализации.

## SysVinit

SysVinit происходит из Unix System V и всё ещё встречается на устаревших системах — именно поэтому экзамен его охватывает.

SysVinit управляет демонами через концепцию **уровней запуска** с номерами от 0 до 6, определяемых дистрибутивом. Единственные уровни с одинаковым значением во всех дистрибутивах — 0, 1 и 6.

Первый процесс: `/sbin/init` с PID 1.

### Уровни запуска

| Уровень запуска | Описание |
|----------|-------------|
| 0 | Останов |
| 1 | Однопользовательский режим |
| 2 | Многопользовательский без NFS (Debian) |
| 3 | Многопользовательский с сетью |
| 4 | Не используется (определяется пользователем) |
| 5 | Многопользовательский с GUI |
| 6 | Перезагрузка |

### Скрипты инициализации

```
/etc/inittab            — основная конфигурация (определяет уровень запуска по умолчанию)
/etc/rc.d/              — каталоги rc-скриптов (RHEL)
/etc/init.d/            — скрипты служб
/etc/rc0.d/ ... /etc/rc6.d/  — символические ссылки на скрипты для каждого уровня запуска
```

Скрипты в `/etc/rcN.d/` именуются: `S20apache2` (S = Start, 20 = порядок) или `K80apache2` (K = Kill). Число задаёт порядок запуска или остановки.

```bash
# Переключить уровень запуска
telinit 3

# Показать текущий уровень запуска
runlevel

# Управление службой
/etc/init.d/apache2 start
/etc/init.d/apache2 stop
```

## Upstart

Upstart был разработан Canonical для Ubuntu. Его главной целью был более быстрый запуск за счёт параллельного старта служб. Ubuntu использовала Upstart в более старых выпусках перед переходом на systemd. Для экзамена: знайте, что Upstart существовал и чем отличался от SysVinit.

Upstart использовал событийную модель: службы запускались и останавливались в ответ на события (например, «появился сетевой интерфейс»). Конфигурационные файлы хранились в `/etc/init/` с расширением `.conf`. Управление службами осуществлялось с помощью `initctl`.

## systemd

systemd — современный менеджер системы и служб с уровнем совместимости с SysV. Из учебных материалов LPI: systemd имеет параллельную архитектуру, использует сокеты и D-Bus для активации служб, поддерживает запуск демонов по требованию, мониторинг процессов через cgroups, снимки состояния, восстановление сессий, управление точками монтирования и обработку зависимостей. Большинство крупных дистрибутивов перешли на systemd в последние годы.

Все основные современные дистрибутивы используют systemd: Ubuntu 15.04+, Debian 8+, RHEL/CentOS 7+, Arch, Fedora.

Первый процесс: `/sbin/init` (символическая ссылка на `/lib/systemd/systemd`) с PID 1.

### Цели

Цели в systemd заменяют уровни запуска SysVinit.

| Цель | Эквивалент уровня запуска | Описание |
|--------|---------------------|-------------|
| poweroff.target | 0 | Выключение |
| rescue.target | 1 | Однопользовательский режим |
| multi-user.target | 3 | Многопользовательский без GUI |
| graphical.target | 5 | Многопользовательский с GUI |
| reboot.target | 6 | Перезагрузка |
| emergency.target | — | Аварийная оболочка |

### Основные команды

```bash
# Установить цель по умолчанию
systemctl set-default multi-user.target

# Немедленно переключиться на цель
systemctl isolate rescue.target

# Показать текущую цель по умолчанию
systemctl get-default

# Управление службами
systemctl start nginx
systemctl stop nginx
systemctl restart nginx
systemctl enable nginx      # запускать при загрузке
systemctl disable nginx     # убрать из автозапуска
systemctl status nginx

# Показать все зависимости цели
systemctl list-dependencies multi-user.target
```

### Юниты

systemd работает с юнитами разных типов:
- `.service` — службы
- `.target` — группы служб
- `.mount` — точки монтирования
- `.socket` — сокеты для активации через сокет
- `.timer` — замена cron

Системные юниты: `/lib/systemd/system/`
Пользовательские и переопределения: `/etc/systemd/system/`

```bash
# Список всех юнитов
systemctl list-units

# Список всех целей
systemctl list-units --type=target
```

## Журналы загрузки

Ошибки загрузки не всегда критичны, но могут влиять на поведение системы. Все они фиксируются в журналах с отметками времени. Даже без ошибок журналы загрузки полезны для настройки и диагностики.

### dmesg

Ядро хранит свои сообщения, включая сообщения загрузки, в **кольцевом буфере**. Сообщения остаются там, даже если вместо них показывалась анимация загрузки. Буфер очищается при выключении и при явном вызове `dmesg --clear`.

Числа в начале каждой строки — секунды с момента начала загрузки ядра.

```bash
# Показать весь буфер
dmesg

# С понятными для человека отметками времени
dmesg -T

# Только ошибки и критические сообщения
dmesg --level=err,crit

# Очистить буфер
dmesg --clear

# Следить за новыми сообщениями в реальном времени
dmesg -w

# Вывод с постраничным просмотром (встроенный пейджер, без pipe)
dmesg -H
```

Буфер очищается только явным `dmesg --clear` и при выключении или перезагрузке. Простое чтение с помощью `dmesg` его **не** очищает.

### journalctl

На системах с systemd `journalctl` показывает сообщения инициализации. Журнал systemd не хранится в виде обычного текста, поэтому требуется `journalctl` — `cat` не подойдёт.

```bash
# Текущая загрузка (оба варианта эквивалентны)
journalctl -b
journalctl -b 0

# Предыдущая загрузка
journalctl -b -1

# Список всех сохранённых загрузок с хэшами и отметками времени
journalctl --list-boots

# Только сообщения ядра (как dmesg)
journalctl -k
journalctl --dmesg

# Только ошибки
journalctl -p err

# Следить в реальном времени
journalctl -f

# Конкретная служба
journalctl -u nginx.service

# За последние 30 минут
journalctl --since "30 min ago"

# Читать журнал из нестандартного каталога
journalctl -D /path/to/directory
journalctl --directory=/path/to/directory
```

По умолчанию журнал systemd хранится в `/var/log/journal/`. Если постоянное хранилище не настроено, он хранится в `/run/log/journal/` (ОЗУ, не переживает перезагрузку).

Флаги `-b`, `--boot`, `-k` и `--dmesg` показывают сообщения загрузки. Флаги `-D` и `--directory` позволяют читать журналы из другой файловой системы, например после загрузки со спасательного диска.

### Файлы в /var/log/

```
/var/log/boot.log      — сообщения загрузки
/var/log/messages      — общий системный журнал (RHEL)
/var/log/syslog        — общий системный журнал (Debian/Ubuntu)
/var/log/dmesg         — вывод dmesg, сохранённый при загрузке
```

## Типовые вопросы экзамена

**В: Что верно относительно последовательности загрузки BIOS?**
BIOS инициирует процесс загрузки после включения питания, и некоторые параметры загрузки настраиваются через него. BIOS **не** загружается с жёсткого диска — он встроен в чип.

**В: Что верно относительно UEFI?**
UEFI может читать таблицы разделов и работать с определёнными файловыми системами. Конфигурация хранится в NVRAM, а не на разделе `/boot`.

**В: Где хранится загрузчик на системе с UEFI?**
На EFI System Partition (ESP). Не в MBR, не в EBR.

**В: Что содержит EFI System Partition?**
Загрузчик первой ступени (файл `.efi`-приложения).

**В: Когда очищается кольцевой буфер ядра?**
При явном `dmesg --clear` и при перезагрузке или выключении. Чтение через `dmesg` его не очищает.

**В: Какой первый процесс запускается ядром в SysVinit?**
`/sbin/init` с PID 1.

**В: Как установить цель systemd по умолчанию?**
`systemctl set-default multi-user.target`

**В: Что верно относительно initramfs?**
initramfs — это сжатый архив cpio, который можно распаковать и изучить. Ядро использует его временно, затем переключается на реальный корень. После загрузки содержимое доступно в `/run/initramfs/`.

**В: Как установить GRUB в MBR третьего жёсткого диска?**
`grub-install /dev/sdc`

**В: Что находится в файловой системе /boot/?**
Образы ядра Linux и образы начального RAM-диска (initramfs). Юниты systemd там не хранятся.

На машине с BIOS: где находится исходный двоичный файл начального загрузчика?

### Упражнение 2 — Расположение EFI-приложений

UEFI поддерживает внешние программы, называемые EFI-приложениями. Где эти приложения хранятся в системе?

<details>
<summary>Ответ</summary>

На EFI System Partition (ESP), на любом доступном блочном устройстве с совместимой файловой системой — обычно FAT32.

</details>

### Упражнение 3 — Передача параметра root ядру

Загрузчики позволяют передавать параметры ядру до его загрузки. Система не загружается из-за неправильно указанного расположения корневой файловой системы. Как передать ядру правильный корневой раздел `/dev/sda3`?

<details>
<summary>Ответ</summary>

Используйте параметр `root`:

```
root=/dev/sda3
```

В меню GRUB нажмите `e`, найдите строку, начинающуюся с `linux`, и добавьте параметр в конец. Чтобы сохранить изменения после перезагрузки, добавьте его в `GRUB_CMDLINE_LINUX` в `/etc/default/grub`, затем выполните:

```bash
grub-mkconfig -o /boot/grub/grub.cfg
```

</details>

### Упражнение 4 — Устройство не найдено при загрузке

Процесс загрузки завершается с:

```
ALERT! /dev/sda3 does not exist. Dropping to a shell!
```

Какова вероятная причина?

<details>
<summary>Ответ</summary>

Ядро не смогло найти устройство `/dev/sda3`, указанное как корневая файловая система.

Чаще всего это происходит, когда в initramfs отсутствует модуль для контроллера диска или файловой системы. Второй вариант — неправильный параметр `root=` в конфигурации загрузчика.

</details>

### Упражнение 5 — UEFI и перезапись MBR

Загрузчик показывает список операционных систем, когда установлено несколько. Однако новая ОС может перезаписать MBR, уничтожив загрузчик первой ступени и сделав другие ОС недоступными. Почему это не происходит на машине с UEFI?

<details>
<summary>Ответ</summary>

Машины с UEFI не используют MBR диска для хранения загрузчика первой ступени.

Каждая ОС помещает свой `.efi`-файл в собственный каталог на ESP (например, `/EFI/ubuntu/` или `/EFI/windows/`) и добавляет собственную запись в NVRAM. Другие записи не затрагиваются, поэтому все установленные системы остаются доступными.

</details>

---

### Упражнение 6 — Новое ядро без initramfs

Каковы типичные последствия установки нового ядра без соответствующего образа initramfs?

<details>
<summary>Ответ</summary>

Корневая файловая система может оказаться недоступной, если её тип скомпилирован как внешний модуль ядра. Без initramfs этот модуль не загрузится и ядро не сможет смонтировать корень.

Это особенно актуально, когда корень находится на LVM, RAID или зашифрованном разделе. Система перейдёт в аварийную оболочку с сообщением вроде `ALERT! /dev/... does not exist`.

</details>

### Упражнение 7 — Постраничный вывод dmesg

Вывод `dmesg` занимает сотни строк и часто передаётся пейджеру через pipe, например `less`. Какой флаг `dmesg` автоматически пагинирует вывод без явного pipe?

<details>
<summary>Ответ</summary>

```bash
dmesg -H
```

Или в полной форме:

```bash
dmesg --human
```

Флаг `-H` включает режим, понятный человеку, который автоматически запускает встроенный пейджер.

</details>

### Упражнение 8 — Чтение журналов с другого диска

Жёсткий диск с полной файловой системой выключенной машины был извлечён и подключён к рабочей машине как второй диск, смонтированный в `/mnt/hd`. Как использовать `journalctl` для чтения журналов из `/mnt/hd/var/log/journal/`?

<details>
<summary>Ответ</summary>

```bash
journalctl -D /mnt/hd/var/log/journal
```

Или в полной форме:

```bash
journalctl --directory=/mnt/hd/var/log/journal
```

Флаг `-D` / `--directory` указывает `journalctl` читать журналы из произвольного каталога вместо стандартного `/var/log/journal/`. Журнал systemd хранится в бинарном формате, поэтому `cat` не подходит для его чтения.

</details>
