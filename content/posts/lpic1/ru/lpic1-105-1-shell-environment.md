## Что нужно знать

- Устанавливать переменные окружения (`PATH`) при входе и при запуске новой оболочки.
- Писать Bash-функции для часто повторяющихся команд и хранить их в startup-файлах.
- Поддерживать skeleton-каталоги для новых учётных записей.
- Настраивать путь поиска команд через корректный каталог.

Файлы и утилиты: `.`, `source`, `/etc/bash.bashrc`, `/etc/profile`, `env`, `export`, `set`, `unset`, `~/.bash_profile`, `~/.bash_login`, `~/.profile`, `~/.bashrc`, `~/.bash_logout`, `function`, `alias`.

## Типы оболочек Bash

### Интерактивная и неинтерактивная

**Интерактивная** оболочка читает команды с терминала и показывает приглашение. Запускается при открытии терминала, входе через консоль или подключении по SSH.

**Неинтерактивная** оболочка стартует для выполнения скрипта или одиночной команды без участия пользователя: `bash script.sh`, задание cron.

### Login и non-login

**Login shell** поднимается при входе в систему: через текстовую консоль, SSH, `su -`, или явно через `bash -l` (`--login`). Загружает полный набор startup-файлов и настраивает окружение с нуля.

**Non-login shell** открывается уже после входа: новый терминал в графической среде, простой вызов `bash`, запуск скрипта. Считает окружение унаследованным от родительской сессии и читает только bashrc-файлы.

### Четыре комбинации

**Интерактивная login:** вход через текстовую консоль, SSH, `su -`, `sudo -i`. Место для пользовательских настроек при старте сессии.

**Интерактивная non-login:** новый терминал в графической среде, простой вызов `bash` после входа, оболочка из IDE. Самый частый случай в работе.

**Неинтерактивная login:** редкая комбинация. Возникает при принудительном запуске скрипта как login (`bash --login script.sh`) или при передаче команды в SSH через канал.

**Неинтерактивная non-login:** автоматические скрипты, cron-задания. Bash не читает ни одного startup-файла — нужные переменные экспортируют из родителя или задают прямо в скрипте.

### Терминалы pts и tty

В графической сессии работают эмуляторы терминалов через псевдотерминалы **pts**: GNOME Terminal, Konsole, XTerm.

Текстовые консоли ядра доступны по `Ctrl + Alt + F1`...`F6` — это настоящие **tty**. `Ctrl + Alt + F7` возвращает в графическую сессию.

```bash
$ ps aux | grep bash
user  5270  ... pts/0  ... bash      # эмулятор в GUI, non-login
user  5411  ... tty1   ... -bash     # консоль, login
```

### Как определить тип оболочки

| Вывод `echo $0` | Тип оболочки |
|---|---|
| `-bash` или `-su` | Интерактивная login |
| `bash` или `/bin/bash` | Интерактивная non-login |
| Имя скрипта | Неинтерактивная |

```bash
$ shopt login_shell
login_shell    off     # off = non-login, on = login
```

## Способы запуска оболочки

### Опции команды bash

| Опция | Действие |
|---|---|
| `-l` / `--login` | Login-оболочка; читает `/etc/profile` и user login-файлы |
| `-i` | Интерактивная оболочка |
| `--noprofile` | Пропустить `/etc/profile` и пользовательские login-файлы |
| `--norc` | Пропустить `/etc/bash.bashrc` и `~/.bashrc` |
| `--rcfile ФАЙЛ` | Использовать ФАЙЛ вместо стандартных rc-файлов |

`bash --noprofile --norc` запускает полностью чистую оболочку для отладки.

### Команда su

С загрузкой окружения целевого пользователя (interactive login):

```bash
$ su - user2
$ su -l user2
$ su --login user2
$ su -           # стать root в login-сессии
```

Без загрузки окружения (interactive non-login):

```bash
$ su user2
$ su             # стать root в non-login сессии
```

Дефис, `-l` и `--login` равнозначны. Без флага входа `su` оставляет окружение исходного пользователя — переменные `HOME` и `PATH` остаются от него, что чревато путаницей.

### Команда sudo

С загрузкой окружения (login):

```bash
$ sudo -i                     # root в login-оболочке
$ sudo -i команда             # выполнить в login-окружении и вернуться
$ sudo su - user2             # login user2 через цепочку
$ sudo su -                   # login root через цепочку
```

Без загрузки окружения (non-login):

```bash
$ sudo -s                     # non-login root
$ sudo -u root -s             # то же явно
$ sudo -u user2 -s            # non-login user2
$ sudo su user2               # то же через su
```

## Файлы инициализации Bash

### Глобальные файлы

`/etc/profile` — главный глобальный файл для login-оболочек. Задаёт `PATH`, `PS1`. Подгружает `/etc/bash.bashrc` и скрипты из `/etc/profile.d/*.sh`.

`/etc/bash.bashrc` — глобальный файл для интерактивных оболочек. Для non-login читается напрямую, для login — через `/etc/profile`. На Red Hat-подобных системах аналог — `/etc/bashrc`.

`/etc/profile.d/*.sh` — каталог для пакетных и пользовательских добавлений. Подхватывается из `/etc/profile`.

### Пользовательские файлы

`~/.bash_profile` — личный login-файл. Приоритет выше `~/.bash_login` и `~/.profile`.

`~/.bash_login` — альтернативный login-файл. Читается только при отсутствии `~/.bash_profile`.

`~/.profile` — универсальный login-файл. Читается, если нет ни `~/.bash_profile`, ни `~/.bash_login`. По умолчанию добавляет `~/bin` в `PATH` и подгружает `~/.bashrc`.

`~/.bashrc` — файл для интерактивных non-login оболочек. Хранит псевдонимы, функции, настройки истории. Часто подгружается из `~/.bash_profile` для login-сессий.

`~/.bash_logout` — выполняется при выходе из login-оболочки.

### Порядок чтения файлов

**Login-оболочка:**

1. `/etc/profile` → подгружает `/etc/bash.bashrc` + `/etc/profile.d/*.sh`
2. Первый найденный из: `~/.bash_profile`, `~/.bash_login`, `~/.profile` → стандартный `~/.profile` подгружает `~/.bashrc`

На Debian login-оболочка фактически читает все четыре файла через цепочку sourcing.

**Интерактивная non-login оболочка:**

1. `/etc/bash.bashrc` (если включён)
2. `~/.bashrc`

**Неинтерактивная оболочка:** не читает startup-файлов. Использует окружение родителя. Исключение: если задан `BASH_ENV`, Bash читает файл по этому пути перед выполнением скрипта.

Локальные файлы перекрывают глобальные — `~/.bash_profile` читается после `/etc/profile` и побеждает.

**Практика:** переменные `PATH` и глобальное окружение → `~/.bash_profile`. Псевдонимы и функции для каждого терминала → `~/.bashrc`. Чтобы login-сессия тоже их видела, добавь `source ~/.bashrc` в `~/.bash_profile`.

## Переменные окружения

### Локальные и экспортированные

Локальная переменная видна только в текущей оболочке. Экспортированная наследуется дочерними процессами.

```bash
$ MYVAR=hello          # локальная переменная
$ export MYVAR         # теперь экспортирована
$ export GREETING=hi   # одна строка делает оба шага
```

Пробелы вокруг `=` недопустимы — Bash попытается выполнить имя как команду.

### env, set, export, unset

| Команда | Назначение |
|---|---|
| `env` | Показать экспортированные переменные; `env VAR=val cmd` — запустить с изменённым окружением; `env -i cmd` — с пустым |
| `printenv` | Значение конкретной переменной: `printenv PATH` |
| `set` | Все переменные и функции, включая неэкспортированные; `set -x` — трассировка команд |
| `export` | Пометить переменную как экспортируемую; `export -f` — экспортировать функцию |
| `unset` | Удалить переменную или функцию; `-f` — функцию, `-v` — переменную |

На экзамене важно: `set` показывает всё, `env` — только то, что унаследуют дочерние процессы.

### Настройка PATH

`PATH` — список каталогов через двоеточие. Bash ищет в них исполняемые файлы.

```bash
$ export PATH="$HOME/bin:$PATH"       # добавить в начало (приоритет)
$ export PATH="$PATH:$HOME/bin"       # добавить в конец
```

Постоянно: вписать в `~/.bashrc` или `~/.bash_profile`. Для всех: `/etc/profile` или файл в `/etc/profile.d/`.

`unset PATH` лишает оболочку поиска команд — всё нужно вызывать по абсолютному пути.

## Псевдонимы alias

```bash
$ alias ll='ls -lah'
$ alias rm='rm -i'      # запрашивать подтверждение
$ alias                 # список всех
$ alias ll              # показать определение
$ unalias ll            # удалить один
$ unalias -a            # удалить все
```

Псевдоним работает только в начале команды. Аргументы передавать нельзя — для этого функции.

Обойти псевдоним:

```bash
$ \rm файл.txt          # обратный слеш отменяет alias
$ command rm файл.txt   # builtin command игнорирует псевдонимы
$ /bin/rm файл.txt      # абсолютный путь
```

Псевдонимы не наследуются дочерними оболочками и не работают в неинтерактивных без `shopt -s expand_aliases`. Для постоянного использования — в `~/.bashrc`.

## Функции Bash

Два равнозначных синтаксиса:

```bash
function greet {
    echo "Привет, $1"
}

greet() {
    echo "Привет, $1"
}
```

Использование: `greet Маркс` → `Привет, Маркс`.

```bash
$ declare -f greet     # показать определение
$ type greet           # тип объекта
$ unset -f greet       # удалить функцию
$ export -f greet      # экспортировать в дочерние процессы
```

`$1`, `$2`, ... — позиционные аргументы. `$@` — все аргументы, `$#` — их количество. `return N` (0–255) задаёт код возврата.

## Команды source и точка

`source файл` и `. файл` равнозначны: читают файл и выполняют его в **текущей** оболочке без порождения дочернего процесса.

Запуск через `./файл.sh` создаёт дочернюю оболочку — изменения теряются при её завершении.

```bash
$ source ~/.bashrc     # применить изменения без перелогина
$ . ~/.bashrc          # то же самое
```

## Skeleton-каталог /etc/skel

`/etc/skel` — шаблоны файлов, которые копируются в домашний каталог при создании нового пользователя.

```bash
$ ls -la /etc/skel
.bash_logout  .bashrc  .profile
```

Добавить файл для всех будущих пользователей:

```bash
# echo 'date' > /etc/skel/.bash_login
```

На существующих пользователей не влияет — только на создаваемых после.

Альтернативный каталог: `useradd -m -k /etc/skel-developer user`.

Настройки skel в Debian:
- `/etc/default/useradd` → для `useradd`
- `/etc/adduser.conf` → для `adduser` (параметр `SKEL=`, фильтр `SKEL_IGNORE_REGEX`)

Без флага `-m` команда `useradd` не создаёт домашний каталог и не копирует из skel. `adduser` всегда создаёт.

## Шпаргалка

```bash
# определение типа оболочки
echo $0                      # -bash = login, bash = non-login
shopt login_shell            # on = login, off = non-login

# опции запуска bash
bash -l / bash --login       # login-оболочка
bash -i                      # интерактивная
bash --noprofile             # без login startup-файлов
bash --norc                  # без rc-файлов
bash --rcfile ФАЙЛ           # свой rc-файл

# su — с окружением (login)
su - user / su -l user / su --login user
su -                         # root в login

# su — без окружения (non-login)
su user / su

# sudo — с окружением (login)
sudo -i / sudo -i команда / sudo su - user / sudo su -

# sudo — без окружения (non-login)
sudo -s / sudo -u user -s / sudo su user

# переменные
env                          # экспортированные
env -i command               # пустое окружение
printenv VAR                 # значение одной переменной
set                          # все переменные и функции
export VAR=value             # создать и экспортировать
export -f func               # экспортировать функцию
unset VAR / unset -f func

# псевдонимы
alias name='cmd'
alias / unalias name / unalias -a

# функции
function name { ... } / name() { ... }
declare -f name / type name / unset -f name / export -f name

# применение конфигурации
source файл / . файл

# PATH
export PATH="$PATH:/новый"   # добавить в конец
export PATH="/новый:$PATH"   # добавить в начало

# создание пользователя
useradd -m name              # с домашним каталогом из /etc/skel
useradd -m -k /шаблон name  # альтернативный skel
useradd -D                   # показать значения по умолчанию
adduser name                 # Debian-обёртка
grep SKEL /etc/adduser.conf
grep SKEL /etc/default/useradd

# BASH_ENV для скриптов
export BASH_ENV=/path/init.sh
```

## Типичные вопросы экзамена

1. Порядок чтения init-файлов login-оболочкой → `/etc/profile`, затем первый из `~/.bash_profile`, `~/.bash_login`, `~/.profile`.
2. Что читает non-login интерактивная оболочка → `/etc/bash.bashrc` и `~/.bashrc`.
3. Что читает неинтерактивная оболочка → Никаких startup-файлов; окружение из родителя или из скрипта.
4. Вывод `echo $0` для login/non-login → Login: `-bash`. Non-login: `bash` или `/bin/bash`. Скрипт: имя файла.
5. Что означает дефис в `-bash` → Это login-оболочка.
6. Разница `su user` и `su - user` → Без дефиса: non-login без загрузки окружения. С дефисом: login с чтением всех startup-файлов.
7. Разница `sudo -i` и `sudo -s` → `-i`: login-оболочка root с полной инициализацией. `-s`: non-login, окружение от исходного пользователя.
8. Что делает `bash --noprofile` → Login-оболочка без `/etc/profile` и пользовательских login-файлов.
9. Что делает `bash --norc` → Интерактивная оболочка без `/etc/bash.bashrc` и `~/.bashrc`.
10. Разница `source файл` и `bash файл` → `source` выполняет в текущей оболочке; `bash` — в дочерней, изменения теряются.
11. Разница `set` и `env` → `set` показывает всё включая локальные; `env` — только экспортированные.
12. Где определять псевдонимы и функции постоянно → `~/.bashrc` для одного; `/etc/bash.bashrc` или `/etc/profile.d/` для всех.
13. Как сделать файл для каждого нового пользователя → Положить в `/etc/skel`. Существующих не затрагивает.
14. Что произойдёт после `unset PATH` → Оболочка теряет поиск команд; нужен абсолютный путь.
15. Как экспортировать функцию → `export -f имя_функции`.
16. Эквивалент `.` (точка) → `source`.
17. Файл при выходе из login-оболочки → `~/.bash_logout`.
18. Связь `-m` у `useradd` со skel → Без `-m` каталог не создаётся и skel не копируется.
19. Как обойти псевдоним → `\rm`, `command rm` или `/bin/rm`.
20. Назначение `BASH_ENV` → Путь к startup-файлу для неинтерактивных скриптов.
21. Разница pts и tty → `pts`: псевдотерминал GUI. `tty`: текстовая консоль ядра (`Ctrl+Alt+F1`...`F6`).
22. Файл настроек skel для `adduser` в Debian → `/etc/adduser.conf`.
23. Файл настроек skel для `useradd` → `/etc/default/useradd`.

| Интерактивная login как user2 | Да | Да | Да | Да |
| Интерактивная login как root | Да | Да | Нет | Нет |
| Интерактивная non-login как root | Нет | Да | Нет | Нет |
| Интерактивная non-login как user2 | Нет | Да | Нет | Да |

Пояснения:
- Строки root составлены из расчёта, что в `/root/` нет `.profile` и `.bashrc`. Если администратор положит их туда, они тоже будут читаться.
- `/etc/bash.bashrc` применяется к любой интерактивной оболочке: для non-login напрямую, для login через цепочку `/etc/profile`.
- `~/.profile` — login-файл; non-login его не читает.
- `~/.bashrc` для login user2 читается косвенно: стандартный `~/.profile` содержит вызов `source ~/.bashrc`.

</details>

### Упражнение 4 — Функция hello world

Дан файл с функцией:

```bash
function hello() {
    echo "Hello world!"
}
```

**а)** Что нужно сделать, чтобы функция стала доступна текущей оболочке?

**б)** Как вызвать функцию после подгрузки?

**в)** В какой файл положить функцию, чтобы она выполнялась при каждом открытии терминала из X Window сессии пользователем user2? Какой тип оболочки?

**г)** В какой файл положить функцию, чтобы root получал её в любой интерактивной оболочке?

<details>

```bash
$ source файл_с_функцией
$ . файл_с_функцией
```

**б)** Ввести имя функции:

```bash
$ hello
Hello world!
```

**в)** `/home/user2/.bashrc`. Тип оболочки: интерактивная non-login (эмулятор терминала в X Window открывает non-login оболочку).

**г)** `/etc/bash.bashrc`. Этот файл читается для всех интерактивных оболочек: для non-login напрямую, для login через цепочку `/etc/profile`.

</details>

### Упражнение 5 — Интерактивность скрипта

Дан скрипт:

```bash
#!/bin/bash
echo "Hello world!"
```

**а)** Скрипт сделан исполняемым и запущен. Будет ли он интерактивным и почему?

**б)** Что делает скрипт интерактивным?

<details>
<summary>Ответ</summary>

**а)** Нет. В скрипте нет ввода от пользователя — команды не печатаются человеком в реальном времени. Оболочка выполняет скрипт от начала до конца без ожидания.

**б)** Необходимость пользовательского ввода. Если скрипт ждёт ответа через `read`, он становится интерактивным:

```bash
#!/bin/bash
read -p "Введите имя: " name
echo "Привет, $name"
```

</details>

### Упражнение 6 — Применение изменений ~/.bashrc

После правки переменных в `~/.bashrc` нужно применить изменения без перезапуска терминала. Из домашнего каталога это делается двумя способами.

<details>
<summary>Ответ</summary>

```bash
$ source .bashrc
```

или

```bash
$ . .bashrc
```

Обе команды читают файл и выполняют его в текущей оболочке. Без них новая конфигурация подхватится только при следующем открытии оболочки.

</details>

### Упражнение 8 — ~/.bash_login для всех новых пользователей

Linda попросила создать `~/.bash_login` для вывода даты при входе. Другим пользователям идея понравилась. Администратор хочет автоматизировать создание этого файла для всех будущих пользователей.

**а)** Что нужно сделать?

**б)** Затронет ли это существующих пользователей?

<details>
<summary>Ответ</summary>

**а)** Положить файл `.bash_login` в `/etc/skel`. При создании нового пользователя через `useradd -m` или `adduser` файл автоматически скопируется в его домашний каталог:

```bash
# echo 'date' > /etc/skel/.bash_login
```

**б)** Нет. Правка `/etc/skel` касается только тех, кто будет создан после. Для уже существующих пользователей файл нужно копировать вручную.

Важно: Bash читает login-файлы по приоритету `~/.bash_profile`, `~/.bash_login`, `~/.profile`. Если у пользователей уже есть `~/.bash_profile`, то `~/.bash_login` будет проигнорирован.

</details>
