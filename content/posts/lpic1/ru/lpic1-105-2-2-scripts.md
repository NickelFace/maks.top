## Что нужно знать

Из официальных objectives курса LPIC-1:

- Применять стандартный синтаксис sh: циклы, проверки.
- Использовать подстановку команд.
- Проверять коды возврата команд для определения успеха или ошибки.
- Соединять команды в цепочки.
- Отправлять условные уведомления администратору root.
- Правильно выбирать интерпретатор скрипта через строку шебанга `#!`.
- Управлять расположением, владельцем, правами на запуск и SUID скриптов.

Файлы, термины и утилиты из objectives: `for`, `while`, `test`, `if`, `read`, `seq`, `exec`, `||`, `&&`.

## Шебанг и выбор интерпретатора

Первая строка скрипта начинается с символов `#!` (шебанг), за которыми идёт абсолютный путь к интерпретатору. Ядро читает эту строку при запуске файла и передаёт его указанной программе. Без шебанга скрипт всё равно запустится, но через текущую оболочку.

Типичные варианты:

```bash
#!/bin/bash       # Bash, наиболее распространённый выбор
#!/bin/sh         # POSIX-совместимая оболочка (на разных системах разные реализации)
#!/usr/bin/env python3   # вызов Python через env, удобно для переносимости
```

Скрипты Bash используют builtin команды самой оболочки и связку с утилитами командной строки. Полная мощность скриптов раскрывается именно в комбинации встроенных команд с утилитами Linux.

## Базовый синтаксис sh

### Переменные

Присвоение, без пробелов вокруг знака равно:

```bash
NAME="Alice"
COUNT=42
```

Обращение через `$NAME` или `${NAME}`. Двойные кавычки сохраняют подстановки переменных, одинарные подавляют их. Это критично при работе с тестами и условиями.

### Параметры скрипта

Внутри скрипта аргументы доступны как позиционные параметры:

`$0`: имя самого скрипта.

`$1`, `$2`, `$3` ...: первый, второй, третий аргумент.

`$#`: количество переданных аргументов.

`$@`: все аргументы как отдельные слова.

`$*`: все аргументы как одна строка.

`$?`: код выхода последней выполненной команды.

`$$`: PID текущего процесса.

`$!`: PID последнего запущенного фонового процесса.

## Подстановка команд

Вывод команды можно подставить в строку или присвоить переменной двумя способами:

```bash
TODAY=$(date +%Y-%m-%d)
TODAY=`date +%Y-%m-%d`
```

Современный синтаксис `$(...)` предпочтительнее: его можно вкладывать друг в друга, обратные кавычки этого не позволяют.

## Возвращаемые значения и команда test

### Код выхода

Каждая команда после завершения возвращает целое число. Ноль означает успех, любое другое значение, ошибку. Код доступен в переменной `$?` сразу после выполнения команды:

```bash
ls /etc
echo $?       # 0
ls /несуществующий
echo $?       # 2
```

Скрипт может явно вернуть свой код через `exit N`, где N это число от 0 до 255.

### Две формы записи теста

Bash builtin команда `test` имеет много опций для оценки свойств объектов файловой системы. У теста две эквивалентные формы записи: либо аргументом команде `test`, либо в квадратных скобках, где `test` подразумевается. Обе формы дают одинаковый результат:

```bash
$ test -d /etc
$ echo $?
0
$ [ -d /etc ]
$ echo $?
0
```

Внутри квадратных скобок обязательны пробелы после `[` и перед `]`. Это синтаксическая особенность, которую часто забывают.

### Проверки файлов и каталогов

Полный набор файловых тестов. Везде ниже считаем, что путь к файлу хранится в переменной `$VAR`:

`-a "$VAR"`: путь существует и это файл.

`-b "$VAR"`: путь это блочное специальное устройство.

`-c "$VAR"`: путь это символьное специальное устройство.

`-d "$VAR"`: путь это каталог.

`-e "$VAR"`: путь существует в файловой системе.

`-f "$VAR"`: путь существует и это обычный файл.

`-g "$VAR"`: путь имеет SGID-право.

`-h "$VAR"`: путь это символическая ссылка.

`-L "$VAR"`: путь это символическая ссылка (синоним `-h`).

`-k "$VAR"`: путь имеет sticky bit.

`-p "$VAR"`: путь это именованный канал (pipe).

`-r "$VAR"`: путь читаем текущим пользователем.

`-s "$VAR"`: путь существует и не пустой.

`-S "$VAR"`: путь это сокет.

`-t "$VAR"`: путь открыт в терминале.

`-u "$VAR"`: путь имеет SUID-право.

`-w "$VAR"`: путь записываем текущим пользователем.

`-x "$VAR"`: путь исполняем текущим пользователем.

`-O "$VAR"`: путь принадлежит текущему пользователю.

`-G "$VAR"`: путь принадлежит эффективной группе текущего пользователя.

`-N "$VAR"`: путь изменялся с момента последнего обращения.

Двойные кавычки вокруг тестируемой переменной нужны не для красоты. Если переменная окажется пустой, без кавычек получится синтаксическая ошибка теста.

### Сравнение файлов между собой

`"$VAR1" -nt "$VAR2"`: путь в VAR1 новее, чем в VAR2 (по времени модификации).

`"$VAR1" -ot "$VAR2"`: путь в VAR1 старше, чем в VAR2.

`"$VAR1" -ef "$VAR2"`: путь в VAR1 это hardlink на VAR2 (тот же inode).

### Сравнение строк

`-z "$TXT"`: переменная пустая (нулевая длина).

`-n "$TXT"` или просто `test "$TXT"`: переменная не пустая.

`"$TXT1" = "$TXT2"` или `"$TXT1" == "$TXT2"`: строки равны.

`"$TXT1" != "$TXT2"`: строки не равны.

`"$TXT1" < "$TXT2"`: TXT1 идёт раньше TXT2 в алфавитном порядке.

`"$TXT1" > "$TXT2"`: TXT1 идёт позже TXT2 в алфавитном порядке.

В разных языках правила алфавитного порядка отличаются. Чтобы получить устойчивый результат независимо от настроек локали системы, перед операцией сравнения строк ставят:

```bash
LANG=C
```

### Сравнение чисел

`$NUM1 -lt $NUM2`: меньше (less than).

`$NUM1 -gt $NUM2`: больше (greater than).

`$NUM1 -le $NUM2`: меньше или равно (less or equal).

`$NUM1 -ge $NUM2`: больше или равно (greater or equal).

`$NUM1 -eq $NUM2`: равно (equal).

`$NUM1 -ne $NUM2`: не равно (not equal).

Численные тесты работают только с целыми числами и только с этими буквенными операторами. Если использовать `<`, `>`, `=` для чисел, Bash сравнит их как строки. Поэтому `test 1 > 2` возвращает истину: символ "1" в строковом сравнении идёт раньше "2", и оператор `>` дополнительно интерпретируется как перенаправление вывода в файл с именем `2`.

### Модификаторы и логические операторы

`! EXPR`: истина, если выражение EXPR ложно.

`EXPR1 -a EXPR2`: оба выражения истинны (логическое И).

`EXPR1 -o EXPR2`: хотя бы одно выражение истинно (логическое ИЛИ).

Пример:

```bash
if [ ! -d "$FROM" -o ! -d "$TO" ]
then
    echo "Один из каталогов не существует"
    exit 1
fi
```

## Условные конструкции

### if then elif else

```bash
if УСЛОВИЕ
then
    КОМАНДЫ
elif ДРУГОЕ_УСЛОВИЕ
then
    КОМАНДЫ
else
    КОМАНДЫ
fi
```

Условием служит любая команда, проверяется её код выхода. Конструкция закрывается ключевым словом `fi`.

### case

Условную развилку с большим количеством вариантов компактнее писать через `case`. Пример скрипта, определяющего формат пакета по имени дистрибутива:

```bash
#!/bin/bash
DISTRO=$1
echo -n "Distribution $DISTRO uses "
case "$DISTRO" in
  debian | ubuntu | mint)
    echo -n "the DEB"
    ;;
  centos | fedora | opensuse )
    echo -n "the RPM"
    ;;
  *)
    echo -n "an unknown"
    ;;
esac
echo " package format."
```

Каждый список шаблонов и связанных команд завершается одним из терминаторов: `;;`, `;&` или `;;&`. Звёздочка в последнем шаблоне срабатывает, если ни один из предыдущих вариантов не подошёл. Конструкция закрывается `esac`.

Вывод при запуске:

```
$ ./script.sh opensuse
Distribution opensuse uses the RPM package format.
```

### Опция nocasematch

У Bash есть опция `nocasematch`, которая включает сопоставление шаблонов без учёта регистра в `case` и других условных командах:

```bash
shopt -s nocasematch    # включить
shopt -u nocasematch    # выключить
```

Изменения через `shopt` действуют только в текущем сеансе. Скрипты обычно запускаются в субшелле, поэтому `shopt` внутри скрипта не повлияет на опции родительской сессии.

## Циклы for, while, until

Скрипты часто автоматизируют повторяющиеся задачи. У Bash три инструкции цикла: `for`, `until` и `while`.

### for

`for` проходит по заданному списку элементов, выполняя для каждого один и тот же набор команд. Синтаксис:

```bash
for VARNAME in LIST
do
    КОМАНДЫ
done
```

VARNAME это произвольное имя переменной, LIST это последовательность разделённых элементов. Разделители определяются переменной `IFS`, по умолчанию пробел, табуляция и перевод строки.

Пример с проверкой чётности:

```bash
#!/bin/bash
for NUM in 1 1 2 3 5 8 13
do
    echo -n "$NUM is "
    if [ $(( $NUM % 2 )) -ne 0 ]
    then
        echo "odd."
    else
        echo "even."
    fi
done
```

Вывод:

```
1 is odd.
1 is odd.
2 is even.
3 is odd.
5 is odd.
8 is even.
13 is odd.
```

### Стиль for из языка C

Bash поддерживает альтернативный формат `for` с двойными скобками, удобный для работы с массивами:

```bash
#!/bin/bash
SEQ=( 1 1 2 3 5 8 13 )
for (( IDX = 0; IDX < ${#SEQ[*]}; IDX++ ))
do
    echo -n "${SEQ[$IDX]} is "
    if [ $(( ${SEQ[$IDX]} % 2 )) -ne 0 ]
    then
        echo "odd."
    else
        echo "even."
    fi
done
```

### until

`until` выполняет команды, пока тестовая команда возвращает ненулевой статус. Как только тест вернёт 0, цикл останавливается. Тот же пример с массивом:

```bash
#!/bin/bash
SEQ=( 1 1 2 3 5 8 13 )
IDX=0
until [ $IDX -eq ${#SEQ[*]} ]
do
    echo -n "${SEQ[$IDX]} is "
    if [ $(( ${SEQ[$IDX]} % 2 )) -ne 0 ]
    then
        echo "odd."
    else
        echo "even."
    fi
    IDX=$(( $IDX + 1 ))
done
```

Важно включать в тело действия, гарантирующие достижение условия остановки (инкремент счётчика), иначе цикл будет крутиться бесконечно.

### while

`while` похож на `until`, но повторяет команды, пока тест возвращает 0 (успех). Условие `until [ $IDX -eq ${#SEQ[*]} ]` из примера выше эквивалентно `while [ $IDX -lt ${#SEQ[*]} ]`.

### Подробный пример: sync.sh

Пользователь хочет периодически синхронизировать набор своих файлов и каталогов с внешним накопителем. Список держим отдельным файлом `~/.sync.list`, по одному элементу на строку:

```
Documents
To do
Work
Family Album
.config
.ssh
.bash_profile
.vimrc
```

В файле смесь файлов и каталогов, некоторые с пробелами в именах. Это удобно обрабатывать через builtin Bash команду `mapfile`, которая разбирает любой текст и создаёт массив, помещая каждую строку как отдельный элемент.

Сам скрипт `sync.sh`:

```bash
#!/bin/bash
set -ef

# Список элементов для синхронизации
FILE=~/.sync.list

# Исходный каталог
FROM=$1

# Каталог назначения
TO=$2

# Проверка корректности обоих каталогов
if [ ! -d "$FROM" -o ! -d "$TO" ]
then
    echo Usage:
    echo "$0 <SOURCEDIR> <DESTDIR>"
    exit 1
fi

# Создание массива из файла
mapfile -t LIST < $FILE

# Синхронизация элементов
for (( IDX = 0; IDX < ${#LIST[*]}; IDX++ ))
do
    echo -e "$FROM/${LIST[$IDX]} \u2192 $TO/${LIST[$IDX]}";
    rsync -qa --delete "$FROM/${LIST[$IDX]}" "$TO";
done
```

Первое действие, `set -ef`, переопределяет два параметра оболочки. Опция `-e` прерывает выполнение, если команда вернёт ненулевой статус, а `-f` отключает раскрытие имён файлов через wildcards.

Прикладные инструкции делятся на три части:

**Сбор и проверка параметров.** `FILE` содержит путь к списку. `FROM` и `TO` это исходный и целевой пути от пользователя. Если хоть один из путей не каталог, скрипт печатает краткую справку и завершается с кодом 1.

**Загрузка списка.** Команда `mapfile -t LIST < $FILE` создаёт массив `LIST` из строк файла `~/.sync.list`. Опция `-t` убирает завершающий перевод строки из каждой строки.

**Копирование и информирование пользователя.** Цикл `for` со скобочной нотацией обходит массив. `echo` сообщает о каждом копируемом элементе. В сообщении используется экранированный unicode-символ `\u2192` (стрелка вправо), поэтому нужна опция `-e` команды `echo`. `rsync` копирует только изменённые куски файлов. Опции `-qa` подавляют сообщения и включают архивный режим. Опция `--delete` удаляет в пункте назначения то, чего больше нет в источнике.

Запуск:

```
$ sync.sh /home/carol /media/carol/backup
/home/carol/Documents → /media/carol/backup/Documents
/home/carol/"To do" → /media/carol/backup/"To do"
/home/carol/Work → /media/carol/backup/Work
/home/carol/"Family Album" → /media/carol/backup/"Family Album"
/home/carol/.config → /media/carol/backup/.config
/home/carol/.ssh → /media/carol/backup/.ssh
/home/carol/.bash_profile → /media/carol/backup/.bash_profile
/home/carol/.vimrc → /media/carol/backup/.vimrc
```

## Команды read, seq, exec

### read

`read` читает строку из стандартного ввода и присваивает её переменной (или нескольким):

```bash
read -p "Имя: " NAME
read -s PASSWORD          # тихий ввод (скрытый, для паролей)
read -t 5 ANSWER          # таймаут 5 секунд
read FIRST SECOND REST    # три переменные, разделение по IFS
```

### seq

`seq` печатает последовательность чисел. Удобно для циклов:

```bash
seq 5            # 1 2 3 4 5
seq 2 2 10       # 2 4 6 8 10 (шаг 2)
seq -w 1 10      # с ведущими нулями: 01 02 ... 10
```

Применение в `for`:

```bash
for i in $(seq 1 10); do echo "Шаг $i"; done
```

### exec

`exec` заменяет текущий процесс оболочки указанной командой. Никакого нового процесса, никакого возврата:

```bash
exec /usr/bin/python3 myscript.py    # текущий bash превращается в python
```

Без команды `exec` используется для перенаправления потоков всего скрипта:

```bash
exec > /tmp/log.txt 2>&1     # весь дальнейший вывод скрипта пойдёт в файл
```

## Цепочки команд

Несколько команд в одной строке с разными правилами связи:

`;` (точка с запятой): выполнить команды по порядку независимо от результата.

`&&` (логическое И): выполнить вторую команду только если первая успешна (код 0).

`||` (логическое ИЛИ): выполнить вторую команду только если первая упала (код не 0).

Примеры:

```bash
make && make install                    # установить только при успешной сборке
test -f /tmp/lock || touch /tmp/lock    # создать lock-файл, если его нет
mkdir backup && cp -r data backup/      # скопировать только в созданный каталог
```

Цепочки заменяют простой `if`:

```bash
[ -d /backup ] && rsync -a /data/ /backup/
```

## Условная отправка почты администратору

Из objectives: уметь условно отправлять почту суперпользователю. Типичный сценарий, скрипт мониторинга. Команды `mail` или `mailx`:

```bash
df -h | grep -E '9[0-9]%|100%' && \
    df -h | mail -s "Disk usage critical on $(hostname)" root
```

Или с `if`:

```bash
if ! systemctl is-active --quiet apache2
then
    echo "Apache упал в $(date)" | mail -s "ALERT: apache2 down" root
fi
```

В реальности на сервере должна работать MTA (postfix, sendmail, exim), иначе письмо никуда не уйдёт.

## Расположение и права скриптов

Скрипты обычно складывают в каталоги:

`~/bin` или `~/.local/bin`: личные скрипты пользователя.

`/usr/local/bin`: скрипты, доступные всем пользователям (рекомендуется для системных).

`/usr/local/sbin`: административные скрипты, требующие прав root.

`/etc/cron.daily/`, `/etc/cron.hourly/`: скрипты, запускаемые cron по расписанию.

Чтобы скрипт можно было запустить, на нём должно быть право исполнения:

```bash
chmod +x myscript.sh         # для всех
chmod 755 myscript.sh        # rwxr-xr-x
chmod 700 myscript.sh        # только владельцу
```

Запуск по короткому имени работает только если каталог скрипта в `PATH`. Иначе указывают полный или относительный путь.

### SUID на скриптах

SUID на shell-скрипте чаще всего игнорируется ядром Linux ради безопасности (гонки состояния при интерпретации файла). Если действительно нужно дать обычному пользователю выполнение чего-то от root, варианты:

Настроить sudo через `/etc/sudoers` для конкретной команды без пароля.

Написать обёртку на C, которая вызовет скрипт.

Использовать `setcap` для конкретных capabilities.

## Полезные команды для экзамена

`test EXPR` или `[ EXPR ]`: проверить выражение, вернуть код 0 или 1.

`echo $?`: посмотреть код выхода последней команды.

`exit N`: завершить скрипт с кодом N.

`set -e`: прервать скрипт на любой ошибке.

`set -f`: отключить раскрытие шаблонов имён файлов.

`set -x`: трассировка, печатать каждую команду перед выполнением.

`shopt -s OPT` / `shopt -u OPT`: включить или выключить опцию оболочки.

`shopt -s nocasematch`: case-insensitive сопоставление в `case` и `[[ ]]`.

`mapfile -t ARR < file`: загрузить файл в массив, по строке на элемент.

`read VAR`: прочитать строку в переменную.

`seq N`: вывести числа от 1 до N.

`exec CMD`: заменить процесс оболочки командой.

`source FILE` или `. FILE`: выполнить файл в текущей оболочке (без субшелла).

## Типичные вопросы экзамена

Какой код выхода означает успех? → Ноль.

Чем `==` в `test` отличается от `-eq`? → `==` сравнивает строки, `-eq` сравнивает целые числа.

Как проверить, что переменная пустая? → `[ -z "$VAR" ]`.

Как проверить, что путь это каталог? → `[ -d "$PATH" ]`.

Как проверить, что один файл новее другого? → `[ "$A" -nt "$B" ]`.

Что такое шебанг и где он стоит? → Строка `#!интерпретатор` в самом начале скрипта.

Как сравнить две строки без учёта регистра в `case`? → `shopt -s nocasematch` перед конструкцией.

Как закрыть `case`? → Ключевым словом `esac`.

Как закрыть `if`? → Ключевым словом `fi`.

Чем `&&` отличается от `;`? → `&&` запускает следующую команду только при успехе предыдущей, `;` запускает в любом случае.

Почему `test 1 > 2` возвращает истину? → Оператор `>` предназначен для строкового сравнения, а не числового. Плюс сам `>` интерпретируется как перенаправление вывода в файл `2`.

Как обеспечить устойчивый алфавитный порядок независимо от локали? → Установить `LANG=C` перед операцией.

Что делает `mapfile -t LIST < file`? → Читает файл и создаёт массив `LIST`, по одной строке на элемент, без завершающих переводов строки.

Какую опцию `set` используют для немедленного выхода при ошибке? → `-e`.

Действуют ли изменения `shopt` внутри скрипта на родительскую оболочку? → Нет, скрипт обычно запускается в субшелле.

## Упражнения

### Практическое упражнение 1 — test для проверки, что один файл новее другого

Как командой `test` проверить, что путь к файлу в переменной `FROM` новее, чем путь к файлу в переменной `TO`?

<details>
<summary>Ответ</summary>

```bash
test "$FROM" -nt "$TO"
```

Вернёт код 0, если файл из `FROM` новее файла из `TO`. Здесь `-nt` это сокращение от newer than.

> Двойные кавычки вокруг переменных нужны на случай, если одна из них окажется пустой: без них тест словит синтаксическую ошибку из-за отсутствующего операнда.

</details>

### Практическое упражнение 2 — Бесконечный while и потерянный инкремент

Скрипт должен напечатать числа от 0 до 9, но вместо этого бесконечно печатает 0. Что нужно исправить?

```bash
#!/bin/bash
COUNTER=0
while [ $COUNTER -lt 10 ]
do
    echo $COUNTER
done
```

<details>
<summary>Ответ</summary>

В теле цикла нет инкремента переменной `COUNTER`, поэтому условие `[ $COUNTER -lt 10 ]` остаётся истинным навсегда. Нужно увеличивать счётчик:

```bash
COUNTER=$(( $COUNTER + 1 ))
```

Исправленный скрипт:

```bash
#!/bin/bash
COUNTER=0
while [ $COUNTER -lt 10 ]
do
    echo $COUNTER
    COUNTER=$(( $COUNTER + 1 ))
done
```

> Альтернативные варианты инкремента: `(( COUNTER++ ))` или `let COUNTER++`. Все три формы делают одно и то же.

</details>

### Практическое упражнение 3 — Разный порядок сортировки на двух машинах

Один и тот же скрипт сортирует список имён пользователей по-разному на двух машинах. На первой:

```
carol
Dave
emma
Frank
Grace
henry
```

На второй:

```
Dave
Frank
Grace
carol
emma
henry
```

В чём разница?

<details>
<summary>Ответ</summary>

Сортировка опирается на текущую локаль системы. На разных машинах настроены разные локали, и правила сравнения букв разного регистра отличаются. Чтобы избежать таких различий, операцию сортировки выполняют с переменной окружения `LANG`, установленной в `C`:

```bash
LANG=C sort users.txt
```

> На первой машине, скорее всего, активна UTF-8 локаль (например, `en_US.UTF-8`), где `a` и `A` приравниваются и упорядочиваются вперемешку. На второй работает `C` (или `POSIX`), где сначала идут все заглавные ASCII-буквы, потом все строчные.

</details>

### Исследовательское упражнение 1 — Аргументы скрипта в массив Bash

Как использовать все аргументы командной строки скрипта для инициализации массива Bash?

<details>
<summary>Ответ</summary>

```bash
PARAMS=( $* )
```

Или эквивалентно:

```bash
PARAMS=( "$@" )
```

Оба варианта создают массив `PARAMS` со всеми аргументами.

> Между формами есть разница в обработке кавычек. `"$@"` сохраняет каждый аргумент как отдельный элемент даже при наличии пробелов внутри. `$*` (без кавычек) разбивает аргументы по `IFS` и теряет границы. Для надёжности стоит писать `( "$@" )`.

</details>

### Исследовательское упражнение 2 — Почему test 1 > 2 возвращает истину

Команда `test 1 > 2` возвращает истину, хотя интуитивно ожидается ложь. Почему?

<details>
<summary>Ответ</summary>

Оператор `>` в `test` предназначен для строкового сравнения, а не для численного. Для чисел используются `-lt`, `-gt`, `-le`, `-ge`, `-eq`, `-ne`.

> Дополнительно: символ `>` в shell сам по себе перенаправляет стандартный вывод. Поэтому `test 1 > 2` фактически выполняет `test 1` с перенаправлением stdout в файл `2`. Команда `test 1` истинна (непустая строка), а перенаправление просто создаёт пустой файл `2` в текущем каталоге. Для правильного строкового сравнения нужно экранировать оператор: `test 1 \> 2`.

</details>

### Исследовательское упражнение 3 — Временная смена IFS на newline

Как временно изменить разделитель полей по умолчанию на символ перевода строки и потом вернуть исходное значение?

<details>
<summary>Ответ</summary>

Сохранить текущее значение `IFS` в другой переменной:

```bash
OLDIFS=$IFS
```

Установить новый разделитель (только перевод строки):

```bash
IFS=$'\n'
```

Вернуть исходное значение:

```bash
IFS=$OLDIFS
```

> Запись `$'\n'` это специальный синтаксис ANSI-C quoting в Bash, где экранированные последовательности (`\n`, `\t`, `\r`) интерпретируются как соответствующие управляющие символы. Без `$` строка `'\n'` была бы двумя обычными символами: обратной косой и буквой `n`.

</details>
