- Загружать разделяемые библиотеки.

---

## Что такое разделяемые библиотеки

Когда программа скомпилирована, она может использовать код двумя способами: статически или динамически. При статической компоновке весь нужный код включается прямо в исполняемый файл. При динамической компоновке программа хранит только ссылки, а реальный код загружается из внешних файлов во время запуска. Эти внешние файлы и есть разделяемые библиотеки.

Динамический компоновщик (dynamic linker/loader) отвечает за то, чтобы найти и подгрузить нужные библиотеки в память при запуске программы. В Linux его имя обычно выглядит как `/lib/ld-linux.so.2` (для 32-битных систем) или `/lib64/ld-linux-x86-64.so.2` (для 64-битных).

Разделяемые библиотеки выгодны по двум причинам: несколько программ используют один и тот же код в памяти, что экономит ресурсы, а обновление библиотеки сразу затрагивает все программы, которые её используют.

---

## Соглашение об именовании

Разделяемые библиотеки следуют строгому соглашению об именах.

**Реальное имя файла (realname):** содержит полный номер версии.
```
libname.so.X.Y.Z
```
Пример: `libz.so.1.2.11`

**soname:** содержит только major-версию. Это символическая ссылка на реальный файл.
```
libname.so.X
```
Пример: `libz.so.1 -> libz.so.1.2.11`

**Linker name:** имя без версии, используется при компиляции.
```
libname.so
```
Пример: `libz.so -> libz.so.1`

Такая цепочка символических ссылок позволяет обновлять библиотеку без пересборки программ: достаточно изменить, на что указывает soname.

---

## Где хранятся библиотеки

Стандартные расположения библиотек в Linux:

| Путь | Назначение |
|------|-----------|
| `/lib` | Библиотеки, нужные для загрузки системы (до монтирования /usr) |
| `/lib32` | 32-битные библиотеки на 64-битных системах |
| `/lib64` | 64-битные библиотеки |
| `/usr/lib` | Основные библиотеки пользовательских программ |
| `/usr/local/lib` | Библиотеки программ, собранных вручную |

---

## Как программа находит библиотеки

При запуске программы динамический компоновщик ищет библиотеки в следующем порядке:

1. Пути, жёстко зашитые в сам бинарный файл (rpath, записан во время компиляции).
2. Пути из переменной окружения `LD_LIBRARY_PATH`.
3. Пути из кэша `/etc/ld.so.cache` (построен из `/etc/ld.so.conf`).
4. Пути по умолчанию: `/lib`, `/lib64`, `/usr/lib`, `/usr/lib64`.

### ldconfig и кэш библиотек

`ldconfig` читает конфигурацию и создаёт кэш `/etc/ld.so.cache`, чтобы компоновщик быстро находил библиотеки. Запускать `ldconfig` нужно каждый раз после установки новой библиотеки или изменения конфигурации.

```bash
# Обновить кэш
sudo ldconfig

# Показать содержимое кэша
ldconfig -p

# Пересканировать каталоги и обновить кэш с подробным выводом
sudo ldconfig -v
```

Вывод `ldconfig -p` показывает список всех библиотек в кэше с их путями:
```
libz.so.1 (libc6,x86-64) => /lib/x86_64-linux-gnu/libz.so.1
```

### /etc/ld.so.conf

Этот файл содержит список каталогов, которые `ldconfig` сканирует при построении кэша. Современные дистрибутивы обычно хранят основную конфигурацию так:

```
# /etc/ld.so.conf
include /etc/ld.so.conf.d/*.conf
```

А в `/etc/ld.so.conf.d/` лежат отдельные файлы для каждого набора библиотек. Если устанавливается новый пакет с библиотеками в нестандартный каталог, добавь путь туда и запусти `ldconfig`.

Пример добавления каталога:
```bash
echo "/usr/local/lib/myapp" | sudo tee /etc/ld.so.conf.d/myapp.conf
sudo ldconfig
```

### LD_LIBRARY_PATH

Переменная окружения `LD_LIBRARY_PATH` позволяет временно добавить каталоги в путь поиска библиотек. Она имеет более высокий приоритет, чем кэш, поэтому удобна для тестирования или запуска программ с нестандартными библиотеками.

```bash
# Запустить программу с кастомной библиотекой
LD_LIBRARY_PATH=/home/user/mylibs ./myprogram

# Или задать переменную в сессии
export LD_LIBRARY_PATH=/usr/local/mylibs:$LD_LIBRARY_PATH
```

Чтобы убрать переменную из текущей сессии:

```bash
unset LD_LIBRARY_PATH
```

Чтобы сделать изменение постоянным, добавь строку `export LD_LIBRARY_PATH=/usr/local/mylib` в один из файлов инициализации Bash: `/etc/bash.bashrc` (для всех пользователей) или `~/.bashrc` (для конкретного пользователя).

Постоянно использовать `LD_LIBRARY_PATH` в production не рекомендуется: лучше добавить путь в `/etc/ld.so.conf.d/` и обновить кэш. Переменная полезна именно для разработки и отладки.

## Команда ldd

`ldd` показывает, какие разделяемые библиотеки нужны конкретному бинарному файлу. Это удобно для диагностики ошибок "library not found" и для проверки зависимостей перед деплоем.

```bash
ldd /bin/ls
```

Типичный вывод:
```
	linux-vdso.so.1 (0x00007ffd1a9b0000)
	libselinux.so.1 => /lib/x86_64-linux-gnu/libselinux.so.1 (0x00007f...)
	libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x00007f...)
	/lib64/ld-linux-x86-64.so.2 (0x00007f...)
```

Если библиотека не найдена, `ldd` покажет `not found` рядом с её именем:
```
	libmissing.so.1 => not found
```

Важный момент: `ldd` на самом деле запускает программу в специальном режиме. Не запускай `ldd` на бинарных файлах из ненадёжных источников — это угроза безопасности. Для этих случаев лучше использовать `objdump -p binary | grep NEEDED` или `readelf -d binary | grep NEEDED`.

`ldd` работает и с самими разделяемыми библиотеками, а не только с исполняемыми файлами:

```bash
ldd /lib/x86_64-linux-gnu/libc.so.6
```

С флагом `-u` `ldd` выводит прямые зависимости, которые по факту не используются:

```bash
ldd -u /usr/bin/git
# Unused direct dependencies:
#   /lib/x86_64-linux-gnu/libz.so.1
#   /lib/x86_64-linux-gnu/libpthread.so.0
```

Неиспользуемые зависимости появляются из-за опций компоновщика при сборке: библиотека помечена как NEEDED в ELF-заголовке, хотя фактически не нужна.

## Полезные команды для экзамена

```bash
# Показать зависимости бинарного файла
ldd /path/to/binary

# Обновить кэш библиотек (нужны права root)
ldconfig

# Просмотреть содержимое кэша
ldconfig -p

# Обновить кэш с подробным выводом
ldconfig -v

# Временно указать путь к библиотекам
LD_LIBRARY_PATH=/my/lib/dir ./program

# Найти файл библиотеки по имени
find /lib /usr/lib /usr/lib64 -name "libz.so*"

# Показать зависимости без запуска программы (безопаснее ldd)
objdump -p /bin/ls | grep NEEDED
readelf -d /bin/ls | grep NEEDED
```

## Типичные вопросы экзамена

**Какой файл конфигурации читает ldconfig?**
`/etc/ld.so.conf` (и включённые через `include` файлы из `/etc/ld.so.conf.d/`).

**Где хранится кэш разделяемых библиотек?**
`/etc/ld.so.cache`. Это бинарный файл, созданный `ldconfig`.

**Какую команду нужно запустить после добавления новой библиотеки в /usr/local/lib?**
`ldconfig` (с правами root).

**Какая переменная окружения позволяет временно добавить путь к библиотекам?**
`LD_LIBRARY_PATH`.

**Для чего нужна команда ldd?**
Она показывает список разделяемых библиотек, от которых зависит бинарный файл, и пути, по которым эти библиотеки будут найдены.

**На 64-битной системе библиотеки обычно лежат в...**
`/lib64/` и `/usr/lib64/` (а также в `/lib/` и `/usr/lib/` для 32-битных библиотек).

**Почему soname содержит только major-версию?**
Потому что совместимость по ABI гарантируется внутри одной major-версии. Программа, скомпонованная с `libfoo.so.2`, будет работать с любой `libfoo.so.2.x.y`, не требуя перекомпиляции.

## Упражнения

### Практические упражнения (Guided Exercises)

**1. Разбей следующие имена разделяемых библиотек на составные части.**

| Полное имя файла | Имя библиотеки | Суффикс so | Номер версии |
|---|---|---|---|
| `linux-vdso.so.1` | linux-vdso | so | 1 |
| `libprocps.so.6` | libprocps | so | 6 |
| `libdl.so.2` | libdl | so | 2 |
| `libc.so.6` | libc | so | 6 |
| `libsystemd.so.0` | libsystemd | so | 0 |
| `ld-linux-x86-64.so.2` | ld-linux-x86-64 | so | 2 |

Обрати внимание на `ld-linux-x86-64`: это динамический компоновщик, и его имя не начинается с `lib`. Он попадает под то же соглашение именования, но выполняет особую роль, загружая все остальные библиотеки.

Также стоит запомнить `linux-vdso.so.1`: это виртуальная библиотека (vDSO, virtual Dynamic Shared Object), которую ядро само отображает в адресное пространство каждого процесса. Файла на диске у неё нет.

---

**2. Ты разработал программу и хочешь добавить в систему новый каталог разделяемых библиотек `/opt/lib/mylib`. Ты записал абсолютный путь в файл `mylib.conf`.**

**В какой каталог нужно поместить этот файл?**

<details>
<summary>Ответ</summary>

`/etc/ld.so.conf.d`

Все `.conf`-файлы в этом каталоге автоматически включаются основным `/etc/ld.so.conf` через директиву `include /etc/ld.so.conf.d/*.conf`.

</details>

**Какую команду нужно запустить, чтобы изменения вступили в силу?**

<details>
<summary>Ответ</summary>

```bash
sudo ldconfig
```

`ldconfig` перечитает конфигурацию, обновит символические ссылки и пересоберёт кэш `/etc/ld.so.cache`. Без этого шага компоновщик не увидит новый каталог.

</details>

---

**3. Какую команду использовать, чтобы вывести список разделяемых библиотек, нужных программе `kill`?**

<details>
<summary>Ответ</summary>

```bash
ldd /bin/kill
```

Путь `/bin/kill` или `/usr/bin/kill` зависит от дистрибутива. Точный путь можно найти командой `which kill`. Главное — передавать `ldd` абсолютный путь к бинарнику.

</details>

### Исследовательские упражнения (Explorational Exercises)

**1. objdump: изучаем зависимости и soname через заголовки ELF.**

`objdump` — утилита для вывода информации из объектных файлов. Проверь, установлена ли она:

```bash
which objdump
```

Если не установлена, поставь пакет `binutils`.

**Вывести зависимости glibc:**

```bash
objdump -p /lib/x86_64-linux-gnu/libc.so.6 | grep NEEDED
```

У glibc зависимостей почти нет, так как она сама является базовой библиотекой системы. Вывод может быть пустым или содержать только `ld-linux`.

**Вывести soname glibc:**

```bash
objdump -p /lib/x86_64-linux-gnu/libc.so.6 | grep SONAME
```

Вывод покажет: `SONAME      libc.so.6`

Soname прописан прямо в ELF-заголовке библиотеки. Именно это имя динамический компоновщик использует при создании символической ссылки и при поиске во время запуска программы.

**Вывести зависимости Bash:**

```bash
objdump -p /bin/bash | grep NEEDED
```

Вывод покажет список библиотек, от которых зависит Bash, например `libtinfo`, `libdl`, `libc`.

> В отличие от `ldd`, `objdump -p | grep NEEDED` только читает ELF-заголовок, не запуская исполняемый файл. Это безопаснее при анализе бинарников из ненадёжных источников. `readelf -d /bin/bash | grep NEEDED` делает то же самое через другой инструмент.
