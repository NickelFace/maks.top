## Обзор виртуализации

Виртуализация — это технология, где специальная платформа, называемая гипервизором, запускает процессы с полностью эмулированными компьютерными системами. Гипервизор управляет физическими ресурсами хоста и распределяет их между гостевыми машинами.

Виртуальная машина может быть перенесена с одного гипервизора на другой. Этот процесс называется миграцией. Горячая миграция (live migration) выполняется без остановки гостя.

### Типы гипервизоров

**Type-1 (bare-metal)** запускается напрямую на железе, без операционной системы под собой. Пример: Xen.

**Type-2** требует операционную систему хоста для работы. Пример: VirtualBox. KVM занимает промежуточное положение: технически ему нужна Linux-система под собой, но он интегрируется в ядро и работает как Type-1.

### Популярные гипервизоры

**Xen** — открытый Type-1 гипервизор. Компьютер загружается прямо в него.

**KVM (Kernel Virtual Machine)** — модуль ядра Linux. Управление ВМ идёт через демон libvirt и утилиты вроде `virsh`. Для эмуляции устройств использует проект QEMU. Конфигурации хранятся как XML-файлы в `/etc/libvirt/qemu/`.

**VirtualBox** — популярное приложение для рабочего стола от Oracle. Работает на Linux, macOS и Windows. Type-2 гипервизор.

## Типы виртуальных машин

### Полная виртуализация

Гость (HardwareVM) не знает, что работает внутри ВМ. Никаких специальных драйверов внутри гостя нет, поэтому все инструкции должны выполняться напрямую. На x86-платформе для этого требуются расширения процессора: Intel VT-x или AMD-V. Их включают в настройках BIOS/UEFI.

### Паравиртуализация

Гость (PVM) знает, что работает в виртуальной среде, и использует специальные гостевые драйверы для взаимодействия с гипервизором. Модифицированное ядро и эти драйверы дают заметный прирост производительности по сравнению с полной виртуализацией.

### Гибридная виртуализация

Сочетает полную виртуализацию и паравиртуализацию. Немодифицированная ОС запускается в полностью виртуальной среде, но при этом использует паравиртуальные драйверы для хранилища и сети. Это даёт производительность, близкую к нативной.

---

## Гостевые драйверы

Разные гипервизоры поставляют свои пакеты гостевых драйверов:

- **KVM** использует драйверы из проекта **Virtio**.
- **VirtualBox** предоставляет **Guest Extensions** в виде загружаемого ISO-образа.

## Хранилище виртуальных машин

Виртуальная машина в libvirt/KVM, как правило, состоит из двух частей: XML-файла с конфигурацией и файла образа диска. Образ диска лежит по умолчанию в `/var/lib/libvirt/images/`.

Гость видит весь заявленный размер диска, хотя реально на хосте может быть занято значительно меньше места.

### Типы образов дисков

**COW (Copy-on-Write)** — тонкое выделение места. Файл создаётся с заданным верхним пределом, а реальный размер растёт по мере записи данных. Формат `qcow2` — стандартный COW-формат для QEMU/KVM.

**RAW** — весь объём диска выделяется сразу. Запись идёт быстрее, потому что гипервизору не нужно следить за расширением файла. Занимает больше места на хосте с первого момента.

## Работа с шаблонами ВМ

Шаблон — это ВМ с базовой установкой системы и предварительно настроенными параметрами, который потом копируется для быстрого развёртывания новых гостей. При клонировании шаблона новый гость получает другое имя хоста, новый MAC-адрес сетевого интерфейса и ряд других уникальных идентификаторов.

### D-Bus Machine ID

Каждая Linux-система при установке получает уникальный машинный идентификатор — D-Bus Machine ID. Если клонировать ВМ без изменения этого ID, два гостя окажутся с одинаковым идентификатором, что приведёт к непредсказуемому поведению.

Файл ID: `/var/lib/dbus/machine-id`, символически ссылается на `/etc/machine-id`.

Менять ID на работающей системе опасно: возможна нестабильность и краши. Если всё же нужно создать новый ID, делают это так:

```bash
# Проверить, что ID существует
dbus-uuidgen --ensure

# Посмотреть текущий ID
dbus-uuidgen --get

# Сгенерировать новый ID (только на выключенной или только что склонированной ВМ)
sudo rm -f /etc/machine-id
sudo dbus-uuidgen --ensure=/etc/machine-id
```

Если `/var/lib/dbus/machine-id` не является символической ссылкой на `/etc/machine-id`, его тоже нужно удалить.

### SSH Host Keys

При клонировании ВМ SSH-ключи хоста тоже надо пересоздать. Если два клона имеют одинаковые host keys, SSH-клиенты при подключении получат предупреждение о конфликте ключей.

Ключи хоста хранятся в `/etc/ssh/` и имеют вид `ssh_host_*_key`.

## Развёртывание ВМ в облаке

IaaS (Infrastructure as a Service) — модель, где провайдер предоставляет инфраструктуру как услугу. Провайдер запускает гипервизоры и предоставляет инструменты для развёртывания гостевых образов.

### Элементы IaaS

**Вычислительные экземпляры (Computing Instances)** — единицы измерения потребляемых CPU-ресурсов. Чем больше экземпляров и чем дольше они работают, тем выше счёт.

**Блочное хранилище (Block Storage)** — внешнее хранилище для ВМ. Стоимость зависит от объёма и скорости доступа: быстрые SSD дороже, архивное хранилище дешевле.

**Сеть (Networking)** — настройка маршрутизации, подсетей и брандмауэров через веб-интерфейс провайдера. Многие провайдеры предлагают DNS, публичные IP-адреса и гибридные решения с VPN для объединения корпоративной и облачной инфраструктур.

### Доступ к облачным гостям по SSH

Основной способ доступа к облачным гостям — ключи SSH. Пароли в облаке обычно не используют.

```bash
# Создать пару ключей (публичный и приватный)
ssh-keygen

# Скопировать публичный ключ на удалённый сервер
ssh-copy-id -i <public_key> user@cloud_server
```

Если в директории `~/.ssh/` находится только один публичный ключ, флаг `-i` можно не указывать: `ssh-copy-id` найдёт файл с расширением `.pub` самостоятельно. Права на файлы ключей: приватный ключ — `0600`, публичный ключ — `0644`.

Некоторые облачные провайдеры автоматически генерируют ключевую пару при создании ВМ и предоставляют приватный ключ для скачивания.

## cloud-init

cloud-init — утилита для автоматической настройки облачных ВМ и контейнеров при первом запуске. Читает YAML-файлы формата `cloud-config` и применяет настройки: сеть, пакеты, SSH-ключи, учётные записи, локаль и многое другое. Работает один раз при начальной загрузке системы.

Пример `cloud-config` файла:

```yaml
#cloud-config
timezone: Africa/Dar_es_Salaam
hostname: test-system

# Обновить систему при первом запуске
apt_update: true
apt_upgrade: true

# Установить веб-сервер Nginx
packages:
  - nginx
```

Первая строка `#cloud-config` обязательна, пробел между `#` и `cloud-config` не допускается.

## Контейнеры

Контейнер — это изолированная среда для запуска приложения, которая использует только тот минимум программного обеспечения, который нужен для работы. В отличие от ВМ, здесь не эмулируется весь компьютер, поэтому накладные расходы значительно меньше.

Контейнеры могут мигрировать между хостами на ходу, без остановки приложения. Новые версии приложений можно развёртывать параллельно со старыми.

В Linux контейнеры опираются на механизм `cgroups` (control groups) ядра. `cgroups` позволяют ограничивать потребление памяти, процессорного времени, дискового и сетевого трафика для конкретного приложения.

Примеры контейнерных технологий: Docker, Kubernetes, LXD/LXC, systemd-nspawn, OpenShift. Детали их реализации выходят за рамки LPIC-1.

Различают два вида контейнеров:
- **Linux container** — контейнер, изолирующий системное окружение (LXC/LXD).
- **Application container** — контейнер для одного приложения (Docker, Podman).

| Команда | Описание |

| `dbus-uuidgen --ensure` | Проверить, что D-Bus Machine ID существует |

## Типичные вопросы экзамена

**Какие CPU-расширения нужны для полной виртуализации на x86?**
VT-x (Intel) или AMD-V.

**Где хранится D-Bus Machine ID?**
`/var/lib/dbus/machine-id` (ссылается на `/etc/machine-id`).

**Что изменить при клонировании ВМ?**
D-Bus Machine ID и SSH host keys.

**Какой формат диска используют ВМ KVM по умолчанию?**
`qcow2` (COW).

**Что делает cloud-init?**
Автоматически настраивает систему при первом запуске по YAML-конфигу.

**Чем контейнер отличается от ВМ?**
Контейнер не эмулирует весь компьютер — он использует ядро хоста и потребляет меньше ресурсов.

**Какая команда покажет, одинаковые ли D-Bus ID у двух клонов?**
`dbus-uuidgen --get` (выполнить на каждой машине и сравнить вывод).

## Упражнения

### Практические упражнения

**1. Какие CPU-расширения необходимы для запуска гостей с полной виртуализацией на x86-платформе?**

<details>
<summary>Ответ</summary>

Intel VT-x для процессоров Intel или AMD-V для процессоров AMD. Эти расширения включаются в настройках BIOS или UEFI.

</details>

---

**2. Критически важный сервер требует максимальной производительности. Какой тип виртуализации он, скорее всего, использует?**

<details>
<summary>Ответ</summary>

Паравиртуализацию. Гостевая ОС в таком случае знает о своём виртуальном окружении и использует специальные программные драйверы для работы с ресурсами гипервизора. Это даёт заметный прирост производительности по сравнению с полной виртуализацией.

> Паравиртуализированная система типа Xen позволяет гостю эффективнее использовать аппаратные ресурсы именно благодаря драйверам, разработанным специально для взаимодействия с гипервизором.

</details>

---

**3. Две ВМ клонированы из одного шаблона и используют D-Bus. Они работают нестабильно, хотя имена хостов и сетевые настройки у них разные. Какая команда поможет проверить, одинаковые ли у них D-Bus Machine ID?**

<details>
<summary>Ответ</summary>

```bash
dbus-uuidgen --get
```

Эту команду нужно выполнить на каждой из машин и сравнить вывод. Если строки совпадают — обе ВМ имеют одинаковый Machine ID, что и вызывает проблемы.

</details>

### Исследовательские упражнения

**1. Проверить, поддерживает ли ваш CPU расширения виртуализации.**

<details>
<summary>Ответ</summary>

```bash
grep --color -E "vmx|svm" /proc/cpuinfo
```

В выводе будет выделено `vmx` для процессоров Intel с включённым VT-x или `svm` для процессоров AMD с включённым AMD-V. Если вывод пустой, нужно зайти в BIOS/UEFI и включить поддержку виртуализации вручную.

Пример вывода на машине с Intel и включённым VT-x:

```
flags : fpu vme de pse tsc msr pae mce cx8 apic sep mtrr pge mca cmov pat pse36
clflush dts acpi mmx fxsr sse sse2 ss ht tm pbe syscall nx pdpe1gb rdtscp lm constant_tsc
art arch_perfmon pebs bts rep_good nopl xtopology nonstop_tsc cpuid aperfmperf pni
pclmulqdq dtes64 monitor ds_cpl vmx smx est tm2 ...
```

</details>

---

**2. Если ваш процессор поддерживает виртуализацию, установить KVM-гипервизор и создать тестовую ВМ.**

<details>
<summary>Ответ</summary>

Пакеты для установки зависят от дистрибутива:
- Ubuntu: `sudo apt install qemu-kvm libvirt-daemon-system virt-manager`
- Fedora: `sudo dnf install @virtualization`
- Arch Linux: `sudo pacman -S qemu libvirt virt-manager`

После установки для управления ВМ через графику можно запустить `virt-manager`. Для создания ВМ из командной строки используется `virt-install`. Рекомендуется попробовать оба способа.

</details>
