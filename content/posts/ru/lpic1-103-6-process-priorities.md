## Планировщик Linux

Операционные системы, запускающие несколько процессов одновременно, называют многозадачными. Истинный параллелизм возникает только при наличии нескольких процессорных ядер, но даже однопроцессорные системы имитируют его, быстро переключаясь между задачами.

В каждый момент времени CPU управляет ровно одним процессом. Большинство операций процесса — чтение файлов, обращение к сети, вывод на экран — это системные вызовы: процесс передаёт управление ядру, и пока ядро обрабатывает запрос, CPU может переключиться на другой процесс.

Планировщик Linux делит процессы на два класса:

- **Процессы реального времени** (real-time). Получают статические приоритеты от 0 до 99. Они всегда вытесняют обычные процессы.
- **Обычные процессы** (normal scheduling). Получают статические приоритеты от 100 до 139. Меньшее значение означает более высокий приоритет.

Статические приоритеты реального времени иногда называют статическими приоритетами планирования, чтобы не путать с динамическими, которые зависят от nice-значения.

В **вытесняющей многозадачности** (preemptive multitasking) процесс с более высоким приоритетом прерывает работу менее приоритетного, как только появляется в очереди.

## Чтение приоритетов

### Приоритеты в /proc

Реальный статический приоритет процесса хранится в `/proc/<PID>/sched`:

```bash
$ grep ^prio /proc/1/sched
prio : 120
```

Стандартный приоритет обычного процесса равен 120. Диапазон для обычных процессов: 100 (выше) до 139 (ниже).

### Приоритеты в ps

Команды `ps -Al` или `ps -el` показывают приоритеты в колонке `PRI`. Из-за исторических особенностей `ps` отображает значения в диапазоне от -40 до 99. Чтобы получить реальный приоритет, нужно прибавить 40:

```
PRI (ps) + 40 = реальный приоритет
80 + 40 = 120  (стандартный)
```

Колонка `NI` показывает nice-число процесса.

Пример вывода:

```
$ ps -el
F S UID  PID PPID C PRI NI ADDR    SZ WCHAN TTY      TIME CMD
4 S   0    1    0 0  80  0 -     9292 -     ?    00:00:00 systemd
1 S   0    3    2 0  60 -20 -       0 -     ?    00:00:00 rcu_gp
```

Для сортировки по загрузке CPU (от меньшей к большей):

```bash
$ ps -el --sort=pcpu
```

Расширенный формат с выбором колонок:

```bash
$ ps -e -o user,uid,comm,tty,pid,ppid,pri,pmem,pcpu --sort=-pcpu | head
```

### Приоритеты в top

`top` отображает приоритеты иначе: вычитает 100 из реального значения. Это делает приоритеты реального времени отрицательными, их легко заметить. Обычные процессы в `top` показываются в диапазоне от 0 до 39. Процессы реального времени отображаются с меткой `rt` или отрицательным числом.

Таблица соответствия:

| Реальный приоритет | ps PRI | top PR |
|--------------------|--------|--------|
| 120 (стандарт)     | 80     | 20     |
| 100 (макс. обычный)| 60     | 0      |
| 139 (мин. обычный) | 99     | 39     |

## Niceness: вежливость процесса

Niceness (вежливость) позволяет влиять на приоритет обычных процессов. Логика проста: чем «вежливее» процесс, тем охотнее он пропускает других вперёд.

- Диапазон nice-чисел: от **-20** (менее вежливый, высокий приоритет) до **19** (более вежливый, низкий приоритет).
- По умолчанию все процессы стартуют с nice = 0, что соответствует реальному приоритету 120.
- Только `root` может уменьшать nice ниже нуля (повышать приоритет).
- Обычный пользователь может только увеличивать nice (снижать приоритет).
- Linux позволяет назначать разные nice-значения отдельным потокам одного процесса.

### Запуск с изменённым приоритетом: nice

Команда `nice` запускает программу с заданным nice-числом. Без флагов `nice` устанавливает niceness = 10.

```bash
# Запустить с niceness 15
$ nice -n 15 tar czf home_backup.tar.gz /home

# Запустить с niceness -5 (только root)
# nice -n -5 ./critical_task
```

### Изменение приоритета запущенного процесса: renice

`renice` меняет nice-число уже работающего процесса.

```bash
# По PID (только root для отрицательных значений)
# renice -10 -p 2164
2164 (process ID) old priority 0, new priority -10

# По группе процессов
$ renice +5 -g users

# По пользователю
$ renice +3 -u john
```

Флаг `-p` указывает PID, `-g` применяет изменение ко всем процессам группы, `-u` — ко всем процессам пользователя.

### Изменение приоритета через top

В интерактивном режиме `top` нажми `r`. Появится строка `PID to renice [default pid = ...]`. Введи PID и нажми Enter, затем укажи новое nice-значение.

## Шпаргалка

```bash
# Посмотреть реальный приоритет процесса
grep ^prio /proc/<PID>/sched

# Список процессов с приоритетами
ps -el
ps -Al

# Запустить с нестандартным приоритетом
nice -n <значение> <команда>

# Изменить приоритет запущенного процесса по PID
renice <новое_значение> -p <PID>

# Изменить приоритет для всех процессов пользователя
renice <значение> -u <пользователь>

# Изменить приоритет через top
# В top нажать: r -> ввести PID -> ввести новое nice-значение

# Сортировать ps по загрузке CPU (по возрастанию)
ps -el --sort=pcpu

# schedtool: показать параметры планирования
schedtool <PID>

# schedtool: перевести в реальное время с приоритетом
schedtool -R -p <приоритет> <PID>
```

## Типичные вопросы экзамена

**Какой nice-номер получает процесс по умолчанию?**
0 (реальный приоритет 120).

**Диапазон nice-чисел?**
От -20 до 19.

**Кто может устанавливать отрицательные nice-числа?**
Только root.

**Как в `ps` определить реальный приоритет?**
Прибавить 40 к значению колонки PRI.

**Как `top` обозначает процессы реального времени?**
Отображает `rt` или отрицательное число в колонке PR.

**В `ps -el` у процесса PRI = -40. Что это означает?**
Процесс реального времени: реальный приоритет = -40 + 40 = 0 (диапазон RT).

**Как запустить команду `backup.sh` с низким приоритетом (niceness 10)?**
`nice backup.sh` или `nice -n 10 backup.sh`.

**Что делает `renice +5 -g users`?**
Повышает nice-число (снижает приоритет) на 5 для всех процессов группы `users`.

**При попытке изменить приоритет через `renice` возникает ошибка:**

```

```

**Какова вероятная причина?**

<details><summary>Ответ</summary>

Уменьшить nice-число ниже нуля (то есть повысить приоритет выше стандартного) может только `root`. Команда выполнялась от имени обычного пользователя.

</details>

#### Упражнение 2

**Команда `schedtool` позволяет просматривать и задавать параметры планирования CPU для процессов. Как с её помощью вывести параметры планирования процесса 1750? И как перевести процесс 1750 в режим реального времени с приоритетом -90 (в нотации `top`)?**

<details><summary>Ответ</summary>

```bash
# Показать параметры планирования
$ schedtool 1750

# Перевести в реальное время с приоритетом -90 (top) = 89 (schedtool)
$ schedtool -R -p 89 1750
```

`top` показывает приоритет реального времени как `реальный_приоритет - 100`. Значит, -90 в `top` соответствует реальному приоритету 10, а schedtool принимает значение 0–99 напрямую. Флаг `-R` указывает политику SCHED_RR (round-robin реального времени), `-p` задаёт числовой приоритет.

</details>
