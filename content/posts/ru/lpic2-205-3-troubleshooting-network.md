## Введение в сетевую диагностику

Универсального «каталога ошибок» с готовыми ответами не существует. Но почти любую проблему можно решить логически, если понять, как трафик должен проходить от источника к месту назначения и обратно. Трафик в обоих направлениях часто идёт по разным маршрутам, и это само по себе нередко является источником проблем.

**Пошаговый алгоритм диагностики:**

```
1. Перечислите все сетевые компоненты: ПК, коммутатор, маршрутизатор, межсетевой экран, провайдер, целевой хост.
2. Определите, как компоненты взаимодействуют.
3. Проверяйте последовательно от ближнего к дальнему.
```

Пример пошаговой диагностики при недоступности сайта:

**Шаг 1.** Проверьте, доступны ли другие машины в интернете. Попробуйте другой URL или пропингуйте другой IP. Если всё остальное работает, проблема в конкретном целевом хосте.

```bash
ping -c 4 8.8.8.8
ping -c 4 1.1.1.1
```

**Шаг 2.** Проверьте, не упал ли сам целевой хост. Попросите кого-нибудь проверить его с другой сети или свяжитесь с администратором сервера.

**Шаг 3.** Проверьте доступность межсетевого экрана. Пропингуйте его IP, попробуйте войти через SSH.

```bash
ping -c 4 192.168.1.1
```

**Шаг 4.** Проверьте, не упал ли один из хопов на пути. Запустите `traceroute` с флагом `-I` (ICMP вместо UDP):

```bash
traceroute -I www.lpi.org
# Каждая строка — один хоп. *** означает потерю пакетов на этом узле.
```

**Шаг 5.** Проверьте, доступен ли межсетевой экран с других машин в сети. Попробуйте `ping` или откройте страницу с другого ПК.

**Шаг 6.** Проверьте, не блокирует ли межсетевой экран трафик к конкретному хосту. Возможно, кто-то добавил блокирующее правило.

**Шаг 7.** Если проблема на межсетевом экране, проверьте настройки его интерфейсов, правила и кабели.

**Шаг 8.** Проверьте, поднят ли интерфейс eth0:

```bash
ifconfig eth0
# или
ip link show eth0
```

**Шаг 9.** Проверьте таблицу маршрутизации, в частности наличие шлюза по умолчанию:

```bash
route -n
# или
ip route show
```

**Шаг 10.** Проверьте физическую причину. Незащищённый кабель рядом с силовыми проводами вызывает непредсказуемые, трудновоспроизводимые ошибки передачи данных.

> **Внимание:** Межсетевые экраны могут блокировать ICMP echo-request. Если `ping` не получает ответа, это не означает, что хост недоступен. Проверьте TCP-подключение с помощью `nc` или `telnet`.

## Файлы управления доступом: TCP Wrappers

TCP Wrappers перехватывает входящие соединения к службам, скомпилированным с поддержкой `libwrap`, и решает, кому разрешить доступ. Проверить, поддерживает ли служба libwrap:

```bash
ldd /sbin/rpcbind | grep libwrap
```

### Порядок проверки

1. Сначала читается `/etc/hosts.allow`. Если адрес клиента найден, доступ разрешается немедленно.
2. Затем читается `/etc/hosts.deny`. Если адрес найден, доступ блокируется.
3. Если адрес не найден ни в одном файле, доступ разрешается по умолчанию.

> **Важно:** Запомните порядок: allow → deny → разрешить. Экзамен часто проверяет этот порядок. `hosts.allow` всегда проверяется первым.

### Формат записей

```
# /etc/hosts.allow
sshd: 192.168.1.0/24
rpcbind: 192.168.56.101, 192.168.56.106
ALL: localhost

# /etc/hosts.deny
sshd: ALL
rpcbind: ALL
```

Поля записи: `служба: список_клиентов`. Список может содержать IP-адреса, имена хостов, подсети и подстановочные знаки (`ALL`, `LOCAL`).

> **Совет:** Лучшая практика — написать `ALL: ALL` в `hosts.deny` и перечислить только разрешённые адреса в `hosts.allow`. После изменения этих файлов перезапуск служб не нужен; изменения вступают в силу немедленно.

## Управление сетевыми интерфейсами

### ifconfig (устаревшее)

```bash
ifconfig -a                                         # просмотр всех интерфейсов
ifconfig eth0                                       # просмотр конкретного интерфейса
ifconfig eth0 192.168.1.100 netmask 255.255.255.0   # назначить IP и маску
ifconfig eth0 up                                    # поднять интерфейс
ifconfig eth0 down                                  # опустить интерфейс
```

### ip (современное, рекомендуется)

```bash
ip addr show                          # показать все интерфейсы и адреса
ip link show                          # показать информацию канального уровня
ip addr add 192.168.1.100/24 dev eth0 # назначить IP-адрес
ip addr del 192.168.1.100/24 dev eth0 # удалить IP-адрес
ip link set eth0 up                   # поднять интерфейс
ip link set eth0 down                 # опустить интерфейс
ip -s link show eth0                  # показать статистику интерфейса
```

> **Важно:** `ip` (из пакета `iproute2`) постепенно вытесняет `ifconfig` (из `net-tools`). Знайте оба для экзамена.

## Управление таблицей маршрутизации

### route (устаревшее)

```bash
route -n                                                        # показать таблицу маршрутизации
route add -net 192.168.10.0 netmask 255.255.255.0 dev eth0     # добавить маршрут к сети
route add -net 192.168.20.0 netmask 255.255.255.0 gw 192.168.1.10  # добавить маршрут через шлюз
route add default gw 192.168.1.1 eth1                          # установить шлюз по умолчанию
route del -net 192.168.10.0 netmask 255.255.255.0              # удалить маршрут
```

Флаги в выводе `route -n`:

| Флаг | Значение |
|---|---|
| U | Маршрут активен (Up) |
| H | Цель — конкретный хост |
| G | Маршрут использует шлюз (Gateway) |
| D | Добавлен динамически (Dynamically) |
| M | Изменён демоном маршрутизации (Modified) |

### ip route (современное)

```bash
ip route show                                     # показать таблицу маршрутизации
ip route add 192.168.10.0/24 via 192.168.1.1 dev eth0  # добавить маршрут
ip route add default via 192.168.1.1              # установить шлюз по умолчанию
ip route del 192.168.10.0/24                      # удалить маршрут
ip route get 8.8.8.8                              # найти маршрут к конкретному хосту
```

> **Внимание:** Две распространённые ошибки маршрутизации. Первая: нет сетевой записи для интерфейса — при настройке интерфейса ядро должно автоматически добавить маршрут для его подсети. Вторая: два шлюза по умолчанию. Должен быть ровно один. Наличие двух может долго не замечаться, потому что трафик случайно идёт через правильный.

## Просмотр состояния сети

### netstat (устаревшее, но встречается на экзамене)

```bash
netstat              # все активные сокеты
netstat -aln --tcp   # TCP-соединения с числовыми адресами
netstat -l           # только слушающие порты
netstat -rn          # таблица маршрутизации (то же что route -n)
netstat -s           # статистика по протоколам
netstat -tulnp       # TCP/UDP слушающие с PID, числовой вывод
```

### ss (современное, рекомендуется)

```bash
ss -t -a    # все TCP-соединения
ss -tlnp    # слушающие TCP-порты с PID
ss -u -a    # все UDP-сокеты
ss -t       # показать очереди отправки/получения
ss -t dst :80  # фильтр по порту назначения
```

Разница между `ss` и `netstat`: `ss` читает данные напрямую из пространства ядра и показывает очереди Recv-Q и Send-Q, что помогает понять, успевает ли приложение обрабатывать поток данных.

> **Совет:** Для экзамена — `netstat -tulnp` и `ss -tlnp` оба показывают слушающие TCP/UDP-порты с именами процессов. Знайте оба.

## Диагностика подключений и трассировка

### ping / ping6

```bash
ping -c 4 192.168.1.1         # отправить 4 пакета
ping6 -c 4 fe80::1            # IPv6
ping -s 1400 8.8.8.8          # указать размер пакета
ping -i 0.2 192.168.1.1       # интервал между пакетами
```

Хороший результат: 0% потерь пакетов, стабильное RTT. Растущее RTT или потери пакетов указывают на перегрузку или деградацию канала.

### traceroute / traceroute6

```bash
traceroute 8.8.8.8                    # стандартная трассировка
traceroute -I 8.8.8.8                 # использовать ICMP вместо UDP
traceroute6 2001:4860:4860::8888      # IPv6
traceroute -6 2001:4860:4860::8888    # эквивалентно traceroute6
```

Использует поле TTL в заголовке IP: каждый хоп уменьшает TTL и отправляет ICMP Time Exceeded при TTL=0, строя карту пути.

> **Примечание:** `traceroute6` и `traceroute -6` — синонимы. Знайте оба для экзамена.

### mtr (лучший инструмент для живой диагностики)

```bash
mtr 8.8.8.8              # интерактивный режим
mtr -n 8.8.8.8           # без разрешения имён
mtr -c 20 8.8.8.8        # отправить 20 пакетов и остановиться
mtr --report 8.8.8.8     # текстовый отчёт
```

`mtr` объединяет `ping` и `traceroute`: непрерывно отправляет пакеты и показывает потери и задержки по каждому хопу в реальном времени.

> **Важно:** Для экзамена — если вопрос касается инструмента, объединяющего ping и traceroute, ответ всегда `mtr`.

## Файлы конфигурации

### /etc/resolv.conf

Содержит адреса DNS-серверов и параметры поиска:

```
domain example.com           # домен по умолчанию
search example.com test.com  # список доменов для поиска
nameserver 192.168.1.1       # основной DNS
nameserver 8.8.8.8           # вторичный DNS
```

> **Внимание:** NetworkManager перезаписывает `resolv.conf`. Если NetworkManager активен, не редактируйте `/etc/resolv.conf` напрямую — изменения исчезнут после переподключения. Используйте `nmcli` или файлы в `/etc/NetworkManager/`.

### /etc/hosts

Локальная таблица сопоставления IP и имён. Проверяется перед DNS-запросами:

```
127.0.0.1   localhost
::1         localhost ip6-localhost
192.168.1.10  server01 server01.example.com
```

### Инструменты разрешения имён: dig и host

Если хост не разрешается, сначала проверьте `/etc/resolv.conf` и `/etc/hosts`, затем используйте `dig` или `host` для ручной проверки:

```bash
# host: быстрый способ получить IP из имени
host ns12.zoneedit.com
# ns12.zoneedit.com has address 209.62.64.46

# dig: подробный вывод с секциями QUESTION, ANSWER, AUTHORITY
dig zonetransfer.me
# Проверьте ANSWER SECTION и строку SERVER: (какой DNS ответил)

# Запрос к конкретному DNS-серверу
dig @8.8.8.8 example.com

# Запрос MX-записей
dig example.com MX

# Обратный поиск (PTR)
dig -x 192.168.1.10
```

`dig` — «швейцарский нож» разрешения имён: показывает всё, что вернул DNS-сервер. `host` проще и быстрее, когда нужен только IP.

> **Важно:** Для экзамена — `host` даёт краткий вывод `hostname has address X.X.X.X`. `dig` даёт полный ответ DNS со всеми секциями. Оба умеют разрешать имена и проверять записи.

### /etc/hostname и /etc/HOSTNAME

Хранит имя хоста. Debian/Ubuntu: `/etc/hostname`; старый Red Hat: `/etc/HOSTNAME`. На системах Red Hat имя также указывается в `/etc/sysconfig/network` как `HOSTNAME=myserver`.

```bash
hostname                          # просмотр текущего имени хоста
hostname newname                  # установить имя хоста (временно)
hostnamectl set-hostname newname  # установить имя хоста (постоянно через systemd)
```

### Каталоги конфигурации интерфейсов

```bash
# Debian/Ubuntu: /etc/network/interfaces
auto eth0
iface eth0 inet static
  address 192.168.1.100
  netmask 255.255.255.0
  gateway 192.168.1.1
  dns-nameservers 8.8.8.8

# Red Hat/CentOS: /etc/sysconfig/network-scripts/ifcfg-eth0
DEVICE="eth0"
BOOTPROTO=static
IPADDR=192.168.1.77
NETMASK=255.255.255.0
GATEWAY=192.168.1.254
ONBOOT=yes
NM_CONTROLLED="no"
```

## Журналы и информация об оборудовании

### dmesg

Показывает кольцевой буфер сообщений ядра. Полезен при первой загрузке для проверки загрузки драйвера сетевой карты:

```bash
dmesg                          # показать полный буфер ядра
dmesg | grep -i eth            # фильтр по сетевой карте
dmesg | grep -i network

# Пример успешной инициализации сетевой карты:
# [2.06] e1000 0000:00:03.0 eth0: Intel(R) PRO/1000 Network Connection
# [21.89] e1000: eth0 NIC Link is Up 1000 Mbps Full Duplex
```

> **Совет:** Если система работает давно, старые загрузочные сообщения могут быть вытеснены из кольцевого буфера. Посмотрите в `/var/log/dmesg`, `/var/log/syslog` или `/var/log/messages`.

### /var/log/syslog и /var/log/messages

```bash
# Debian/Ubuntu использует /var/log/syslog
tail -f /var/log/syslog
grep -i "eth0\|network" /var/log/syslog

# Red Hat/CentOS использует /var/log/messages
tail -f /var/log/messages
grep -i "eth0\|network" /var/log/messages

# С systemd: journalctl
journalctl -u NetworkManager
journalctl -k   # только сообщения ядра
```

## Некорректная инициализация системы

Если сетевой интерфейс вообще не поднялся, причиной может быть ошибка загрузки — ядро не загрузило нужный модуль сетевой карты или не смогло инициализировать интерфейс.

```bash
dmesg | grep -i eth
dmesg | grep -i "link is up\|link is down\|error"

# Если буфер перезаписан, проверьте журналы
grep -i eth /var/log/messages  # Red Hat
grep -i eth /var/log/syslog    # Debian/Ubuntu
```

Ошибка инициализации сетевой карты выглядит так: нет строки `NIC Link is Up`, или есть строка вида `probe of 0000:xx:xx.x failed`. В этом случае проверьте, установлен ли нужный драйвер, и загрузите модуль вручную с помощью `modprobe`.

## NetworkManager

NetworkManager управляет сетевыми соединениями в большинстве современных дистрибутивов. Запускается автоматически при загрузке. Настройки хранятся в двух местах:

```
# Пользовательские настройки
/home/$USER/.gconf/system/networking/connections/

# Системные настройки
/etc/NetworkManager/
/etc/NetworkManager/system-connections/
```

```bash
systemctl status NetworkManager             # статус NetworkManager
nmcli connection show                       # показать все соединения
nmcli connection show --active              # показать активные соединения
nmcli device status                         # показать состояние устройств
nmcli device show eth0                      # показать детали интерфейса
nmcli device connect eth0                   # подключить интерфейс
nmcli device disconnect eth0                # отключить интерфейс

# Установить статический IP
nmcli connection modify "eth0" ipv4.addresses "192.168.1.100/24"
nmcli connection modify "eth0" ipv4.gateway "192.168.1.1"
nmcli connection modify "eth0" ipv4.method manual
nmcli connection up "eth0"
```

> **Внимание:** Если NetworkManager активен, он может перезаписать настройки, сделанные через `ip` или `ifconfig`, при следующем переподключении или перезагрузке. Чтобы отключить управление конкретным интерфейсом, добавьте `NM_CONTROLLED="no"` в его конфигурацию (Red Hat) или раздел `unmanaged-devices` в `/etc/NetworkManager/NetworkManager.conf`.

### Инициализация сети: Systemd против SysV

```bash
# Systemd
systemctl start networking
systemctl enable networking
systemctl start NetworkManager
systemctl enable NetworkManager

# SysV (устаревшее)
/etc/init.d/networking start
/etc/init.d/network start   # Red Hat

# Просмотр сетевых systemd-юнитов
systemctl list-units --type=service | grep -i net
```

### Типичные ошибки конфигурации

1. Нет записи маршрута для интерфейса после его настройки.
2. Два шлюза по умолчанию.
3. Неправильный DNS в `/etc/resolv.conf`.
4. `/etc/hosts` переопределяет DNS: старая запись указывает на неправильный IP.
5. TCP Wrappers блокирует доступ: адрес в `hosts.deny`, но не в `hosts.allow`.
6. NetworkManager перезаписывает ручные настройки.

## Практические вопросы

**В1.** Администратор запустил `ping 8.8.8.8` и не получил ответа. Это означает, что 8.8.8.8 недоступен.

А. Верно  
Б. Неверно — межсетевой экран может блокировать ICMP  
В. Верно, но только для IPv4  
Г. Неверно — используйте ping6

**Ответ:** Б. Межсетевой экран на промежуточном узле или на самом 8.8.8.8 может блокировать ICMP echo-request/reply. Отсутствие ответа на ping не доказывает недоступность хоста.

**В2.** В каком порядке TCP Wrappers проверяет файлы доступа для входящего соединения?

А. hosts.deny → hosts.allow → разрешить  
Б. hosts.allow → hosts.deny → разрешить  
В. hosts.allow → hosts.deny → запретить  
Г. hosts.deny → hosts.allow → запретить

**Ответ:** Б. Сначала `hosts.allow` (разрешить и выйти), затем `hosts.deny` (заблокировать); если не найдено ни в одном файле, доступ разрешается.

**В3.** Какая команда показывает путь к хосту с отображением потерь и задержки по каждому хопу в реальном времени?

А. traceroute  
Б. ping  
В. mtr  
Г. netstat

**Ответ:** В. `mtr` объединяет `ping` и `traceroute`, непрерывно отправляя пакеты и показывая статистику по каждому хопу.

**В4.** Файл `/etc/resolv.conf` был отредактирован вручную, но изменения исчезли после перезагрузки. В чём причина?

А. Файл доступен только для чтения и не принимает изменения  
Б. NetworkManager автоматически перезаписывает файл  
В. `/etc/hosts` переопределяет `/etc/resolv.conf`  
Г. DNS-сервер нужно перезапустить

**Ответ:** Б. NetworkManager управляет `/etc/resolv.conf` при активном соединении. Для постоянных изменений используйте `nmcli` или настройки в `/etc/NetworkManager/`.

**В5.** `route -n` показывает два маршрута с Destination `0.0.0.0`. Что это означает?

А. Нормально — один маршрут для IPv4, другой для IPv6  
Б. Проблема — два шлюза по умолчанию могут вызвать непредсказуемую маршрутизацию  
В. Нормально — один основной, другой резервный  
Г. Синтаксическая ошибка в команде route

**Ответ:** Б. Должен быть ровно один шлюз по умолчанию. Два шлюза по умолчанию создают непредсказуемое поведение, которое может долго не замечаться, если трафик случайно идёт по правильному пути.

**В6.** Какая команда выводит все слушающие TCP-порты с именами процессов и без разрешения имён?

А. `netstat -l`  
Б. `ss -tlnp`  
В. `ip route show`  
Г. `lsof -i tcp`

**Ответ:** Б. `ss -tlnp`: `-t` только TCP, `-l` только слушающие, `-n` без разрешения имён, `-p` с именами процессов. `netstat -tlnp` тоже подходит, но `ss` предпочтительнее.

**В7.** Администратор хочет запретить весь доступ через TCP Wrappers, кроме подсети 192.168.1.0/24. Как настроить файлы?

А. Добавить `ALL: ALL` в `hosts.allow` и `ALL: 192.168.1.` в `hosts.deny`  
Б. Добавить `ALL: 192.168.1.` в `hosts.allow` и `ALL: ALL` в `hosts.deny`  
В. Добавить только `ALL: 192.168.1.` в `hosts.allow`  
Г. Добавить только `ALL: ALL` в `hosts.deny`

**Ответ:** Б. Разрешите нужную подсеть в `hosts.allow`, запретите всех остальных через `ALL: ALL` в `hosts.deny`. Порядок проверки: сначала allow, поэтому 192.168.1.x проходит, все остальные попадают под правило deny.

**В8.** Где хранятся файлы конфигурации сетевых интерфейсов на Red Hat/CentOS?

А. `/etc/network/interfaces`  
Б. `/etc/sysconfig/network-scripts/ifcfg-*`  
В. `/etc/NetworkManager/system-connections/`  
Г. `/etc/netplan/`

**Ответ:** Б. На совместимых с Red Hat системах конфигурация eth0 находится в `/etc/sysconfig/network-scripts/ifcfg-eth0`. Вариант А — Debian/Ubuntu. Вариант Г — Ubuntu с Netplan.
