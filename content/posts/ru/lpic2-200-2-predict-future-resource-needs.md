## Диагностика использования ресурсов

Диагностика — первый шаг перед любым прогнозированием. Она опирается на инструменты, рассмотренные в теме 200.1.

### Ключевой инструмент — sar

`sar` (System Activity Reporter) — один из важнейших инструментов, поскольку **записывает измерения на длительных промежутках времени**, делая накопленные данные доступными для **анализа тенденций**.

```bash
# Показать использование CPU за сегодня (каждые 10 минут)
sar

# Показать использование CPU: 4 выборки с интервалом 1 секунда
sar 1 4

# Показать использование памяти
sar -r

# Показать активность диска
sar -d

# Показать статистику сети
sar -n DEV
```

Данные собирает **`sadc`** (System Activity Data Collector) и хранит в `/var/log/sa/` в виде файлов `saDD` (DD = день месяца).

- **`sa1`** — сохраняет данные в бинарный файл (запускается из `cron`)
- **`sa2`** — создаёт ежедневный отчёт из данных `sa1` (запускается из `cron`)

```bash
# Включить sadc на Debian/Ubuntu
/etc/default/sysstat    # установить ENABLE="true"

# Настроить sadc на RHEL/CentOS
/etc/sysconfig/sysstat  # SADC_OPTIONS="-S DISK"
```

### Установление базовой линии

Перед диагностикой проблемы установите **базовую линию** — «нормальное» поведение системы при корректной работе. Если ресурс достигает **100% загрузки**, источник проблемы очевиден, но для его точного определения может потребоваться дополнительный анализ.

Вспомогательные утилиты диагностики:

```bash
top        # интерактивный мониторинг процессов и ресурсов
vmstat     # статистика виртуальной памяти
iostat     # статистика дискового ввода/вывода
netstat    # сетевые соединения и статистика
```

## Мониторинг ИТ-инфраструктуры

**Планирование ёмкости** — официальное название процесса прогнозирования потребностей в ресурсах.

### Этапы планирования ёмкости

```
1. Понять текущие потребности пользователей системы
2. Отслеживать текущее использование ресурсов
3. Собрать информацию о будущих направлениях и ожидаемых потребностях
4. Принимать решения и делать прогнозы на основе собранных данных
```

> **Предупреждение:** Точное прогнозирование требует **документально подтверждённых данных** как о текущем использовании ресурсов, так и о **темпах его роста с течением времени**. Без этих данных любой прогноз будет крайне неточным.

### Архитектура решения мониторинга

```
┌─────────────────┐         ┌──────────────────┐
│  Сборщик        │  ──────►│  Представление   │
│  (Регистратор   │         │  (Визуализация)  │
│   данных)       │         │                  │
│  collectd       │         │  Cacti           │
│  MRTG           │         │  MRTG HTML       │
│  sadc/sar       │         │  Nagios UI       │
└─────────────────┘         └──────────────────┘
```

Решения мониторинга выполняют две роли:

- **Сборщик** — собирает и регистрирует метрики
- **Представление** — строит графики и дашборды

Многие продукты объединяют обе роли или предназначены для совместного использования.

## Инструменты мониторинга

### collectd

**collectd** — демон, который периодически собирает статистику производительности системы.

```
Назначение:    Сбор системных метрик (CPU, память, диск, сеть и т.д.)
Язык:          C (переносимость, высокая производительность)
Файл настройки: /etc/collectd/collectd.conf  (или /etc/collectd.conf)
Особенность:   Работает БЕЗ cron или языков сценариев — подходит для встраиваемых систем
```

> **Важно:** collectd только собирает данные. **collectd НЕ отображает** собранные данные — для визуализации нужны дополнительные инструменты (например, Cacti, Grafana, RRDTool).

```bash
# Установить (Debian/Ubuntu)
apt install collectd

# Основной файл конфигурации
/etc/collectd/collectd.conf

# Запустить / проверить статус
systemctl start collectd
systemctl status collectd
```

#### Конфигурация collectd — загрузка плагинов

```apache
# /etc/collectd/collectd.conf

# Интервал сбора данных (секунды)
Interval 10

# Загрузить плагины
LoadPlugin cpu
LoadPlugin memory
LoadPlugin disk
LoadPlugin interface
LoadPlugin load
LoadPlugin df
LoadPlugin network    # для удалённых хостов

# Конфигурация плагина сети (отправка данных на сервер)
<Plugin network>
    Server "monitoring.example.com" "25826"
</Plugin>
```

> **Совет:** Ключевая директива — `LoadPlugin`. Она определяет **что** будет собирать collectd. Без загрузки плагина данные по этой метрике не собираются.

### Nagios и Icinga2

#### Nagios

```
Тип:       Система мониторинга сетей и хостов
Редакции:  Nagios Core (FOSS) / Nagios XI (коммерческий)
Функции:
  - Мониторинг систем, сетевых устройств и служб
  - Пользовательские проверки через плагины
  - Уведомления об отказах
```

**Ключевые пути Nagios:**

```
/etc/nagios/               # основной каталог конфигурации
/etc/nagios/nagios.cfg     # основной файл конфигурации
/etc/nagios/objects/       # определения хостов и служб
/usr/lib/nagios/plugins/   # плагины проверок
/var/log/nagios/nagios.log # файл журнала
```

```bash
# Проверить конфигурацию
nagios -v /etc/nagios/nagios.cfg

# Запустить
systemctl start nagios
```

#### Icinga2

> **Примечание:** Icinga2 — это **форк Nagios Core** с расширенными возможностями: язык конфигурации DSL, поддержка кластеризации и современный интерфейс Icinga Web 2.

```
Тип:     Система мониторинга (форк Nagios)
Настройка: /etc/icinga2/icinga2.conf
Объекты: /etc/icinga2/conf.d/
Журнал:  /var/log/icinga2/icinga2.log
```

```icinga2
# Пример определения хоста в Icinga2
object Host "web-server" {
    address = "192.168.1.10"
    check_command = "hostalive"
}

# Пример проверки службы
object Service "http" {
    host_name = "web-server"
    check_command = "http"
}
```

### MRTG

**MRTG** (Multi Router Traffic Grapher) — собирает и визуализирует сетевой трафик.

```
Полное название: Multi Router Traffic Grapher
Язык:            Perl (переносимость)
Назначение:      Сбор и построение графиков сетевого трафика (SNMP)
Вывод:           HTML-страницы с динамическими графиками
Совместимость:   Работает с RRDTool
```

```bash
# Установить
apt install mrtg

# Сгенерировать базовую конфигурацию через cfgmaker
cfgmaker public@router.example.com > /etc/mrtg/mrtg.cfg

# Сгенерировать HTML-отчёт
indexmaker /etc/mrtg/mrtg.cfg > /var/www/html/mrtg/index.html

# Запустить (обычно из cron каждые 5 минут)
mrtg /etc/mrtg/mrtg.cfg
```

> **Совет:** MRTG использует **SNMP** для опроса сетевых устройств. Он может строить графики практически любой метрики, доступной через сетевое оборудование.

---

### Cacti

**Cacti** — решение для визуализации метрик, построенное на RRDTool.

```
Тип:           Программа для представления данных (фронтенд к RRDTool)
База данных:   MySQL
Фронтенд:      PHP
Специализация: Графики сетевого трафика + шаблоны для любых метрик
Совместимость: Графики MRTG, RRDTool
```

> **Примечание:** Cacti поддерживает использование данных, собранных MRTG, в собственных графиках. Это позволяет совместить сбор данных MRTG с богатыми возможностями визуализации Cacti.

### Сравнение инструментов

| Инструмент | Тип | Язык | Специализация | Хранение данных |
|---|---|---|---|---|
| **collectd** | Сборщик | C | Системные метрики | RRD/файлы |
| **Nagios Core** | Монитор + оповещения | C | Хосты/службы | Файлы |
| **Icinga2** | Монитор + оповещения | C++ | Хосты/службы (форк Nagios) | БД/файлы |
| **MRTG** | Сборщик + представление | Perl | Сетевой трафик (SNMP) | RRD/HTML |
| **Cacti** | Представление | PHP | Графики (фронтенд RRDTool) | MySQL |

## Прогнозирование роста

Анализируя накопленные данные мониторинга, можно **статистически прогнозировать рост** потребностей в ресурсах.

> **Предупреждение:** Прогнозирование называется **статистическим**, поскольку реальные потребности определяются внешними факторами. Например, электронная почта снизила спрос на факсы и телефонные линии. Рост не всегда линейный — при расширении инфраструктуры разные службы растут с разной скоростью.

### Алгоритм прогнозирования

```
Шаг 1: Определить, что измерять
        └─► CPU, ОЗУ, диск, сеть, ввод/вывод...

Шаг 2: Выбрать подходящие инструменты для измерения и записи данных
        └─► sar, collectd, Nagios, MRTG, Cacti...

Шаг 3: Проанализировать результаты измерений
        └─► Начать с наибольших отклонений

Шаг 4: Предсказать будущие потребности на основе анализа
        └─► Экстраполировать тенденцию → точка исчерпания ёмкости
```

### Точка исчерпания ёмкости

```
Текущее использование + темп роста → прогнозируемая дата исчерпания ресурса

Например: диск заполнен на 70%, растёт на 5% в месяц
          → примерно через 6 месяцев диск будет заполнен на 100% (точка исчерпания)
```

**Ключевые термины для экзамена:**

- **точка исчерпания ёмкости** — момент, когда ресурс перестаёт справляться с нагрузкой
- **темп роста** — скорость увеличения использования ресурса
- **график тенденций** — график для визуализации прогноза
- **базовая линия** — эталонная линия «нормального» поведения

## Исчерпание ресурсов

**Исчерпание ресурсов** — состояние, при котором ресурс **не может нормально обслуживать запросы**. Предложение и спрос рассинхронизированы.

### Последствия

```
Исчерпание ресурсов
       │
       ├──► Отказ в обслуживании (DoS)
       │
       └──► Уязвимость «fail open»:
            Устройства в режиме «fail open» могут быть
            взломаны через преднамеренное исчерпание ресурсов
```

> **Предупреждение — реальный пример: ARP-флуд на коммутаторах.** Некоторые **сетевые коммутаторы** переходят в режим **«fail open»** при переполнении таблицы ARP и начинают **пересылать весь трафик на все порты** (флуд), фактически превращаясь из коммутатора в концентратор. Это позволяет злоумышленнику перехватывать трафик других пользователей (пассивный перехват).

### Узкое место

> **Важно:** **Узкое место** — единственная точка в системе, ограничивающая пропускную способность и **замедляющая всё остальное**. Исчерпание ресурсов одного компонента обычно видно в собранных данных мониторинга.

```
Система с узким местом:

  [CPU 30%] → [ОЗУ 40%] → [DISK I/O 100% ← УЗКОЕ МЕСТО] → [СЕТЬ 20%]
                                    ↑
                          Весь поток «стопорится» здесь
```

> **Предупреждение:** Устранение одного узкого места не решает проблему полностью. После устранения конкретного узкого места (например, замены диска) **следующий наиболее нагруженный компонент становится новым ограничением**. Поэтому при анализе важно выявить **как можно больше потенциальных узких мест одновременно**, а не только самое очевидное.

```
До замены диска:              После замены диска:
[CPU 30%]                     [CPU 95% ← новое УЗКОЕ МЕСТО]
[ОЗУ 40%]                     [ОЗУ 40%]
[DISK 100% ← узкое место]     [DISK 50%]
                                   ↑
                    Проблема сместилась, а не исчезла
```

### Типичные сценарии исчерпания

| Ресурс | Симптом | Инструмент диагностики |
|---|---|---|
| CPU | 100% нагрузка, деградация служб | `top`, `sar -u`, `uptime` |
| ОЗУ | переполнение swap, OOM killer | `free`, `vmstat`, `sar -r` |
| Диск | нет места для записи, сбои служб | `df`, `iostat`, `sar -d` |
| Сеть | потери пакетов, высокая задержка | `netstat`, `sar -n DEV` |
| Файловые дескрипторы | «too many open files» | `lsof`, `ulimit -n` |
| Inodes | нет места для новых файлов | `df -i` |
| Таблица ARP | коммутатор → режим флуда (fail open) | `arp -n`, `ip neigh` |

## Шпаргалка для экзамена

### Ключевые команды

```bash
sar                          # просмотр исторических данных CPU
sar -r                       # статистика памяти
sar -d                       # статистика диска
sar -n DEV                   # статистика сети
sar 1 5                      # 5 выборок с интервалом 1 секунда

# Хранение данных sar:
/var/log/sa/saDD             # бинарные файлы (DD = день месяца)
/etc/default/sysstat         # настройка sysstat (Debian): ENABLE="true"
/etc/sysconfig/sysstat       # настройка sysstat (RHEL)

# collectd
/etc/collectd/collectd.conf  # основная конфигурация
LoadPlugin cpu               # ключевая директива загрузки плагина
systemctl start collectd

# MRTG
cfgmaker public@router > /etc/mrtg/mrtg.cfg
mrtg /etc/mrtg/mrtg.cfg

# Nagios
/etc/nagios/nagios.cfg
nagios -v /etc/nagios/nagios.cfg   # проверить конфигурацию

# Icinga2
/etc/icinga2/icinga2.conf
```

### Сводка инструментов

```
collectd  → демон, C, системные метрики, /etc/collectd/collectd.conf
Cacti     → PHP-фронтенд, MySQL, RRDTool, сетевой трафик
MRTG      → Perl, SNMP, сетевой трафик, HTML-страницы
Nagios    → мониторинг хостов/служб, Core (FOSS) + XI (коммерческий)
Icinga2   → форк Nagios, конфигурация DSL, /etc/icinga2/
sar/sadc  → встроен в Linux, /var/log/sa/, исторические данные
```

### Типичные ловушки на экзамене

1. **collectd** — только **собирает** данные, без визуализации
2. **Cacti** — только **визуализация** (фронтенд), не собирает данные
3. **MRTG** — специализируется на **сетевом трафике** (не системных метриках)
4. **Icinga2** — является **форком Nagios**, а не самостоятельной разработкой
5. Конфигурация collectd: ключевая директива — `LoadPlugin`, а не `Plugin` или `Enable`
6. Данные `sar` хранятся в `/var/log/sa/`, а не в `/var/log/sysstat/`

## Практические вопросы

**В1.** Какой инструмент является C-демоном, собирающим статистику производительности, но **не отображающим** её?

**Ответ:** `collectd`

---

**В2.** В каком каталоге `sadc` хранит собранные данные по умолчанию?

**Ответ:** `/var/log/sa/`

---

**В3.** Как официально называется процесс прогнозирования будущих потребностей в ресурсах?

**Ответ:** **Планирование ёмкости** (capacity planning)

---

**В4.** Какой инструмент является фронтендом к RRDTool, использует MySQL для хранения и PHP для веб-интерфейса?

**Ответ:** **Cacti**

---

**В5.** Что происходит с устройством в режиме «fail open» при исчерпании его ресурсов?

**Ответ:** Устройство начинает пропускать весь трафик/запросы без проверки, что может использоваться как вектор атаки (обход защиты).

---

**В6.** Какой инструмент является форком Nagios с поддержкой конфигурации DSL и кластеризации?

**Ответ:** **Icinga2**

---

**В7.** Перечислите четыре шага алгоритма прогнозирования будущих потребностей в ресурсах.

**Ответ:**
1. Определить, что измерять
2. Выбрать подходящие инструменты для измерения и записи данных
3. Проанализировать результаты (начиная с наибольших отклонений)
4. Предсказать будущие потребности на основе анализа

---

**В8.** Какая директива в `collectd.conf` отвечает за загрузку плагинов сбора данных?

**Ответ:** `LoadPlugin` (например, `LoadPlugin cpu`, `LoadPlugin memory`)

---

**В9.** Что такое **узкое место** и почему его устранение не всегда решает проблему?

**Ответ:** Узкое место — единственная точка в системе, ограничивающая пропускную способность и замедляющая всё остальное. После устранения одного узкого места **следующий наиболее нагруженный компонент становится новым ограничением**. Поэтому при анализе нужно выявить как можно больше потенциальных узких мест, а не только самое очевидное.

---

**В10.** Как ведёт себя сетевой коммутатор при переполнении таблицы ARP и почему это опасно?

**Ответ:** Коммутатор переходит в режим **«fail open»** и начинает пересылать трафик **на все порты** (флуд), превращаясь из коммутатора в концентратор. Злоумышленник в той же сети может **перехватывать трафик других пользователей** (пассивный перехват).
