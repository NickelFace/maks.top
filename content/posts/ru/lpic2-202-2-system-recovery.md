## Процесс загрузки Linux

```
BIOS/UEFI  →  GRUB (MBR/ESP)  →  Kernel  →  initramfs  →  init/systemd
```

| Фаза | Описание |
|---|---|
| 1 | Прошивка (BIOS/UEFI) выполняет POST и находит загрузчик |
| 2 | Загрузчик (GRUB) загружается из MBR или ESP, показывает меню, передаёт управление ядру |
| 3 | Ядро распаковывается в памяти, инициализирует оборудование, монтирует корневую файловую систему |
| 4 | initrd/initramfs предоставляет минимальное окружение для загрузки модулей перед монтированием `/` |
| 5 | init/systemd запускает пользовательские службы и цели |

> **Exam tip:** Знайте точную последовательность и какой инструмент отвечает за каждую фазу.

## MBR и размещение загрузчика

**MBR (Master Boot Record)** — первый 512-байтный сектор первого диска. Для кода загрузчика доступно лишь 446 байт; остаток занимают таблица разделов и сигнатура.

Из-за этого ограничения GRUB использует двухэтапный подход:

- **Stage 1** (446 байт в MBR) — указывает на Stage 1.5 / Stage 2
- **Stage 1.5 / core.img** — хранится в зоне DOS Compatibility Region (первый цилиндр диска); содержит код для работы с файловой системой
- **Stage 2** — основная логика загрузчика, хранится в `/boot/grub/`

```bash
# Установить GRUB на MBR первого диска
grub-install /dev/sda

# Установить на MBR в формате GRUB Legacy
grub-install '(hd0)'

# Установить на загрузочный сектор раздела (цепочечная загрузка)
grub-install /dev/sda1
```

> **Внимание:** `grub-install /dev/sda` устанавливает загрузчик в MBR всего диска. `grub-install /dev/sda1` — в загрузочный сектор раздела. Перепутаете — сломаете загрузку.

## GRUB Legacy

GRUB Legacy разработан в 1999 году и заморожен. Встречается на старых системах.

### Настройка

Файл конфигурации: `/boot/grub/menu.lst` (или симлинк `/boot/grub/grub.conf`).

```bash
# /boot/grub/menu.lst — пример
default 0          # загружать первую запись по умолчанию
timeout 10         # ждать 10 секунд
color white/blue yellow/blue

title CentOS Linux
root (hd1,0)       # второй диск, первый раздел (нумерация с 0!)
kernel (hd1,0)/boot/vmlinuz root=/dev/sdb1
initrd /boot/initrd

title Windows
rootnoverify (hd0,0)
chainloader +1
```

> **Важно:** В GRUB Legacy **и** диски, **и** разделы нумеруются с 0. Первый диск, первый раздел: `(hd0,0)`. В GRUB 2 диски нумеруются с 0, разделы — с 1: `(hd0,1)`.

### Основные директивы GRUB Legacy

| Директива | Назначение |
|---|---|
| `default N` | Загружать N-ю запись по умолчанию |
| `timeout N` | Таймаут в секундах |
| `title` | Начало пункта меню |
| `root (hdX,Y)` | Раздел, содержащий `/boot` |
| `kernel` | Путь к ядру и параметры |
| `initrd` | Путь к образу initrd |
| `rootnoverify` | Не-Linux раздел (Windows) |
| `chainloader +1` | Передать управление следующему загрузчику |

### Интерактивный режим GRUB Legacy

В меню загрузки:

- Стрелки для выбора записи
- `e` — редактировать выбранную запись
- `b` — загрузиться после редактирования
- `c` — открыть командную строку GRUB

Чтобы загрузиться в однопользовательский режим, добавьте `single` в конец строки `kernel`:

```
kernel /boot/vmlinuz root=/dev/sda1 single
```

### Оболочка GRUB (Legacy)

Оболочка GRUB Legacy запускается командой `grub` из работающей системы. Она эмулирует поведение загрузчика: позволяет просматривать конфигурацию, устанавливать загрузчик и вручную загружать систему.

```bash
grub
grub> root (hd0,0)
grub> kernel /boot/vmlinuz root=/dev/sda1
grub> initrd /boot/initrd
grub> boot
```

Основные команды оболочки GRUB Legacy:

| Команда | Назначение |
|---|---|
| `boot` | Загрузить систему |
| `cat FILE` | Вывести содержимое файла |
| `chainloader FILE` | Загрузить другой загрузчик |
| `configfile FILE` | Загрузить файл конфигурации |
| `find FILENAME` | Найти файл на всех разделах |
| `halt` | Выключить систему |
| `initrd FILE` | Загрузить образ initrd |
| `kernel FILE` | Загрузить ядро |
| `reboot` | Перезагрузить систему |
| `root [DEVICE]` | Выбрать корневое устройство |
| `rootnoverify [DEVICE]` | Выбрать устройство без монтирования (для Windows) |
| `setup (hd0)` | Установить GRUB на MBR |
| `quit` | Выйти из оболочки GRUB |

> **Примечание:** Команда `grub` доступна только в GRUB Legacy. Аналогичного отдельного инструмента для GRUB 2 не существует — вместо него используется интерактивная командная строка в меню загрузки.

## GRUB 2

GRUB 2 разработан около 2011 года. Он модульный: поддержка файловых систем (ext4, btrfs, zfs, NTFS), LVM и программных RAID-массивов загружается в виде отдельных модулей.

### Настройка

| Файл | Назначение |
|---|---|
| `/boot/grub/grub.cfg` | Основной конфиг (не редактировать вручную!) |
| `/etc/default/grub` | Глобальные настройки (GRUB_TIMEOUT и т.д.) |
| `/etc/grub.d/` | Скрипты для генерации grub.cfg |

```bash
# Сгенерировать grub.cfg из скриптов в /etc/grub.d/
grub-mkconfig -o /boot/grub/grub.cfg

# update-grub — обёртка Debian/Ubuntu вокруг grub-mkconfig
update-grub
```

Скрипты в `/etc/grub.d/` управляют записями меню. `/etc/grub.d/30_os_prober` автоматически обнаруживает установленные ОС, включая Windows. Добавляйте собственные записи в `/etc/grub.d/40_custom`.

Пример записи в `grub.cfg`:

```bash
menuentry "CentOS Linux" {
    set root=(hd1,1)       # второй диск, первый раздел (GRUB 2: разделы с 1!)
    linux /boot/vmlinuz root=/dev/sdb1 ro quiet
    initrd /boot/initramfs.img
}

menuentry "Windows" {
    set root=(hd0,1)
    chainloader +1
}
```

### Сравнение GRUB Legacy и GRUB 2

| Параметр | GRUB Legacy | GRUB 2 |
|---|---|---|
| Файл конфигурации | `/boot/grub/menu.lst` | `/boot/grub/grub.cfg` |
| Редактирование | Напрямую | Только через `/etc/grub.d/` + `grub-mkconfig` |
| Нумерация разделов | С 0: `(hd0,0)` | Диски с 0, разделы с 1: `(hd0,1)` |
| Загрузка ядра | `kernel` | `linux` |
| Установка корня | `root (hd0,0)` | `set root=(hd0,1)` |
| Загрузка модуля | `module` | `insmod` |
| Оболочка | `grub` (отдельная) | Интерактивная командная строка только в меню загрузки |

### device.map

GRUB использует BIOS для обнаружения дисков, но не всегда может сопоставить имена BIOS с именами устройств Linux. Это сопоставление хранится в `/boot/grub/device.map`:

```bash
(fd0)  /dev/fd0
(hd0)  /dev/sda
(hd1)  /dev/sdb
```

> **Внимание:** При программном RAID-1 (зеркалировании) устанавливайте GRUB на **оба** диска. В момент загрузки программный RAID ещё не активен, поэтому система читает только один физический диск. Если GRUB установлен только на первый диск и он выйдет из строя, система не загрузится.
> ```bash
> grub-install /dev/sda
> grub-install /dev/sdb
> ```

### Интерактивный режим GRUB 2

В меню загрузки:

- `e` — редактировать запись (добавьте `single` или другие параметры в строку `linux`)
- `Ctrl+x` — загрузиться после редактирования
- `c` — войти в командную строку GRUB 2

Если дистрибутив скрывает меню GRUB, нажмите `Shift` сразу после инициализации BIOS/UEFI.

```bash
# Ручная загрузка из командной строки GRUB 2
grub> set root=(hd0,1)
grub> linux /boot/vmlinuz root=/dev/sda1 ro
grub> initrd /boot/initramfs.img
grub> boot
```

### Команды оболочки GRUB 2

| Команда | Назначение |
|---|---|
| `ls` | Список устройств и файлов |
| `set root=(hdX,Y)` | Выбрать корневой раздел |
| `linux /path/vmlinuz` | Загрузить ядро |
| `initrd /path/initramfs` | Загрузить initramfs |
| `boot` | Загрузить систему |
| `insmod ext2` | Загрузить модуль файловой системы |
| `search --label myroot` | Найти раздел по метке |
| `search --fs-uuid UUID` | Найти раздел по UUID |
| `chainloader +1` | Цепочечная загрузка другого загрузчика |
| `configfile /boot/grub/grub.cfg` | Загрузить файл конфигурации |
| `reboot` / `halt` | Перезагрузить / выключить систему |

> **Важно:** Устройства в GRUB записываются в скобках: `(fd0)` = дискета, `(hd0,1)` = первый диск, первый раздел (GRUB 2). Без скобок команда не выполнится.

## initrd и initramfs

**initrd** (initial RAM disk) и **initramfs** (initial RAM filesystem) — временные корневые файловые системы, которые ядро монтирует перед переключением на настоящую `/`.

Нужны для загрузки модулей (драйверов), необходимых для доступа к реальной корневой файловой системе (например, драйверы ext4, LVM, RAID, шифрования).

| | initrd | initramfs |
|---|---|---|
| Формат | Образ блочного устройства (ext2) | cpio-архив, сжатый gzip |
| Монтирование | Как блочное устройство | Распаковывается напрямую в tmpfs |
| Статус | Устаревший подход | Действующий стандарт |

```bash
# Создать initramfs (Debian/Ubuntu)
update-initramfs -u

# Создать initramfs (Red Hat/CentOS)
dracut --force /boot/initramfs-$(uname -r).img $(uname -r)

# Создать с помощью mkinitrd (RPM-дистрибутивы, устаревший инструмент)
mkinitrd /boot/initrd-$(uname -r).img $(uname -r)
```

> **Примечание:** В Debian-based системах файл называется `initrd.img-VERSION`; в Red Hat-based — `initramfs-VERSION.img`. Оба загрузчика — GRUB Legacy и GRUB 2 — используют директиву `initrd` для загрузки образа.

## UEFI и NVMe

### UEFI vs BIOS

Ограничения BIOS: хранит загрузчик в MBR (446 байт), не поддерживает диски объёмом более 2 ТБ, нет нативной поддержки GPT. Intel разработала EFI в 1998 году; в 2005 году производители приняли стандарт под названием UEFI.

| Параметр | BIOS | UEFI |
|---|---|---|
| Расположение загрузчика | MBR (446 байт) | Раздел ESP (FAT, любой размер) |
| Таблица разделов | MBR (макс. 2 ТБ, 4 раздела) | GPT (до 9,4 ЗБ, 128 разделов) |
| Безопасность | Отсутствует | Secure Boot |
| Сетевая загрузка | Отсутствует | IPv4/IPv6, TFTP, HTTP (UEFI 2.5+) |
| Точка монтирования ESP | — | `/boot/efi` |

**ESP (EFI System Partition)** — специальный раздел, отформатированный в FAT. Загрузочные файлы имеют расширение `.efi` и располагаются в `/boot/efi/EFI/`.

```
/boot/efi/EFI/BOOT/BOOTX64.EFI      # универсальный резервный загрузчик
/boot/efi/EFI/fedora/shim.efi       # загрузчик первого уровня для Secure Boot
/boot/efi/EFI/fedora/MokManager.efi # управление ключами Secure Boot
/boot/efi/EFI/fedora/fwupx64.efi    # обновление прошивки
```

Для управления записями загрузки UEFI используйте `efibootmgr`.

> **Важно:** Ядро Linux 3.15+ поддерживает UEFI. На экзамене могут спросить, где монтируется ESP: ответ — `/boot/efi`.

### NVMe

NVMe (Non-Volatile Memory Express) — протокол для SSD-накопителей, подключаемых через PCIe. Он заменяет устаревший AHCI (1 очередь, 32 команды) на NVMHCI (65 000 очередей по 65 000 команд каждая).

Именование устройств NVMe в Linux:

```
/dev/nvme0        # первый контроллер NVMe
/dev/nvme0n1      # первое пространство имён на первом контроллере
/dev/nvme0n1p1    # первый раздел первого пространства имён
```

> **Важно:** Нумерация NVMe: контроллер нумеруется с 0, пространство имён и раздел — с 1. `/dev/nvme0n1p1` = первый раздел первого диска. Частый вопрос на экзамене.

## Сценарии восстановления системы

### Сценарий 1: ядро не загружается

Используйте стрелки для выбора записи в меню GRUB и нажмите Enter. Если новое ядро не загружается, выберите предыдущее из списка.

### Загрузка в однопользовательский режим через GRUB

**GRUB 2:**
1. Нажмите `e` на нужной записи.
2. Найдите строку, начинающуюся с `linux`.
3. Добавьте `single` в конец строки.
4. Нажмите `Ctrl+x` для загрузки.

**GRUB Legacy:**
1. Нажмите `e` на нужной записи.
2. Выберите строку, начинающуюся с `kernel`.
3. Добавьте `single` в конец.
4. Нажмите `Enter`, затем `b` для загрузки.

### Передача параметров ядра через GRUB

Если устройство не обнаруживается при загрузке, можно вручную передать параметры драйверу ядра. Это работает только если поддержка устройства **скомпилирована в ядро**, а не загружается как модуль.

Пример: два сетевых адаптера, скомпилированных в ядро. Ядро находит только один по умолчанию. Добавьте в строку `linux` / `kernel`:

```
ether=5,0x300,eth0 ether=11,0x340,eth1
```

### Сценарий 2: повреждённая корневая файловая система

```bash
# 1. Загрузиться с диска восстановления (CD/USB) — система работает полностью в RAM

# 2. Проверить файловую систему на /dev/sda2
fsck -y /dev/sda2
# -y автоматически отвечает "да" на все запросы fsck

# 3. Смонтировать восстановленный раздел
mount /dev/sda2 /target

# 4. Проверить содержимое
ls /target

# 5. Размонтировать перед перезагрузкой
umount /target

# 6. Перезагрузить систему
reboot
```

> **Внимание:** Всегда выполняйте `umount` для всех смонтированных разделов перед перезагрузкой. Иначе ядро увидит некорректно размонтированные тома и запустит fsck снова при следующей загрузке.

### Коды возврата fsck

| Код | Значение |
|---|---|
| 0 | Ошибок нет |
| 1 | Ошибки найдены и исправлены |
| 2 | Рекомендуется перезагрузка |
| 4 | Ошибки найдены, но не исправлены |
| 8 | Операционная ошибка |
| 16 | Ошибка синтаксиса или использования |
| 128 | Ошибка разделяемой библиотеки |

> **Важно:** Никогда не запускайте fsck на смонтированной файловой системе. Для проверки `/` — загрузитесь с диска восстановления или сначала перейдите в однопользовательский режим.

### Сценарий 3: сбой загрузки (fsck не прошёл)

В Debian файловые системы проверяются через `/etc/rcS.d/S30check.fs` с использованием `/etc/fstab`. Если `fsck` возвращает код больше 1, загрузка останавливается и система отображает:

```
fsck failed. Please repair manually

"CONTROL-D" will exit from this shell and
continue system startup.
```

Два варианта действий:

- Нажать `Ctrl+D` — система продолжит загрузку без исправлений (опасно при серьёзных повреждениях ФС)
- Ввести пароль root — запустится `/sbin/sulogin` и предоставит командную строку для восстановления

На этом этапе корневая файловая система монтируется **только для чтения** — это правильно и ожидаемо для запуска fsck на корневом разделе.

```bash
fsck -y /dev/sda1
reboot
```

### Сценарий 4: восстановление GRUB

```bash
# Переустановить GRUB на MBR (с диска восстановления)
grub-install /dev/sda

# Сгенерировать grub.cfg заново (GRUB 2)
grub-mkconfig -o /boot/grub/grub.cfg

# На Debian/Ubuntu
update-grub
```

## Структура /boot/ и /boot/grub/

```
/boot/
├── vmlinuz-5.15.0-91-generic    # сжатый образ ядра
├── System.map-5.15.0-91-generic # таблица символов ядра
├── initrd.img-5.15.0-91-generic # образ initramfs
├── config-5.15.0-91-generic     # конфигурация сборки ядра
├── grub/
│   ├── grub.cfg                 # конфиг GRUB 2 (не редактировать вручную)
│   ├── menu.lst                 # конфиг GRUB Legacy
│   ├── device.map               # сопоставление дисков BIOS с устройствами Linux
│   └── i386-pc/                 # модули GRUB 2 для BIOS
└── efi/                         # только для систем с UEFI
    └── EFI/
        ├── BOOT/
        │   └── BOOTX64.EFI      # универсальный резервный загрузчик
        └── fedora/
            ├── shim.efi
            ├── MokManager.efi
            └── fwupx64.efi
```

> **Важно:** `/boot/grub/grub.cfg` автоматически генерируется командой `grub-mkconfig`. Редактировать его напрямую опасно — следующий запуск `update-grub` перезапишет все изменения. Редактируйте `/etc/default/grub` и файлы в `/etc/grub.d/`.

## Шпаргалка для экзамена

### Файлы и пути

| Файл / Путь | Назначение |
|---|---|
| `/boot/grub/menu.lst` | Конфиг GRUB Legacy |
| `/boot/grub/grub.cfg` | Конфиг GRUB 2 (генерируется автоматически) |
| `/etc/default/grub` | Глобальные настройки GRUB 2 |
| `/etc/grub.d/` | Скрипты для генерации grub.cfg |
| `/boot/efi/EFI/` | Загрузочные файлы UEFI (.efi) |
| `/etc/inittab` | Конфигурация SysV init |
| `/etc/fstab` | Таблица файловых систем (fsck читает её при загрузке) |
| `/sbin/telinit` | Переключение уровня выполнения (симлинк на init) |

### Основные команды

```bash
grub-install /dev/sda                          # установить GRUB на MBR
grub-mkconfig -o /boot/grub/grub.cfg           # сгенерировать grub.cfg заново
update-grub                                    # Debian/Ubuntu: то же самое
fsck -y /dev/sda1                              # проверить и восстановить файловую систему
mount /dev/sda1 /mnt                           # смонтировать раздел
umount /dev/sda1                               # размонтировать перед перезагрузкой
telinit 1                                      # перейти в однопользовательский режим
init q                                         # повторно прочитать /etc/inittab
```

### Типичные ловушки на экзамене

1. GRUB Legacy: разделы с 0 (`hd0,0`). GRUB 2: диски с 0, разделы с 1 (`hd0,1`).
2. `grub.cfg` нельзя редактировать вручную — только через `grub-mkconfig`.
3. Никогда не запускайте fsck на смонтированной файловой системе.
4. После fsck и монтирования в режиме восстановления всегда выполняйте `umount` перед перезагрузкой.
5. `telinit` — обычно симлинк на `init`, а не отдельный бинарный файл.
6. NVMe: `/dev/nvme0n1p1` = первый раздел. Контроллер с 0, пространство имён и раздел с 1.
7. ESP монтируется в `/boot/efi`; загрузочные файлы имеют расширение `.efi`.
8. Команда `grub` доступна только в GRUB Legacy. GRUB 2 имеет только интерактивную командную строку в меню загрузки.

## Практические вопросы

**В1.** Администратор хочет установить GRUB 2 на MBR первого диска. Какая команда это делает?

A. `grub-install /dev/sda1` B. `grub-install /dev/sda` C. `grub-mkconfig /dev/sda` D. `update-grub /dev/sda`

**Ответ:** B. `grub-install /dev/sda` устанавливает Stage 1 на MBR `/dev/sda`. Вариант A устанавливает на загрузочный сектор раздела (для цепочечной загрузки). `grub-mkconfig` только генерирует файл конфигурации.

---

**В2.** В GRUB 2 нужно указать первый раздел второго диска как корневой. Как это записывается?

A. `root (hd1,0)` B. `set root=(hd2,1)` C. `set root=(hd1,1)` D. `root (hd2,0)`

**Ответ:** C. В GRUB 2 диски нумеруются с 0 (второй диск = hd1), разделы — с 1 (первый раздел = 1). Варианты A и D используют синтаксис GRUB Legacy.

---

**В3.** После замены диска система не загружается. Администратор загружается с USB-диска восстановления и хочет восстановить файловую систему на `/dev/sdb2`. Какая команда правильная?

A. `fsck /dev/sdb2` (раздел смонтирован в `/mnt`)
B. `fsck -y /dev/sdb2` (раздел не смонтирован)
C. `mount -o remount,rw /dev/sdb2`
D. `e2fsck -n /dev/sdb2`

**Ответ:** B. fsck нельзя запускать на смонтированном разделе. Флаг `-y` автоматически подтверждает все исправления.

---

**В4.** Какой файл `grub-mkconfig` читает для получения глобальных настроек GRUB 2?

A. `/boot/grub/grub.cfg` B. `/etc/grub.d/00_header` C. `/etc/default/grub` D. `/boot/grub/menu.lst`

**Ответ:** C. `/etc/default/grub` содержит глобальные настройки, такие как `GRUB_TIMEOUT` и `GRUB_CMDLINE_LINUX`. В `/etc/grub.d/` находятся скрипты для каждого пункта меню.

---

**В5.** Администратор хочет переключить работающую систему SysV init в однопользовательский режим без перезагрузки. Какая команда это делает?

A. `init 0` B. `telinit 1` C. `systemctl rescue` D. `shutdown -s`

**Ответ:** B. `telinit 1` переключает систему на уровень выполнения 1 (однопользовательский режим) без перезагрузки. `init 0` выключает систему.

---

**В6.** Где расположены загрузочные файлы UEFI на Linux-системе?

A. `/boot/grub/` B. `/etc/grub.d/` C. `/boot/efi/EFI/` D. `/var/lib/efi/`

**Ответ:** C. ESP монтируется в `/boot/efi`; загрузочные файлы с расширением `.efi` хранятся в `/boot/efi/EFI/`.

---

**В7.** Как называется первый раздел первого пространства имён первого NVMe-диска в Linux?

A. `/dev/nvme0n0p0` B. `/dev/nvme1n1p1` C. `/dev/nvme0n1p1` D. `/dev/nvme0p1`

**Ответ:** C. Контроллер нумеруется с 0 (`nvme0`), пространство имён и раздел — с 1 (`n1p1`).

---

**В8.** Администратор отредактировал `/etc/inittab`. Как применить изменения без перезагрузки?

A. `telinit q` B. `init reload` C. `init q` D. `systemctl daemon-reload`

**Ответ:** C (также A, поскольку telinit является симлинком на init). `init q` или `telinit q` даёт команду init повторно прочитать `/etc/inittab`.
