> **Тема экзамена 206.3** — Уведомление пользователей о системных проблемах (вес: 1). Охватывает статические сообщения при входе, динамические широковещательные сообщения и управление службами и целями systemd.

---

## Обзор

Все методы уведомления пользователей делятся на два типа:

- **Статические** (logon messaging): файлы, отображаемые при входе в систему. Содержимое меняется только при обновлении самого сообщения.
- **Динамические** (fluid messaging): инструменты для мгновенной рассылки сообщений всем активным пользователям.

| Инструмент | Тип | Когда отображается | Кому |
|---|---|---|---|
| `/etc/issue` | Статический | До входа (pre-login) | Локальные tty-терминалы |
| `/etc/issue.net` | Статический | До входа (pre-login) | Telnet, SSH (если настроен) |
| `/etc/motd` | Статический | После входа (post-login) | Локальные tty-терминалы |
| `wall` | Динамический | Немедленно | Все активные терминалы |
| `shutdown` | Динамический | По расписанию + немедленно | Все активные терминалы |

## Статические сообщения при входе

### /etc/issue

`/etc/issue` отображается на экране входа tty **до** того, как пользователь вводит имя и пароль. Типичное использование: политика доступа компании, предупреждение о несанкционированном доступе, объявление о плановом обслуживании.

По умолчанию файл показывает системную информацию через управляющие последовательности:

```bash
cat /etc/issue
# \S
# Kernel \r on an \m
```

Пример изменённого файла с объявлением о техническом обслуживании:

```
########################################################
 УВЕДОМЛЕНИЕ
 Система будет недоступна для технического обслуживания
 Когда: 26 декабря с 1:00 до 1:30
########################################################
```

> **Внимание:** Отображение дистрибутива и версии ядра в `/etc/issue` считается плохой практикой. Злоумышленник получает ценную информацию до аутентификации. С юридической точки зрения лучше добавить уведомление о запрете несанкционированного доступа.

### /etc/issue.net

`/etc/issue.net` выполняет ту же роль, что и `/etc/issue`, но предназначен для **удалённых подключений**. По умолчанию активен только для Telnet. Чтобы включить для SSH, отредактируйте конфигурацию SSH:

```bash
# /etc/ssh/sshd_config
# Раскомментировать строку:
Banner /etc/issue.net
```

После редактирования перезапустите SSH-сервер:

```bash
sudo systemctl restart sshd
```

> **Примечание:** При использовании `/etc/issue.net` через SSH специальные управляющие последовательности (например, `\r` для версии ядра) могут работать некорректно. Надёжнее использовать простой текст.

### /etc/motd

`/etc/motd` (Message of the Day — сообщение дня) отображается пользователю **после** успешной аутентификации, но до появления приглашения командной строки. Традиционно используется для неформальных объявлений; сегодня чаще применяется для уведомлений о предстоящем обслуживании, а `/etc/issue` и `/etc/issue.net` оставляются для юридических предупреждений.

На некоторых дистрибутивах файл существует, но пуст. На других его нет и нужно создать:

```bash
sudo nano /etc/motd
```

Пример содержимого:

```
Добро пожаловать на server01.example.com

Плановое обслуживание: воскресенье 2:00 - 4:00
Пожалуйста, сохраните свою работу заблаговременно.

Поддержка: helpdesk@example.com
```

> **Совет:** В Ubuntu и Debian каталог `/etc/update-motd.d/` содержит скрипты, которые динамически генерируют `/etc/motd` при каждом входе, автоматически отображая актуальную системную информацию.

## Динамические сообщения активным пользователям

### wall

`wall` (write all) отправляет сообщение **всем активным терминалам** одновременно. Длина сообщения ограничена 22 строками.

```bash
wall "Перезагрузка сервера через 30 минут. Пожалуйста, сохраните свою работу."

# Отправить сообщение из файла
wall /tmp/maintenance_notice.txt

# Подавить стандартный заголовок широковещательного сообщения (только root)
wall -n "Срочно: диск /dev/sdb отказывает, создайте резервную копию СЕЙЧАС"
```

По умолчанию любой пользователь может запускать `wall`. На практике администраторы часто ограничивают его использование только root.

Пользователь, не желающий получать широковещательные сообщения, может отключить запись в свой терминал:

```bash
mesg n    # запретить запись в терминал
mesg y    # снова разрешить
```

Какие терминалы запретили запись, можно проверить с помощью `finger`.

> **Примечание:** Опция `-n` команды `wall` работает только для root и подавляет стандартный заголовок широковещательного сообщения. Она имеет смысл только при использовании через демон `rpc.walld`.

### shutdown

`shutdown` автоматически отправляет широковещательные сообщения по мере приближения времени выключения и в момент самого выключения.

```bash
# Выключить через 10 минут с сообщением
shutdown -h +10 "Остановка сервера для замены оборудования. Ожидаемое время возобновления: 00:00:00."

# Перезагрузка в указанное время
shutdown -r 02:00 "Плановая перезагрузка для обновления ядра."

# Немедленное выключение
shutdown -h now

# Репетиция: отправляет предупреждения, но НЕ выключает систему
sudo shutdown -k +20 "Пожалуйста, выйдите из системы, скоро начнётся обслуживание."

# Выключить без широковещательных сообщений
shutdown --no-wall +5

# Отменить запланированное выключение
shutdown -c "Обслуживание перенесено на следующую неделю."
```

> **Внимание:** `shutdown -k` создаёт `/etc/nologin`, что временно блокирует новые входы пользователей — даже несмотря на то, что система на самом деле не выключается. Файл удаляется по окончании периода репетиции, но пока он существует, новые входы невозможны.

> **Внимание:** Если вы запустили `shutdown --no-wall +5`, а затем отменили с помощью `shutdown -c`, пользователи всё равно получат сообщение об отмене. Чтобы подавить и его, используйте `shutdown -c --no-wall`.

**Опции shutdown, связанные с уведомлениями:**

| Опция | Действие |
|---|---|
| `+N` | Выполнить через N минут |
| `HH:MM` | Выполнить в указанное время |
| `-k` | Репетиция: предупреждения без реального выключения |
| `--no-wall` | Выключить без широковещательных сообщений |
| `-c` | Отменить запланированное выключение |
| `-h` | Остановить после выключения |
| `-r` | Перезагрузить после выключения |

## systemctl — управление состоянием системы

В системах на базе systemd `systemctl` заменяет несколько устаревших инструментов. Это центральный инструмент управления системой инициализации systemd: службами, состояниями системы (целями/уровнями выполнения) и файлами конфигурации юнитов.

### Запуск и остановка служб

```bash
sudo systemctl start application.service
sudo systemctl stop application.service

# Суффикс .service можно опустить
sudo systemctl start application
```

Несколько экземпляров одной службы с разными конфигурациями используют синтаксис `@`:

```bash
sudo systemctl start openvpn@config1.service
sudo systemctl start openvpn@config2.service
```

Несколько служб в одной команде:

```bash
sudo systemctl stop application1.service application2.service
```

### Перезапуск и перезагрузка конфигурации

```bash
sudo systemctl restart application.service              # полный перезапуск
sudo systemctl reload application.service               # перезагрузить конфигурацию без остановки
sudo systemctl reload-or-restart application.service    # перезагрузить если возможно, иначе перезапустить
```

### Включение и отключение автозапуска

```bash
sudo systemctl enable application.service   # включить автозапуск при загрузке
sudo systemctl disable application.service  # отключить автозапуск
```

`enable` создаёт симлинк от файла службы (в `/lib/systemd/system/` или `/etc/systemd/system/`) в каталог автозапуска (`/etc/systemd/system/some_target.target.wants/`). `disable` удаляет симлинк.

> **Примечание:** `systemctl enable` только настраивает автозапуск при следующей загрузке. Чтобы запустить службу немедленно И включить автозапуск, выполните оба: `systemctl start` и `systemctl enable`.

### Проверка статуса службы

```bash
systemctl status sshd.service                 # полный статус с журналами
systemctl is-active application.service       # код выхода 0 = активна
systemctl is-enabled application.service      # код выхода 0 = автозапуск включён
systemctl is-failed application.service       # код выхода 0 = завершилась с ошибкой
```

Пример вывода `systemctl status sshd.service`:

```
sshd.service - OpenSSH server daemon
   Loaded: loaded (/usr/lib/systemd/system/sshd.service; enabled)
   Active: active (running) since Fri 2016-12-16 08:18:17 CET; 4h 58min ago
  Process: 1033 ExecStart=/usr/sbin/sshd $OPTIONS (code=exited, status=0/SUCCESS)
 Main PID: 1067 (sshd)
```

`is-active`, `is-enabled` и `is-failed` особенно полезны в shell-скриптах, где важнее код выхода, а не текстовый вывод.

### Обзор состояния системы

```bash
systemctl list-units                          # все активные юниты
systemctl                                     # то же (поведение по умолчанию)
systemctl list-units --all                    # все юниты включая неактивные
systemctl list-units --all --state=inactive   # фильтр по состоянию
systemctl list-units --type=service           # фильтр по типу
```

Столбцы вывода `list-units`:

| Столбец | Содержимое |
|---|---|
| `UNIT` | Имя юнита в systemd |
| `LOAD` | Загружена ли конфигурация в память |
| `ACTIVE` | Общее состояние (active / inactive) |
| `SUB` | Подробное состояние (running, exited, failed и т.д.) |
| `DESCRIPTION` | Краткое описание юнита |

### Просмотр файлов юнитов

```bash
systemctl list-unit-files    # все файлы юнитов в системе
```

Значения столбца `STATE`:

| Значение | Смысл |
|---|---|
| `enabled` | Автозапуск включён |
| `disabled` | Автозапуск отключён |
| `static` | Нет секции `[Install]`; не может быть включён через `enable`. Используется как зависимость или одноразовое действие |
| `masked` | Полностью отключён; не может быть запущен ни вручную, ни автоматически |

### Управление файлами юнитов

**Просмотр файла юнита:**

```bash
systemctl cat atd.service
```

Пример вывода:

```ini
[Unit]
Description=ATD daemon

[Service]
Type=forking
ExecStart=/usr/bin/atd

[Install]
WantedBy=multi-user.target
```

**Просмотр зависимостей:**

```bash
systemctl list-dependencies sshd.service            # дерево зависимостей
systemctl list-dependencies sshd.service --all      # все зависимости рекурсивно
systemctl list-dependencies sshd.service --reverse  # обратные (кто зависит от этого)
systemctl list-dependencies sshd.service --before   # что запускается до этого юнита
systemctl list-dependencies sshd.service --after    # что запускается после этого юнита
```

**Просмотр свойств юнита:**

```bash
systemctl show sshd.service               # все свойства в формате ключ=значение
systemctl show sshd.service -p Conflicts  # одно конкретное свойство
```

**Маскировка и демаскировка:**

Маскировка, в отличие от `disable`, делает запуск юнита полностью невозможным. systemd создаёт симлинк на `/dev/null`. Служба не может быть запущена ни вручную, ни автоматически.

```bash
sudo systemctl mask nginx.service     # полностью заблокировать службу
sudo systemctl unmask nginx.service   # снять блокировку
```

Попытка запустить замаскированную службу даст:

```
Failed to start nginx.service: Unit nginx.service is masked.
```

> **Примечание:** `disable` убирает автозапуск, но ручной `systemctl start` всё ещё работает. `mask` блокирует любой запуск — как ручной, так и автоматический — до выполнения `unmask`.

**Редактирование файлов юнитов:**

```bash
sudo systemctl edit nginx.service          # открыть фрагмент для частичного переопределения
sudo systemctl edit --full nginx.service   # открыть полный файл юнита для редактирования
```

`systemctl edit` без флага создаёт `/etc/systemd/system/nginx.service.d/` и открывает пустой файл фрагмента. Настройки фрагмента переопределяют значения из системного файла юнита.

`--full` загружает весь файл юнита в редактор и сохраняет изменённую копию в `/etc/systemd/system/nginx.service`. Эта копия имеет приоритет над системным файлом в `/lib/systemd/system/`.

Удаление изменений:

```bash
sudo rm -r /etc/systemd/system/nginx.service.d   # удалить фрагмент
sudo rm /etc/systemd/system/nginx.service         # удалить переопределённый файл юнита
sudo systemctl daemon-reload                      # перезагрузить конфигурацию
```

> **Внимание:** После любого ручного редактирования или удаления файлов юнитов запустите `systemctl daemon-reload`. Без этого systemd продолжит использовать старую конфигурацию из памяти.

### Управление целями и уровнями выполнения

Цели — это специальные файлы юнитов с суффиксом `.target`, описывающие состояния системы. Их назначение — группировать другие юниты. Это аналог уровней выполнения SysV, но более гибкий — одновременно могут быть активны несколько целей.

**Получение и установка цели по умолчанию:**

```bash
systemctl get-default                          # получить текущую цель по умолчанию
sudo systemctl set-default graphical.target    # установить графический режим по умолчанию
sudo systemctl set-default multi-user.target   # вернуться к многопользовательскому без GUI
```

**Просмотр целей:**

```bash
systemctl list-unit-files --type=target   # все файлы целей
systemctl list-units --type=target        # только текущие активные цели
```

**Isolate: переключение между целями:**

`isolate` запускает все юниты, принадлежащие цели, и останавливает все юниты, не входящие в неё. Аналогично `init N` или `telinit N` в SysV.

```bash
systemctl list-dependencies multi-user.target   # сначала проверить зависимости
sudo systemctl isolate multi-user.target        # переключиться в многопользовательский без GUI
sudo systemctl isolate graphical.target         # переключиться в графический режим
```

> **Внимание:** `isolate` останавливает службы. Перед изоляцией проверьте зависимости с помощью `list-dependencies`.

**Быстрые команды для распространённых событий:**

```bash
sudo systemctl rescue    # режим восстановления (аналог однопользовательского)
sudo systemctl halt      # остановить систему без отключения питания
sudo systemctl poweroff  # полное отключение питания
sudo systemctl reboot    # перезагрузка
```

**Соответствие уровней выполнения SysV целям systemd:**

| Уровень выполнения SysV | Цель systemd | Описание |
|---|---|---|
| 0 | `poweroff.target` | Выключение |
| 1 | `rescue.target` | Однопользовательский / режим восстановления |
| 2, 3, 4 | `multi-user.target` | Многопользовательский, без GUI |
| 5 | `graphical.target` | Многопользовательский с GUI |
| 6 | `reboot.target` | Перезагрузка |

## Управляющие последовательности в /etc/issue

`/etc/issue` поддерживает управляющие последовательности, обрабатываемые `getty`:

| Последовательность | Значение |
|---|---|
| `\r` | Версия ядра (uname -r) |
| `\m` | Архитектура процессора |
| `\n` | Имя хоста системы |
| `\o` | Доменное имя |
| `\d` | Текущая дата |
| `\t` | Текущее время |
| `\s` | Имя ОС |
| `\S` | Название дистрибутива (из /etc/os-release) |
| `\l` | Имя текущей линии tty |

> **Примечание:** Эти последовательности работают в `/etc/issue` через `getty`. В `/etc/issue.net` при SSH-соединении они, как правило, не интерпретируются.

## Шпаргалка для экзамена

**Файлы и их назначение:**

| Файл | Когда отображается | Для кого |
|---|---|---|
| `/etc/issue` | До входа | Локальный tty |
| `/etc/issue.net` | До входа | Telnet / SSH (требует настройки) |
| `/etc/motd` | После входа | Локальный tty |

**Команды:**

```bash
wall "сообщение"                         # широковещательная рассылка всем активным терминалам
shutdown -h +10 "текст"                  # остановить через 10 минут с сообщением
shutdown -r 02:00 "текст"                # перезагрузить в 02:00
shutdown -k +20 "текст"                  # репетиция (без реального выключения)
shutdown --no-wall +5                    # выключить без широковещательных сообщений
shutdown -c "текст"                      # отменить запланированное выключение
mesg n                                   # запретить запись в терминал
mesg y                                   # разрешить запись в терминал
systemctl restart sshd                   # перезапустить SSH после изменения sshd_config
```

**Включение /etc/issue.net для SSH:**

```
# /etc/ssh/sshd_config
Banner /etc/issue.net
```

### Типичные ошибки на экзамене

- `/etc/motd` отображается ПОСЛЕ входа, а не до.
- `/etc/issue.net` по умолчанию работает только с Telnet; для SSH требуется директива `Banner` в `sshd_config`.
- `wall -n` работает только для root.
- `shutdown -k` создаёт `/etc/nologin` и блокирует новые входы, даже несмотря на то, что система не выключается.
- Управляющие последовательности (`\r`, `\m` и т.д.) не гарантированно работают в `/etc/issue.net` через SSH.

## Практические вопросы

**В1.** Какой файл отображает сообщение пользователю ДО аутентификации на локальном tty-терминале?

A. `/etc/motd`  B. `/etc/issue.net`  C. `/etc/issue`  D. `/etc/login.msg`

**Ответ:** C. `/etc/issue` отображается до ввода имени пользователя и пароля на tty-терминале.

---

**В2.** Какое изменение нужно внести в `/etc/ssh/sshd_config`, чтобы SSH отображал содержимое `/etc/issue.net` при входе?

A. `Issue /etc/issue.net`  B. `Banner /etc/issue.net`  C. `Motd /etc/issue.net`  D. `LoginMessage /etc/issue.net`

**Ответ:** B. Директива `Banner` в `sshd_config` указывает путь к файлу сообщения перед входом.

**В3.** Администратор хочет немедленно разослать срочное сообщение всем вошедшим пользователям, не выключая систему. Какую команду использовать?

A. `broadcast "сообщение"`  B. `notify "сообщение"`  C. `wall "сообщение"`  D. `write all "сообщение"`

**Ответ:** C. `wall` рассылает сообщение на все активные терминалы.

**В4.** Администратор запустил `shutdown -k +30 "Обслуживание через 30 минут"`. Что произойдёт?

A. Система выключится через 30 минут, пользователи получат предупреждение  
B. Пользователи получат предупреждение, система не выключится, но новые входы будут заблокированы на 30 минут  
C. Только root получит предупреждение; система не выключится  
D. Никакого эффекта — флаг `-k` устарел

**Ответ:** B. Флаг `-k` — это репетиция: предупреждения отправляются, но система не выключается. Однако создаётся `/etc/nologin`, временно блокируя новые входы до окончания периода репетиции.

**В5.** Пользователь не хочет получать широковещательные сообщения от `wall`. Какую команду ему выполнить?

A. `mesg n`  B. `wall --ignore`  C. `block wall`  D. `chmod 000 /dev/tty`

**Ответ:** A. `mesg n` запрещает запись в терминал пользователя, включая сообщения от `wall`.

**В6.** Какой файл отображается пользователю tty ПОСЛЕ успешного входа, но ДО появления приглашения командной строки?

A. `/etc/issue`  B. `/etc/issue.net`  C. `/etc/motd`  D. `/etc/profile`

**Ответ:** C. `/etc/motd` (Message of the Day — сообщение дня) отображается после аутентификации, перед приглашением shell.

**В7.** Администратор запланировал выключение с `shutdown --no-wall +5` и теперь хочет отменить его так, чтобы пользователи НЕ получили сообщение об отмене. Какая команда верна?

A. `shutdown -c`  B. `shutdown -c --no-wall`  C. `shutdown --cancel --silent`  D. `systemctl cancel shutdown`

**Ответ:** B. Использование `shutdown -c` без `--no-wall` всё равно отправит пользователям сообщение об отмене. Чтобы подавить его, используйте `shutdown -c --no-wall`.

**В8.** В `/etc/issue` используется управляющая последовательность `\r`. Что она отображает?

A. Имя хоста  B. Версию ядра  C. Текущую дату  D. Архитектуру процессора

**Ответ:** B. Последовательность `\r` в `/etc/issue` подставляет версию ядра (эквивалент `uname -r`).
