## Структура каталогов

| Путь | Назначение |
|---|---|
| `/usr/src/linux/` | Символическая ссылка на текущее дерево исходного кода ядра |
| `/usr/src/linux-<version>/` | Фактический каталог с исходниками конкретной версии |
| `/usr/src/linux/.config` | Файл конфигурации ядра |
| `/boot/vmlinuz-<version>` | Скомпилированный образ ядра |
| `/boot/System.map-<version>` | Таблица символов для отладки ядра |
| `/boot/initrd-<version>.img` | Образ начального RAM-диска |
| `/lib/modules/<version>/` | Модули ядра для конкретной версии |
| `/lib/modules/<version>/modules.dep` | Файл зависимостей модулей, генерируется командой `depmod` |

> **Примечание:** `/usr/src/linux` — символическая ссылка. Создайте её вручную командой `ln -s /usr/src/linux-4.x.x /usr/src/linux`. Никогда не размещайте исходники непосредственно в `/usr/src/linux`.

## Получение исходного кода ядра

Все версии ядра доступны на [kernel.org](https://www.kernel.org/). Распакуйте скачанный архив в `/usr/src/`:

```bash
# Скачать архив (пример для версии 4.3.3)
wget https://cdn.kernel.org/pub/linux/kernel/v4.x/linux-4.3.3.tar.xz

# Распаковать
tar xvf linux-4.3.3.tar.xz -C /usr/src/

# Создать символическую ссылку
ln -s /usr/src/linux-4.3.3 /usr/src/linux

# Перейти в каталог с исходниками
cd /usr/src/linux
```

> **Подсказка:** Для компиляции нужно не менее 1 ГБ свободного места. Перед сборкой убедитесь, что установлены `gcc`, `make` и заголовочные файлы разработки: `apt-get install build-essential` (Debian) или `yum groupinstall "Development Tools"` (RHEL).

## Очистка дерева исходников

Очистка происходит на трёх уровнях:

```bash
# Уровень 1: удаляет большинство объектных файлов, но сохраняет достаточно для сборки внешних модулей
make clean

# Уровень 2: удаляет текущую конфигурацию (.config) и все объектные файлы
make mrproper

# Уровень 3: то же, что mrproper, плюс резервные копии редактора, остатки патчей и прочий мусор
make distclean
```

Запуск `make mrproper` перед каждой новой конфигурацией и сборкой является хорошей практикой.

> **Внимание:** `make mrproper` полностью удаляет `.config`. Если на создание конфигурации ушло значительное время, скопируйте `.config` в надёжное место перед выполнением этой команды.

## Создание файла .config

Перед компиляцией ядро необходимо настроить: выбрать, какие функции включить непосредственно в ядро (`=y`), какие скомпилировать как модули (`=m`), а какие исключить (`=n`). Результат сохраняется в `/usr/src/linux/.config`.

`.config` — это обычный текстовый файл, где каждая строка описывает один параметр:

```
CONFIG_64BIT=y
CONFIG_SMP=y
CONFIG_MODULES=y
CONFIG_EXT4_FS=m
# CONFIG_BTRFS_FS is not set
```

> **Внимание:** Редактировать `.config` вручную крайне не рекомендуется. Вместо этого используйте одну из целей конфигурации `make`.

---

## Цели конфигурации make

| Цель | Интерфейс | Зависимости | Примечания |
|---|---|---|---|
| `make config` | Последовательные текстовые подсказки | Нет | Нельзя вернуться назад. Сотни вопросов подряд. Работает на любом терминале |
| `make menuconfig` | Псевдографика ncurses | libncurses | Меню с навигацией. Можно сохранить в любой момент |
| `make nconfig` | ncurses (тёмная тема) | libncurses | Альтернатива `menuconfig` с другой цветовой схемой |
| `make xconfig` | Графический интерфейс Qt | Qt dev libs, X11 | Навигация мышью. Требует работающей X Window System |
| `make gconfig` | Графический интерфейс GTK | GTK+ 2.x dev libs, X11 | То же, что `xconfig`, но на GTK |
| `make oldconfig` | Текст, только новые параметры | Существующий `.config` | Сохраняет старые настройки, спрашивает только о новых параметрах |
| `make defconfig` | Без вопросов | Нет | Создаёт конфигурацию по умолчанию для определённого типа системы |
| `make mrproper` | Нет | Нет | Удаляет `.config` и все объектные файлы |

```bash
# Перейти в корень исходников ядра
cd /usr/src/linux

# Запустить текстовое меню (наиболее распространённый вариант)
make menuconfig

# Обновить конфигурацию при обновлении с 4.3 до 4.4
cp /path/to/old/.config /usr/src/linux/.config
make oldconfig
```

### Как make oldconfig работает с .config

При запуске `make oldconfig` старый `.config` копируется в `.config.old`, а на его основе создаётся новый `.config`. Для каждого параметра, отсутствующего в старом файле, задаётся вопрос. Все ранее выбранные значения сохраняются без запроса.

> **Примечание:** `make xconfig`, `make gconfig` и `make menuconfig` также автоматически подхватывают существующий `.config` из `/usr/src/linux/`. Новые параметры получают значения по умолчанию.

> **Подсказка к экзамену:** Часто задаваемый вопрос — разница между `make config`, `make menuconfig` и `make oldconfig`. Запомните: `menuconfig` — интерфейс ncurses с навигацией; `oldconfig` обновляет существующую конфигурацию, спрашивая только о новых параметрах; `make config` — самый примитивный вариант, без возможности вернуться назад.

## Компиляция ядра

После создания `.config` запустите сборку:

```bash
# Скомпилировать сжатый образ ядра (рекомендуется)
make bzImage

# Скомпилировать несжатый образ ядра
make zImage

# Скомпилировать только модули
make modules

# Скомпилировать всё: ядро + модули
make all

# Создать RPM-пакет с исходниками
make rpm-pkg

# Создать бинарный RPM-пакет
make binrpm-pkg

# Создать .deb-пакет
make deb-pkg
```

Итоговый `bzImage` находится по пути:

```
/usr/src/linux/arch/x86/boot/bzImage
```

> **Примечание:** `bzImage` — сжатый образ ядра (b = «big», z = zipped). Он отличается от `zImage` тем, что поддерживает размеры более 512 КБ. Всегда используйте `bzImage` на современных системах.

## Цели сборки make

| Цель | Описание |
|---|---|
| `all` | Собрать ядро и все модули |
| `bzImage` | Сжатый образ ядра (Big zImage) |
| `zImage` | Несжатый образ ядра (ограничен 512 КБ) |
| `modules` | Скомпилировать только модули |
| `modules_install` | Установить модули в `/lib/modules/<version>/` |
| `clean` | Удалить объектные файлы (`.config` сохраняется) |
| `mrproper` | Полная очистка, включая `.config` |
| `rpm-pkg` | Создать RPM-пакет с исходниками и ядром |
| `binrpm-pkg` | Создать бинарный RPM-пакет |
| `deb-pkg` | Создать .deb-пакет для систем Debian |

## Установка нового ядра

Полная последовательность сборки и установки:

```bash
# 1. Очистить артефакты предыдущей сборки
make clean

# 2. Собрать образ ядра
make bzImage       # или make zImage для очень старых систем

# 3. Собрать модули
make modules

# 4. Установить модули в /lib/modules/<version>/
#    Каталог создаётся автоматически, если не существует
make modules_install
```

После сборки скопируйте ядро и связанные файлы в `/boot/`. Путь к `bzImage` зависит от архитектуры:

```bash
# x86_64
cp /usr/src/linux/arch/x86_64/boot/bzImage /boot/vmlinuz-2.6.31

# i386
cp /usr/src/linux/arch/i386/boot/bzImage /boot/vmlinuz-2.6.31

# Скопировать таблицу символов (необязательно, нужна для отладки)
cp /usr/src/linux/System.map /boot/System.map-2.6.31

# Пересобрать файл зависимостей модулей
depmod -a
```

Включение версии в имя файла позволяет нескольким ядрам сосуществовать в `/boot/`, чтобы при необходимости можно было откатиться.

После копирования файлов обновите загрузчик (GRUB):

```bash
# GRUB 2
update-grub                               # Debian/Ubuntu
grub2-mkconfig -o /boot/grub2/grub.cfg   # RHEL/CentOS
```

> **Важно:** Если загрузчик не видит новое ядро, проверьте два момента: файл ядра находится в `/boot/` и конфигурация GRUB обновлена. Это проверяется на экзамене.

## Модули ядра

Модули ядра — это объектные файлы `.ko`, загружаемые в ядро по требованию. Все модули хранятся в иерархии каталогов под `/lib/modules/<kernel-version>/`.

```
/lib/modules/4.3.3/
├── kernel/
│   ├── drivers/
│   │   ├── net/        # Сетевые драйверы
│   │   ├── scsi/       # SCSI-драйверы
│   │   └── block/      # Блочные устройства
│   └── fs/             # Файловые системы
└── modules.dep         # Файл зависимостей
```

`modules.dep` содержит список зависимостей каждого модуля. Он генерируется командой `depmod`:

```bash
# Пересгенерировать modules.dep для работающего ядра
depmod -a

# Пересгенерировать для конкретной версии ядра
depmod -a 4.3.3
```

Пример строки из `modules.dep`:

```
/lib/modules/2.6.31/kernel/drivers/acpi/thermal.ko: \
  /lib/modules/2.6.31/kernel/drivers/thermal/thermal_sys.ko
```

> **Примечание:** `modprobe` читает `modules.dep` и автоматически загружает все зависимости модуля. `insmod` зависимостями не управляет.

## Начальный RAM-диск — initrd и initramfs

При загрузке ядро не может получить доступ к диску, если драйвер диска скомпилирован как модуль. Возникает замкнутый круг: для загрузки модуля нужен диск, но для доступа к диску нужен модуль. Решением является начальный RAM-диск (initrd).

initrd — это образ минимальной корневой файловой системы в оперативной памяти. Загрузчик (GRUB или LILO) передаёт его ядру вместе с образом ядра. Ядро монтирует initrd как временную корневую файловую систему, загружает необходимые модули, затем монтирует реальную корневую файловую систему с диска. После переключения корня старый initrd либо перемещается в `/initrd`, либо отмонтируется.

### Проверка формата существующего initrd

Старые ядра используют внутри tar-архив; более новые — cpio. Для определения формата:

```bash
zcat /boot/initrd-2.6.18-348.6.1.el5.img | file -
# /dev/stdin: ASCII cpio archive (SVR4 with no CRC)
```

### Сборка initrd вручную

Если `mkinitrd` недоступен, можно собрать образ вручную с помощью блочного устройства — RAM-диска (`/dev/ram0`) или loopback-устройства:

```bash
# 1. Создать файловую систему на RAM-диске (300 блоков)
mke2fs -m0 /dev/ram0 300

# 2. Смонтировать его
mount -t ext2 /dev/ram0 /mnt

# 3. Создать необходимые узлы устройств и каталоги
mkdir /mnt/dev
mknod /mnt/dev/tty1 c 4 1

# 4. Создать /linuxrc — ядро запускает его первым
#    Для теста достаточно символической ссылки на /bin/sh
ln -s /bin/sh /mnt/linuxrc

# 5. Скопировать необходимые модули, скрипты, бинарные файлы

# 6. Отмонтировать RAM-диск
umount /dev/ram0

# 7. Скопировать его содержимое в файл образа
dd if=/dev/ram0 bs=1k count=300 of=/boot/initrd

# 8. Освободить RAM-диск
freeramdisk /dev/ram0
```

Подключите образ к загрузчику. Пример записи в конфигурации GRUB Legacy:

```
title=initrd test entry
root (hd0,0)
kernel /vmlinuz-2.6.28
initrd /initrd
```

> **Примечание:** `/linuxrc` является точкой входа для initrd. Ядро находит и запускает его сразу после монтирования образа. В реальных системах это скрипт, загружающий модули и монтирующий реальный корень.

### mkinitrd (RPM-based системы)

```bash
# Создать образ initrd
mkinitrd /boot/initrd-4.3.3.img 4.3.3

# Полезные опции
mkinitrd --version                    # Показать версию
mkinitrd -f /boot/initrd.img 4.3.3   # Перезаписать существующий файл
mkinitrd --omit-scsi-modules ...      # Исключить модули SCSI
```

### mkinitramfs (Debian-based системы)

```bash
# Создать образ initramfs
mkinitramfs -o /boot/initrd.img-4.3.3 4.3.3

# Указать альтернативный каталог конфигурации
mkinitramfs -d /etc/initramfs-tools -o /boot/initrd.img 4.3.3
```

Файл конфигурации `mkinitramfs` находится в `/etc/initramfs-tools/initramfs.conf`.

> **Важно:** На экзамене проверяются знания обоих инструментов: `mkinitrd` для RPM-based систем (Red Hat, SUSE), `mkinitramfs` для Debian-based систем (Debian, Ubuntu). Оба создают сжатый gzip архив cpio.

## Патчинг ядра

> **Примечание:** Патчинг ядра больше не входит в программу экзамена LPIC-2, но является полезной базой для работы с исходниками ядра.

Файл патча — это вывод команды `diff -u old_file new_file`. Он состоит из блоков изменений, называемых hunks (кусками). Каждый hunk содержит контекстные строки — неизменённые строки вокруг правки. Инструмент `patch` использует этот контекст для нахождения нужного места в файле. Строки для удаления отмечены `-`, строки для добавления — `+`. Если контекст в файле не совпадает с контекстом в патче, `patch` не может применить этот hunk и сохраняет его в файл `.rej`.

```diff
--- a/drivers/net/foo.c   2024-01-01
+++ b/drivers/net/foo.c   2024-01-02
@@ -42,7 +42,7 @@
     int ret;
     /* init */
-    ret = old_init();
+    ret = new_init();
     if (ret < 0)
```

```bash
# 1. Поместить файл патча в /usr/src/
# 2. Перейти в /usr/src/
cd /usr/src

# 3. Распаковать патч (если в формате xz)
unxz patch-4.3.4.xz

# 4. Применить патч
patch -p1 < patch-4.3.4

# 5. Проверить наличие .rej-файлов (отклонённые hunks)
find /usr/src/linux -name "*.rej"
```

Опция `-p<n>` управляет количеством удаляемых ведущих компонентов пути. `git diff` записывает пути в виде `a/drivers/net/foo.c` — без удаления `patch` ищет этот буквальный путь и завершается с ошибкой. `-p1` удаляет первый компонент (`a/` или `b/`), оставляя реальный относительный путь:

| Опция | Путь из патча | Результат |
|---|---|---|
| `-p0` | `a/drivers/net/foo.c` | `a/drivers/net/foo.c` — без изменений |
| `-p1` | `a/drivers/net/foo.c` | `drivers/net/foo.c` — `a/` удалено |
| `-p2` | `a/drivers/net/foo.c` | `net/foo.c` — `a/drivers/` удалено |

Для патчей ядра стандартом является `-p1`.

Полезные опции `patch`:

| Опция | Описание |
|---|---|
| `-p<n>` | Удалить `n` ведущих компонентов пути |
| `-s` | Тихий режим, показывать только ошибки |
| `-R` | Откатить патч (отменить изменения) |
| `-E` | Удалять пустые файлы после патчинга |

### Откат патча

```bash
patch -R -p1 < patch-2.6.28
# Если patch обнаружит, что патч уже был применён:
# Reversed (or previously applied) patch detected! Assume -R? [n] y
```

## DKMS

DKMS (Dynamic Kernel Module Support) был разработан компанией Dell в 2003 году. Проблема была конкретной: Dell поддерживала клиентов, работающих на Linux, и каждое обновление ядра ломало модули для проприетарного оборудования. DKMS решает это, отделяя исходники модулей от дерева ядра и автоматически пересобирая их при обновлениях ядра.

Когда менеджер пакетов обновляет ядро в системе с DKMS, хук проверяет, какие модули нужно пересобрать для нового ядра. Это работает и в обратную сторону: новый модуль устанавливается без привязки к конкретной версии ядра.

```bash
# 1. Установить пакет DKMS (исходники попадают в /usr/src/<module-version>/)
apt-get install flashcache-dkms

# 2. Добавить модуль в дерево DKMS
dkms add -m flashcache -v 3.1.1

# 3. Собрать модуль для текущего ядра
dkms build -m flashcache -v 3.1.1

# 4. Установить модуль
dkms install -m flashcache -v 3.1.1

# 5. Проверить статус
dkms status

# 6. Удалить установленный модуль
dkms uninstall -m flashcache -v 3.1.1
```

Каждый модуль DKMS описывается файлом `dkms.conf` в `/usr/src/<module-version>/`:

```bash
BUILT_MODULE_NAME="flashcache"
DEST_MODULE_LOCATION="/kernel/drivers/block"
PACKAGE_NAME="flashcache"
PACKAGE_VERSION="3.1.1+git20140801"
AUTOINSTALL="yes"
REMAKE_INITRD="yes"
MAKE="KERNEL_TREE=$kernel_source_dir make modules"
```

Основные директивы `dkms.conf`:

| Директива | Обязательна | Назначение |
|---|:-:|---|
| `BUILT_MODULE_NAME` | При 2+ модулях | Имя скомпилированного модуля |
| `DEST_MODULE_LOCATION` | Да* | Путь установки, всегда начинается с `/kernel` |
| `PACKAGE_NAME` | Да | Имя пакета |
| `PACKAGE_VERSION` | Да | Версия пакета |
| `AUTOINSTALL` | Нет | `yes` — устанавливать модуль при каждой загрузке нового ядра |
| `REMAKE_INITRD` | Нет | `yes` — пересобирать initrd после установки модуля (по умолчанию `no`) |
| `MAKE` | Нет | Команда сборки; если не задана, DKMS использует стандартный `make` |

\* `DEST_MODULE_LOCATION` не обязательна для Fedora Core 6+, RHEL 5+, SuSE Linux ES 10+ и Ubuntu — они используют каталоги дистрибутива.

> **Внимание:** Директива `REMAKE_INITRD` читает только первый символ значения — остальные игнорируются. Первый символ интерпретируется как «yes» только если это `y` или `Y`. `yes` и `YES` ведут себя одинаково; так же как `no` и `nope`.

### Работа с архивами модулей

DKMS умеет работать с tar-архивами, что удобно для переноса модулей между системами:

```bash
# Создать tarball из текущих модулей на этой системе
dkms mktarball -m flashcache -v 3.1.1

# Импортировать tarball в дерево DKMS на другой системе
dkms ldtarball /path/to/flashcache-3.1.1.tar.gz
```

Архив должен содержать корректный `dkms.conf`.

### Расположение примеров конфигурации

```
Debian:    /usr/share/doc/dkms/examples/
Red Hat:   /usr/share/doc/dkms/
```

> **Важно:** DKMS требует установленных заголовков ядра: `apt-get install linux-headers-$(uname -r)` или `yum install kernel-devel`.

> **Примечание:** Глобальная конфигурация DKMS находится в `/etc/dkms/framework.conf`. Переменная `source_tree` там указывает DKMS, где искать каталоги с `dkms.conf`.

## Dracut

Dracut — утилита для создания образов initramfs, поддерживающая несколько дистрибутивов и способная работать в событийно-управляемом режиме. Подобно тому, как DKMS реагирует на обновления ядра и пересобирает модули, dracut реагирует на изменения и автоматически пересобирает образ initramfs.

```bash
# Создать initramfs для работающего ядра
dracut /boot/initramfs-$(uname -r).img $(uname -r)

# Принудительное создание (перезапись существующего образа)
dracut --force /boot/initramfs-$(uname -r).img $(uname -r)
```

> **Примечание:** На экзамене LPIC-2 требуется только знать, что dracut существует и что он делает — автоматически создаёт initramfs при необходимости. Детальная настройка не проверяется.

## Шпаргалка к экзамену

### Ключевые пути

```
/usr/src/linux/           — символическая ссылка на исходники ядра
/usr/src/linux/.config    — файл конфигурации ядра
/boot/vmlinuz-<ver>       — образ ядра
/boot/System.map-<ver>    — таблица символов
/boot/initrd-<ver>.img    — начальный RAM-диск
/lib/modules/<ver>/       — модули ядра
/lib/modules/<ver>/modules.dep  — файл зависимостей
/etc/dkms/framework.conf  — конфигурация DKMS
/etc/initramfs-tools/initramfs.conf — конфигурация mkinitramfs
```

### Ключевые команды

```bash
make mrproper              # Полная очистка + удаление .config
make clean                 # Удалить только объектные файлы
make menuconfig            # Текстовый конфигуратор (ncurses)
make oldconfig             # Обновить .config для новых параметров ядра
make bzImage               # Собрать сжатый образ ядра
make modules               # Собрать только модули
make modules_install       # Установить модули в /lib/modules/
depmod -a                  # Пересгенерировать modules.dep
mkinitrd /boot/img.img <ver>          # Создать initrd (RPM)
mkinitramfs -o /boot/img.img <ver>    # Создать initramfs (Debian)
dkms status                # Статус модулей DKMS
dkms add/build/install     # Жизненный цикл модуля DKMS
update-grub / grub2-mkconfig          # Обновить конфигурацию GRUB
```

### Типичные ошибки на экзамене

- `make mrproper` удаляет `.config`; `make clean` — нет.
- `mkinitrd` для RPM-based систем; `mkinitramfs` для Debian.
- После `make modules_install` запустите `depmod -a`.
- Скомпилированный `bzImage` находится в `arch/x86/boot/bzImage` относительно корня дерева исходников.
- `DEST_MODULE_LOCATION` в `dkms.conf` должен начинаться с `/kernel`.

## Практические вопросы

**Q1.** Администратор скачал исходники ядра 4.3.3 и распаковал их в `/usr/src/linux-4.3.3`. Какая команда заставит `/usr/src/linux` указывать на эти исходники?

A) `cp -r /usr/src/linux-4.3.3 /usr/src/linux`
B) `ln -s /usr/src/linux-4.3.3 /usr/src/linux`
C) `mount --bind /usr/src/linux-4.3.3 /usr/src/linux`
D) `mv /usr/src/linux-4.3.3 /usr/src/linux`

**Ответ:** B. Используйте символическую ссылку. Вариант A дублирует файлы; D переименовывает каталог и нарушает привязку к версии.

---

**Q2.** Администратор хочет обновить конфигурацию ядра с 4.3 до 4.4, сохранив все существующие настройки и отвечая только на вопросы о новых параметрах. Какую цель `make` следует использовать?

A) `make config` B) `make defconfig` C) `make oldconfig` D) `make menuconfig`

**Ответ:** C. `make oldconfig` читает существующий `.config` и спрашивает только о новых параметрах. `make config` задаёт все вопросы с нуля.

---

**Q3.** После компиляции ядра на системе x86_64, где находится итоговый `bzImage`?

A) `/usr/src/linux/bzImage`
B) `/usr/src/linux/arch/x86/boot/bzImage`
C) `/boot/bzImage`
D) `/usr/src/linux/kernel/bzImage`

**Ответ:** B. Файл создаётся по пути `arch/x86/boot/bzImage` относительно корня дерева исходников.

---

**Q4.** Какая команда создаёт `/lib/modules/<version>/modules.dep`?

A) `modprobe --update` B) `insmod --deps` C) `depmod -a` D) `ldconfig`

**Ответ:** C. `depmod -a` сканирует все модули в `/lib/modules/<version>/` и записывает файл зависимостей `modules.dep`.

---

**Q5.** Администратор на системе Debian хочет создать образ initramfs для ядра версии 4.3.3. Какую команду следует использовать?

A) `mkinitrd /boot/initrd-4.3.3.img 4.3.3`
B) `dracut /boot/initrd.img 4.3.3`
C) `mkinitramfs -o /boot/initrd.img-4.3.3 4.3.3`
D) `mkinitrfs 4.3.3 /boot/initrd.img`

**Ответ:** C. `mkinitramfs` — инструмент для Debian. `mkinitrd` предназначен для RPM-based систем (Red Hat, SUSE).

---

**Q6.** Какой файл обязательно должен присутствовать в каталоге модуля DKMS под `/usr/src/`?

A) `Makefile` B) `dkms.conf` C) `module.spec` D) `.config`

**Ответ:** B. `dkms.conf` содержит директивы для DKMS и является обязательным для работы фреймворка.

---

**Q7.** Администратор хочет полностью сбросить конфигурацию ядра и удалить все ранее скомпилированные объектные файлы. Какую команду следует использовать?

A) `make clean` B) `make distclean` C) `make mrproper` D) `make reset`

**Ответ:** C. `make mrproper` удаляет и объектные файлы, и `.config`. `make clean` удаляет только объектные файлы, сохраняя `.config`.

---

**Q8.** После установки нового ядра и выполнения `make modules_install` администратор замечает, что `modprobe` не может загрузить новый модуль, жалуясь на отсутствующий файл зависимостей. Что исправит ситуацию?

A) Перезапустить `udevd`
B) Выполнить `depmod -a`
C) Выполнить `ldconfig`
D) Пересобрать ядро с `make all`

**Ответ:** B. `depmod -a` создаёт файл `modules.dep`, который `modprobe` читает для разрешения зависимостей.
