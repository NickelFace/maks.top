## LILO

LILO (Linux Loader) загружается в два этапа. Во время загрузки он выводит буквы "L", "I", "L", "O" по одной — каждая буква обозначает завершённый этап. Неполная последовательность вроде `LIL-` или `LIL?` точно указывает, на каком этапе произошёл сбой.

В интерактивном режиме (требует директивы `prompt` в конфигурации) LILO отображает меню из не более чем 16 записей. Без `timeout=nnn` (единица — 0,1 с) загрузчик ждёт бесконечно. По истечении таймаута LILO загружает первый образ из списка, если не задан `default=name`.

Файл конфигурации: `/etc/lilo.conf`. После любых изменений **необходимо** запустить `/sbin/lilo`, чтобы перезаписать MBR. Установщик читает конфигурацию и записывает карту файлов в `/boot/map`. LILO не поддерживает UEFI и в значительной мере устарел.

```bash
# /etc/lilo.conf — пример конфигурации
boot=/dev/hda        # устройство для записи загрузчика
prompt               # показывать меню выбора
timeout=150          # ждать 15 секунд (150 × 0,1 с)
lba32                # использовать адресацию LBA
compact
vga=normal
root=/dev/hda1
read-only
menu-title=" My Computer "

# Ядра Linux
image=/boot/zImage-1.5.99
    label=try
image=/boot/zImage-1.0.9
    label=1.0.9
image=/tamu/vmlinuz
    label=tamu
    initrd=initramdisk.img
    root=/dev/hdb2
    vga=ask

# Другие ОС
other=/dev/hda3
    label=dos
    boot-as=0x80
other=/dev/hdb1
    label=Win98
    boot-as=0x80
```

> **Ловушка на экзамене:** В отличие от GRUB, LILO **не читает** `lilo.conf` при загрузке — файл читает только установщик `/sbin/lilo`. Если отредактировать файл без запуска `/sbin/lilo`, изменения **не вступят в силу**.

## Семейство Syslinux

Проект Syslinux (syslinux.org) объединяет несколько загрузчиков под одним именем. Каждый решает свою задачу, но все используют одинаковый формат файла конфигурации.

| Загрузчик | Файловая система / среда | Типичный сценарий |
|---|---|---|
| **SYSLINUX** | FAT (MS-DOS/Windows) | Загрузочная USB-флешка |
| **ISOLINUX** | ISO 9660 (CD/DVD) | Дистрибутив LiveCD/LiveDVD |
| **EXTLINUX** | ext2/3/4, btrfs | Встраиваемые системы |
| **PXELINUX** | Сеть через PXE/TFTP | Сетевая загрузка |
| **MEMDISK** | Любая (через другой загрузчик) | Запуск DOS-образов |

### SYSLINUX

SYSLINUX загружает Linux с FAT-разделов. Работает только на Intel/AMD. Установщик поставляется в трёх вариантах: `syslinux` для Linux, `syslinux.exe` для Windows, `syslinux.com` для DOS.

Основные опции установщика:

| Опция | Краткая | Описание |
|---|---|---|
| `--install` | `-i` | Установить, заменив текущий загрузочный сектор |
| `--update` | `-U` | Обновить предыдущую установку |
| `--directory` | `-d` | Каталог для установки |
| `--offset` | `-t` | Смещение файловой системы на устройстве |
| `--raid` | `-r` | При сбое загрузки попробовать следующее устройство |
| `--once=...` | | Выполнить команду один раз при следующей загрузке |
| `--clear-once` | `-O` | Очистить команду однократной загрузки |
| `--stupid` | `-s` | Медленный безопасный режим |
| `--force` | `-f` | Игнорировать предупреждения |
| `--menu-save=` | `-M` | Установить метку по умолчанию для следующей загрузки |

Только для MS-DOS:
```
-m    MBR: записать загрузочный MBR-сектор в начало диска
-a    Active: пометить раздел как активный (загрузочный)
```

```bash
# Установить на USB-диск: загрузчик + MBR + активный раздел
syslinux --install -m -a /dev/sdb1
```

### ISOLINUX

ISOLINUX загружается с CD/DVD. В отличие от `syslinux`, установщик ISOLINUX не пишет напрямую на устройство — он генерирует ISO-образ из каталога с помощью `xorriso`. Каталог должен содержать папку `isolinux/` с файлом `isolinux.bin`.

```
# Минимальная структура каталогов для создания ISO
my-iso/
├── isolinux/
│   ├── isolinux.bin    # сам загрузчик (обязателен)
│   ├── isolinux.cfg    # конфигурация
│   └── ldlinux.c32     # библиотека модулей
└── boot/
    └── vmlinuz
```

Для загрузки с USB дополнительно требуется файл `isohdpfx.bin` — гибридный MBR-образ, генерируемый `xorriso`.

### EXTLINUX

EXTLINUX устанавливается непосредственно на смонтированную файловую систему ext2/3/4 или btrfs.

```bash
extlinux --install /boot
```

Ищет конфигурацию по путям `/boot/extlinux/extlinux.conf` или `/extlinux/extlinux.conf`.

### PXELINUX

PXELINUX выступает вторым этапом PXE-загрузки. TFTP-сервер предоставляет `pxelinux.0` вместе с необходимой библиотекой `ldlinux.c32`. Конфигурации клиентов хранятся в `/tftpboot/pxelinux.cfg/`.

PXELINUX определяет имя файла конфигурации тремя методами (в порядке приоритета):

**Метод 1: опции DHCP 209 и 210**

Опция 209 (`pxelinux.config-file`) задаёт имя файла конфигурации. Опция 210 (`pxelinux.pathprefix`) задаёт префикс пути на TFTP-сервере.

**Метод 2: жёстко прописано в образе pxelinux.0**

Утилита `pxelinux-options` встраивает опции непосредственно в образ `pxelinux.0`. Опции могут быть `before-options` (DHCP имеет приоритет) или `after-options` (они переопределяют DHCP).

Поддерживаемые встроенные опции:

| Номер | Опция |
|---|---|
| 6 | domain-name-servers |
| 15 | domain-name |
| 54 | next-server |
| 209 | config-file |
| 210 | path-prefix |
| 211 | reboottime |

**Метод 3: автоматически — по UUID, MAC или IP**

Если конфигурация не задана явно, PXELINUX перебирает имена файлов на TFTP-сервере в следующем порядке:

1. UUID клиента (в нижнем регистре)
2. MAC-адрес клиента (в нижнем регистре, байты разделены `-`, например `01-aa-bb-cc-dd-ee-ff`)
3. IP-адрес в виде шестнадцатеричного числа в верхнем регистре с уменьшающимся префиксом: `C0A80101`, `C0A8010`, `C0A801`, ..., `C`
4. Файл `default`

## Конфигурация Syslinux

Все члены семейства ищут файл конфигурации `syslinux.cfg` в трёх местах по порядку: `/boot/syslinux/`, `/syslinux/`, `/`. ISOLINUX сначала проверяет `/boot/isolinux/` и `/isolinux/`. EXTLINUX — `/boot/extlinux/` и `/extlinux/`.

```ini
# Пример syslinux.cfg / isolinux.cfg
DEFAULT linux
TIMEOUT 50         # 5 секунд (единица — 1/10 с)
PROMPT 1

LABEL linux
  KERNEL /boot/vmlinuz
  APPEND root=/dev/sda1 ro quiet
  INITRD /boot/initrd.img
```

Основные директивы конфигурации:

| Директива | Назначение |
|---|---|
| `DEFAULT` | Метка для загрузки по умолчанию (переопределяет `LABEL linux`) |
| `TIMEOUT` | Время ожидания ввода в единицах 1/10 секунды |
| `LABEL` | Имя пункта меню |
| `KERNEL` | Путь к образу ядра или загрузчику следующего этапа |
| `APPEND` | Параметры, добавляемые к cmdline ядра (применяется только последняя) |
| `INITRD` | Путь к начальному RAM-диску — эквивалент `APPEND initrd=filename` |
| `CONFIG` | Перезапустить загрузку с другим файлом конфигурации |
| `SYSAPPEND` | Битовая маска для автоматического добавления строк к cmdline ядра |

> **Примечание:** По умолчанию загрузчик ищет запись `LABEL linux` в конфигурации. Директива `DEFAULT` переопределяет эту метку. Директива `CONFIG` с двумя путями переопределяет и файл конфигурации, и каталог по умолчанию для последующих путей.

Расширения файлов для `KERNEL`:

| Расширение | Тип образа |
|---|---|
| `.0` | PXE NBP (Network Boot Program), по умолчанию `pxelinux.0` |
| `.bin` | Загрузочный сектор CD (ISOLINUX) |
| `.bs` / `.bss` | Загрузочные секторы DOS (SYSLINUX) |
| `.com` / `.cbt` | Образы COMBOOT (DOS / не-DOS) |
| `.c32` | COM32: COMBOOT до v5.00, ELF-бинарник начиная с v5.00 |
| `.img` | Образ диска (ISOLINUX) |
| без расширения | Ядро Linux |

Тип файла можно задать явно с помощью псевдонимов `KERNEL`: `LINUX`, `BOOT`, `BSS`, `PXE`, `FDIMAGE`, `COMBOOT`, `COM32`.

`SYSAPPEND` принимает числовую битовую маску. Каждый бит добавляет автоматически сгенерированную строку к cmdline ядра:

| Бит | Добавляемая строка |
|---|---|
| 1 | `ip=client:bootserver:gw:netmask` — сетевые параметры клиента |
| 2 | `BOOTIF=...` — MAC-адрес активного сетевого интерфейса |
| 4 | `SYSUUID=...` — UUID системы |
| 8 | `CPU=...` — информация о процессоре |
| 0x10–0x10000 | Дополнительные строки из DMI/SMBIOS |

> **Совет:** `SYSAPPEND 3` (биты 1+2) в конфигурации PXELINUX одновременно добавляет к cmdline ядра как IP-параметры, так и MAC-адрес интерфейса — полезно при массовых сетевых установках, когда ядру нужно знать, откуда оно было загружено.

## PXE: сетевая загрузка

PXE (Pre-boot Execution Environment) — спецификация Intel, расширяющая BOOTP/DHCP и TFTP. Позволяет машине загружаться без локального диска, получая всё необходимое с сетевого сервера.

### PXE и BIOS

DHCP Discover для PXE содержит как минимум пять обязательных опций, три из которых специфичны для PXE:

| Опция | Назначение |
|---|---|
| (53) | Тип DHCP-сообщения: Discover |
| (55) | Список запрашиваемых параметров (включая опции 128–135) |
| (93) | Client System Architecture — тип архитектуры клиента |
| (94) | Client Network Device Interface — версия UNDI |
| (97) | UUID/GUID-based Client Identifier |

Опции 93, 94 и 97 идентифицируют клиента как PXE-совместимый. Опция 55 должна запрашивать диапазон 128–135, даже если сервер не обязан его предоставлять.

Последовательность загрузки для BIOS:

```
1. Клиент отправляет DHCP Discover с опциями PXE (53, 55, 93, 94, 97)
2. DHCP-сервер возвращает IP-адрес и имя файла загрузчика
3. Клиент загружает pxelinux.0 и ldlinux.c32 через TFTP
4. pxelinux.0 запрашивает файл конфигурации из pxelinux.cfg/
5. Ядро и initrd загружаются через TFTP и запускаются
```

Структура TFTP-сервера:

```
/tftpboot/
├── pxelinux.0                 # основной загрузчик PXELINUX
├── ldlinux.c32                # обязательная библиотека
└── pxelinux.cfg/
    ├── default                # резервная конфигурация
    ├── 01-aa-bb-cc-dd-ee-ff   # конфигурация по MAC-адресу клиента
    └── C0A80001               # конфигурация по IP-адресу (hex, уменьшающийся префикс)
```

> **Примечание — Proxy DHCP:** Если основной DHCP-сервер не поддерживает опции PXE, рядом с ним можно запустить Proxy DHCP-сервер. Он слушает на **UDP-порту 4011** и добавляет только PXE-параметры поверх основного DHCP-ответа, не конфликтуя с ним.

### PXE и UEFI

При использовании UEFI загрузка по PXE работает иначе: вместо `pxelinux.0` клиент запрашивает EFI-загрузчик. Поле `Client System Architecture` (опция 93) сообщает серверу архитектуру клиента.

```
# Значения опции 93 (Client System Architecture):
0  = x86 BIOS
7  = EFI BC (64-bit x86 UEFI)
9  = x86 UEFI (32-bit)
```

Для UEFI-клиентов TFTP-сервер предоставляет `uefi/shim.efi` или `uefi/grubx64.efi`. Пример конфигурации ISC DHCP с разграничением BIOS и UEFI:

```
# /etc/dhcp/dhcpd.conf: фрагмент PXE
if option arch = 00:07 {
    # UEFI 64-bit
    filename "uefi/shim.efi";
} else {
    # Legacy BIOS
    filename "pxelinux.0";
}
next-server 192.168.1.10;  # IP TFTP-сервера
```

## UEFI и Secure Boot

UEFI (Unified Extensible Firmware Interface) заменяет BIOS и хранит загрузчики на выделенном разделе ESP (EFI System Partition), отформатированном в FAT. В Linux ESP обычно монтируется в `/boot/efi`. Загрузчики используют расширение `.efi`.

Secure Boot требует цифровой ключ: прошивка UEFI проверяет подпись загрузчика перед его запуском. Крупные дистрибутивы — Red Hat, Ubuntu и SUSE — приобрели такие ключи и работают с Secure Boot из коробки. Есть три способа запустить Linux с включённым Secure Boot:

1. Отключить Secure Boot в настройках UEFI — просто, но снижает защиту.
2. Приобрести собственный ключ подписи и подписать каждый образ.
3. Использовать подписанный shim от Linux Foundation или Fedora.

Если Secure Boot мешает загрузке, сначала попробуйте отключить следующие функции прошивки:

```
- Quickboot / Fastboot
- Intel Smart Response Technology (ISRT)
- FastStartUp (если установлена Windows 8)
```

На системах с Windows 8 настройки UEFI можно открыть через «Дополнительные параметры запуска» прямо из Windows. Некоторые прошивки поддерживают оба режима — EFI и Legacy BIOS; переключение в Legacy решает проблему.

### shim и grubx64.efi

Shim — небольшой подписанный промежуточный загрузчик. UEFI запускает `shim.efi`, который затем запускает `grubx64.efi`. Это удобно: shim подписан третьей стороной, а GRUB можно обновлять без повторной подписи.

```
ESP (/boot/efi/)
└── EFI/
    └── uefi/
        ├── shim.efi      # подписанный посредник, запускается UEFI
        └── grubx64.efi   # GRUB2 для UEFI, запускается shim
```

> **Важно для экзамена:** UEFI запускает `shim.efi`, а `shim.efi` затем находит `grubx64.efi` в том же каталоге. Имена файлов фиксированы.

## systemd-boot

systemd-boot (ранее gummiboot) встроен в systemd и работает **только с EFI-системами**. Может запускать любой EFI-исполняемый файл: ядро Linux через EFISTUB, UEFI Shell, GRUB или Windows Boot Manager.

Управляется командой `bootctl`:

```bash
# Установить systemd-boot на ESP
bootctl install

# Проверить статус
bootctl status

# Обновить загрузчик
bootctl update
```

ESP должен быть смонтирован, желательно в `/boot`. Конфигурация хранится в `/boot/loader/`:

```
/boot/loader/
├── loader.conf           # основная конфигурация загрузчика
└── entries/
    ├── arch.conf         # запись для Arch Linux
    └── windows.conf      # запись для Windows
```

```ini
# /boot/loader/loader.conf
default  arch.conf
timeout  5
editor   no

# /boot/loader/entries/arch.conf
title   Arch Linux
linux   /vmlinuz-linux
initrd  /initramfs-linux.img
options root=/dev/sda2 rw quiet
```

> **Внимание:** systemd-boot не умеет читать конфигурации с ext4 и других не-FAT разделов. Ядро и initrd должны находиться непосредственно на ESP или на отдельном разделе `/boot` в формате FAT.

## U-Boot

Das U-Boot расшифровывается как Universal Boot Loader. Это загрузчик с открытым исходным кодом для встраиваемых устройств, поддерживающий широкий спектр архитектур: 68k, ARM, AVR32, Blackfin, MicroBlaze, MIPS, Nios, SuperH, PPC и x86.

Ключевые особенности U-Boot:

- Загружает ядра из флэш-памяти, по сети, с SD-карты или через последовательный порт.
- Требует явного указания адресов памяти для копирования ядра и RAM-диска — другие загрузчики выбирают адреса автоматически; U-Boot — нет.
- Может быть разделён на несколько этапов при ограниченном размере прошивки.
- Низкоуровневые команды дают гибкость, но загрузка ядра требует нескольких шагов вместо одного.

```bash
# Загрузить ядро через TFTP и запустить
setenv ipaddr 192.168.1.100
setenv serverip 192.168.1.10
tftp 0x80000000 uImage
bootm 0x80000000

# Загрузить с SD-карты
mmc dev 0
load mmc 0:1 0x80000000 /boot/uImage
bootm 0x80000000
```

> **Примечание для LPIC-2:** Для экзамена достаточно понимать, что U-Boot предназначен для встраиваемых устройств, является гибким и низкоуровневым инструментом и требует явного управления адресами памяти. Глубокое знание команд не проверяется.

| `ldlinux.c32` | Обязательная библиотека рядом с `pxelinux.0` на TFTP |

### Пути поиска файлов конфигурации Syslinux

| Загрузчик | Места поиска конфигурации |
|---|---|
| SYSLINUX | `/boot/syslinux/`, `/syslinux/`, `/` |
| ISOLINUX | `/boot/isolinux/`, `/isolinux/`, `/` |
| EXTLINUX | `/boot/extlinux/`, `/extlinux/`, `/` |
| PXELINUX | `/tftpboot/pxelinux.cfg/` (UUID → MAC → IP → `default`) |

### PXE: основные опции DHCP

| Опция DHCP | Роль в PXE |
|---|---|
| (53) | Тип DHCP-сообщения: Discover |
| (55) | Список запрашиваемых параметров (также запрашивает 128–135) |
| (93) | Client System Architecture (0=BIOS, 7=UEFI 64-bit) |
| (94) | Client Network Device Interface |
| (97) | Client UUID/GUID |
| (209) | pxelinux.config-file — имя файла конфигурации |
| (210) | pxelinux.pathprefix — префикс пути TFTP |
| UDP 4011 | Порт Proxy DHCP |

### Биты SYSAPPEND

| Бит | Добавляется к cmdline ядра |
|---|---|
| 1 | `ip=client:bootserver:gw:netmask` |
| 2 | `BOOTIF=` + MAC активного интерфейса |
| 4 | `SYSUUID=` + UUID системы |
| 8 | `CPU=` + информация о процессоре |

### Типичные ошибки

> **Внимание — частые ловушки:**
>
> - Отредактировали `/etc/lilo.conf`, но забыли запустить `/sbin/lilo` — изменения не применились.
> - `pxelinux.0` требует `ldlinux.c32` рядом на TFTP-сервере — без него загрузка не пройдёт.
> - systemd-boot требует монтирования ESP в `/boot`, а не в `/boot/efi`.
> - `isohdpfx.bin` нужен только для загрузки ISO с USB — для обычного CD не требуется.
> - При Secure Boot UEFI запускает `shim.efi`, а не `grubx64.efi` напрямую.
> - Применяется только **последняя** директива `APPEND` в конфигурации — предыдущие игнорируются.
> - `INITRD filename` — лишь синтаксический сахар для `APPEND initrd=filename`.

## Практические вопросы

**В1.** Вы настраиваете PXE-сервер для сетевой загрузки. Какой файл PXELINUX ищет на TFTP-сервере **первым**?

**Ответ:** Файл с именем по **UUID клиента** (в нижнем регистре). PXELINUX перебирает имена по порядку: UUID → MAC-адрес → уменьшающийся hex-префикс IP → `default`.

---

**В2.** Какой загрузчик из семейства Syslinux используется для загрузки с FAT-раздела (например, USB-флешки)?

**Ответ:** **SYSLINUX**. ISOLINUX работает с CD/DVD, EXTLINUX — с ext/btrfs, PXELINUX — с сетевой загрузкой.

---

**В3.** Какой основной файл конфигурации у ISOLINUX?

**Ответ:** `isolinux.cfg`. (`isolinux.bin` — сам бинарный файл загрузчика; `isohdpfx.bin` генерирует гибридный MBR для USB.)

---

**В4.** Что делает `shim.efi` в цепочке UEFI Secure Boot?

**Ответ:** Выступает **подписанным посредником** между UEFI и `grubx64.efi`. UEFI запускает подписанный `shim.efi`, который затем запускает `grubx64.efi`. Это позволяет использовать Secure Boot без покупки собственного ключа подписи.

---

**В5.** Чем systemd-boot отличается от GRUB?

**Ответ:** systemd-boot работает **исключительно с UEFI-системами** и умеет запускать только EFI-исполняемые файлы. Его конфигурация хранится в `/boot/loader/`, и он не умеет читать конфигурации с ext4 и других не-FAT разделов.

---

**В6.** Для каких устройств предназначен U-Boot?

**Ответ:** **Встраиваемые устройства** с разнообразными архитектурами (ARM, MIPS, PPC, x86 и др.). Отличительная черта — требование явного управления адресами памяти; другие загрузчики выбирают адреса автоматически.

---

**В7.** Вы отредактировали `/etc/lilo.conf`, но после перезагрузки изменения не вступили в силу. Почему?

**Ответ:** LILO **не читает** `lilo.conf` при загрузке — файл читает только установщик `/sbin/lilo`, который записывает результат в MBR. Без запуска `/sbin/lilo` изменения в `lilo.conf` не имеют никакого эффекта.

---

**В8.** Какая команда используется для управления systemd-boot?

**Ответ:** `bootctl` (`bootctl install`, `bootctl status`, `bootctl update`). `efibootmgr` управляет записями UEFI Boot Manager, а не самим загрузчиком.

---

**В9.** Какая опция DHCP в запросе PXE Discover сообщает серверу архитектуру клиента (BIOS или UEFI)?

**Ответ:** **Опция (93) Client System Architecture**. Значение 0 = x86 BIOS, значение 7 = EFI BC (64-bit UEFI). DHCP-сервер использует это поле для выбора правильного имени файла загрузчика.

---

**В10.** Администратор хочет, чтобы ядро автоматически получало сетевые IP-параметры и MAC-адрес интерфейса в cmdline при загрузке по PXE. Какую директиву следует добавить в конфигурацию PXELINUX?

**Ответ:** `SYSAPPEND 3` — устанавливает биты 1 и 2 одновременно. Бит 1 добавляет `ip=client:bootserver:gw:netmask`; бит 2 добавляет `BOOTIF=` с MAC-адресом активного интерфейса.
