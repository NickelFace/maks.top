## BIOS и UEFI: настройка до загрузки ОС

До запуска операционной системы оборудование можно настроить непосредственно в утилите BIOS или UEFI. Вход осуществляется при включении питания нажатием соответствующей клавиши — обычно Del, F2 или F12. Конкретная клавиша зависит от производителя и, как правило, отображается на экране во время POST.

BIOS (Basic Input/Output System) использовался на материнских платах x86 вплоть до середины 2000-х годов. С конца 2000-х его начал вытеснять UEFI (Unified Extensible Firmware Interface), обладающий более широкими возможностями: идентификация устройств, тестирование, настройка и обновление прошивки. На практике оба по-прежнему часто называют «BIOS», поскольку выполняют одну и ту же функцию.

Что можно сделать в BIOS/UEFI:

- Включить или отключить встроенные периферийные устройства (NIC, Bluetooth, последовательные порты)
- Изменить назначения IRQ и DMA для устройств
- Задать порядок загрузки, чтобы система загружалась с нужного диска
- Включить или отключить функции CPU (отключение экономит энергию и устраняет ряд уязвимостей)
- Установить правильные тайминги и частоту оперативной памяти согласно спецификации производителя

Если ОС не загружается без явной причины, первым делом следует проверить порядок загрузочных устройств в BIOS/UEFI.

## Файловые системы и псевдофайловые системы

Linux предоставляет три псевдофайловые системы, через которые можно считывать информацию об оборудовании в реальном времени.

### /proc/

`/proc/` не хранит на диске никаких реальных файлов. Это виртуальная файловая система, которую ядро генерирует на лету. В ней содержится информация о запущенных процессах и состоянии системы.

Полезные файлы:

```
/proc/cpuinfo       — информация о CPU
/proc/meminfo       — информация о памяти
/proc/interrupts    — таблица прерываний (IRQ)
/proc/ioports       — диапазоны портов ввода/вывода
/proc/dma           — используемые каналы DMA
/proc/bus/pci/      — устройства PCI
```

Например, просмотр модели CPU:

```bash
cat /proc/cpuinfo | grep "model name"
```

### /sys/

`/sys/` появился после `/proc/` и называется sysfs. Здесь ядро экспортирует иерархию устройств и драйверов в виде дерева каталогов. Каждое устройство в системе получает собственный каталог.

Например, найти информацию о сетевой карте можно здесь:

```bash
ls /sys/class/net/
```

Прочитать конкретный параметр устройства, например состояние подключения:

```bash
cat /sys/class/net/eth0/operstate
```

### /dev/

`/dev/` содержит файлы устройств. При открытии `/dev/sda` вы взаимодействуете с первым жёстким диском. При записи в `/dev/null` данные просто исчезают.

Типы файлов устройств:
- **Блочные (b):** жёсткие диски, SSD, USB-накопители (`/dev/sda`, `/dev/nvme0n1`)
- **Символьные (c):** терминалы, последовательные порты (`/dev/tty0`, `/dev/ttyS0`)

Список всех блочных устройств:

```bash
ls -l /dev/sd*
```

## sysfs, udev и dbus

### sysfs

sysfs, смонтированный в `/sys/`, экспортирует информацию о ядре, устройствах и драйверах. Это основной интерфейс между пространством пользователя и подсистемой устройств ядра.

### udev

udev управляет динамическими файлами устройств в `/dev/`. Ядро обнаруживает событие устройства и передаёт его процессу udev, который создаёт соответствующие файлы в `/dev/` согласно заранее определённым правилам.

udev обрабатывает два сценария:

- **Coldplug** — устройства, уже подключённые в момент включения машины. udev обрабатывает их при запуске системы.
- **Hotplug** — устройства, подключаемые во время работы системы (например, USB-флэшка). Ядро поддерживает hotplug начиная с версии 2.6.

udev опирается на sysfs (`/sys/`) для получения информации об устройствах. Правила идентификации устройств хранятся в `/etc/udev/rules.d/`. Базовые правила предоставляются дистрибутивом; вы можете добавлять свои для конкретных случаев.

Мониторинг событий устройств в реальном времени:

```bash
udevadm monitor
```

### dbus

D-Bus — шина сообщений, через которую процессы взаимодействуют друг с другом. Например, Network Manager использует D-Bus для уведомления рабочего стола об изменении сети. systemd активно использует D-Bus для управления службами.

## Типы устройств массового хранения

Все устройства хранения в Linux называются блочными, поскольку данные считываются и записываются в буферизованных блоках.

Начиная с ядра 2.4, большинство устройств хранения представлены как SCSI-устройства независимо от фактического типа подключения. IDE, SATA, SSD и USB-накопители — все получают префикс `sd`.

| Тип | Интерфейс | Пример в /dev/ | Примечания |
|------|-----------|------------------|-------|
| HDD/SSD (SATA, IDE, USB) | SATA/IDE/USB | `/dev/sda`, `/dev/sdb` | Префикс `sd`, буква зависит от порядка обнаружения |
| NVMe SSD | PCIe | `/dev/nvme0n1` | Префикс `nvme`, разделы: `nvme0n1p1`, `nvme0n1p2` |
| SD-карты | MMC | `/dev/mmcblk0` | Разделы: `mmcblk0p1`, `mmcblk0p2` |
| Оптические приводы (устаревший IDE) | IDE | `/dev/hdc` | Второй канал IDE, master |
| Дискеты | FDC | `/dev/fd0`, `/dev/fd1` | Устаревший тип |
| Виртуальные диски | Virtio | `/dev/vda` | Внутри виртуальных машин |

Устаревшая нумерация IDE: первый канал IDE master `/dev/sda`, slave `/dev/sdb`; второй канал master `/dev/sdc`, slave `/dev/sdd`. Оптический привод на втором канале получает имя `/dev/hdc` в старой нотации.

Именование разделов: для дисков `sd` просто добавляется номер: `/dev/sda1`, `/dev/sda2`. Для NVMe и MMC между именем диска и номером раздела вставляется `p`: `/dev/nvme0n1p1`, `/dev/mmcblk0p1`.

## Аппаратные ресурсы

Каждое устройство использует один или несколько ресурсов:

**IRQ (Interrupt Request)** — линия прерывания, используемая устройством для сигнализации CPU. Просмотр текущих назначений:

```bash
cat /proc/interrupts
```

**I/O Ports** — диапазоны адресов для обмена данными с устройством:

```bash
cat /proc/ioports
```

**DMA (Direct Memory Access)** — каналы для прямого доступа к памяти, минуя CPU:

```bash
cat /proc/dma
```

**Memory-mapped I/O** — диапазоны физической памяти, зарезервированные для устройств.

## Работа с модулями ядра

Драйверы устройств в Linux загружаются как модули ядра. Ниже перечислены основные инструменты.

### lsmod

Выводит список всех загруженных модулей:

```bash
lsmod
```

Вывод содержит три столбца:

| Столбец | Описание |
|--------|-------------|
| Module | Имя модуля |
| Size | Оперативная память, используемая модулем (байты) |
| Used by | Модули, зависящие от данного |

Столбец `Used by` важен при диагностике: нельзя выгрузить модуль, от которого зависят другие модули. Например, `snd_hda_intel` подтягивает `snd_hda_codec`, `snd_pcm` и другие аудиомодули.

Фильтрация вывода по имени:

```bash
lsmod | grep snd_hda_intel
```

### modprobe

Загружает или выгружает модуль вместе со всеми его зависимостями:

```bash
# Загрузить модуль
sudo modprobe bluetooth

# Выгрузить модуль и все зависимые (если не используются)
sudo modprobe -r snd-hda-intel
```

В отличие от `insmod`, `modprobe` разрешает зависимости автоматически. Флаг `-r` выгружает модуль только если ни один запущенный процесс его не использует.

Параметры модуля можно передавать непосредственно при загрузке:

```bash
sudo modprobe nouveau modeset=0
```

Чтобы параметры применялись при каждой загрузке, добавьте их в файл в `/etc/modprobe.d/`:

```bash
# /etc/modprobe.d/nouveau.conf
options nouveau modeset=0
```

### modinfo

Показывает описание модуля, путь к файлу, автора, лицензию, зависимости и доступные параметры:

```bash
modinfo bluetooth
```

Флаг `-p` выводит только доступные параметры и больше ничего:

```bash
modinfo -p nouveau
```

Пример вывода:

```
modeset:enable driver (default: auto, 0 = disabled, 1 = enabled) (int)
noaccel:disable kernel/abi16 acceleration (int)
```

Это полезно перед записью параметров в `/etc/modprobe.d/`.

## Просмотр устройств PCI: lspci

`lspci` выводит список всех устройств, подключённых к шине PCI/PCIe — видеокарты, сетевые карты, контроллеры SATA, карты расширения USB. Одни устройства распаяны на материнской плате, другие вставляются в слоты расширения.

> **Примечание:** многие аппаратные команды требуют привилегий root или показывают ограниченный вывод для обычных пользователей. Используйте `sudo`.

Базовый вывод:

```bash
lspci
```

Например:

```
01:00.0 VGA compatible controller: NVIDIA Corporation GM107 [GeForce GTX 750 Ti]
04:02.0 Network controller: Ralink corp. RT2561/RT61 802.11g PCI
```

Шестнадцатеричные числа в начале каждой строки (например, `04:02.0`) — уникальный PCI-адрес устройства на шине.

Подробная информация об устройстве по его адресу:

```bash
lspci -s 04:02.0 -v
```

В выводе отображаются флаги, IRQ, диапазон памяти и строка `kernel driver in use` с именем загруженного модуля.

Другие полезные флаги:

```bash
lspci -s 01:00.0 -k   # модуль в использовании + все доступные модули
lspci -v               # подробный вывод для всех устройств
lspci -vv              # очень подробный вывод
lspci -nn              # добавить числовые ID производителя и устройства
```

Пример вывода `lspci -s 01:00.0 -k`:

```
01:00.0 VGA compatible controller: NVIDIA Corporation GM107 [GeForce GTX 750 Ti]
    Kernel driver in use: nvidia
    Kernel modules: nouveau, nvidia_drm, nvidia
```

`Kernel driver in use` — активный драйвер; `Kernel modules` — все доступные модули для данного устройства.

Три признака корректной работы устройства:
1. Устройство обнаружено (`lspci` его видит)
2. Загружен подходящий модуль ядра (присутствует `Kernel driver in use`)
3. Устройство готово к использованию

## Просмотр USB-устройств: lsusb

`lsusb` выводит список всех USB-устройств в системе, включая концентраторы. Клавиатуры, мыши и съёмные накопители чаще всего используют интерфейс USB.

Базовый вывод:

```bash
lsusb
```

Например:

```
Bus 001 Device 028: ID 093a:2521 Pixart Imaging, Inc. Optical Mouse
Bus 001 Device 020: ID 1131:1001 Integrated System Solution Corp. KY-BT100 Bluetooth Adapter
Bus 001 Device 001: ID 1d6b:0002 Linux Foundation 2.0 root hub
```

Формат строки: шина (`Bus 001`), номер устройства (`Device 028`), ID в виде `vendorid:productid`, описание.

Выбор устройства по номеру шины и устройства (из вывода `lsusb -t`):

```bash
lsusb -s 01:20
```

Подробная информация об устройстве по его ID:

```bash
lsusb -v -d 1781:0c9f
```

Это отображает полный дескриптор устройства: класс, протокол, размер пакета, версию USB и многое другое. Если устройство отображается в `lsusb -t`, но поле `Driver=` пустое, значит модуль не загружен или устройство управляется напрямую приложением без модуля ядра.

Показать дерево устройств с портами и драйверами:

```bash
lsusb -t
```

Пример вывода `lsusb -t`:

```
/:  Bus 01.Port 1: Dev 1, Class=root_hub, Driver=dwc_otg/1p, 480M
    |__ Port 3: Dev 20, If 0, Class=Wireless, Driver=btusb, 12M
    |__ Port 1: Dev 7, If 0, Class=Vendor Specific Class, Driver=lan78xx, 480M
```

Дерево показывает, к какому порту подключено каждое устройство, его класс и активный драйвер.

## Управление USB-устройствами

Помимо `lsusb`, существует несколько способов взаимодействия с USB-устройствами.

Мониторинг событий подключения через udev:

```bash
udevadm monitor --subsystem-match=usb
```

Получение подробной информации об устройстве через udev:

```bash
udevadm info -a -n /dev/sdb
```

Установка правила udev для конкретного USB-устройства (файл `/etc/udev/rules.d/99-my-usb.rules`):

```
SUBSYSTEM=="usb", ATTR{idVendor}=="046d", ATTR{idProduct}=="c52b", MODE="0666"
```

## Включение и отключение периферийных устройств

Встроенные периферийные устройства (Wi-Fi, Bluetooth, последовательные порты) часто настраиваются в BIOS/UEFI. Из Linux ими можно управлять через:

1. Выгрузку модуля ядра: `sudo modprobe -r iwlwifi` отключает Wi-Fi
2. Чёрный список в `/etc/modprobe.d/`: запрещает загрузку модуля при загрузке системы

Два подхода к блокировке. Можно добавить строку в существующий `/etc/modprobe.d/blacklist.conf`:

```
blacklist nouveau
```

Предпочтительный подход — создать отдельный файл для конкретного модуля:

```
# /etc/modprobe.d/nouveau.conf
blacklist nouveau
```

Это необходимо, например, когда установлен проприетарный драйвер `nvidia` и нужно не допустить загрузки модуля `nouveau` по умолчанию.

После внесения изменений пересоберите initramfs:

```bash
sudo update-initramfs -u
```

## Типовые вопросы экзамена

1. Какой преемник BIOS на системах x86? → UEFI
2. Какой файл показывает назначения IRQ? → `/proc/interrupts`
3. Какая команда загружает модуль с зависимостями? → `modprobe`
4. Что такое udev? → Менеджер динамических файлов устройств
5. Где хранятся файлы устройств? → `/dev/`
6. Чем `/sys/` отличается от `/proc/`? → `/sys/` специализирован для структуры устройств; `/proc/` охватывает процессы и общее состояние ядра
7. Какая команда показывает драйвер для PCI-устройства? → `lspci -k`
8. Как просмотреть параметры модуля ядра? → `modinfo -p module_name`
9. Где хранятся постоянные параметры модулей? → `/etc/modprobe.d/`
10. Что такое coldplug? → Обнаружение устройств, уже подключённых в момент запуска системы
11. Как называются разделы NVMe? → `/dev/nvme0n1p1`, `/dev/nvme0n1p2`
12. Как называются разделы SD-карты? → `/dev/mmcblk0p1`, `/dev/mmcblk0p2`
13. Каков предпочтительный способ добавить модуль в чёрный список? → Создать отдельный файл `/etc/modprobe.d/module_name.conf` с директивой `blacklist`

## Упражнения

### Упражнение 1 — Порядок загрузки с SATA-диска

Система перестала загружаться после добавления второго SATA-диска. Все компоненты исправны. Какова вероятная причина?

<details>
<summary>Ответ</summary>

Скорее всего, BIOS/UEFI начал пытаться загружаться с нового диска вместо того, на котором установлена операционная система. При добавлении второго SATA-диска он может получить более высокий приоритет в списке загрузочных устройств, и BIOS попытается загрузиться с него. Поскольку на новом диске нет загрузчика, система зависает.

Исправление: войдите в BIOS/UEFI и восстановите правильный диск на верхней позиции в порядке загрузки.

</details>

### Упражнение 2 — Просмотр видеокарты с помощью lspci

Нужно проверить характеристики дискретной видеокарты, подключённой к шине PCI, не открывая корпус. Какую команду вы используете?

<details>
<summary>Ответ</summary>

```bash
lspci -v
```

`lspci` выводит список всех устройств на шине PCI/PCIe, включая видеокарты. Флаг `-v` добавляет подробности: производитель, модель, подсистема (торговое название) и другие атрибуты.

Если карт несколько или нужно указать конкретную, сначала найдите её адрес с помощью `lspci`, затем запросите подробности:

```bash
lspci -s 01:00.0 -v
```

Строка `Subsystem` в выводе показывает именно то, что написано на коробке.

</details>

### Упражнение 3 — Модуль ядра для RAID-контроллера

Вывод `lspci` содержит такую строку:

```
03:00.0 RAID bus controller: LSI Logic / Symbios Logic MegaRAID SAS 2208 [Thunderbolt] (rev 05)
```

Какая команда показывает загруженный модуль ядра для этого устройства?

<details>
<summary>Ответ</summary>

```bash
lspci -s 03:00.0 -k
```

Флаг `-s` указывает конкретный адрес устройства; `-k` показывает `Kernel driver in use` (активный драйвер) и `Kernel modules` (все доступные модули для данного устройства).

</details>

### Упражнение 4 — Ошибка выгрузки модуля

Администратор пытается выгрузить модуль `bluetooth` с помощью `modprobe -r bluetooth` и получает:

```
modprobe: FATAL: Module bluetooth is in use.
```

Какова причина?

<details>
<summary>Ответ</summary>

Модуль `bluetooth` используется запущенным процессом. `modprobe -r` выгружает модуль только если его никто не использует.

Проверьте, кто удерживает модуль, с помощью `lsmod | grep bluetooth` и посмотрите столбец `Used by`. Сначала нужно выгрузить зависимые модули (например, `btusb`), а затем сам `bluetooth`. Либо остановите службу `bluetoothd` и повторите попытку.

</details>

### Упражнение 5 — Безголовая загрузка на устаревшем BIOS

На старых серверах с устаревшим BIOS система отказывается загружаться, если не подключена клавиатура. Как это исправить?

<details>
<summary>Ответ</summary>

В настройках BIOS найдите опцию, управляющую поведением при отсутствии клавиатуры — обычно называется `Halt on` или `Boot with keyboard errors`. Установите значение `All, But Keyboard` (или аналогичное), чтобы игнорировать отсутствие клавиатуры и продолжить загрузку. Стандартная настройка для безголовых серверов.

</details>

### Упражнение 6 — Нет lspci на ARM

На одноплатных компьютерах, таких как Raspberry Pi (ARM), команда `lspci` отсутствует. Почему?

<details>
<summary>Ответ</summary>

Потому что у ARM-плат нет шины PCI/PCIe. `lspci` опрашивает именно шину PCI, которая является стандартом для x86. На ARM периферийные устройства подключены через внутреннюю шину SoC напрямую, без интерфейса PCI. Используйте `lsusb` для USB-устройств или читайте из `/sys/bus/` вместо этого.

</details>

### Упражнение 7 — USB-накопитель на Linux-роутере

К роутеру на базе Linux подключён внешний USB-жёсткий диск. Других блочных устройств нет. Как он будет называться в `/dev/`?

<details>
<summary>Ответ</summary>

```
/dev/sda
```

Начиная с ядра 2.4, USB-накопители представлены как SCSI-устройства и получают префикс `sd`. Поскольку других блочных устройств нет, первый диск получает букву `a`. Его разделы: `/dev/sda1`, `/dev/sda2` и т.д.

</details>

### Упражнение 8 — Проверка уязвимости Meltdown

Как проверить, подвержена ли текущая система уязвимости Meltdown?

<details>
<summary>Ответ</summary>

```bash
grep bugs /proc/cpuinfo
```

Если CPU уязвим, в выводе будет `bugs: cpu_meltdown`. Современные ядра также предоставляют детальную проверку через sysfs:

```bash
cat /sys/devices/system/cpu/vulnerabilities/meltdown
```

На экзамене ожидается подход через `/proc/cpuinfo`.

</details>
