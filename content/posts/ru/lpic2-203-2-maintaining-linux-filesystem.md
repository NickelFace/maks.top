## fsck — проверка и восстановление файловых систем

`fsck` — это обёртка для всех утилит проверки файловых систем. Реальную работу выполняет `fsck.fstype` — где `fstype` — это `ext4`, `btrfs`, `xfs` и т.д.

> **Внимание — критично для экзамена:** `fsck.xfs` и `fsck.btrfs` — **пустые заглушки**. Они ничего не делают. Для XFS используйте `xfs_repair`; для Btrfs — `btrfs check`.

`fsck` запускается автоматически при загрузке, если файловая система помечена как «нечистая», превышено максимальное количество монтирований или истёк интервал проверки.

### Часто используемые опции fsck

| Опция | Описание |
|---|---|
| `-s` | Сериализовать операции (запускать проверки последовательно) |
| `-A` | Проверить все файловые системы из `/etc/fstab` за один проход |
| `-R` | При использовании с `-A` пропустить корневую файловую систему |
| `-n` | Без изменений, только проверка (сухой прогон) |

### Порядок проверки через /etc/fstab

Файловые системы с `fs_passno` равным `0` не проверяются вовсе. Корневой `/` проверяется первым; остальные проверяются в порядке возрастания `fs_passno`. Файловые системы с одинаковым значением `fs_passno` могут проверяться параллельно, но не если они расположены на одном физическом диске.

### e2fsck — проверщик ext-файловых систем

`e2fsck` — реальный проверщик для ext2/3/4; `fsck` просто вызывает его.

```bash
# Принудительная проверка, даже если файловая система чистая
e2fsck -f /dev/sda1

# Автоматическое исправление без запросов
e2fsck -p /dev/sda1

# Ответить "да" на все вопросы
e2fsck -y /dev/sda1

# Ответить "нет" на всё (только чтение)
e2fsck -n /dev/sda1

# Найти и пометить плохие блоки
e2fsck -c /dev/sda1
```

> **Примечание — lost+found:** Когда `fsck` находит файлы без записей в каталогах (inode существует, но имя в каталоге отсутствует), он помещает их в каталог `lost+found` соответствующей файловой системы. Периодически проверяйте этот каталог.

## mkfs — создание файловых систем

`mkfs` — обёртка, аналогичная `fsck`. Реальную работу выполняет `mkfs.fstype`.

```bash
# Создать ext4 на разделе
mkfs -t ext4 /dev/sdb1
# Или напрямую:
mkfs.ext4 /dev/sdb1

# Создать Btrfs на нескольких устройствах
mkfs.btrfs /dev/sdb /dev/sdc

# Создать XFS
mkfs.xfs /dev/sdb1

# Btrfs с RAID0 для данных
mkfs.btrfs -d raid0 /dev/sdb /dev/sdc

# Btrfs с RAID10 для данных и метаданных
mkfs.btrfs -m raid10 -d raid10 /dev/sdb /dev/sdc /dev/sdd /dev/sde
```

## tune2fs — настройка ext-файловых систем

`tune2fs` — многофункциональный инструмент для тонкой настройки файловых систем ext2/3/4. Чаще всего используется для управления периодичностью проверок и UUID.

### Управление интервалами проверки

```bash
# Показать текущие параметры суперблока
tune2fs -l /dev/sda1

# Задать максимальное количество монтирований до принудительной проверки
tune2fs -c 10 /dev/sda1

# Задать интервал между проверками (в секундах; 0 = отключить)
tune2fs -i 30d /dev/sda1

# Установить текущий счётчик монтирований (для распределения проверок)
tune2fs -C 15 /dev/sda1
```

> **Совет — распределение счётчиков монтирований:** Если у вас 5 разделов и система перезагружается раз в месяц, назначьте каждому разделу разные значения счётчика. Это предотвратит одновременную проверку всех разделов на 20-й перезагрузке и сократит время запуска.

### Изменение UUID и метки

```bash
# Сгенерировать новый UUID
uuidgen

# Присвоить новый UUID разделу
tune2fs /dev/sdc1 -U b77a195a-e5a8-4810-932e-5d9adb97adc6

# Установить метку файловой системы
tune2fs -L "my_data" /dev/sda1
```

> **Внимание:** После изменения UUID обновите `/etc/fstab` и `/boot/grub/grub.cfg`, если раздел является загрузочным — иначе система не загрузится.

## dumpe2fs — информация суперблока

`dumpe2fs` выводит подробную информацию о параметрах файловой системы ext2/3/4, хранящихся в суперблоке.

```bash
# Показать всю информацию о файловой системе
dumpe2fs /dev/sda1

# Показать только информацию суперблока (без описания групп блоков)
dumpe2fs -h /dev/sda1
```

Вывод включает: UUID, метку, размер блока, количество inode, время последней проверки, максимальное количество монтирований и многое другое.

## badblocks — поверхностное сканирование

`badblocks` находит повреждённые секторы и помечает их так, чтобы файловая система больше не использовала их для хранения данных.

> **Внимание:** Не запускайте `badblocks` напрямую. Официальная рекомендация — вызывать его через опцию `-c` команды `e2fsck` или `mke2fs`. Прямой запуск может конфликтовать с существующим списком плохих блоков в inode.

```bash
# Правильный способ: через e2fsck
e2fsck -c /dev/sda1

# Сохранить список плохих блоков в файл
badblocks -o /tmp/bad_blocks.txt /dev/sda1

# Неразрушающий тест чтения (безопасен для смонтированных дисков)
badblocks -n /dev/sda1

# Разрушающий тест записи (только для несмонтированных разделов!)
badblocks -w /dev/sda1
```

## debugfs — работа с ext-файловыми системами на низком уровне

`debugfs` — интерактивный инструмент для низкоуровневых операций с ext-файловыми системами. По умолчанию открывает файловую систему только для чтения; режим записи требует флага `-w`.

```bash
# Открыть файловую систему только для чтения
debugfs /dev/sda1

# Открыть в режиме записи (использовать с осторожностью!)
debugfs -w /dev/sda1

# Использовать резервный суперблок при повреждении основного
debugfs -b 1024 -s 8193 /dev/sda1
# (-b = размер блока, -s = номер резервного суперблока)
# Расположение резервных суперблоков можно узнать через dumpe2fs
```

### Полезные команды внутри debugfs

```
params              # показать текущий режим и файловую систему
open /dev/sdb1      # переключиться на другую файловую систему без выхода
close               # закрыть текущую файловую систему
stats               # статистика суперблока по группам блоков
testb <block>       # проверить, используется ли блок
icheck <block>      # получить номер inode для блока
ncheck <inode>      # получить имя файла для inode
stat <inode>        # показать информацию inode
ls /                # вывести список файлов в каталоге
dump <inode> /tmp/recovered_file   # извлечь файл из повреждённой ФС
?                   # вывести список всех доступных команд
```

> **Совет — работа с плохими блоками через debugfs:** Найдите плохой блок с помощью `testb`. Получите inode через `icheck`, затем имя файла через `ncheck`. Лучшее решение: пометить блок как плохой и восстановить файл из резервной копии.

> **Внимание:** Используйте `-w` только если точно знаете, что делаете. Некорректные изменения метаданных могут сделать файловую систему неработоспособной.

## Btrfs — подтома, снимки и конвертация

Btrfs — современная файловая система с Copy-on-Write (COW), встроенными контрольными суммами, RAID и снимками.

> **Примечание:** Подтом Btrfs — это не блочное устройство (в отличие от логического тома LVM). Это POSIX-пространство имён, которое можно смонтировать отдельно через опцию `subvol=`.

### Работа с подтомами

```bash
# Создать подтом
btrfs subvolume create /home

# Вывести список всех подтомов
btrfs subvolume list /mnt/btrfs-test/

# Смонтировать конкретный подтом
mount -o subvol=subvol_z /dev/sdb /mnt/point
```

### Работа со снимками

```bash
# Создать снимок /home
btrfs subvolume snapshot /home/ /home-snap

# Создать снимок только для чтения
btrfs subvolume snapshot -r /home/ /home-snap-ro

# Удалить снимок
btrfs subvolume delete /home-snap/
```

> **Совет:** После удаления снимка Btrfs в фоновом режиме очищает дерево файлов. На простаивающей системе может наблюдаться активность ввода-вывода — это нормально.

### Утилиты проверки и восстановления Btrfs

| Утилита | Описание |
|---|---|
| `btrfs check` | Проверить и при необходимости восстановить несмонтированную файловую систему |
| `btrfs rescue` | Восстановить повреждённую файловую систему |
| `btrfs restore` | Восстановить файлы из повреждённой файловой системы (наиболее мощный инструмент) |
| `btrfs scrub` | Прочитать все данные с диска и проверить контрольные суммы |
| `btrfsck` | Устаревший псевдоним для `btrfs check` |

```bash
# Запустить scrub (проверку целостности)
btrfs scrub start /mnt/data

# Проверить статус scrub
btrfs scrub status /mnt/data

# Информация о файловой системе
btrfs filesystem show
btrfs filesystem df /mnt/data
```

> **Внимание:** `btrfs scrub` читает весь диск последовательно. На загруженных серверах запускайте его ночью или ограничивайте с помощью `btrfs scrub start -B -d`.

### Конвертация ext → Btrfs

```bash
# Конвертировать ext4 в Btrfs (раздел должен быть несмонтирован)
btrfs-convert /dev/sdb1

# Конвертировать обратно в ext4
btrfs-convert -r /dev/sdb1
```

> **Важно:** `btrfs-convert` сохраняет образ исходной ext-файловой системы как подтом с именем `ext2_subvol` (по умолчанию). Пока вы его не удалили, можно выполнить откат командой `btrfs-convert -r`. После удаления подтома откат невозможен.

## XFS — утилиты

XFS — журналируемая файловая система, используемая по умолчанию на RHEL/CentOS. Имеет собственный набор утилит.

### Информация и проверка

```bash
# Показать геометрию файловой системы
xfs_info /dev/sda1
# Эквивалентно:
xfs_growfs -n /dev/sda1

# Проверить целостность (сухой прогон, без исправлений)
xfs_check /dev/sda1

# Восстановить повреждённую файловую систему (должна быть несмонтирована)
xfs_repair /dev/sda1

# Сухой прогон xfs_repair (когда xfs_check недоступен)
xfs_repair -n /dev/sda1
```

> **Внимание:** `xfs_check` отсутствует во многих современных дистрибутивах. Используйте `xfs_repair -n` для сухого прогона.

### Резервное копирование и восстановление для XFS

```bash
# Создать дамп файловой системы
xfs_dump -f /backup/sda1.dump /dev/sda1

# Восстановить из дампа
xfs_restore -f /backup/sda1.dump /mnt/restore

# Инкрементальный дамп (уровень 1)
xfs_dump -l 1 -f /backup/sda1-inc.dump /dev/sda1
```

> **Примечание:** `xfs_dump` сохраняет расширенные атрибуты, ACL и другие метаданные, специфичные для XFS. Обычный `tar` этого не делает.

### Сводная таблица утилит XFS

| Утилита | Назначение |
|---|---|
| `xfs_info` | Показать геометрию файловой системы |
| `xfs_check` | Проверить без исправлений (устарел) |
| `xfs_repair` | Проверить и восстановить |
| `xfs_dump` | Резервное копирование |
| `xfs_restore` | Восстановление из дампа |
| `xfs_admin` | Настройка атрибутов файловой системы (UUID, метка) |
| `xfs_fsr` | Дефрагментация файлов |
| `xfs_growfs` | Расширить файловую систему |
| `xfs_metadump` | Скопировать метаданные в файл |

## ZFS — обзор

ZFS разработана компанией Sun Microsystems и в настоящее время принадлежит Oracle. В Linux нативная поддержка на уровне ядра есть только в Ubuntu; другие дистрибутивы используют ZFS через FUSE в пространстве пользователя.

> **Примечание для LPIC-2:** Требуется только общее понимание ZFS. Глубокие знания команд не проверяются — только концепции и архитектура пулов.

### Ключевые возможности ZFS

- 128-битная адресация (практически неограниченный объём хранилища)
- Copy-on-Write с транзакционной записью: **ueberblock** обновляется только при успешном завершении полной транзакции
- 128 предыдущих копий ueberblock хранятся в кольцевом буфере
- **Метки Vdev** — метки дисков в пуле, хранятся в 4 копиях: 2 в начале и 2 в конце каждого диска
- Встроенный RAID-Z (эквивалент RAID5) и RAID-Z2 (эквивалент RAID6)
- Снимки и клоны (создаются мгновенно, не занимают место до изменения данных)
- Сжатие и дедупликация данных (настраивается для каждой файловой системы)
- Контрольные суммы для всех данных и метаданных
- Выделение томов (zvols — блочные устройства внутри ZFS)
- Отдельные устройства для кэша (L2ARC) и журнала (ZIL/SLOG)

> **Почему в ZFS нет fsck:** ZFS работает транзакционно. Ueberblock обновляется только если всё было успешно записано. Система хранит 128 предыдущих ueberblock в кольцевом буфере. Для проверки целостности и исправления используйте `zpool scrub` — он исправляет ошибки на лету при наличии нескольких дисков в пуле.

### zpool — управление пулами хранилищ

Пул в ZFS аналогичен менеджеру логических томов. Пул может состоять из одного или нескольких дисков.

```bash
# Создать простой однодисковый пул
zpool create tank /dev/sdb

# Создать зеркальный пул
zpool create tank mirror /dev/sdb /dev/sdc

# Показать список пулов с подробностями
zpool list -v

# Показать статус пула
zpool status -v tank

# Запустить проверку целостности
zpool scrub tank

# Показать историю операций с пулом
zpool history tank
```

### zfs — управление файловыми системами внутри пула

По умолчанию при создании пула внутри него создаётся одна файловая система с таким же именем.

```bash
# Показать все файловые системы ZFS
zfs list

# Создать файловую систему с включённым сжатием
zfs create -o compression=on tank/documents

# Включить сжатие на существующей файловой системе
zfs set compression=on tank/documents

# Проверить коэффициент сжатия
zfs get compressratio tank/documents

# Показать все атрибуты файловой системы
zfs get all tank/documents

# Создать снимок (мгновенно, не занимает место)
zfs snap tank/documents@backup

# Вывести список снимков
zfs list -t snapshot

# Получить доступ к содержимому снимка через специальный каталог
ls /tank/documents/.zfs/snapshot/backup/
```

> **Совет:** Снимок ZFS создаётся мгновенно и изначально занимает 0 байт. Место потребляется только по мере изменения исходных данных. Если данные не изменились, снимок бесплатен.

## SMART — мониторинг состояния дисков

SMART (Self-Monitoring, Analysis and Reporting Technology) — технология самодиагностики, встроенная в большинство современных HDD и SSD. Поддержка в Linux реализована пакетом `smartmontools`.

### smartctl — прямое взаимодействие со SMART

```bash
# Показать информацию об устройстве
smartctl -i /dev/sda

# Включить SMART на устройстве
smartctl -s on /dev/sda

# Быстрая проверка здоровья (вердикт PASSED / FAILED)
smartctl -H /dev/sda

# Полная информация об устройстве и атрибутах
smartctl -a /dev/sda

# Запустить короткий самотест
smartctl -t short /dev/sda

# Запустить длительный расширенный самотест
smartctl -t long /dev/sda

# Прервать выполняющийся тест
smartctl -X /dev/sda

# Просмотреть журнал ошибок
smartctl -l error /dev/sda

# Показать поддерживаемые типы устройств
smartctl -P showall | less
```

> **Важно:** Результат `SMART overall-health self-assessment test result: PASSED` означает, что диск исправен. `FAILED` означает, что диск уже вышел из строя или вот-вот выйдет.

> **Внимание:** SMART не гарантирует предупреждение заранее. Устройство может показывать PASSED вплоть до момента отказа. Необычные значения атрибутов (высокий `Reallocated_Sector_Ct`, ненулевой `Current_Pending_Sector`) могут указывать на надвигающийся отказ даже при статусе PASSED.

### smartd — демон мониторинга

`smartd` работает как системная служба и автоматически включает мониторинг SMART на всех подключённых устройствах. По умолчанию проверяет каждые **30 минут**.

Файлы конфигурации (путь зависит от дистрибутива):
- `/etc/smartd.conf`
- `/etc/smartmontools/smartd.conf`
- `/usr/local/etc/smartd.conf`

Файлы журналов:
- `/var/log/smartd.log`
- `/var/log/messages`
- `/var/log/syslog`

```bash
# Отправить USR1 для немедленной проверки всех дисков
kill -USR1 $(pidof smartd)
```

### Конфигурация smartd.conf

```
# Мониторить все SMART-устройства; длительный тест каждое воскресенье с 1 до 2 ночи
DEVICESCAN -s L/../../7/01

# Мониторить конкретный диск и отправлять email при ошибке
/dev/sda -m admin@example.com -M exec /usr/bin/my_alert_script
```

> **Примечание — DEVICESCAN:** Применяется только первая незакомментированная строка, начинающаяся с `DEVICESCAN`. Все последующие строки `DEVICESCAN` игнорируются.

## Шпаргалка для экзамена

### Основные пути и файлы

| Путь | Описание |
|---|---|
| `/etc/fstab` | Настройки монтирования; поле `fs_passno` управляет порядком fsck |
| `/etc/smartd.conf` | Конфигурация демона smartd (основной путь) |
| `/etc/smartmontools/smartd.conf` | Альтернативный путь конфигурации smartd |
| `/usr/local/etc/smartd.conf` | Ещё один возможный путь конфигурации smartd |
| `/var/log/smartd.log` | Журнал smartd |
| `lost+found/` | Каталог для восстановленных файлов без имён |

### Важные команды

```bash
fsck -A                          # проверить все файловые системы из fstab
e2fsck -f /dev/sda1              # принудительная проверка ext4
tune2fs -l /dev/sda1             # показать параметры ext4
tune2fs -c 10 /dev/sda1          # установить максимальное количество монтирований
btrfs subvolume snapshot /home/ /home-snap   # создать снимок Btrfs
btrfs-convert /dev/sdb1          # конвертировать ext4 в Btrfs
xfs_info /dev/sda1               # информация о файловой системе XFS
xfs_repair /dev/sda1             # восстановить XFS (должна быть несмонтирована)
smartctl -H /dev/sda             # вердикт о состоянии диска
smartctl -a /dev/sda             # полная информация SMART
```

### Таблица соответствия инструментов файловым системам

| Утилита | ext2/3/4 | Btrfs | XFS | ZFS |
|---|---|---|---|---|
| `fsck` | Да (через e2fsck) | Заглушка | Заглушка | Нет |
| `tune2fs` | Да | Нет | Нет | Нет |
| `dumpe2fs` | Да | Нет | Нет | Нет |
| `debugfs` | Да | Нет | Нет | Нет |
| `btrfs check` | Нет | Да | Нет | Нет |
| `xfs_repair` | Нет | Нет | Да | Нет |
| `zpool scrub` | Нет | Нет | Нет | Да |

### Типичные ловушки на экзамене

- `fsck.xfs` и `fsck.btrfs` — заглушки, они ничего не делают. Используйте `xfs_repair` и `btrfs check`.
- `xfs_repair` и `e2fsck` нельзя запускать на смонтированных разделах.
- `tune2fs -c 0` отключает проверки на основе счётчика монтирований — рискованно, ошибки могут накапливаться незаметно.
- `btrfs-convert` сохраняет старую ext-файловую систему как подтом `ext2_subvol`. Удаляйте его явно для освобождения места.
- `xfs_check` отсутствует во многих современных дистрибутивах; используйте `xfs_repair -n` вместо него.
- `smartctl -H` даёт только общий вердикт. Используйте `smartctl -a` для полной картины.

## Практические вопросы

**В1.** Вы запускаете `fsck.xfs /dev/sdb1`. Что произойдёт?

**Ответ:** **Ничего** — `fsck.xfs` является пустой заглушкой. Для реального восстановления XFS используйте `xfs_repair`.

---

**В2.** Как изменить максимальное количество монтирований для ext4-раздела `/dev/sda2` до 15?

**Ответ:** `tune2fs -c 15 /dev/sda2` — устанавливает максимальное количество монтирований до принудительной проверки.

---

**В3.** Нужно создать снимок подтома Btrfs `/data` с именем `/data-backup`. Какая команда правильная?

**Ответ:** `btrfs subvolume snapshot /data /data-backup`

---

**В4.** Какая команда показывает общий вердикт о состоянии SMART (PASSED/FAILED) для `/dev/sda`?

**Ответ:** `smartctl -H /dev/sda` — выводит результат общего теста самооценки. `-a` даёт полную картину; `-i` показывает только информацию об устройстве; `-t` запускает тест.

---

**В5.** Вы конвертировали ext4-раздел в Btrfs с помощью `btrfs-convert`. Как называется подтом, содержащий старый образ ext4?

**Ответ:** `ext2_subvol` — имя по умолчанию, используемое `btrfs-convert`. Откат (`btrfs-convert -r`) возможен только пока этот подтом существует.

---

**В6.** Как выполнить сухой прогон проверки файловой системы XFS на системе, где `xfs_check` не установлен?

**Ответ:** `xfs_repair -n /dev/sda1` — выполняет проверку без внесения каких-либо изменений.

---

**В7.** Какое значение `fs_passno` в `/etc/fstab` заставляет `fsck -A` пропустить эту файловую систему?

**Ответ:** **`0`** — файловая система не проверяется.

---

**В8.** С каким интервалом `smartd` по умолчанию опрашивает SMART-устройства?

**Ответ:** Каждые **30 минут**. Это можно изменить в файле конфигурации.
