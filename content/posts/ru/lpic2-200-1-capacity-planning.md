> **Вес на экзамене: 6 баллов** — одна из самых высокоприоритетных тем раздела 200.

Основной принцип: **«Нельзя управлять тем, что нельзя измерить.»** Мониторинг ресурсов делится на три категории:

- **Оперативный** — наблюдение за текущим состоянием системы
- **Диагностический** — поиск первопричины конкретной проблемы
- **Планирование ёмкости** — сбор данных для планирования ресурсов

Ключевые метрики для отслеживания на каждой системе: uptime, средняя нагрузка CPU, использование ОЗУ и swap, дисковый ввод/вывод, сетевой ввод/вывод, пропускная способность брандмауэра и маршрутизатора.

## Измерение CPU

### uptime

Показывает текущее время, время работы системы, количество вошедших пользователей и **среднюю нагрузку** за 1, 5 и 15 минут.

```bash
uptime          # стандартный вывод
uptime -V       # версия утилиты
```

Пример вывода:

```
$ uptime
03:03:12 up  1:00,  2 users,  load average: 0.17, 0.18, 0.16
```

> **Примечание:** Средняя нагрузка — среднее число процессов в очереди на выполнение (выполняющихся + ожидающих ввода/вывода). Значение **≤ числа ядер CPU** считается нормальным. Если средняя нагрузка превышает число ядер, система перегружена.

---

### top

Динамический монитор реального времени для CPU, памяти и состояний процессов.

```bash
top
```

Пример вывода:

```
top - 03:01:24 up 59 min,  2 users,  load average: 0.15, 0.19, 0.16
Tasks: 117 total,   2 running, 115 sleeping,   0 stopped,   0 zombie
%Cpu(s):  0.9 us,  4.5 sy,  0.1 ni, 94.3 id,  0.1 wa,  0.0 hi,  0.1 si,  0.0 st
KiB Mem:    514332 total,   497828 used,    16504 free,    63132 buffers
KiB Swap:   392188 total,        0 used,   392188 free,   270552 cached

  PID USER      PR  NI  VIRT  RES  SHR S  %CPU %MEM    TIME+  COMMAND
 4041 root      20   0  106m  31m 9556 R  30.4  6.3   3:05.58 Xorg
 4262 user      20   0  527m  71m  36m S  18.2 14.3   2:04.42 gnome-shell
```

**Поля CPU (`%Cpu(s)`):**

| Поле | Значение |
|---|---|
| `us` | Пространство пользователя (не ядро) |
| `sy` | Пространство ядра |
| `ni` | Процессы с изменённым значением nice |
| `id` | Простой |
| `wa` | Ожидание ввода/вывода — **высокое значение = узкое место диска** |
| `hi` | Аппаратные прерывания |
| `si` | Программные прерывания |
| `st` | Время кражи (гипервизор забрал CPU у этой ВМ) |

**Горячие клавиши `top`:**

| Клавиша | Действие |
|---|---|
| `P` | Сортировать по CPU |
| `M` | Сортировать по памяти |
| `k` | Завершить процесс |
| `1` | Показать каждое ядро CPU отдельно |
| `h` / `?` | Справка |
| `q` | Выйти |

### htop

Расширенный `top` с вертикальной и горизонтальной прокруткой. Показывает все процессы с полными командными строками. Завершение процессов и изменение приоритета без ввода PID.

```bash
htop            # запустить
htop -u user    # фильтрация по пользователю
```

> **Совет:** Главное преимущество `htop` перед `top`: управление процессами (завершение, изменение приоритета) без знания PID — всё через интерактивный интерфейс с функциональными клавишами.

### sar — System Activity Reporter

Собирает, хранит и отображает активность системы. Данные хранятся в `/var/log/sa/` (демоном `sadc`). Пакет: `sysstat`.

```bash
sar                    # статистика CPU за сегодня (с интервалом 10 минут)
sar 2 20               # 20 выборок с интервалом 2 секунды
sar -u 1 5             # использование CPU
sar -d 1 5             # статистика дискового ввода/вывода
sar -b 1 5             # скорости передачи ввода/вывода
sar -r 1 5             # свободная память и swap
sar -n DEV 1 5         # сетевые интерфейсы
sar -q 1 5             # очередь выполнения
sar -c 1 5             # системные вызовы
sar -p                 # активность подкачки страниц
sar -w                 # активность свопинга
sar -f /var/log/sa/sa15  # данные из файла (15-е число месяца)
```

Например — CPU (без параметров):

```
$ sar
Linux 3.2.0-4-686-pae (debian)  05/07/2013  _i686_  (2 CPU)

02:02:34      LINUX RESTART

02:05:01    CPU     %user     %nice   %system   %iowait    %steal     %idle
02:15:01    all      0.15      0.00      1.06      0.23      0.00     98.56
02:25:01    all      0.98      0.83      3.84      0.04      0.00     94.31
Average:    all      0.64      0.14      4.19      0.06      0.00     94.98
```

Например — статистика диска (`sar -d`):

```
$ sar -d
06:45:01     DEV      tps  rd_sec/s  wr_sec/s  avgrq-sz avgqu-sz   await   svctm   %util
06:55:01  dev8-0     6.89    227.01     59.67     41.59     0.02    2.63    1.38    0.95
Average:  dev8-0     3.49     85.63     30.14     33.15     0.01    2.36    1.19    0.42
```

**Ключевые параметры для экзамена:**

| Параметр | Показывает |
|---|---|
| _(без параметра)_ | Использование CPU |
| `-u` | Использование CPU (явно) |
| `-d` | Дисковый ввод/вывод (`tps`, `rd_sec/s`, `wr_sec/s`, `%util`) |
| `-b` | Скорости передачи ввода/вывода (`tps`, `bread/s`, `bwrtn/s`) |
| `-r` | Свободная память и swap |
| `-n DEV` | Сетевые интерфейсы |
| `-q` | Очередь выполнения |
| `-c` | Системные вызовы |
| `-p`, `-w` | Активность подкачки страниц и свопинга |

> **Важно:** Данные `sar` хранятся в `/var/log/sa/`. Строка `LINUX RESTART` в выводе означает перезагрузку системы в тот период.

### ps и pstree

`ps` поддерживает три синтаксических стиля, которые можно смешивать:

- **UNIX** — с одним дефисом: `ps -ef`
- **BSD** — без дефиса: `ps aux`
- **GNU long** — с двумя дефисами: `ps --forest`

```bash
ps aux              # все процессы (синтаксис BSD)
ps -ef              # все процессы (синтаксис UNIX)
ps -eo pid,ppid,cmd,%cpu,%mem  # настраиваемый вывод
ps -u username      # процессы конкретного пользователя
```

**Состояния процессов (`STAT`):**

| Код | Значение |
|---|---|
| `R` | Running — выполняется или в очереди |
| `S` | Sleeping — прерываемое ожидание |
| `D` | **Непрерываемый сон — ожидание ввода/вывода** |
| `Z` | Zombie — завершился, но не собран родительским процессом |
| `T` | Stopped — остановлен сигналом |
| `s` | Лидер сессии |
| `+` | В группе процессов переднего плана |

> **Предупреждение:** Процессы в состоянии **`D`** заблокированы на вводе/выводе. Их количество отражается в столбце **`b`** команды `vmstat`. Много процессов в состоянии D = проблема с диском или сетевой файловой системой.

### pstree

Отображает процессы в виде дерева с корнем в PID (или в `init`, если не указан). Идентичные ветви сворачиваются с числовым префиксом. Дочерние потоки отображаются в `{}`.

```bash
pstree              # дерево от init
pstree -p           # с PID в каждом узле
pstree -u           # с именами пользователей
pstree 3655         # дерево от конкретного PID
```

Например (`pstree 3655`):

```
gnome-terminal---bash---pstree
               |-bash---man---pager
               |-gnome-pty-helpe
               `-3*[{gnome-terminal}]
```

> **Примечание:** `3*[{gnome-terminal}]` означает 3 одинаковых дочерних потока `gnome-terminal`. Используйте `pstree` для отслеживания PPID (родительского процесса) любого процесса.

---

### w

Показывает информацию о вошедших пользователях, их процессах и те же метрики uptime/средней нагрузки, что и `uptime`.

```bash
w           # все пользователи (полный формат)
w -s        # краткий формат
w username  # только конкретный пользователь
```

> **Примечание:** `-s` (краткий формат) убирает столбцы `LOGIN@`, `JCPU` и `PCPU`.

## Измерение памяти и swap

### vmstat

Отчёты о виртуальной памяти, процессах, подкачке страниц, блочном вводе/выводе, прерываниях и CPU.

```bash
vmstat              # статистика с момента последней загрузки
vmstat 2 5          # 5 отчётов с интервалом 2 секунды
vmstat -a           # показать активную/неактивную память
vmstat -s           # сводная таблица статистики
vmstat -d           # статистика дисков
```

Например (`vmstat 2 2`):

```
procs ---memory-- ---swap-- -----io---- -system-- ----cpu----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa
 0  0      0 109112  33824 242204    0    0   603    26  196  516  7 11 81  1
 0  0      0 109152  33824 242204    0    0     0     2  124  239  0  1 98  0
```

> **Предупреждение:** **Первая строка** показывает средние значения с момента загрузки — **игнорируйте её** при анализе текущей нагрузки.

**Справочник по столбцам:**

| Раздел | Столбец | Значение |
|---|---|---|
| **procs** | `r` | Процессы в очереди выполнения — ожидают CPU |
| **procs** | `b` | Процессы в очереди блокировки — **непрерываемый сон** (ожидание ввода/вывода) |
| **memory** | `swpd` | Использованная виртуальная (swap) память, КБ |
| **memory** | `free` | Свободное ОЗУ (не кэш, не буферы), КБ |
| **memory** | `buff` | Буферы — содержат **необработанные блоки диска** |
| **memory** | `cache` | Кэш — содержит **файлы** |
| **swap** | `si` | Swap in — страниц/сек **с диска в ОЗУ** |
| **swap** | `so` | Swap out — страниц/сек **из ОЗУ на диск** |
| **io** | `bi` | Blocks in — блоков/сек **от** блочного устройства |
| **io** | `bo` | Blocks out — блоков/сек **к** блочному устройству |
| **system** | `in` | Прерываний/сек (включая таймер) |
| **system** | `cs` | Переключений контекста/сек |
| **cpu** | `us` | % пространства пользователя |
| **cpu** | `sy` | % пространства ядра |
| **cpu** | `id` | % простоя |
| **cpu** | `wa` | % ожидания ввода/вывода |
| **cpu** | `st` | % времени кражи (виртуализация) |

> **Важно:** `vmstat` **не может** показывать статистику CPU по ядрам. Для этого используйте `mpstat` или `ps`.

**Swap:** ОЗУ делится на страницы по 4 КБ. Когда памяти не хватает, ядро выгружает страницы неактивных процессов на выделенный раздел. Интенсивный свопинг (высокие `si`/`so`) резко снижает производительность.

---

### free

Показывает общий объём физической памяти и swap, свободное пространство, использованную память и буферы ядра. Столбец `shared` (устаревший) теперь показывает память `tmpfs` (shmem).

```bash
free -h             # в понятном для человека виде (К/М/Г)
free -m             # в мегабайтах
free -g             # в гигабайтах
free -s 2           # обновлять каждые 2 секунды
```

Например (`free -h`):

```
             total       used       free     shared    buffers     cached
Mem:          502M       489M        13M        50B        44M       290M
-/+ buffers/cache:       154M       347M
Swap:         382M       3.9M       379M
```

> **Примечание:** Строка `-/+ buffers/cache` показывает реальное использование памяти без буферов и кэша. В этом примере реально используется 154 МБ, а 347 МБ доступны приложениям (кэш можно освободить при необходимости).

## Измерение дискового ввода/вывода

### iostat

Мониторинг нагрузки на устройства ввода/вывода. Без параметров показывает статистику **с момента последней перезагрузки**. С `interval` и `count` добавляет периодические выборки.

```bash
iostat              # статистика с момента загрузки
iostat 1 3          # 3 выборки с интервалом 1 секунда
iostat -c           # только статистика CPU
iostat -d           # только устройства
iostat -x           # расширенная статистика устройств
iostat -y 1 5       # подавить первую строку (среднее с момента загрузки)
```

Например — базовый вывод:

```
$ iostat
avg-cpu:  %user   %nice %system %iowait  %steal   %idle
           1.25    0.32    3.76    0.20    0.00   94.46

Device:            tps    kB_read/s    kB_wrtn/s    kB_read    kB_wrtn
sda              12.21       214.81        17.38     333479      26980
```

| Поле | Значение |
|---|---|
| `tps` | Транзакций в секунду на устройство |
| `kB_read/s` | КБ/с читается |
| `kB_wrtn/s` | КБ/с записывается |
| `%iowait` | % CPU в ожидании ввода/вывода — **высокое значение = узкое место** |

> **Совет:** `watch` превращает любую статичную утилиту в динамическую:
> ```bash
> watch -n 5 iostat    # обновлять каждые 5 секунд
> ```

---

### iotop

Аналог `top` для дискового ввода/вывода — показывает нагрузку по процессам в реальном времени. Отображает пропускную способность ввода/вывода на процесс/поток, приоритет ввода/вывода и процент времени, затраченного на swap-in и ожидание ввода/вывода.

```bash
iotop               # интерактивный режим
iotop -b            # пакетный режим (для скриптов/pipe)
iotop -o            # только процессы с активным вводом/выводом
iotop -p PID        # конкретный процесс
```

Например (`iotop -b | head`):

```
Total DISK READ :       0.00 B/s | Total DISK WRITE :     213.38 M/s
  TID  PRIO  USER     DISK READ  DISK WRITE  SWAPIN      IO    COMMAND
    6 be/4 root        0.00 B/s    0.00 B/s  0.00 % 99.99 % [kworker/u8:0]
 2976 be/4 root        0.00 B/s  213.38 M/s  0.00 %  0.00 % dd if=/dev/zero of=/tmp/foo
```

| Столбец | Значение |
|---|---|
| `PRIO` | Приоритет ввода/вывода (класс/уровень), например `be/4` = best effort |
| `DISK READ` / `DISK WRITE` | Пропускная способность чтения/записи для процесса |
| `SWAPIN` | % времени в swap-in |
| `IO` | % времени в ожидании ввода/вывода |

---

### lsof

Выводит информацию об открытых файлах и процессах, их использующих. Обрабатывает обычные файлы, каталоги, блочные/символьные специальные файлы, библиотеки, потоки и сетевые файлы.

```bash
lsof                    # все открытые файлы всех процессов
lsof -p PID             # файлы конкретного процесса
lsof /path/to/file      # какой процесс использует этот файл
lsof +d /var/log        # открытые файлы в каталоге (один уровень)
lsof +D /var/log        # рекурсивно
lsof -i                 # все сетевые соединения
lsof -i :80             # процессы на порту 80
lsof -i TCP             # только TCP-соединения
lsof -u username        # файлы пользователя
lsof -F                 # машиночитаемый вывод
```

Например — кто открыл файл:

```
$ sudo lsof /var/run/utmp
COMMAND    PID USER   FD   TYPE DEVICE SIZE/OFF NODE NAME
gdm-simpl 4040 root   10u   REG   0,14     5376  636 /run/utmp
```

| Столбец | Значение |
|---|---|
| `FD` | Файловый дескриптор — `r`=чтение, `w`=запись, `u`=чтение+запись |
| `TYPE` | Тип файла: `REG`=файл, `DIR`=каталог, `IPv4`/`IPv6`=сеть |
| `DEVICE` | Номера устройства (major, minor) |
| `NODE` | Номер inode |

> **Совет:** `lsof -i` быстро показывает все активные сетевые службы. `lsof | grep deleted` находит файлы, которые были удалены, но ещё открыты процессами (частая причина исчерпания дискового пространства).

## Измерение сетевого ввода/вывода

### netstat

> **Предупреждение:** `netstat` считается **устаревшим**. Вместо него используйте `ss` и `ip`.

Показывает сетевые соединения, таблицы маршрутизации, статистику интерфейсов, маскарадные соединения и членство в группах многоадресной рассылки.

```bash
netstat             # все активные сокеты
netstat -r          # таблица маршрутизации (= route -e)
netstat -i          # статистика интерфейсов
netstat -s          # статистика по протоколам
netstat -tulnp      # прослушивающие TCP/UDP с PID (без разрешения имён)
netstat -an         # все соединения без разрешения имён
```

---

### ss

Современная замена `netstat` — читает данные напрямую из пространства ядра и предоставляет больше информации о состояниях TCP.

```bash
ss                  # все соединения
ss -t               # только TCP
ss -u               # только UDP
ss -l               # только прослушивающие сокеты
ss -n               # без разрешения имён
ss -p               # показать процессы
ss -a               # все (включая прослушивающие)
ss -tulnp           # аналог netstat -tulnp
ss -s               # сводная статистика
ss -o state ESTABLISHED  # только установленные соединения
```

Например (`ss -t -a`):

```
State      Recv-Q Send-Q    Local Address:Port    Peer Address:Port
LISTEN     0      5         192.168.122.1:domain             *:*
LISTEN     0      128                   *:ssh                *:*
LISTEN     0      100           127.0.0.1:smtp               *:*
```

> **Важно:** `ss` быстрее и информативнее `netstat`, поскольку читает данные напрямую из пространства ядра. Отдавайте предпочтение ему при диагностике сетевой производительности.

---

### iptraf / iptraf-ng

Интерактивный сетевой монитор с меню. Захватывает пакеты и показывает текущий трафик.

```bash
iptraf-ng           # запустить (требует root)
```

Собирает: счётчики TCP-пакетов и байт на соединение, статистику интерфейсов, разбивку трафика TCP/UDP, счётчики пакетов на LAN-станцию, информацию о флагах TCP / ICMP / OSPF.

> **Примечание:** `iptraf` — **динамический** инструмент с меню, особенно полезный для мониторинга нагрузки на IP-сеть в реальном времени.

## Корреляция процессов и ресурсов

```bash
# Найти процессы, заблокированные на вводе/выводе
ps aux | awk '$8 == "D"'
ps -eo pid,stat,cmd | grep " D"

# Дерево процессов с PID
pstree -p

# Файлы, открытые процессом
lsof -p PID

# Карта памяти процесса
pmap PID
```

> **Важно:** Процессы в состоянии **`D`** (непрерываемый сон) заблокированы на вводе/выводе. Их количество отображается в столбце **`b`** команды `vmstat`.

**Цепочка зависимостей ресурсов:**

```
Нехватка ОЗУ
    ↓
Swap увеличивается (si/so в vmstat)
    ↓
Дисковый ввод/вывод увеличивается (bi/bo в vmstat, iostat показывает высокий %iowait)
    ↓
Процессы блокируются в состоянии D (столбец b в vmstat)
    ↓
Искусственно низкая нагрузка CPU (CPU простаивает в ожидании ввода/вывода)
```

> **Предупреждение:** Нехватка ОЗУ **искусственно занижает** нагрузку CPU. После добавления ОЗУ нагрузка CPU возрастёт — это нормально.

## Корреляция симптомов и проблем

> **Важно:** Все проблемы с ресурсами имеют общую черту — один или несколько ресурсов не справляются с нагрузкой при определённых условиях. Ресурсы взаимосвязаны: CPU, физическая/виртуальная память, хранилище, сетевые интерфейсы, ввод/вывод между компонентами.

| Симптом | Вероятная причина | Инструменты диагностики |
|---|---|---|
| Высокий `%iowait` в CPU | Перегрузка диска / медленный ввод/вывод | `iostat -x`, `iotop`, `lsof` |
| Высокие `si`/`so` в vmstat | Нехватка ОЗУ, активный swap | `free`, `vmstat`, `sar -r` |
| Много процессов в `b` (vmstat) | Блокировка ввода/вывода, проблемы с диском | `ps -eo pid,stat,cmd \| grep " D"`, `iotop` |
| Высокая средняя нагрузка, низкий %CPU | Процессы ожидают ввода/вывода (состояние D) | `vmstat`, `iostat`, `ps` |
| Ресурс на 100% | Явное узкое место | `top`, `sar`, `iostat` |
| Много TIME_WAIT в netstat/ss | Высокий трафик, исчерпание сокетов | `ss -s`, `netstat -s` |
| Высокий `st` (кража) | Перегрузка хост-системы (ВМ) | `top`, `vmstat` |
| Неожиданное исчерпание дискового пространства | Открытые удалённые файлы | `lsof \| grep deleted` |

### Алгоритм диагностики

1. `uptime` → проверить среднюю нагрузку
2. `top` / `vmstat` → CPU, память, swap, заблокированные процессы
3. `iostat` / `iotop` → проблема с диском?
4. `free` / `sar -r` → нехватка памяти?
5. `ss` / `netstat` → сетевые соединения
6. `ps aux` → конкретные проблемные процессы
7. `lsof -p PID` → что делает процесс

### Наиболее часто проверяемые команды

```bash
# Статистика CPU sar: 20 выборок с интервалом 2 секунды
sar 2 20

# Процессы, заблокированные на вводе/выводе
ps -eo pid,stat,cmd | grep " D"
# ИЛИ: столбец b в vmstat

# Что прослушивает порт 80
ss -tulnp | grep :80
lsof -i :80

# Главные потребители ввода/вывода
iotop -o

# Расширенная статистика диска
iostat -x 1 5

# Статистика swap
vmstat -s | grep -i swap
free -h

# Только прослушивающие сокеты
ss -lntp
netstat -tulnp

# Таблица маршрутизации
netstat -r
ip route show
```

### Ключевые факты для экзамена

- Столбец `b` в `vmstat` = процессы в **непрерываемом сне** (ожидание ввода/вывода)
- Процессы в состоянии D в `ps` = заблокированы на вводе/выводе
- `si`/`so` в vmstat = swap in/out (высокое = нехватка ОЗУ)
- `bi`/`bo` в vmstat = блоки in/out (ввод/вывод блочного устройства)
- `wa` в CPU = % ожидания ввода/вывода (высокое = проблемы с диском)
- `ss` заменяет `netstat` (читает напрямую из пространства ядра)
- Данные `sar` хранятся в `/var/log/sa/`
- Пакет `sysstat` включает: `iostat`, `sar`, `mpstat`, `sadf`
- Пакет `procps` включает: `free`, `uptime`, `vmstat`, `w`
- `netstat` **устарел** (замена — `ss` и `ip`)

## Практические вопросы

**В1.** В выводе `vmstat` столбец `b` показывает 8. Что это означает?

**Ответ:** 8 процессов находятся в непрерываемом сне, заблокированные в ожидании ввода/вывода (очередь блокировки). Это может указывать на проблему с производительностью диска.

---

**В2.** Какая команда позволяет видеть в реальном времени, какие процессы создают наибольшую нагрузку на диск?

**Ответ:** `iotop` (или `iotop -o` только для активных процессов).

---

**В3.** Администратор видит высокий `%iowait` в `top`. Какие утилиты помогут определить проблемный процесс?

**Ответ:** `iotop` (показывает нагрузку по процессам), `lsof` (показывает открытые файлы), `ps aux | grep " D"` (процессы в состоянии D).

---

**В4.** Столбцы `si` и `so` в `vmstat` стабильно показывают высокие значения. Что это означает и какова вероятная причина?

**Ответ:** Интенсивный swap in/out — система активно перемещает страницы памяти между ОЗУ и swap. Вероятная причина — недостаточный объём физической ОЗУ. Следствие — замедление системы из-за медленного дискового ввода/вывода вместо памяти.

---

**В5.** Чем `ss` отличается от `netstat` и почему предпочтительнее?

**Ответ:** `ss` читает данные напрямую из пространства ядра, работает быстрее и предоставляет больше информации о состояниях TCP и сокетах. `netstat` считается устаревшим.

---

**В6.** Какой пакет нужно установить на Debian для использования `iostat` и `sar`?

**Ответ:** `sysstat` (`apt install sysstat`).

---

**В7.** Администратор замечает высокую среднюю нагрузку при низкой загрузке CPU (высокий `%idle`). Как объяснить это?

**Ответ:** Процессы ожидают ввода/вывода (непрерываемый сон, состояние D). Средняя нагрузка учитывает не только процессы на CPU, но и заблокированные на вводе/выводе. Проверьте `vmstat` (столбец `b`), `iostat` (высокий `%iowait`), `iotop`.

---

**В8.** Какая команда `sar` показывает активность свопинга за текущий день, 10 выборок с интервалом 1 секунда?

**Ответ:** `sar -w 1 10` (активность свопинга) или `sar -r 1 10` (память и swap).
