> **Тема экзамена 204.2** — Настройка доступа к устройствам хранения. Охватывает hdparm/sdparm, tune2fs, sysctl, SSD/NVMe, настройку iSCSI, идентификаторы хранилищ (WWID, WWN, LUN, IQN) и протоколы SAN.

## Типы дисков и интерфейсы

| Тип диска | Файл устройства | Протокол/интерфейс |
|---|---|---|
| IDE/PATA | `/dev/hd*` | ATAPI |
| SATA / SCSI / SAS | `/dev/sd*` | AHCI / libata |
| NVMe SSD | `/dev/nvme*` | NVMe (PCIe) |
| iSCSI | `/dev/sd*` | iSCSI over IP |

> **Примечание:** Старые диски PATA именовались `/dev/hda`, `/dev/hdb` и т.д. Современные диски SATA и SCSI используют общий префикс `/dev/sd*`, поскольку ядро обрабатывает их через единый драйвер libata.

## hdparm

`hdparm` читает, тестирует и изменяет параметры дисков SATA/IDE. Работает через интерфейс libata ядра и поддерживает диски PATA, ATAPI, SATA и SCSI с поддержкой SAT (SCSI-ATA Command Translation).

> **Внимание:** Большинство опций `hdparm` опасны и могут повредить диск. Тестируйте только на непроизводственных дисках и читайте `man hdparm` перед использованием.

### Синтаксис

```bash
hdparm [options] [device]
```

### Основные команды

| Опция | Описание |
|---|---|
| `-I` | Подробная информация о диске (DMA, кэш, модель) |
| `-i` | Краткая идентификационная информация |
| `-g` | Геометрия диска (цилиндры, головки, секторы) |
| `-d [0\|1]` | Получить или установить флаг DMA |
| `-a` | Упреждающее чтение: количество секторов для предвыборки |
| `-r [0\|1]` | Флаг только для чтения (1 = только чтение) |
| `-t` | Тест скорости чтения диска (буферизованное чтение) |
| `-T` | Тест скорости чтения кэша |
| `-W [0\|1]` | Показать или установить кэширование записи |
| `-X` | Установить режим DMA (опасно, обычно уже настроен) |
| `-v` | Показать все настройки, кроме `-i` |

### Пример: тест производительности

```bash
sudo hdparm -tT /dev/sde
# /dev/sde:
#  Timing cached reads:   11810 MB in  2.00 seconds = 5916.78 MB/sec
#  Timing buffered disk reads:  1024 MB in  2.69 seconds =  380.94 MB/sec
```

> **Совет:** Запускайте тест несколько раз и усредняйте результаты — единственное измерение ненадёжно.

### Сохранение настроек

Изменения `hdparm` теряются при перезагрузке. Для их постоянного сохранения создайте или отредактируйте:

```
/etc/udev/rules.d/50-hdparm.rules
```

## sdparm

`sdparm` взаимодействует с устройствами SCSI через таблицы VPD (Vital Product Data) и страницы режимов. Это не «hdparm для SCSI» — это отдельный инструмент с иным подходом.

`sdparm` может читать серийные номера, коды деталей и описания устройств. Он также может управлять поведением диска: останавливать шпиндель, изменять параметры кэширования с обратной записью.

### Основные команды

| Опция | Описание |
|---|---|
| (без опций) | Показать все общие параметры режима для устройства |
| `--all` | Показать все распознанные поля для данного типа устройства |
| `--command cmd` | Выполнить команду: `stop`, `start`, `sync`, `load` |
| `--hex` | Показать страницы VPD в шестнадцатеричном виде (когда sdparm не может их декодировать) |
| `--page` | Показать конкретную страницу VPD |
| `--vendor` | Показать информацию о производителе |

> **Важно для экзамена:** Запуск `sdparm` без опций показывает все **общие параметры режима**. Опция `--all` показывает ВСЕ распознанные поля — это не одно и то же.

## tune2fs

`tune2fs` настраивает взаимодействие ядра с файловыми системами ext2 и ext3.

```bash
tune2fs [options] /dev/sdXY
```

### Основные команды

| Опция | Описание |
|---|---|
| `-e continue` | При ошибке ФС: игнорировать, вернуть ошибку приложению |
| `-e remount-ro` | При ошибке ФС: перемонтировать только для чтения |
| `-e panic` | При ошибке ФС: вызвать панику ядра и остановить систему |
| `-m <percent>` | Процент зарезервированных блоков (по умолчанию 5%) |
| `-O [^]mount_option` | Установить или сбросить параметр монтирования (^ = сбросить) |
| `-s [0\|1]` | Включить/выключить разреженный суперблок (для больших файловых систем) |

> **Совет:** На больших дисках стандартные 5% зарезервированных блоков расточительны. На диске 2 ТБ это 100 ГБ неиспользуемого пространства. Установите 1% командой `tune2fs -m 1 /dev/sda1`.

> **Внимание:** После изменения флага разреженного суперблока (`-s`) запустите `e2fsck`, чтобы привести файловую систему в корректное состояние.

## sysctl и /proc

`sysctl` изменяет параметры ядра во время работы. Все настраиваемые параметры находятся в дереве `/proc/sys/`.

### Опции sysctl

| Опция | Описание |
|---|---|
| `-a`, `-A` | Показать все доступные значения |
| `-w` | Изменить значение параметра |
| `-p` | Загрузить настройки из файла (по умолчанию `/etc/sysctl.conf`) |
| `-e` | Игнорировать ошибки о неизвестных ключах |
| `-n` | Скрыть имя ключа, показать только значение |

```bash
sysctl -a
sysctl -w vm.swappiness=10
sysctl -p /etc/sysctl.d/99-sysctl.conf

# Чтение/запись напрямую через /proc
cat /proc/sys/vm/swappiness
echo 10 > /proc/sys/vm/swappiness

# Включить IP-форвардинг
echo 1 > /proc/sys/net/ipv4/ip_forward
```

> **Внимание:** Изменения через `echo` в `/proc/sys/` или через `sysctl` **не сохраняются** после перезагрузки. Для постоянных изменений добавьте строку в `/etc/sysctl.conf` или файл в `/etc/sysctl.d/`.

### Мониторинг прерываний

```bash
cat /proc/interrupts
```

Пример вывода на однопроцессорной системе:

```
           CPU0
  0:   16313284   XT-PIC  timer
  1:     334558   XT-PIC  keyboard
 14:     123824   XT-PIC  ide0
 15:          7   XT-PIC  ide1
```

На многоядерных системах появляются дополнительные столбцы CPU. По умолчанию все прерывания обрабатывает CPU0, что становится узким местом, когда он перегружен IRQ от сетевого хранилища, а другие ядра простаивают.

### Управление smp_affinity

Каждая запись прерывания в `/proc/interrupts` имеет подкаталог в `/proc/irq/`. Файл `smp_affinity` внутри содержит битовую маску, где каждый бит соответствует ядру процессора.

| Значение маски | CPU |
|---|---|
| 1 | CPU0 |
| 2 | CPU1 |
| 4 | CPU2 |
| 8 | CPU3 |
| 32 | CPU5 |

```bash
# Направить прерывание #13 на CPU5
echo 32 > /proc/irq/13/smp_affinity

# Направить прерывание #12 на CPU1
echo 2 > /proc/irq/12/smp_affinity
```

> **Совет:** Правильное распределение прерываний по ядрам может заметно ускорить сильно нагруженные системы с сетевым хранилищем — CPU0 больше не будет узким местом.

## SSD, AHCI и NVMe

### AHCI

AHCI (Advanced Host Controller Interface) — интерфейс для дисков SATA. Спроектирован для механических накопителей: 1 очередь из 32 команд. SSD на базе SATA используют тот же AHCI, создавая узкое место.

> **Важно:** Если SSD не распознаётся, проверьте настройки BIOS/UEFI. Режим SATA должен быть установлен в **AHCI**, а не IDE.

### NVMe

NVMe (Non-Volatile Memory Express) подключается через PCIe и использует протокол NVMHCI. Вместо 1 очереди из 32 команд NVMe поддерживает **65 000 очередей по 65 000 команд каждая**. До 7 раз быстрее SSD на базе AHCI.

Поддержка NVMe добавлена в ядро Linux 3.3.

### Именование устройств NVMe

```
/dev/nvme0n1p1
       ^  ^ ^
       |  | └── p1 = раздел 1
       |  └──── n1 = пространство имён 1
       └─────── nvme0 = первый диск NVMe (нумерация с 0)
```

Диски нумеруются с 0; пространства имён и разделы — с 1. Третий диск, четвёртое пространство имён, второй раздел = `/dev/nvme2n4p2`.

### SSD TRIM

SSD хранит данные в блоках, стирает целые блоки и записывает постранично. Это приводит к усилению записи: изменение одной страницы требует перезаписи целого блока. Команда TRIM сообщает контроллеру SSD, какие блоки свободны, что позволяет проводить внутреннюю дефрагментацию.

```bash
# Автоматический TRIM (через параметр монтирования)
/dev/sda1 / ext4 discard,noatime 0 1

# Ручной TRIM (для всех свободных блоков в ФС)
fstrim /
```

### nvme-cli

Для управления дисками NVMe требуется утилита `nvme` из пакета `nvme-cli`. Стандартные инструменты SMART (`smartctl`) не работают с NVMe напрямую, так как нет интерфейса AHCI или SCSI.

```bash
nvme help
nvme smart-log /dev/nvme0
```

## iSCSI

iSCSI (Internet Small Computer System Interface) — сетевая реализация протокола SCSI поверх IP. Клиент отправляет команды SCSI (CDB, Command Descriptor Blocks) по сети; сервер получает и выполняет их. В отличие от Fibre Channel, iSCSI работает на обычной сетевой инфраструктуре через LAN, WAN или Интернет.

| Термин | Роль |
|---|---|
| initiator | Клиент, импортирующий хранилище |
| target | Сервер, экспортирующий хранилище |
| CDB | Command Descriptor Block — пакет команды SCSI |

### Настройка инициатора (клиента)

Файл конфигурации (RHEL и производные):

```
/etc/iscsi/iscsid.conf
```

Управление службой:

```bash
/etc/init.d/iscsi status
/etc/init.d/iscsi start
/etc/init.d/iscsi restart
```

### Обнаружение целей

```bash
iscsiadm -m discovery -t sendtargets -p 192.168.1.100
```

После обнаружения войдите в том или перезапустите службу для автоматического монтирования всех обнаруженных томов.

### /etc/fstab для iSCSI

```
/dev/sdb1 /mnt/iscsi ext4 _netdev 0 0
```

> **Важно:** Опция `_netdev` является **обязательной** для томов iSCSI в fstab. Она гарантирует, что сеть будет запущена до того, как система попытается смонтировать устройство.

### Пример /etc/iscsi/iscsid.conf

```ini
# Настройки iSNS
isns.address = 192.168.1.200
isns.port = 3260

# Аутентификация CHAP
node.session.auth.authmethod = CHAP
node.session.auth.username = myuser
node.session.auth.password = mypassword

# CHAP для сессий обнаружения
discovery.sendtargets.auth.authmethod = CHAP
discovery.sendtargets.auth.username = discuser
discovery.sendtargets.auth.password = discpassword
```

### Цель iSCSI (сервер)

До ядра 2.6.38 стандартом был STGT/TGT. Начиная с 2.6.38 стандартной целью Linux является **LIO** (linux-iscsi.org), поддерживающая Fibre Channel, FCoE, iSCSI, iSER, SRP, USB и другие транспортные протоколы.

#### Устаревшее: tgtd / tgt

Файл конфигурации цели:

```
/etc/tgt/targets.conf
```

Пример с двумя LUN:

```xml
<target iqn.2008-09.com.example:server.target1>
    backing-store /srv/images/iscsi-share.img
    direct-store /dev/sdd
</target>
```

`backing-store` указывает на файл или блочное устройство; `direct-store` передаёт параметры локального устройства SCSI (серийный номер, производитель) напрямую в LUN iSCSI.

Управление службой:

```bash
service tgtd start
service tgtd stop
service tgtd force-stop   # немедленно разрывает все сессии
```

#### Современное: LIO + targetcli

```bash
systemctl enable target
targetcli
```

Основной рабочий процесс в `targetcli`:

```bash
# 1. Создать хранилище (backstore)
/> cd /backstores/block
/backstores/block> create iscsidisk1 dev=/dev/sde

# 2. Создать IQN для цели
/backstores/block> cd /iscsi
/iscsi> create iqn.2016-02.com.example.server07:iscsidisk1

# 3. Ограничить портал конкретным IP (удалить 0.0.0.0)
/iscsi> cd iqn.2016-02.com.example.server07:iscsidisk1/tpg1/portals
/iscsi/.../portals> delete 0.0.0.0 3260
/iscsi/.../portals> create 192.168.56.103

# 4. Создать LUN
/iscsi/.../portals> cd ../luns
/iscsi/.../luns> create /backstores/block/iscsidisk1

# 5. Выйти (конфигурация сохраняется автоматически)
/> exit
```

> **Совет:** targetcli сохраняет конфигурацию в `/etc/target/saveconfig.json` при выходе. Последние 10 резервных копий конфигурации хранятся в `/etc/target/backup/`.

> **Внимание:** Используйте только строчные буквы в именах хранилищ — targetcli ведёт себя непредсказуемо с заглавными буквами.

После настройки цели подключитесь с инициатора:

```bash
iscsiadm -m discovery -t sendtargets -p 192.168.56.103
iscsiadm -m node -T iqn.2016-02.com.example.server07:iscsidisk1 \
         -p 192.168.56.103 -l
iscsiadm -m session -P3
```

Обнаруженные записи iSCSI хранятся в:

```
/var/lib/iscsi/nodes/
/var/lib/iscsi/send_targets/
```

IQN инициатора задаётся в:

```
/etc/iscsi/initiatorname.iscsi
```

## scsi_id и персистентность LUN

Имена устройств Linux (`/dev/sda`, `/dev/sdb`) меняются между перезагрузками, если порядок обнаружения дисков изменился. Персистентность LUN решает эту проблему: диск всегда адресуется по стабильному имени независимо от порядка подключения.

### WWID через /dev/disk/by-id/

Linux автоматически создаёт символические ссылки в `/dev/disk/by-id/`, где имя ссылки содержит WWID устройства:

```bash
ls -l /dev/disk/by-id/
# scsi-3600508b400105e210000900000490000 -> ../../sda   (страница 0x83)
# scsi-SSEAGATE_ST373453LW_3HW1RHM6     -> ../../sda   (страница 0x80)
```

Существует два типа WWID: страница 0x83 и страница 0x80. Любое устройство SCSI имеет один из них.

### scsi_id

Утилита `scsi_id` получает WWID непосредственно с устройства через команду запроса.

По умолчанию `scsi_id` не показывает устройства, отсутствующие в его белом списке. Чтобы получить WWID любого устройства, используйте флаг `-g`.

```bash
# Получить WWID (современный синтаксис)
/sbin/scsi_id -g -u -d /dev/sdc
# 3200049454505080f

# Старый синтаксис sysfs (RHEL5 и старше)
scsi_id -g -u -s /block/sdc
```

Чтобы не указывать `-g` каждый раз, отредактируйте `/etc/scsi_id.conf`: удалите строку `options=-b` и добавьте `options=-g`.

### Персистентность LUN через udev

Получив WWID, создайте правило в `/etc/udev/rules.d/20-names.rules`:

```
KERNEL="sd*", BUS="scsi", PROGRAM="/sbin/scsi_id -g -u -d /dev/$parent", RESULT="3200049454505080f", NAME="bookone"
```

Теперь при каждом подключении этого диска udev сравнивает результат `scsi_id` с указанным RESULT и создаёт устройство с именем `bookone`, независимо от того, какое имя `/dev/sd*` назначит ядро.

> **Примечание:** На RHEL6 и новее используйте `-d /dev/$parent` вместо `-s /block/$parent`, так как новые версии `scsi_id` не принимают пути sysfs.

### Multipath

Если у устройства несколько путей (разные сетевые карты или HBA), device-mapper-multipath использует WWID для объединения их в одно псевдоустройство:

```
/dev/mapper/3600508b400105df70000e00000ac0000
```

При включённом `user_friendly_names` устройство получает имя вида `/dev/mapper/mpath0`, а соответствие WWID хранится в:

```
/etc/multipath/bindings
```

Показать все пути к устройству:

```bash
multipath -l
```

#### Постоянный псевдоним в /etc/multipath.conf

Назначьте понятное имя через псевдоним. Это работает даже с одним путём — просто добавьте WWID и псевдоним:

```
multipaths {
    multipath {
        wwid    3600a0b80001327510000015427b625e
        alias   backupdisk
    }
}
```

Устройство всегда будет доступно как `/dev/mpath/backupdisk`.

## WWID, WWN, LUN, IQN

| Идентификатор | Расшифровка | Назначение |
|---|---|---|
| WWID | World Wide Identifier | Уникальный идентификатор устройства SCSI (синоним WWN) |
| WWN | World Wide Name | Уникальный идентификатор устройства SCSI (синоним WWID) |
| LUN | Logical Unit Number | Номер логического устройства на цели |
| IQN | iSCSI Qualified Name | Уникальный адрес, идентифицирующий как целевой сервер iSCSI, так и диск |
| scsi_id | Утилита | Получает уникальный идентификатор устройства SCSI |

> **Важно:** IQN идентифицирует как целевой сервер iSCSI, так и предлагаемый им диск. WWID и WWN однозначно идентифицируют только диск, но не сервер.

Пример IQN:

```
iqn.2003-01.org.linux-iscsi.server1.x8664:sn.abc123
```

## SAN

SAN (Storage Area Network) — выделенная высокоскоростная сеть для доступа к дисковому хранилищу. Диски в SAN выглядят для сервера как локально подключённые.

| Протокол | Описание |
|---|---|
| Fibre Channel (FC) | Требует специального оборудования; высокая скорость |
| FCoE | Fibre Channel over Ethernet — FC поверх стандартного Ethernet |
| AoE | ATA over Ethernet — команды ATA напрямую через Ethernet |
| iSCSI | SCSI поверх IP; работает на обычной инфраструктуре |

> **Примечание:** AoE и FCoE работают только в локальных сетях, поскольку не поддерживают маршрутизацию. iSCSI может работать через WAN и Интернет.

### Файлы и пути

| Файл / путь | Назначение |
|---|---|

| `/etc/iscsi/initiatorname.iscsi` | IQN инициатора |
| `/etc/tgt/targets.conf` | Конфигурация цели (устаревший tgtd/TGT) |
| `/etc/target/saveconfig.json` | Конфигурация цели (современный LIO/targetcli) |
| `/var/lib/iscsi/nodes/` | Записи обнаруженных узлов iSCSI |
| `/var/lib/iscsi/send_targets/` | Записи сессий обнаружения |
| `/dev/disk/by-id/` | Символические ссылки WWID → /dev/sd* |
| `/etc/scsi_id.conf` | Конфигурация scsi_id (белый список / флаг -g по умолчанию) |
| `/etc/udev/rules.d/20-names.rules` | Правила udev для персистентности LUN |
| `/etc/multipath/bindings` | Соответствие WWID → имя mpath |
| `/etc/udev/rules.d/50-hdparm.rules` | Постоянные настройки hdparm через udev |
| `/proc/interrupts` | Таблица прерываний (текущее распределение) |
| `/proc/irq/<N>/smp_affinity` | Битовая маска балансировки IRQ |
| `/etc/sysctl.conf` | Постоянные параметры ядра |
| `/etc/multipath.conf` | Конфигурация multipath, псевдонимы |

### Основные команды

```bash
# hdparm
hdparm -I /dev/sda             # подробная информация о диске
hdparm -tT /dev/sda            # тест скорости (диск + кэш)
hdparm -d 1 /dev/sda           # включить DMA
hdparm -W 1 /dev/sda           # включить кэширование записи

# sdparm
sdparm /dev/sda                # общие параметры режима
sdparm --all /dev/sda          # все распознанные поля
sdparm --command stop /dev/sda # остановить шпиндель

# tune2fs
tune2fs -m 1 /dev/sda1         # зарезервировать 1% блоков
tune2fs -e remount-ro /dev/sda1  # поведение при ошибке ФС

# sysctl
sysctl -a                      # все параметры ядра
sysctl -w vm.swappiness=10     # установить параметр
sysctl -p /etc/sysctl.conf     # загрузить из файла
echo 32 > /proc/irq/13/smp_affinity  # направить IRQ13 на CPU5

# iSCSI
iscsiadm -m discovery -t sendtargets -p 192.168.1.100
iscsiadm -m node -T iqn... -p 192.168.1.100 -l
iscsiadm -m session -P3
/etc/init.d/iscsi restart

# targetcli (цель LIO)
targetcli
systemctl enable target

# tgtd (устаревшая цель TGT)
service tgtd start
service tgtd force-stop

# scsi_id
/sbin/scsi_id -g -u -d /dev/sdc

# multipath
multipath -l

# NVMe
nvme help
nvme smart-log /dev/nvme0

# TRIM
fstrim /
```

### Частые ловушки на экзамене

- `sdparm` без опций показывает **общие параметры режима**. `--all` показывает все поля — это разный вывод.
- Изменения `hdparm` являются **временными**. Без `/etc/udev/rules.d/50-hdparm.rules` они исчезнут при перезагрузке.
- Записи iSCSI в `/etc/fstab` **обязательно** должны содержать опцию `_netdev`. Без неё система зависнет при загрузке, пытаясь смонтировать том до поднятия сети.
- WWID и WWN идентифицируют диск; IQN идентифицирует как целевой сервер, так и диск. Разные ответы на экзамене.
- Именование NVMe: диск с 0, пространство имён и раздел с 1. Второй диск, первое пространство имён, первый раздел = `/dev/nvme1n1p1`.

## Практические вопросы

**В1.** Какая команда показывает скорость чтения **диска** (не кэша) для `/dev/sdb`?

**Ответ:** `hdparm -t /dev/sdb` — строчная `-t` тестирует буферизованное чтение диска. Заглавная `-T` тестирует чтение кэша.

---

**В2.** Администратор запускает `sdparm /dev/sdc` без параметров. Что будет отображено?

**Ответ:** Все **общие параметры режима** устройства. `--all` показало бы все распознанные поля (другой вывод).

---

**В3.** Какой файл нужно создать, чтобы настройки `hdparm` сохранялись после перезагрузки?

**Ответ:** `/etc/udev/rules.d/50-hdparm.rules` — настройки hdparm применяются при загрузке через правила udev.

---

**В4.** Что означает `/dev/nvme1n3p2`?

**Ответ:** Второй диск NVMe (`nvme1`, нумерация с 0), третье пространство имён (`n3`, с 1), второй раздел (`p2`, с 1).

---

**В5.** Какой идентификатор одновременно указывает как на целевой сервер iSCSI, так и на предлагаемый им диск?

**Ответ:** **IQN** (iSCSI Qualified Name). WWID и WWN однозначно идентифицируют только диск.

---

**В6.** Администратор добавил диск iSCSI в `/etc/fstab`. Система зависает при загрузке до монтирования диска. В чём вероятная причина?

**Ответ:** Отсутствует опция `_netdev`. Без неё система пытается смонтировать устройство до поднятия сети. Добавление `_netdev` откладывает монтирование до завершения инициализации сети.

---

**В7.** Какая утилита управляет дисками NVMe и получает с них журналы SMART?

**Ответ:** `nvme` (из пакета `nvme-cli`). `smartctl` не работает с NVMe, поскольку нет интерфейса AHCI или SCSI.

---

**В8.** Какой процент блоков резервирует `tune2fs` по умолчанию, и как изменить его на 1%?

**Ответ:** **5%** по умолчанию. Изменить командой `tune2fs -m 1 /dev/sda1`.
