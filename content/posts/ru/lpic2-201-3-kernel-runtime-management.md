## Управление модулями ядра

Модули ядра позволяют загружать и выгружать драйверы во время работы системы без перезагрузки. Файлы модулей хранятся в `/lib/modules/<kernel-version>/` с расширением `.ko`.

### lsmod — список загруженных модулей

`lsmod` отображает все загруженные в данный момент модули. Читает данные непосредственно из `/proc/modules`.

```bash
lsmod
```

Вывод содержит три столбца: имя модуля, размер в байтах и счётчик использования. Четвёртый столбец, если присутствует, показывает, какие другие модули зависят от данного.

```
Module                  Size  Used by
snd_ens1371            28672  0
snd_ac97_codec        131072  1 snd_ens1371
snd                    98304  3 snd_ens1371,snd_ac97_codec
```

> **Важно:** Если счётчик использования (Used by) больше нуля, модуль не может быть выгружен. Сначала выгрузите все модули, которые на него ссылаются.

### insmod — ручная загрузка модуля

`insmod` загружает модуль по полному пути к файлу `.ko`. Не разрешает зависимости автоматически — это его главный недостаток по сравнению с `modprobe`.

```bash
# Успешная загрузка
insmod /lib/modules/$(uname -r)/kernel/drivers/scsi/sym53c8xx.ko

# Завершается ошибкой из-за неразрешённых зависимостей
insmod snd-ens1371.ko
# insmod: error inserting 'snd-ens1371.ko': -1 Unknown symbol in module
```

> **Внимание:** `insmod` требует полного пути к файлу `.ko`. Если зависимости не загружены, вы получите ошибку «Unknown symbol in module». В таких случаях используйте `modprobe`.

| Опция | Описание |
|---|---|
| `-s` | Записывать результат в syslog вместо терминала |
| `-v` | Подробный вывод |

### rmmod — выгрузка модуля

`rmmod` выгружает модуль из работающего ядра. Требуются права root, и счётчик использования должен быть равен нулю.

```bash
# Попытка выгрузить модуль, который используется
rmmod snd_ac97_codec
# ERROR: Module snd_ac97_codec is in use by snd_ens1371

# Правильный порядок: сначала выгрузить зависящий модуль
rmmod snd_ens1371
rmmod snd_ac97_codec
```

> **Подсказка:** `modprobe -r` удобнее, чем `rmmod` — он автоматически определяет правильный порядок выгрузки с учётом зависимостей.

### modprobe — умная загрузка и выгрузка

`modprobe` — рекомендуемый инструмент для загрузки модулей. В отличие от `insmod`, он автоматически загружает все зависимости, читая `/lib/modules/<version>/modules.dep`. Принимает имя модуля без пути и расширения.

```bash
# Загрузить модуль и все его зависимости
modprobe snd_ens1371

# Загрузить с параметрами
modprobe usbcore nousb=1

# Выгрузить модуль и его зависимости
modprobe -r snd_ens1371

# Показать полную текущую конфигурацию
modprobe -c

# Подробный режим: показывает, что происходит
modprobe -v btusb
# insmod /lib/modules/3.13.0-63-generic/kernel/drivers/bluetooth/btusb.ko
```

| Опция | Описание |
|---|---|
| `-r` | Удалить модуль (как `rmmod`) |
| `-a` | Загрузить все перечисленные модули |
| `-c` | Показать полную конфигурацию включая значения по умолчанию |
| `-l` | Список модулей |
| `-n` | Пробный запуск — проверить без фактической загрузки |
| `-v` | Подробный вывод |
| `-s` | Записывать ошибки в syslog |
| `-f` | Принудительная загрузка, игнорировать конфликты версий |

### modinfo — информация о модуле

`modinfo` выводит подробную информацию о модуле: описание, автора, версию, параметры и зависимости.

```bash
modinfo snd_ens1371
# filename:  /lib/modules/.../snd-ens1371.ko
# author:    Thomas Sailer
# description: Ensoniq AudioPCI97 ES1371+
# license:  GPL
# parm:      joystick_port:int

# Показать только параметры модуля
modinfo -p snd_ens1371

# Показать только имя файла
modinfo -n snd_ens1371
```

> **Подсказка к экзамену:** Используйте `modinfo -p`, когда спросят «как узнать, какие параметры принимает модуль».

| Опция | Описание |
|---|---|
| `-a` | Показать автора |
| `-d` | Показать описание |
| `-n` | Показать имя файла |
| `-p` | Показать принимаемые параметры |
| `-V` | Показать версию утилиты |

### depmod — пересчёт зависимостей

`depmod` сканирует все модули в `/lib/modules/<version>/` и пересобирает файл зависимостей `modules.dep`. Большинство дистрибутивов запускают `depmod -a` автоматически при загрузке системы.

```bash
# Пересчитать для работающего ядра
depmod -a

# Пересчитать для конкретной версии
depmod -a 4.15.0-20-generic

# Показать зависимости без записи в файл
depmod -n
```

> **Внимание:** Если вы установили новый модуль вручную и не запустили `depmod -a`, `modprobe` не найдёт его или не сможет разрешить его зависимости.

## Конфигурация модулей

### Конфигурационные файлы в /etc

| Файл/Каталог | Назначение |
|---|---|
| `/etc/modprobe.conf` | Основной файл конфигурации (устаревший) |
| `/etc/modprobe.d/` | Каталог с конфигурационными файлами (современный подход) |
| `/etc/modules` | Список модулей для автозагрузки при старте системы |
| `/etc/modules.conf` | Старый формат, устаревший |

Файлы в `/etc/modprobe.d/` используют расширение `.conf`. Строки, начинающиеся с `#`, и пустые строки игнорируются.

Пример `/etc/modprobe.d/blacklist.conf`:

```
# Запретить автоматическую загрузку модуля
blacklist pcspkr
```

Пример `/etc/modprobe.d/sound.conf`:

```
# Задать параметры модуля при загрузке
options snd_ens1371 joystick_port=1

# Команды, выполняемые при загрузке/выгрузке
install mymodule /sbin/modprobe realmodule
pre-install mymodule /bin/echo "Loading..."
post-remove mymodule /bin/echo "Removed."
```

### Псевдонимы (алиасы)

Псевдоним позволяет загружать модуль под другим именем. Полезно, когда программа ищет драйвер по стандартному имени, а фактический файл называется иначе.

```
# Загрузить модуль "eth0" как "e1000"
alias eth0 e1000

# Псевдоним для всех устройств определённого типа
alias net-pf-10 off
```

```bash
# После добавления псевдонима в конфигурацию
modprobe eth0
# modprobe загружает e1000
```

> **Подсказка к экзамену:** «Как настроить загрузку модуля под именем, отличным от имени файла» — ответ: директива `alias` в `/etc/modprobe.d/*.conf`.

### modules.dep

`/lib/modules/<version>/modules.dep` хранит зависимости между модулями. `modprobe` читает его для определения порядка загрузки.

```bash
cat /lib/modules/$(uname -r)/modules.dep
# kernel/drivers/sound/snd-ens1371.ko: kernel/sound/snd-ac97-codec.ko kernel/sound/snd.ko
```

Файл пересоздаётся командой `depmod -a` при каждой загрузке системы.

## Файловая система /proc

`/proc` — псевдофайловая система: ядро генерирует её содержимое на лету. Никакие данные не хранятся на диске. Через `/proc` можно как читать, так и изменять параметры ядра во время работы системы.

```bash
# Версия ядра
cat /proc/sys/kernel/osrelease

# Имя хоста
cat /proc/sys/kernel/hostname

# Загруженные модули (то же, что lsmod)
cat /proc/modules

# Информация о процессоре
cat /proc/cpuinfo

# Использование памяти
cat /proc/meminfo
```

### /proc/sys/kernel

Каталог `/proc/sys/kernel/` содержит файлы для изменения параметров ядра во время работы. Запись в файл немедленно меняет поведение системы.

```bash
# Прочитать текущее значение
cat /proc/sys/kernel/panic
# 0

# Включить автоперезагрузку через 20 секунд после kernel panic
echo 20 > /proc/sys/kernel/panic

# Проверить максимальное количество открытых файлов
cat /proc/sys/fs/file-max
```

> **Внимание:** Изменения через `/proc/sys/` не сохраняются после перезагрузки. Для постоянных изменений используйте `sysctl` с `/etc/sysctl.conf`.

## sysctl

`sysctl` читает и задаёт параметры ядра. Параметры берутся из иерархии `/proc/sys/`, но записываются через точку: например, `kernel.panic` вместо `/proc/sys/kernel/panic`.

```bash
# Показать все параметры
sysctl -a

# Показать конкретный параметр
sysctl kernel.panic
# kernel.panic = 0

# Изменить параметр
sysctl -w kernel.panic=20

# Применить настройки из файла (используется при запуске системы)
sysctl -p /etc/sysctl.conf

# Применить все файлы из каталога
sysctl -p /etc/sysctl.d/
```

Пример `/etc/sysctl.conf`:

```
# Включить IP-пересылку
net.ipv4.ip_forward = 1

# Автоперезагрузка через 30 секунд после kernel panic
kernel.panic = 30

# Защита от IP-спуфинга
net.ipv4.conf.all.rp_filter = 1
```

Также можно создавать отдельные файлы в `/etc/sysctl.d/`, например `/etc/sysctl.d/99-custom.conf`.

> **Важно:** Модули, загруженные после применения `sysctl`, могут переопределить его настройки. Чтобы избежать этого, запускайте `sysctl -p` после загрузки всех модулей.

## Инструменты анализа оборудования

### uname

`uname` отображает информацию о работающем ядре.

```bash
uname -a
# Linux myhost 5.15.0-76-generic #83-Ubuntu SMP Thu Jun 15 19:16:32 UTC 2023 x86_64 x86_64 x86_64 GNU/Linux
```

| Опция | Описание |
|---|---|
| `-a` | Вся информация |
| `-r` | Выпуск ядра, например `5.15.0-76-generic` |
| `-n` | Сетевое имя хоста |
| `-m` | Архитектура машины |
| `-s` | Имя ядра (всегда `Linux`) |
| `-v` | Версия ядра (дата сборки) |
| `-p` | Тип процессора |
| `-o` | Операционная система |

### dmesg

`dmesg` выводит кольцевой буфер ядра — сообщения о загрузке, обнаружении оборудования и ошибках.

```bash
# Вывести весь буфер
dmesg

# Сохранить в файл
dmesg > /var/log/boot.messages

# Найти информацию о конкретном устройстве
dmesg | grep -i usb

# Найти ошибки
dmesg | grep -i error
```

> **Подсказка:** При диагностике проблем с оборудованием `dmesg` — первое место, куда следует заглянуть. Вывод также сохраняется в `/var/log/dmesg`.

### lspci

`lspci` перечисляет все устройства на PCI-шинах системы. Требует ядро 2.1.82 или новее.

```bash
# Базовый список
lspci

# Подробный вывод (до 3 флагов -v)
lspci -v
lspci -vvv

# Показать числовые идентификаторы вместо имён
lspci -n
```

Пример вывода:

```
00:03.0 Ethernet controller: Intel Corporation 82540EM Gigabit Ethernet Controller
00:05.0 Multimedia audio controller: Intel Corporation 82801AA AC'97 Audio Controller
```

### lsusb

`lsusb` перечисляет USB-устройства — аналог `lspci` для USB. Требует поддержки `/proc/bus/usb` (ядро 2.3.15 или новее).

```bash
# Список USB-устройств
lsusb

# Подробный вывод
lsusb -v

# Древовидное представление топологии USB
lsusb -t
```

Пример вывода:

```
Bus 007 Device 003: ID 03f0:0324 Hewlett-Packard SK-2885 keyboard
Bus 007 Device 002: ID 045e:0040 Microsoft Corp. Wheel Mouse Optical
```

### lsdev

`lsdev` показывает сводную информацию об оборудовании: адреса ввода-вывода, прерывания (IRQ) и каналы DMA. Данные берутся из `/proc/interrupts`, `/proc/ioports` и `/proc/dma`.

```bash
lsdev
```

> **Примечание:** `lsdev` может отсутствовать на некоторых системах. Используйте `procinfo` как альтернативу.

## udev

### Принцип работы udev

`udev` перенёс управление устройствами из пространства ядра в пространство пользователя. Демон `udevd` работает в фоновом режиме и получает сообщения «uevent» от ядра через псевдофайловую систему `sysfs`. Когда вы подключаете USB-флешку или камеру, ядро посылает uevent, `udevd` получает его, сопоставляет с набором правил и выполняет необходимые действия: создаёт файл устройства в `/dev`, загружает соответствующий модуль, монтирует устройство, запускает скрипты.

```
Ядро обнаруживает устройство
       ↓
  Посылает uevent
       ↓
  udevd получает uevent
       ↓
  Сопоставляет с правилами в /etc/udev/rules.d/
       ↓
  Выполняет действия: создаёт /dev/sdX, запускает скрипты и т.д.
```

### Файлы и каталоги udev

| Путь | Описание |
|---|---|
| `/etc/udev/udev.conf` | Основной конфигурационный файл udev (приоритет логирования и т.д.) |
| `/etc/udev/rules.d/` | Пользовательские правила (переопределяют системные) |
| `/lib/udev/rules.d/` | Системные правила по умолчанию |

> **Важно:** Пользовательские правила размещаются в `/etc/udev/rules.d/`; системные правила по умолчанию — в `/lib/udev/rules.d/`. Правила в `/etc/` имеют приоритет.

### Синтаксис правил udev

Каждое правило — это разделённый запятыми список пар key=value, разделённых на ключи совпадения (условия) и ключи назначения (действия).

```
# Назначить имя eth0 сетевой карте с конкретным MAC-адресом
SUBSYSTEM=="net", ACTION=="add", ATTR{address}=="00:21:86:9e:c2:c4", NAME="eth0"
```

Операторы:

| Оператор | Описание |
|---|---|
| `==` | Совпадение: равно |
| `!=` | Совпадение: не равно |
| `=` | Присвоить значение (очищает список) |
| `+=` | Добавить в список |
| `:=` | Присвоить окончательно, запретить дальнейшие изменения |

Основные ключи совпадения:

| Ключ | Описание |
|---|---|
| `SUBSYSTEM` | Подсистема устройства (net, usb, block) |
| `ACTION` | Действие события (add, remove, change) |
| `ATTR{...}` | Атрибут устройства из sysfs |
| `KERNEL` | Имя устройства в ядре |
| `DRIVERS` | Имя драйвера |

Основные ключи назначения:

| Ключ | Описание |
|---|---|
| `NAME` | Имя файла устройства в /dev |
| `SYMLINK` | Символическая ссылка на устройство |
| `MODE` | Права доступа к файлу устройства |
| `OWNER` | Владелец файла устройства |
| `RUN` | Команда для выполнения |

Примеры правил:

```
# Создать символическую ссылку /dev/my-usb для конкретного USB-устройства
SUBSYSTEM=="usb", ATTR{idVendor}=="03f0", ATTR{idProduct}=="0324", SYMLINK+="my-usb"

# Запустить скрипт при подключении USB-накопителя
ACTION=="add", SUBSYSTEM=="block", KERNEL=="sd*", RUN+="/usr/local/bin/usb-mount.sh"
```

> **Внимание:** Следите за конфликтующими правилами: если два правила назначают одно и то же имя устройства разным устройствам, поведение становится непредсказуемым. Используйте `udevadm test` для проверки правила без фактического применения.

### Мониторинг событий udev

```bash
# Современный способ мониторинга событий
udevadm monitor

# Только события ядра
udevadm monitor --kernel

# Только события udev
udevadm monitor --udev

# Проверить применение правил к устройству (без реальных действий)
udevadm test /sys/class/net/eth0

# Перезагрузить правила udev без перезапуска демона
udevadm control --reload-rules

# Получить информацию об устройстве
udevadm info --query=all --name=/dev/sda
```

> **Примечание:** На старых системах `udevmonitor` была отдельной утилитой. В современных дистрибутивах это символическая ссылка на `udevadm`. Если ссылка отсутствует, используйте `udevadm monitor` напрямую.

## Инструменты трассировки

### strace — трассировка системных вызовов

`strace` перехватывает и записывает все системные вызовы, выполняемые процессом, и получаемые им сигналы. Исходный код и перекомпиляция не требуются. Для каждого вызова выводит имя, аргументы и возвращаемое значение в stderr.

```bash
# Трассировать команду
strace cat /dev/null

# Прикрепиться к работающему процессу по PID
strace -p 1234

# Прикрепиться к процессу и всем его дочерним процессам
strace -p 1234 -f

# Сохранить вывод в файл
strace -o /tmp/trace.log cat /dev/null

# Показать только конкретные системные вызовы
strace -e open,read cat /dev/null
```

Пример вывода `strace cat /dev/null`:

```
execve("/bin/cat", ["cat", "/dev/null"], [...]) = 0
open("/dev/null", O_RDONLY)             = 3
read(3, "", 32768)                      = 0
close(3)                                = 0
exit_group(0)                           = ?
```

> **Подсказка:** Флаг `-p` особенно полезен при отладке демонов: подключитесь к зависшему процессу в реальном времени и посмотрите, на каком системном вызове он завис.

### ltrace — трассировка вызовов библиотек

`ltrace` работает как `strace`, но перехватывает вызовы динамических библиотек, а не системные вызовы ядра. Также не требует исходного кода.

```bash
# Трассировать вызовы библиотек
ltrace cat /dev/null

# Прикрепиться к работающему процессу
ltrace -p 1234
```

Пример вывода `ltrace cat /dev/null`:

```
__libc_start_main(...)
setlocale(6, "")        = "en_US.utf8"
open("/dev/null", 0, 02) = 3
read(3, "", 32768)       = 0
exit(0)
```

> **Примечание:** `strace` показывает вызовы ядра (`open`, `read`, `write`); `ltrace` показывает вызовы библиотек (`malloc`, `setlocale`, `printf`). В реальной отладке оба инструмента часто используются совместно.

### strings — чтение строк из бинарных файлов

`strings` выводит все печатаемые строки длиной 4 и более символов из файла. Полезно для изучения бинарных файлов без исходного кода: поиск имён конфигурационных файлов, переменных окружения, путей.

```bash
# Найти строки в бинарном файле
strings /usr/bin/ssh

# Найти конфигурационные файлы, используемые программой
strings /usr/sbin/sshd | grep etc

# Задать минимальную длину строки
strings -n 8 /usr/bin/cat
```

> **Подсказка к экзамену:** `strace` трассирует системные вызовы; `ltrace` трассирует вызовы библиотек; `strings` читает текстовое содержимое бинарных файлов. Все три работают без перекомпиляции программы.

## Процесс загрузки и initrd

`initrd` (начальный RAM-диск) — временная корневая файловая система, которую загрузчик загружает в память вместе с ядром. Он необходим для монтирования настоящего корня, которому могут потребоваться модули (RAID, LVM, шифрование), недоступные до того, как станет доступен `/lib/modules`.

Последовательность загрузки с `initrd`:

1. Загрузчик (GRUB) загружает ядро и образ `initrd` в память
2. Ядро преобразует `initrd` в обычный RAM-диск и освобождает память образа
3. RAM-диск монтируется как корень (чтение-запись)
4. `/linuxrc` запускается с правами root и выполняет начальную настройку
5. После завершения `/linuxrc` монтируется настоящий корень
6. Если существует `/initrd`, RAM-диск перемещается туда; иначе отмонтируется
7. Запускается стандартный процесс инициализации `/sbin/init`

> **Внимание:** Если каталог `/initrd` не существует, RAM-диск не может быть отмонтирован. Любой процесс, использующий файлы из `initrd`, заблокирует это. Поэтому перед переключением на настоящий корень отмонтируйте все файловые системы, включая `/proc`.

Параметры загрузки для `initrd`:

| Параметр | Описание |
|---|---|
| `initrd=<file>` | Загрузить указанный файл как начальный RAM-диск |
| `noinitrd` | Не преобразовывать initrd в RAM-диск; сохранить данные в `/dev/initrd` для отладки |
| `root=/dev/ram` | Использовать RAM-диск как постоянный корень (сжатые файловые системы) |

> **Примечание:** `/dev/initrd` доступен только для чтения и лишь один раз. После того как последний процесс закроет его, память освобождается и устройство становится недоступным.

```bash
# Освободить память RAM-диска после переключения корня
freeramdisk /dev/ram0
```

## Шпаргалка к экзамену

### Ключевые файлы и пути

```
/proc/modules                      # Загруженные модули
/proc/sys/kernel/                  # Редактируемые параметры ядра
/lib/modules/<ver>/modules.dep     # Зависимости модулей
/lib/modules/<ver>/modules.alias   # Псевдонимы модулей
/etc/modprobe.d/                   # Конфигурация modprobe
/etc/modules                       # Автозагружаемые модули
/etc/sysctl.conf                   # Постоянные параметры ядра
/etc/sysctl.d/                     # Дополнительные файлы sysctl
/etc/udev/udev.conf                # Конфигурация udev
/etc/udev/rules.d/                 # Пользовательские правила udev
/lib/udev/rules.d/                 # Системные правила udev
/boot/vmlinuz-<ver>                # Образ ядра
/boot/initrd.img-<ver>             # Начальный RAM-диск
/boot/System.map-<ver>             # Таблица символов ядра
/var/log/dmesg                     # Сохранённый буфер dmesg
```

### Ключевые команды

```bash
uname -r                           # Версия ядра
uname -a                           # Полная информация о ядре

lsmod                              # Загруженные модули
modinfo <module>                   # Информация о модуле
modinfo -p <module>                # Параметры модуля

insmod /path/to/module.ko          # Загрузить (путь к файлу, без зависимостей)
modprobe <module>                  # Загрузить (с зависимостями)
modprobe -r <module>               # Выгрузить (с зависимостями)
rmmod <module>                     # Выгрузить модуль
depmod -a                          # Пересобрать зависимости модулей

sysctl -a                          # Все параметры ядра
sysctl kernel.panic                # Прочитать параметр
sysctl -w kernel.panic=30          # Записать параметр
sysctl -p /etc/sysctl.conf         # Применить конфигурационный файл

dmesg                              # Буфер сообщений ядра
lspci                              # PCI-устройства
lspci -v                           # Подробный вывод PCI
lsusb                              # USB-устройства
lsusb -v                           # Подробный вывод USB
lsdev                              # Сводка IRQ/DMA/IO

udevadm monitor                    # Мониторинг событий udev
udevadm test /sys/class/net/eth0   # Проверить правило
udevadm control --reload-rules     # Перезагрузить правила
udevadm info --query=all --name=/dev/sda  # Информация об устройстве
```

### Типичные ошибки

- `insmod` требует полного пути к файлу `.ko`; `modprobe` работает с именем модуля
- `rmmod` не выгрузит модуль с ненулевым счётчиком использования
- Изменения в `/proc/sys/` не сохраняются после перезагрузки — используйте `/etc/sysctl.conf`
- Пользовательские правила udev помещаются в `/etc/udev/rules.d/`, а не `/lib/udev/rules.d/`
- `modprobe -c` показывает конфигурацию; `modprobe -r` удаляет модуль
- После ручной установки нового модуля всегда запускайте `depmod -a`

## Практические вопросы

**Q1.** Вы хотите узнать, какие параметры принимает модуль `e1000`. Какую команду вы запустите?

A. `lsmod e1000`
B. `modinfo -p e1000`
C. `insmod e1000 --help`
D. `modprobe -c e1000`

**Ответ:** B. `modinfo -p e1000` выводит список параметров, которые принимает модуль.

---

**Q2.** Команда `rmmod snd_ac97_codec` завершается с ошибкой «in use by snd_ens1371». Как правильно его выгрузить?

A. `rmmod -f snd_ac97_codec`
B. `modprobe -r snd_ac97_codec`
C. `rmmod snd_ens1371 && rmmod snd_ac97_codec`
D. Оба варианта B и C верны.

**Ответ:** D. Можно вручную сначала выгрузить `snd_ens1371`, а затем `snd_ac97_codec`, или использовать `modprobe -r snd_ac97_codec`, который автоматически определяет правильный порядок выгрузки.

---

**Q3.** Вы хотите, чтобы `e1000` загружался автоматически под именем `eth_driver`. Как это настроить?

A. Добавить `install e1000 eth_driver` в `/etc/modprobe.d/eth.conf`
B. Добавить `alias eth_driver e1000` в `/etc/modprobe.d/eth.conf`
C. Добавить `e1000 eth_driver` в `/etc/modules`
D. Создать символическую ссылку `/lib/modules/.../eth_driver.ko -> e1000.ko`

**Ответ:** B. Директива `alias` в `/etc/modprobe.d/` загружает модуль под именем, отличным от имени файла.

---

**Q4.** Какой файл нужно отредактировать, чтобы `net.ipv4.ip_forward=1` применялось при каждой загрузке?

A. `/proc/sys/net/ipv4/ip_forward`
B. `/etc/modules`
C. `/etc/sysctl.conf`
D. `/lib/modules/modules.dep`

**Ответ:** C. `/etc/sysctl.conf` сохраняет настройки параметров ядра между перезагрузками. Изменения в `/proc/sys/` теряются при перезагрузке.

---

**Q5.** Куда должны помещаться пользовательские правила udev для переопределения системных правил по умолчанию?

A. `/lib/udev/rules.d/`
B. `/etc/udev/udev.conf`
C. `/etc/udev/rules.d/`
D. `/proc/udev/rules/`

**Ответ:** C. Пользовательские правила помещаются в `/etc/udev/rules.d/`. Системные правила по умолчанию находятся в `/lib/udev/rules.d/`.

---

**Q6.** Зачем нужен `depmod -a` после ручной установки нового модуля?

A. Загружает все модули из `/lib/modules/`
B. Пересобирает файл зависимостей `modules.dep`
C. Проверяет целостность файлов модулей
D. Удаляет устаревшие модули из системы

**Ответ:** B. `depmod -a` сканирует модули и пересобирает `/lib/modules/<version>/modules.dep`, который `modprobe` читает для разрешения зависимостей.

---

**Q7.** Вы наблюдаете вывод `udevadm monitor`. Какая команда позволит проверить, как правила udev применятся к конкретному устройству, без фактического их срабатывания?

A. `udevadm info /dev/sda`
B. `udevadm test /sys/block/sda`
C. `udevadm control --reload-rules`
D. `udevadm trigger /dev/sda`

**Ответ:** B. `udevadm test` обрабатывает путь sysfs и показывает, какие правила совпадут, не применяя их.

---

**Q8.** После выполнения `sysctl -w kernel.panic=30` значение сбрасывается до 0 после перезагрузки. Как сделать изменение постоянным?

A. Выполнить `depmod -a` после изменения
B. Добавить `kernel.panic = 30` в `/etc/sysctl.conf`
C. Добавить `echo 30 > /proc/sys/kernel/panic` в `/etc/rc.local`
D. Добавить строку в `/etc/modules`

**Ответ:** B. `sysctl -w` изменяет значение только до следующей перезагрузки. Для постоянства добавьте параметр в `/etc/sysctl.conf` и примените его с помощью `sysctl -p`.
