## AutoFS — автоматическое монтирование

AutoFS монтирует и размонтирует файловые системы по требованию без вмешательства администратора. Когда пользователь обращается к каталогу, демон `automount` монтирует нужное устройство. Когда устройство простаивает в течение заданного времени, AutoFS автоматически его размонтирует.

AutoFS полезен в двух сценариях: в сетевых окружениях, где NFS-серверы не всегда доступны, и для сменных носителей, таких как CD-ROM или USB.

> **Важно для экзамена:** AutoFS монтирует ресурс только при обращении **непосредственно к псевдокаталогу**. Запуск `ls /var/autofs/floppy` ничего не покажет; запуск `ls /var/autofs/floppy/floppy` инициирует монтирование.

AutoFS состоит из двух частей: компонента ядра и демона `automount`. Демон читает файлы конфигурации и делегирует задачи монтирования ядру.

---

## Конфигурация AutoFS: auto.master

Главный файл конфигурации AutoFS `/etc/auto.master` называется **главной картой**. Он содержит строки с тремя полями, разделёнными пробелами:

```
# Формат: <точка-монтирования> <файл-карты> [опции]
/var/autofs/floppy   /etc/auto.floppy   --timeout=2
/var/autofs/cdrom    /etc/auto.cdrom    --timeout=6
/home                /etc/auto.home     --timeout=60
```

| Поле | Описание |
|---|---|
| Точка монтирования | Каталог, в котором будут появляться псевдокаталоги |
| Файл карты | Путь к подчинённому файлу, описывающему устройства |
| Опции | Например, `--timeout=N` — секунды простоя перед размонтированием |

> **Внимание:** AutoFS не перезагрузится и не перезапустится, если точка монтирования занята. Перед запуском `reload` проверьте наличие активных процессов в каталоге.

Каждая запись в `auto.master` запускает собственный экземпляр демона `automount`. Рекомендуется хранить каждое устройство в отдельном файле карты: если один демон зависнет, остальные устройства продолжат работу.

---

## Подчинённые файлы карты

Подчинённые файлы (`/etc/auto.[dir]`) описывают конкретные устройства. Каждая строка содержит три поля:

```
# Формат: <псевдокаталог> <опции-монтирования> <устройство>
floppy   -user,fstype=auto   :/dev/fd0
cdrom    -ro,fstype=iso9660  :/dev/cdrom
```

```
# /etc/auto.home — пример NFS
*   -fstype=nfs4,rw   server.example.com:/home/&
```

| Поле | Описание |
|---|---|
| Псевдокаталог | Имя, появляющееся внутри точки монтирования |
| Опции | Параметры монтирования, аналогичные опциям `mount -o` |
| Устройство | Локальное устройство (`:/dev/fd0`) или NFS-ресурс (`server:/share`) |

> **Примечание:** Для локальных устройств путь начинается с двоеточия: `:/dev/fd0`. Для NFS двоеточие разделяет хост и путь: `server:/share`.

Файлы конфигурации перечитываются при перезагрузке демона:

```bash
/etc/init.d/autofs reload
# или
systemctl reload autofs
```

---

## Типы файлов карты

`auto.master` поддерживает три типа карт:

| Тип | Описание | Пример записи |
|---|---|---|
| Прямая карта | Монтирование по абсолютным путям | `/- /etc/auto.direct` |
| Косвенная карта | Монтирование в указанном родительском каталоге | `/home /etc/auto.home` |
| Встроенная карта | Встроенные карты AutoFS, например `+auto.master` | `+dir:/etc/auto.master.d` |

Пример прямой карты (`/etc/auto.direct`):

```
/mnt/projectx   -fstype=nfs4,rw   server:/tmp/projectx
/mnt/projecty   -fstype=nfs4,rw   server:/tmp/projecty
```

Включение вложенных файлов главной карты:

```
+dir:/etc/auto.master.d
```

> **Важно:** Если AutoFS сообщает `unable to read auto.master` при запуске, закомментируйте строку `+auto.master` в `/etc/auto.master` символом `#`.

## AutoFS и systemd

В systemd-системах автомонтирование настраивается через файлы юнитов с расширением `.automount`. Имя файла отражает путь точки монтирования (то же соглашение об именовании, что и для юнитов `.mount`). Для точки монтирования `/mnt` файл называется `mnt.automount`.

Юнит `.automount` работает совместно с одноимённым юнитом `.mount`. systemd регистрирует точку монтирования с типом `autofs` и ожидает первого обращения. Когда процесс обращается к каталогу (например, `ls /mnt`), systemd немедленно выполняет монтирование через связанный юнит `.mount`.

```ini
# /etc/systemd/system/mnt.automount
[Unit]
Description=My new automounted file system.

[Automount]
Where=/mnt

[Install]
WantedBy=multi-user.target
```

Связанный юнит `.mount` описывает само монтирование:

```ini
# /etc/systemd/system/mnt.mount
[Unit]
Description=My new file system

[Mount]
What=/dev/sdb1
Where=/mnt
Type=ext4
Options=

[Install]
WantedBy=multi-user.target
```

Полная последовательность настройки:

```bash
# 1. Включить и запустить юнит автомонтирования
systemctl enable mnt.automount
systemctl start mnt.automount

# 2. Проверить статус — юнит будет в состоянии "active (waiting)"
systemctl status mnt.automount

# 3. После запуска mount показывает точку как autofs, но диск ещё не смонтирован
#    systemd-1 on /mnt type autofs (rw,relatime,...)
mount | grep mnt

# 4. Обратиться к каталогу — это инициирует монтирование
ls /mnt

# 5. Теперь lsblk и mount показывают реальный диск на /mnt
lsblk
mount | grep mnt
```

> **Важно:** После `systemctl start mnt.automount` точка монтирования регистрируется как `autofs`, но физический диск ещё не смонтирован. `lsblk` показывает пустой `MOUNTPOINT`. Только при первом обращении к `/mnt` systemd выполняет реальное монтирование.

| Опция `.automount` | Обязательна | Описание |
|---|---|---|
| `Where=` | Да | Путь точки монтирования |
| `DirectoryMode=` | Нет | Права доступа для создаваемых каталогов, по умолчанию 0755 |
| `TimeoutIdleSec=` | Нет | Время простоя до размонтирования; по умолчанию отключено |

## Файловые системы CD-ROM

На экзамене проверяются три типа файловых систем оптических носителей:

| Тип | Описание |
|---|---|
| ISO9660 | Стандарт для CD-ROM; поддерживает имена файлов до 8.3 (формат DOS) |
| UDF (Universal Disk Format) | Современный стандарт OSTA для всех оптических носителей; разработан для замены ISO9660 |
| HFS (Hierarchical File System) | Файловая система Apple, используется на Mac-совместимых CD |

---

## ISO9660 и расширения

ISO9660 ограничен именами файлов 8.3 и не поддерживает права доступа Unix. Для устранения этих ограничений были добавлены расширения:

| Расширение | Назначение |
|---|---|
| **Rock Ridge** | Unix-расширение: длинные имена файлов, права доступа и символические ссылки. Необходимо для чтения Linux-ом CD с именами в стиле Unix. |
| **Joliet** | Расширение Microsoft: имена файлов до 64 символов Unicode; делает диск читаемым в Windows. |
| **El Torito** | Расширение для создания загрузочных CD-ROM. Без него диск нельзя использовать как загрузочный носитель. |

> **Важно для экзамена:** El Torito, Joliet и Rock Ridge — **расширения** ISO9660. UDF — полностью **отдельная и независимая файловая система**. Не путайте их.

---

## UDF

UDF стандартизирован организацией OSTA (Optical Storage Technology Association). Работает с CD-R, CD-RW и DVD, включая видео DVD.

Инструменты из пакета `udftools`:

| Утилита | Назначение |
|---|---|
| `mkudffs` | Создать новую UDF-файловую систему на диске или CD-R/RW |
| `udffsck` | Проверить целостность и исправить ошибки на UDF-разделах |
| `wrudf` | Интерактивная оболочка для обслуживания UDF: cp, rm, mkdir и т.д. |
| `cdrwtool` | Управление приводами CD-RW: форматирование, настройка скорости |

## mkisofs — создание ISO-образов

`mkisofs` создаёт ISO-образы из каталогов. В более новых дистрибутивах заменяется на `genisoimage`, но синтаксис совместим.

```bash
# Создать простой ISO-образ
mkisofs -r -o cd_image.iso /path/to/private_collection/

# С Joliet и Rock Ridge (совместимость с Windows и Linux)
mkisofs -r -J -o image.iso /path/to/files/

# Создать загрузочный ISO (El Torito)
mkisofs -r -b boot/grub/stage2_eltorito \
        -no-emul-boot -boot-load-size 4 \
        -boot-info-table -c boot/boot.catalog \
        -o bootable.iso /path/to/files/
```

Основные опции `mkisofs`:

| Опция | Описание |
|---|---|
| `-o filename` | Имя выходного ISO-файла |
| `-r` | Rock Ridge + изменение прав доступа на root/публичный |
| `-R` | Rock Ridge без изменения прав доступа |
| `-J` | Расширения Joliet (совместимость с Windows) |
| `-b file` | El Torito: загрузочный образ для загрузочного CD |
| `-c file` | El Torito: файл каталога загрузки (обязателен вместе с `-b`) |
| `-no-emul-boot` | El Torito: без режима эмуляции диска при загрузке |
| `-boot-load-size N` | El Torito: размер загрузочного сектора, рекомендуемое значение 4 |
| `-hfs` | Создать гибридный образ ISO9660/HFS для Mac |
| `-udf` | Добавить поддержку UDF (для видео DVD) |

> **Примечание:** `mkisofs` не умеет записывать напрямую на CD-привод. Для записи используйте `cdrecord`. Такое разделение намеренно: сначала создайте образ, проверьте его, затем запишите. Три причины: `mkisofs` ничего не знает о CD-приводах; прямая запись ненадёжна на медленных машинах; образ нужно проверить перед записью на диск.

## Тестирование ISO-образа

Linux может смонтировать ISO-файл как петлевое устройство без записи на диск, позволяя проверить структуру каталогов и права доступа перед записью.

```bash
# Смонтировать ISO-образ как петлевое устройство
mount -t iso9660 -o ro,loop=/dev/loop0 cd_image /cdrom

# Просмотреть содержимое
ls /cdrom

# Размонтировать после проверки
umount /cdrom
```

## Запись образа на CD

`cdrecord` записывает образ на SCSI-привод CD. Сначала найдите координаты устройства с помощью `cdrecord -scanbus` (BUS, ID, LUN), затем используйте их для записи.

```bash
# Найти координаты CD-привода
cdrecord -scanbus

# Записать образ (dev=BUS,ID,LUN)
cdrecord -v speed=2 dev=0,6,0 -data cd_image

# Перезаписать CD-RW (сначала стереть)
cdrecord -v speed=2 dev=0,6,0 blank=fast -data cd_image
```

На достаточно быстрой машине можно передавать вывод `mkisofs` напрямую в `cdrecord` без создания промежуточного файла. Необходимо заранее вычислить размер образа:

```bash
# Шаг 1: определить размер образа (сухой прогон)
IMG_SIZE=`mkisofs -R -q -print-size private_collection/ 2>&1 | sed -e "s/.* = //"`

# Шаг 2: создать образ и записать через канал
mkisofs -r private_collection/ | cdrecord speed=2 dev=0,6,0 tsize=${IMG_SIZE}s -data -
```

> **Внимание:** Запись CD нельзя прерывать. Даже удаление большого файла во время записи может нарушить поток данных и испортить диск: удаление файла объёмом 650 МБ приводит к обновлению метаданных для 650 000 блоков, временно блокируя активность диска.

## Копирование CD

При наличии отдельных приводов CD-ROM и CD-рекордера:

```bash
cdrecord -v dev=0,6,0 speed=2 -isosize /dev/scd0
```

При наличии только одного привода — сначала прочитать в файл, затем записать:

```bash
# Прочитать содержимое CD в файл
dd if=/dev/scd0 of=cdimage

# Записать сохранённый образ на чистый CD
cdrecord -v speed=2 dev=0,6,0 -data cdimage
```

> **Примечание:** Файл, созданный с помощью `dd if=/dev/scd0`, эквивалентен выводу `mkisofs`. Оба метода создают образ, подходящий для записи с помощью `cdrecord`.

## Зашифрованные файловые системы

Linux поддерживает несколько алгоритмов симметричного шифрования:

| Алгоритм | Примечания |
|---|---|
| AES (Rijndael) | Стандарт NIST; 128-битный блок; ключ 128/192/256 бит. Аппаратное ускорение Intel через AES-NI с 2010 года. |
| Twofish | Кандидат AES; 128-битный блок, ключ до 256 бит. Проиграл Rijndael по производительности. |
| DES | Устаревший стандарт; 56-битный ключ. Считается небезопасным — быстро взламывается перебором. |

> **Важно:** Проверить поддержку AES-NI в процессоре: `grep aes /proc/cpuinfo`. Проверить поддержку ядром: `sort -u /proc/crypto | grep module`. Загрузить драйвер: `modprobe aesni-intel`.

На экзамене требуется знание трёх механизмов шифрования:

| Тип | Уровень | Примечания |
|---|---|---|
| dm-crypt | Блочное устройство | Один ключ, нет метаданных, только для экспертов |
| LUKS | Блочное устройство | До 8 ключей, метаданные в заголовке раздела, рекомендуется |
| eCryptfs | Уровень файлов | Накладывается поверх существующей ФС, шифрует каждый файл отдельно, метаданные в заголовке файла |

**Device Mapper** — общий механизм ядра Linux для отображения одного блочного устройства на другое. Используется LVM, программным RAID и шифрованием. Виртуальное устройство появляется в `/dev/mapper`; все данные проходят через фильтр шифрования/дешифрования перед записью на физический диск.

## dm-crypt

dm-crypt шифрует целое блочное устройство. Метаданные не хранятся — при утере пароля данные восстановить невозможно. После шифрования раздел неотличим от диска со случайными данными, что обеспечивает правдоподобное отрицание.

```bash
# Загрузить необходимые модули при запуске
echo aes >> /etc/modules
echo dm_mod >> /etc/modules
echo dm_crypt >> /etc/modules
modprobe -a aes dm_mod dm_crypt

# Создать зашифрованное устройство и задать пароль
cryptsetup -y create crypt /dev/hda3

# Добавить в crypttab и fstab для автоматической разблокировки при загрузке
echo "crypt /dev/hda3 none none" >> /etc/crypttab
echo "/dev/mapper/crypt /crypt reiserfs defaults 0 1" >> /etc/fstab

# Создать файловую систему на зашифрованном устройстве
mkfs.reiserfs /dev/mapper/crypt

# Смонтировать (при каждой загрузке будет запрашиваться пароль)
mkdir /crypt
mount /crypt
```

> **Внимание:** dm-crypt предназначен для опытных пользователей. Для всех остальных рекомендуется LUKS. Потеря пароля означает безвозвратную утрату данных — метаданных для восстановления нет.

---

## LUKS

LUKS (Linux Unified Key Setup) — предпочтительный метод шифрования в современных системах Linux. Хранит метаданные и криптографические параметры в заголовке раздела, что упрощает перенос данных между системами. Поддерживает **до 8 ключей** на раздел (депонирование ключей). На экзамене LUKS иногда называется **dm-crypt/LUKS**.

```bash
# Шаг 1: создать файл-контейнер (512 МиБ)
dd if=/dev/urandom of=/root/encrypted bs=1M count=512

# Шаг 2: отформатировать как LUKS (запросит подтверждение и пароль)
cryptsetup -y luksFormat encrypted

# Шаг 3: открыть контейнер под именем "encrypted"
cryptsetup luksOpen encrypted encrypted

# Шаг 4: создать файловую систему на открытом устройстве
mkfs.ext4 /dev/mapper/encrypted

# Шаг 5: смонтировать для использования
mount /dev/mapper/encrypted /mnt

# Шаг 6: размонтировать и закрыть контейнер
umount /mnt
cryptsetup luksClose /dev/mapper/encrypted
```

Просмотр информации о LUKS-контейнере:

```bash
cryptsetup luksDump /dev/sdb1
```

## eCryptfs

eCryptfs накладывается поверх существующей файловой системы. Требуется два монтирования: сначала стандартное `mount` для раздела, затем `mount` с типом `ecryptfs`.

```bash
# Первое монтирование: смонтировать раздел обычным образом
mount /dev/sdb1 /mnt/data

# Второе монтирование: добавить слой шифрования
mount -t ecryptfs /mnt/data /mnt/data
```

Шифрование выполняется на уровне файлов; метаданные хранятся в заголовке каждого файла. Зашифрованные файлы можно переносить между системами.

> **Примечание:** eCryptfs можно добавить в `/etc/fstab` для автоматического монтирования. Дополнительная документация: `man 7 ecryptfs`.

## Утилиты: dd и mke2fs

`dd` — универсальный низкоуровневый инструмент копирования данных. Работает с блоками фиксированного размера.

```bash
# Создать файл, заполненный случайными данными (512 МиБ)
dd if=/dev/urandom of=/root/encrypted bs=1M count=512

# Создать резервную копию MBR
dd if=/dev/sda of=/backup/mbr.bin bs=512 count=1

# Полное копирование диска
dd if=/dev/sda of=/dev/sdb bs=64K

# Создать пустой контейнер заданного размера
dd if=/dev/zero of=/tmp/container bs=1M count=100
```

`mke2fs` создаёт файловые системы ext2/ext3/ext4:

```bash
# Создать ext2
mke2fs /dev/sdb1

# Создать ext4 (рекомендуется)
mke2fs -t ext4 /dev/sdb1

# Создать ext4 с меткой
mke2fs -t ext4 -L "mydata" /dev/sdb1

# Эквивалентно через mkfs
mkfs.ext4 /dev/sdb1
```

> **Примечание:** `mke2fs -t ext4` и `mkfs.ext4` дают одинаковый результат. Обе формы встречаются на экзамене.

## Шпаргалка для экзамена

### Пути и файлы

| Путь | Назначение |
|---|---|
| `/etc/auto.master` | Основная конфигурация AutoFS (главная карта) |
| `/etc/auto.[dir]` | Подчинённые файлы карты для конкретных точек монтирования |
| `/etc/sysconfig/autofs` | Конфигурация поведения AutoFS (RHEL/CentOS) |
| `/etc/default/autofs` | Конфигурация поведения AutoFS (Debian/Ubuntu) |
| `/etc/auto.master.d/` | Каталог для вложенных файлов главной карты |
| `/etc/crypttab` | Определения зашифрованных разделов для автоматической разблокировки |
| `/etc/systemd/system/` | Каталог для файлов юнитов `.mount` и `.automount` |

### Основные команды

```bash
# AutoFS
systemctl start|stop|reload autofs
/etc/init.d/autofs reload

# ISO-образы
mkisofs -r -J -o image.iso /source/dir/
mount -t iso9660 -o ro,loop=/dev/loop0 cd_image /cdrom
cdrecord -scanbus
cdrecord -v speed=2 dev=0,6,0 -data cd_image
dd if=/dev/scd0 of=cdimage

# UDF
mkudffs /dev/sdb1
udffsck /dev/sdb1

# dd и mke2fs
dd if=/dev/urandom of=file bs=1M count=512
mke2fs -t ext4 /dev/sdb1

# LUKS
cryptsetup luksFormat /dev/sdb1
cryptsetup luksOpen /dev/sdb1 myname
cryptsetup luksClose myname
cryptsetup luksDump /dev/sdb1
```

### Типичные ловушки на экзамене

- Файловые системы AutoFS **не должны** быть перечислены в `/etc/fstab`. Дублирующие записи вызывают проблемы при перезагрузке.
- UDF — **отдельная файловая система**, а не расширение ISO9660. El Torito, Joliet и Rock Ridge являются расширениями ISO9660.
- `cdrecord` записывает образ, но не создаёт его. `mkisofs` создаёт образ, но не записывает его.
- Храните каждое устройство в отдельном файле карты. Зависший демон может заблокировать все устройства из одного файла.
- `mkisofs` без `-r` или `-R` создаёт CD с правами доступа исходного каталога, а не публично читаемыми.

## Практические вопросы

**В1.** Администратор хочет автоматически монтировать NFS-каталог `/export/data` с сервера `fileserver` по пути `/mnt/data` через AutoFS. Какой файл нужно отредактировать первым?

**Ответ:** `/etc/auto.master` — добавить строку, сопоставляющую точку монтирования с файлом карты. Затем создать сам файл карты.

---

**В2.** Файл `/etc/auto.floppy` содержит `floppy -user,fstype=auto :/dev/fd0`. Что произойдёт, когда пользователь выполнит `ls /var/autofs/floppy`?

**Ответ:** **Ничего** — псевдокаталог `floppy` не виден при выводе списка родительского каталога. Монтирование инициируется только при прямом обращении: `ls /var/autofs/floppy/floppy`.

---

**В3.** Какое расширение ISO9660 используется для создания загрузочных CD-ROM?

**Ответ:** **El Torito**. Rock Ridge добавляет Unix-атрибуты; Joliet добавляет совместимость с Windows; UDF — отдельная файловая система.

---

**В4.** Какой метод шифрования хранит метаданные в заголовке тома и поддерживает до 8 ключей?

**Ответ:** **LUKS**. dm-crypt не имеет метаданных вообще. eCryptfs работает на уровне файлов, а не тома.

---

**В5.** Администратор создал ISO-образ и хочет проверить его содержимое без записи на диск. Какая команда это делает?

**Ответ:** `mount -t iso9660 -o ro,loop=/dev/loop0 cd_image /cdrom` — монтирует ISO как петлевое устройство для проверки.

---

**В6.** Что делает опция `-J` в команде `mkisofs -r -J -o disk.iso /data/`?

**Ответ:** Включает **расширения Joliet** для совместимости с Windows и именами файлов Unicode до 64 символов. `-r` включает Rock Ridge; `-b` предназначен для El Torito; `-hfs` — для HFS.

---

**В7.** Файл `/etc/auto.master` содержит `/home /etc/auto.home --timeout=300`. Что произойдёт со смонтированным домашним каталогом после того, как пользователь не обращался к нему 5 минут?

**Ответ:** Он будет **автоматически размонтирован**. `--timeout=300` означает 300 секунд (5 минут) простоя. При следующем обращении AutoFS снова автоматически смонтирует его.

---

**В8.** Что содержит файл юнита `.automount` в systemd?

**Ответ:** Секцию `[Automount]` с обязательным параметром `Where=`. Работает совместно с одноимённым юнитом `.mount`, который описывает что и как монтировать.
