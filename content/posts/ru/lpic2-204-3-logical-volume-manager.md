> **Тема экзамена 204.3** — Менеджер логических томов. Охватывает трёхуровневую архитектуру LVM, команды для физических/групповых/логических томов, изменение размеров, снимки, Device Mapper и настройку lvm.conf.

---

## Что такое LVM

LVM (Logical Volume Manager) объединяет несколько физических разделов в единый пул хранения, а затем нарезает из этого пула логические тома любого размера. Ядро Linux не видит разницы между обычным разделом и логическим томом, поэтому `mount` работает одинаково для обоих.

Ключевое преимущество перед традиционной разметкой: размер тома можно изменить на работающей системе, а новый диск добавляется в пул без переразметки существующих данных.

> **Важно:** Пакет называется `lvm2`. Если `pvcreate`, `vgcreate` и `lvcreate` недоступны, установите его: `apt install lvm2` или `yum install lvm2`.

## Архитектура LVM: три уровня

```
Физический диск/раздел  (/dev/sdb1, /dev/sdc)
          |
    Физический том (PV)      ← pvcreate
          |
    Группа томов (VG)        ← vgcreate
          |
    Логический том (LV)      ← lvcreate
          |
      Файловая система        ← mkfs
          |
    Точка монтирования        ← mount
```

У каждого уровня есть свой набор команд: `pv*` для физических томов, `vg*` для групп томов, `lv*` для логических томов.

> **Примечание:** Физические экстенты (PE) — это минимальные блоки распределения внутри VG, по умолчанию 4 МБ. Логические экстенты (LE) на стороне LV отображаются один-к-одному на PE. Размер PE задаётся при создании VG с помощью `vgcreate -s`.

## Физические тома (PV)

Физический том — это диск или раздел, подготовленный для использования в LVM. `pvcreate` записывает метаданные LVM в начало устройства.

Тип раздела для LVM в `fdisk`: `0x8e` (Linux LVM).

### Создание PV

```bash
pvcreate /dev/sdb1 /dev/sdc1
```

### Просмотр PV

```bash
pvdisplay              # подробный вывод по каждому PV
pvs                    # краткая таблица всех PV
pvscan                 # сканировать диски, найти PV
pvck /dev/sdb1         # проверить метаданные PV
```

Пример вывода `pvs`:

```
  PV         VG       Fmt  Attr PSize  PFree
  /dev/sdb1  vg_data  lvm2 a--  10.00g  5.00g
  /dev/sdc1  vg_data  lvm2 a--   5.00g  2.00g
```

### Управление PV

```bash
pvmove /dev/sdb1          # переместить экстенты на другие PV внутри VG
pvremove /dev/sdb1        # удалить метаданные LVM с устройства
pvchange -x n /dev/sdb1   # запретить выделение новых экстентов на этом PV
pvresize /dev/sdb1        # пересчитать размер PV после изменения раздела
```

> **Внимание:** Перед `pvremove` убедитесь, что на устройстве нет активных экстентов. Сначала перенесите данные с помощью `pvmove`.

## Группы томов (VG)

Группа томов объединяет один или несколько PV в единый пул. Из этого пула создаются логические тома.

### Создание VG

```bash
# Создать группу volume01 из двух PV
vgcreate volume01 /dev/sdb1 /dev/sdc1

# Создать с нестандартным размером PE (8 МБ вместо 4 МБ)
vgcreate -s 8M vg_data /dev/sdb1
```

### Просмотр VG

```bash
vgdisplay volume01    # подробная информация
vgs                   # краткая таблица всех VG
vgscan                # поиск VG на всех дисках
vgck volume01         # проверить метаданные VG
```

### Управление VG

```bash
vgextend volume01 /dev/sdd1      # добавить PV в VG
vgreduce volume01 /dev/sdb1      # удалить PV (данных не должно оставаться)
vgrename volume01 vg_new         # переименовать VG
vgremove volume01                # удалить VG (все LV должны быть удалены сначала)
vgchange -a y volume01           # активировать VG
vgchange -a n volume01           # деактивировать VG
vgexport volume01                # экспортировать VG (для миграции на другую машину)
vgimport volume01                # импортировать ранее экспортированную VG
vgmerge vg_target vg_source      # слить две VG в одну
vgsplit volume01 vg_new /dev/sdb1 # разделить PV в новую VG
vgcfgbackup volume01             # создать резервную копию метаданных VG
vgcfgrestore volume01            # восстановить метаданные VG из резервной копии
```

> **Важно:** `vgchange -a y` активирует все VG; `vgchange -a n` деактивирует их. Флаг `-a` означает `--activate`.

## Логические тома (LV)

Логический том — это виртуальный раздел внутри VG. Ядро видит его как обычное блочное устройство.

### Создание LV

```bash
# Создать LV размером 100 МБ с автоматически сгенерированным именем (lvol0, lvol1...)
lvcreate -L 100M volume01

# Создать с явным именем
lvcreate -L 2G -n mydata volume01

# Создать, используя 50% свободного места в VG
lvcreate -l 50%FREE -n mydata volume01

# Создать с указанием количества экстентов
lvcreate -l 128 -n mydata volume01
```

После создания LV доступен по двум путям:

- `/dev/volume01/mydata` (символическая ссылка)
- `/dev/mapper/volume01-mydata` (Device Mapper)

### Просмотр LV

```bash
lvdisplay /dev/volume01/mydata   # подробная информация
lvs                              # краткая таблица всех LV
lvscan                           # список всех LV во всех VG
lvmdiskscan                      # все устройства, видимые LVM
```

### Форматирование и монтирование

```bash
mkfs -t xfs /dev/volume01/mydata
mkfs.ext4 /dev/volume01/mydata
mount /dev/volume01/mydata /mnt/data

# Запись в /etc/fstab (рекомендуется использовать путь /dev/mapper)
/dev/mapper/volume01-mydata  /mnt/data  xfs  defaults  0  0
```

> **Совет:** В `/etc/fstab` используйте пути `/dev/mapper/`, а не `/dev/VG/LV`. Это обеспечивает корректную активацию LVM при загрузке на некоторых дистрибутивах.

## Полный цикл: от диска до точки монтирования

```bash
# Шаг 1: создать физические тома
pvcreate /dev/hda4 /dev/hda5

# Шаг 2: создать группу томов
vgcreate volume01 /dev/hda4 /dev/hda5

# Шаг 3: создать логический том размером 100 МБ
lvcreate -L 100M -n lvol0 volume01

# Шаг 4: форматировать
mkfs -t xfs /dev/volume01/lvol0

# Шаг 5: смонтировать
mount /dev/volume01/lvol0 /mnt
```

## Расширение VG и LV

Расширьте VG, добавив новый PV, затем расширьте LV, а затем файловую систему внутри него.

### Расширение VG

```bash
pvcreate /dev/hda6
vgextend volume01 /dev/hda6
```

### Расширение LV

```bash
lvextend -L +50M /dev/volume01/lvol0     # добавить 50 МБ
lvextend -L 500M /dev/volume01/lvol0     # изменить размер до 500 МБ
lvextend -l +100%FREE /dev/volume01/lvol0  # использовать всё свободное место в VG
```

### Расширение файловой системы

После `lvextend` файловая система ещё не знает о новом пространстве — измените её размер отдельно.

```bash
# XFS (только расширение; уменьшение не поддерживается — ФС должна быть смонтирована)
xfs_growfs /dev/volume01/lvol0
# или по точке монтирования:
xfs_growfs /mnt

# ext2/ext3/ext4 (ФС должна быть размонтирована для изменения размера)
resize2fs /dev/volume01/lvol0
```

> **Важно:** XFS можно расширять только **в режиме онлайн** (смонтированной). ext2/ext3/ext4 необходимо **размонтировать** для изменения размера. Это частый вопрос на экзамене.

## Уменьшение LV

Уменьшение сложнее расширения: сначала уменьшите файловую систему, затем LV.

```bash
# Только для ext2/ext3/ext4 — XFS не поддерживает уменьшение!

# Шаг 1: размонтировать
umount /dev/volume01/lvol0

# Шаг 2: проверить файловую систему
e2fsck -f /dev/volume01/lvol0

# Шаг 3: сначала уменьшить файловую систему
resize2fs /dev/volume01/lvol0 80M

# Шаг 4: затем уменьшить LV
lvreduce -L 80M /dev/volume01/lvol0

# Альтернатива: lvresize работает в обоих направлениях
lvresize -L -20M /dev/volume01/lvol0
```

> **Внимание:** Всегда уменьшайте в этом порядке: **сначала файловая система, затем LV**. Обратный порядок приводит к потере данных. XFS вообще не поддерживает уменьшение.

## Снимки LVM

Снимок — это виртуальная копия логического тома на определённый момент времени. Используется для резервного копирования без остановки системы.

Снимки LVM используют копирование при записи (CoW): когда блок в исходном LV изменяется, старое значение сначала сохраняется в снимок, а затем оригинал перезаписывается. Снимок использует пространство пропорционально объёму изменений, а не размеру исходного тома.

### Создание снимка

```bash
# Создать снимок lvol0, выделив 50 МБ для данных CoW
lvcreate -L 50M -s -n snapshot0 /dev/volume01/lvol0
```

Флаги: `-s` (или `--snapshot`) создаёт снимок; `-n` задаёт имя.

### Использование снимка

```bash
# Смонтировать снимок только для чтения (например, для резервного копирования)
mount -o ro /dev/volume01/snapshot0 /mnt/backup

# Создать резервную копию данных
tar -czf /backup/data.tar.gz /mnt/backup/

# Размонтировать
umount /mnt/backup
```

### Удаление снимка

```bash
lvremove /dev/volume01/snapshot0
```

> **Внимание:** Удаляйте снимки сразу после использования. На базах данных и системах с интенсивной записью незакрытый снимок снижает производительность — каждая запись в исходный LV требует дополнительной операции копирования.

> **Важно:** Если снимок заполнится (пространство CoW исчерпано), он становится **недействительным**. Контролируйте степень заполнения с помощью `lvs` или `lvdisplay`.

## Device Mapper

Device Mapper — драйвер ядра Linux, создающий виртуальные блочные устройства поверх физических. LVM использует Device Mapper как транспортный уровень; сам Device Mapper ничего не знает о группах томов или форматах метаданных LVM.

При активации логического тома LVM транслирует его в отображённое устройство. Каждый сегмент LV становится строкой в таблице отображения. Device Mapper поддерживает три типа отображения: **linear** (последовательный), **striped** (чередующийся) и **error** (для тестирования отказов).

Каждый LV отображается в `/dev/mapper/` с именем `VG_имя-LV_имя` (через дефис).

```bash
dmsetup info
dmsetup info /dev/mapper/volume01-lvol0
```

Пример вывода:

```
Name:              volume01-lvol0
State:             ACTIVE
Read Ahead:        256
Tables present:    LIVE
Open count:        1
Major, minor:      253, 2
Number of targets: 1
UUID: LVM-abc123...
```

Пример записей в `/etc/fstab`:

```
/dev/mapper/centos-root   /      xfs   defaults  0 0
/dev/mapper/centos-swap   swap   swap  defaults  0 0
```

> **Примечание:** Если имя VG или LV содержит дефис, в пути `/dev/mapper/` он **удваивается**. Например, VG `vg-data` с LV `lv-home` даёт `/dev/mapper/vg--data-lv--home`.

## Настройка lvm.conf

Глобальный файл конфигурации LVM — `/etc/lvm/lvm.conf`. Загружается при запуске системы. Путь по умолчанию можно переопределить переменной окружения `LVM_SYSTEM_DIR`.

```bash
lvm dumpconfig --type default
lvm dumpconfig --type default --withcomments
```

> **Примечание:** В старых дистрибутивах вместо `lvm dumpconfig` использовался `lvmconf`.

### Секция devices: фильтры

`vgscan` и `pvscan` сканируют все блочные устройства в поисках метаданных LVM. В средах с multipath одни и те же метаданные обнаруживаются на каждом пути, что порождает дублирующиеся записи в выводе `pvs`. Опция `filter` в секции `devices` ограничивает список устройств с помощью регулярных выражений.

Формат: `"a|regex|"` (принять) или `"r|regex|"` (отклонить). Правила применяются по порядку; первое совпадение побеждает.

```ini
# Принять все устройства (поведение по умолчанию)
filter = [ "a/.*/" ]

# Исключить CD-ROM
filter = [ "r|/dev/cdrom|" ]

# Принять только устройства multipath, отклонить всё остальное
filter = [ "a|/dev/disk/by-id/dm-uuid-.*-mpath-.*|", "r|.*|" ]

# Принять только /dev/hda8, отклонить всё остальное
filter = [ "a|^/dev/hda8$|", "r/.*/" ]
```

> **Внимание:** Порядок правил имеет значение. Если поставить `"r|.*|"` первым, все устройства будут заблокированы и LVM не найдёт ни одного PV.

> **Важно:** `a` = принять, `r` = отклонить. Разделитель в регулярном выражении может быть любым символом (`|`, `/`, `!`) — открывающий и закрывающий должны совпадать.

### Секция devices: preferred_names

Управляет тем, какой путь LVM отображает в выводе `pvs`, когда у физического устройства несколько путей. Побеждает первый совпавший шаблон.

```ini
# В средах с multipath: показывать пути mpath вместо путей sd
preferred_names = [ "^/dev/mpath/", "^/dev/mapper/mpath", "^/dev/[hs]d" ]
```

### Секция devices: types

Задаёт допустимые типы блочных устройств и номера типов разделов. Необходимо, когда система загружается с локального диска, а другие диски являются multipath.

```ini
# Разрешить устройства device-mapper с номером типа 253
types = [ "device-mapper", 253 ]
```

Полная документация: `man lvm.conf`.

## Шпаргалка для экзамена

### Последовательность создания

```
fdisk (тип 0x8e) → pvcreate → vgcreate → lvcreate → mkfs → mount
```

### Последовательность расширения

```
pvcreate (новый диск) → vgextend → lvextend → xfs_growfs / resize2fs
```

### Последовательность уменьшения (только ext)

```
umount → e2fsck -f → resize2fs → lvreduce → mount
```

### Снимок

```bash
lvcreate -L <size> -s -n <snap_name> /dev/VG/LV
lvremove /dev/VG/snap_name
```

### Пути Device Mapper

```
/dev/mapper/VG_имя-LV_имя
/dev/VG_имя/LV_имя      ← символическая ссылка на /dev/mapper/...
```

Типы отображения Device Mapper: **linear**, **striped**, **error**.

### Фильтры lvm.conf

```ini
filter = [ "a/.*/" ]                     # принять всё
filter = [ "r|/dev/cdrom|" ]             # отклонить cdrom
filter = [ "a|/dev/sda|", "r|.*|" ]     # только /dev/sda
```

Порядок правил важен — побеждает первое совпадение.

### Типичные ошибки на экзамене

| Ошибка | Правильный подход |
|---|---|
| Уменьшить LV до уменьшения ФС | При уменьшении: сначала ФС, затем LV |
| Попытаться уменьшить XFS | XFS нельзя уменьшить, только ext |
| Забыть `xfs_growfs` после `lvextend` | После расширения LV расширить и ФС |
| Смонтировать снимок для записи | Снимки обычно монтируются только для чтения (`-o ro`) |
| Забыть `pvcreate` перед `vgextend` | Диск должен стать PV перед добавлением в VG |

## Практические вопросы

**В1.** Какая команда инициализирует `/dev/sdb` для использования в LVM?

**Ответ:** `pvcreate /dev/sdb`

---

**В2.** Администратор хочет добавить 10 ГБ к логическому тому `/dev/vg0/data` с примонтированной XFS. Какая правильная последовательность команд?

**Ответ:** `lvextend -L +10G /dev/vg0/data`, затем `xfs_growfs /dev/vg0/data`. Размонтирование не нужно — XFS расширяется в режиме онлайн.

---

**В3.** В чём разница между `/dev/mapper/vg0-data` и `/dev/vg0/data`?

**Ответ:** Это два пути к одному устройству. `/dev/vg0/data` — символическая ссылка, указывающая на `/dev/mapper/vg0-data`. Рекомендуется использовать `/dev/mapper/` в `/etc/fstab`.

**В4.** Администратор выполнил `lvcreate -L 200M -s -n snap0 /dev/vg0/data`. Через неделю снимок стал недействительным. Почему?

**Ответ:** 200 МБ пространства CoW оказалось недостаточно для хранения всех изменённых блоков. Когда выделенное место заполнилось, снимок потерял целостность. Выделяйте не менее 15–20% от размера исходного LV для снимков или контролируйте заполнение с помощью `lvs`.

**В5.** Какова правильная последовательность команд для миграции LV на другой сервер с помощью `vgexport`/`vgimport`?

**Ответ:**
1. `umount /dev/vg0/data`
2. `vgchange -a n vg0`
3. `vgexport vg0`
4. Физически перенести диск на новый сервер
5. `vgscan`
6. `vgimport vg0`
7. `vgchange -a y vg0`
8. `mount /dev/vg0/data /mnt/data`

**В6.** Файловая система ext4 на `/dev/vg0/logs` занимает 5 ГБ и должна быть уменьшена до 3 ГБ. Какие команды нужны и в каком порядке?

**Ответ:**
1. `umount /dev/vg0/logs`
2. `e2fsck -f /dev/vg0/logs`
3. `resize2fs /dev/vg0/logs 3G`
4. `lvreduce -L 3G /dev/vg0/logs`
5. `mount /dev/vg0/logs /var/log`

**В7.** Какая команда показывает текущие настройки `lvm.conf` с описаниями?

**Ответ:** `lvm dumpconfig --type default --withcomments`

**В8.** Администратор выполнил `lvextend -L +1G /dev/vg0/data`, но `df -h` по-прежнему показывает старый размер. Что нужно сделать?

**Ответ:** Расширить файловую систему. Для XFS (смонтированной): `xfs_growfs /dev/vg0/data`. Для ext4: `resize2fs /dev/vg0/data`. `lvextend` расширяет только блочное устройство — файловую систему внутри нужно изменить отдельно.
