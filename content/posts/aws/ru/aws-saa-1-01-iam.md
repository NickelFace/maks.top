## Основные компоненты IAM

AWS Identity and Access Management (IAM) — это **глобальный сервис** (не привязан к региону), управляющий тем, кто и что может делать в вашем аккаунте AWS.

| Компонент | Описание | Ключевой момент |
|---|---|---|
| **Root-аккаунт** | Создаётся вместе с AWS-аккаунтом; неограниченный доступ | Никогда не использовать для повседневных задач; защитить MFA |
| **IAM User** | Долгосрочная идентичность для человека или сервиса | Получает ключи доступа (programmatic) или пароль (консоль) |
| **IAM Group** | Набор пользователей; политики прикрепляются к группе | Нельзя вкладывать группы в группы |
| **IAM Role** | Временная идентичность, принимаемая принципалом | Нет постоянных credentials; используются токены STS |
| **IAM Policy** | JSON-документ, определяющий разрешения | Прикрепляется к пользователям, группам или ролям |

## Политики IAM

### Типы политик

| Тип | К чему прикрепляется | Примечания |
|---|---|---|
| **Identity-based** | User / Group / Role | Даёт разрешения идентичности |
| **Resource-based** | Ресурс (S3 bucket, KMS key…) | Даёт доступ К ресурсу; только inline |
| **Permission boundary** | User / Role | Задаёт максимально допустимые разрешения |
| **Organizations SCP** | OU / Account | Организационный ограничитель |
| **Session policy** | Вызов AssumeRole | Сужает разрешения для конкретной сессии |
| **ACL** | S3, VPC | Устаревший формат; ограниченный; только кросс-аккаунт |

### Структура политики (JSON)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowS3Read",
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:ListBucket"],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ],
      "Condition": {
        "StringEquals": { "aws:RequestedRegion": "us-east-1" }
      }
    }
  ]
}
```

> **📌 Tip:** Всегда используйте `"Version": "2012-10-17"` — эта версия включает переменные политики, например `${aws:username}`. Дата 2008 года — распространённая ловушка на экзамене.

## Логика вычисления разрешений

AWS оценивает политики в следующем порядке:

1. **Явный Deny** — любой совпадающий Deny в любой политике → **ЗАПРЕТ** (окончательный, без исключений)
2. **SCP** (если используется Organizations) — если в SCP нет Allow → ЗАПРЕТ
3. **Resource-based policy** — кросс-аккаунт: Allow нужен и в identity, и в resource policy
4. **Identity-based policy** — если нет Allow → ЗАПРЕТ (неявный)
5. **Permission boundary** — Allow и в identity policy, и в boundary → РАЗРЕШЕНИЕ
6. **Session policy** — тоже должна разрешать

```
По умолчанию = ЗАПРЕТ → проверяем явный DENY → проверяем ALLOW → выдаём результат
```

> **📌 Tip:** Главное правило: **явный Deny всегда побеждает**, даже если другая политика разрешает то же действие. «Deny перебивает Allow» — вопрос, который почти наверняка будет на экзамене.

## IAM Roles и Instance Profiles

### Почему роли лучше ключей доступа для EC2

| | Ключи доступа на EC2 | IAM Role (Instance Profile) |
|---|---|---|
| **Хранение** | Зашиты в код / переменные среды | Никогда не хранятся; берутся из IMDS |
| **Ротация** | Ручная; легко забыть | Автоматическая (STS ротирует) |
| **Риск утечки** | Высокий (утечка = постоянный доступ) | Низкий (токены истекают ≤ 1 ч) |
| **Аудит** | Сложнее (статичный key ID) | Полный CloudTrail по роли |

**Instance Profile** — контейнер, в котором хранится одна IAM Role и который прикрепляется к инстансу EC2. SDK/CLI автоматически получают credentials с `http://169.254.169.254/latest/meta-data/iam/security-credentials/<role-name>`.

```bash
# Получить credentials роли из метаданных EC2 (IMDSv2)
TOKEN=$(curl -sX PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
curl -sH "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/iam/security-credentials/MyRole
```

> **📌 Tip:** Экзамен часто спрашивает: «Приложение на EC2 не может обратиться к S3 — что лучше сделать?» Ответ: прикрепить IAM-роль с нужной политикой, а не создавать IAM-пользователя и вставлять ключи в код.

## Границы разрешений (Permission Boundaries)

Граница разрешений задаёт **максимум** того, что может разрешить identity-based политика. Сама по себе она ничего не разрешает — только ограничивает.

```
Эффективные разрешения = identity_policy ∩ permission_boundary
```

**Сценарий:** разрешить разработчикам создавать IAM-роли для Lambda, но запретить выходить за пределы определённого скоупа.

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:*", "cloudwatch:*", "logs:*"],
    "Resource": "*"
  }]
}
```

Если назначить это как границу — роль никогда не получит доступ к EC2 или IAM, даже если identity-политика это разрешает.

> **📌 Tip:** Границы разрешений применяются только к **IAM-пользователям и ролям** — не к группам и не к resource-based политикам. На уровне организации аналогичную роль играет SCP.

## Кросс-аккаунтный доступ через AssumeRole

Стандартная схема предоставления доступа между AWS-аккаунтами:

1. **Аккаунт B (ресурсный)** создаёт роль с trust policy, разрешающей аккаунту A её принять.
2. **Принципал аккаунта A** вызывает `sts:AssumeRole` → получает временные credentials.
3. Принципал использует credentials для действий в аккаунте B.

```json
// Trust policy роли в аккаунте B
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "AWS": "arn:aws:iam::111122223333:root" },
    "Action": "sts:AssumeRole",
    "Condition": {
      "StringEquals": { "sts:ExternalId": "unique-external-id" }
    }
  }]
}
```

**External ID** защищает от атаки «confused deputy» — третья сторона не сможет обманом заставить ваш аккаунт принять роль в её интересах.

## STS — Security Token Service

STS выдаёт **временные credentials** (Access Key ID + Secret Access Key + Session Token).

| API STS | Сценарий использования |
|---|---|
| `AssumeRole` | Кросс-аккаунтное или внутреннее принятие роли |
| `AssumeRoleWithSAML` | Федерация SAML 2.0 (корпоративный IdP) |
| `AssumeRoleWithWebIdentity` | OIDC-федерация (Cognito, Google, GitHub Actions) |
| `GetFederationToken` | Долгосрочная федеративная сессия для брокер-приложений |
| `GetSessionToken` | Принудительный MFA; короткие credentials для IAM-пользователя |

**Срок жизни токена:** от 15 мин до 12 ч (по умолчанию 1 ч для AssumeRole). Токены нельзя отозвать, но они истекают автоматически.

> **📌 Tip:** `AssumeRoleWithWebIdentity` используется в **GitHub Actions OIDC** и **EKS IRSA**. Если экзамен описывает контейнеры, которым нужен доступ к AWS, — ответ: роли + OIDC, а не ключи доступа.

## Лучшие практики IAM (чеклист для экзамена)

| Практика | Почему важна |
|---|---|
| Заблокировать root; включить MFA на root | Root нельзя ограничить через SCP или политики |
| Включить MFA для всех привилегированных пользователей | Компрометация пароля не равна компрометации аккаунта |
| Использовать роли вместо долгосрочных ключей | Ключи ротируются автоматически; нет риска хранения |
| Принцип наименьших привилегий | Уменьшить масштаб ущерба при компрометации |
| Permission boundaries для делегированного администрирования | Предотвратить эскалацию привилегий разработчиками |
| Использовать Access Analyzer | Выявляет ресурсы с внешним доступом |
| Регулярно ротировать ключи доступа | Рекомендуемый цикл — 90 дней |
| Не хранить credentials в коде | Использовать Secrets Manager, Parameter Store или роли |
| Управлять разрешениями через группы | Проще, чем per-user политики |
| Включить CloudTrail | Полный аудит всех API-вызовов |

## Типичные ловушки на экзамене

| Ловушка | Правильное понимание |
|---|---|
| «Добавить Allow, чтобы перебить Deny» | Невозможно — явный Deny всегда финален |
| «Root ограничивается SCP» | Неверно — root всегда исключён из SCP |
| «IAM — региональный сервис» | Неверно — IAM глобален |
| «Группы можно вкладывать» | Неверно — группы не могут содержать другие группы |
| «Ключи доступа на EC2 — best practice» | Неверно — всегда используйте IAM-роли |
| «Граница разрешений выдаёт разрешения» | Неверно — она только ограничивает; нужна ещё identity-политика |
| «Токены STS постоянны» | Неверно — они истекают (15 мин – 12 ч) |
| «Resource-based политика одна даёт кросс-аккаунтный доступ» | Частично верно для S3; для большинства сервисов нужна ещё identity-политика |

## Шпаргалка: дерево решений IAM

```
Нужен доступ к AWS для...
├── Человека (консоль/CLI)          → IAM User + MFA, или SSO через Identity Center
├── EC2 / Lambda / ECS task         → IAM Role (instance profile / execution role)
├── Внешней идентичности (корп. AD) → SAML 2.0 через IAM Identity Center
├── Пользователей моб./веб-приложения → Amazon Cognito Identity Pools → IAM Role
├── CI/CD (GitHub Actions)          → OIDC + AssumeRoleWithWebIdentity
└── Кросс-аккаунтная автоматизация  → IAM Role в целевом аккаунте + AssumeRole
```

> **📌 Tip:** Когда экзамен говорит «наименьшие привилегии» + «временные credentials» + «никаких долгосрочных ключей» — ответ всегда **IAM Role**. Запомните: роли — для машин; пользователи — для людей; группы — для организации пользователей.
