## Route 53

Route 53 — высокодоступный масштабируемый DNS-сервис AWS. Также поддерживает регистрацию доменов и проверки состояния.

---

## Типы DNS-записей

| Запись | Назначение | Пример |
|---|---|---|
| **A** | Имя хоста → IPv4 | `api.example.com → 1.2.3.4` |
| **AAAA** | Имя хоста → IPv6 | `api.example.com → 2001:db8::1` |
| **CNAME** | Имя хоста → другое имя хоста | `www.example.com → lb.example.com` |
| **MX** | Почтовый сервер (с приоритетом) | `example.com → 10 mail.example.com` |
| **NS** | Серверы имён для hosted zone | Делегированные серверы имён |
| **SOA** | Start of Authority — метаданные зоны | Управляется Route 53 автоматически |
| **TXT** | Произвольный текст — SPF, DKIM, верификация домена | `"v=spf1 include:...` |
| **CAA** | Авторизованные УЦ для домена | `example.com 0 issue "letsencrypt.org"` |
| **SRV** | Расположение сервиса (порт + хост) | Используется SIP, XMPP, Minecraft |
| **PTR** | Обратный DNS (IP → имя хоста) | Управляется отдельно, используется для email |

## Alias vs CNAME

Критическое различие — встречается почти в каждом экзамене.

| Параметр | Alias-запись | CNAME-запись |
|---|---|---|
| Работает на вершине зоны (корневой домен) | **Да** — `example.com` | **Нет** — только поддомены |
| Разрешается в | Имя хоста AWS-ресурса | Любое имя хоста |
| Стоимость | **Бесплатно** — без платы за запросы | Стандартная плата за DNS-запрос |
| Видна клиентам | **Нет** — прозрачно; клиент видит IP | Да — клиенты видят цель CNAME |
| Интеграция с health check | Поддерживается | Не напрямую |
| TTL | Задаётся Route 53 (не пользователем) | Настраивается пользователем |

**Поддерживаемые цели Alias:**
- ELB (ALB, NLB, CLB)
- Дистрибутивы CloudFront
- Endpoint статического сайта S3
- API Gateway
- Elastic Beanstalk
- VPC Interface Endpoints
- Global Accelerator
- **Другая запись Route 53 в той же hosted zone**

> **📌 Tip:** CNAME **нельзя** задать для корневого домена (`example.com`). Всегда используйте Alias для корневого домена, указывающего на AWS-ресурсы. Кроме того: Alias-записи бесплатны; разрешение CNAME добавляет стоимость запроса.

## Политики маршрутизации

| Политика | Health Check | Применение | Примечания |
|---|---|---|---|
| **Simple** | Опционально | Один ресурс | Несколько значений → клиент выбирает случайно |
| **Weighted** | Опционально | A/B-тестирование, blue/green | Вес 0 = нет трафика; все 0 = равное распределение |
| **Latency** | Опционально | Мультирегион, минимальная задержка | Основан на измерениях задержки AWS, а не географии |
| **Failover** | **Обязателен** | Активно-пассивный DR | Основной + резервный; резервный при недоступности основного |
| **Geolocation** | Опционально | Правовое соответствие, контентные ограничения | По стране или континенту; нужна запись для неопределённых |
| **Geoproximity** | Опционально | Сдвиг трафика через bias | Требует Traffic Flow; bias от -99 до +99 |
| **Multi-Value** | **Обязателен** | Балансировка на стороне клиента | До 8 здоровых записей; НЕ замена ELB |
| **IP-Based** | Опционально | Маршрутизация по ISP/CIDR | Маршрутизация по CIDR клиента |

### Weighted-маршрутизация

```
api.example.com → server-A  Вес: 70  (70% трафика)
api.example.com → server-B  Вес: 20  (20%)
api.example.com → server-C  Вес: 10  (10%)
```

Веса относительные — проценты рассчитываются пропорционально.

### Failover-маршрутизация

```
Основная:  api.example.com → ALB-in-us-east-1  (с health check)
Резервная: api.example.com → ALB-in-eu-west-1  (цель failover)
```

При сбое health check основного Route 53 направляет весь трафик на резервный.

### Geolocation vs Latency

| | Geolocation | Latency |
|---|---|---|
| Основание | **Географическое** расположение пользователя | Измеренная AWS **сетевая задержка** |
| Детерминированность | Да — пользователь в Германии → EU | Нет — пользователь в Германии может получить us-east при меньшей задержке |
| Применение | Правовые требования, язык, контентные ограничения | Лучший пользовательский опыт (скорость) |
| Запись по умолчанию | Нужна для несовпадающих локаций | Н/Д |

> **📌 Tip:** Geolocation ≠ Latency. Требование «пользователи в Германии должны использовать EU-серверы» → Geolocation. «Пользователи должны получать самый быстрый ответ» → Latency-based.

### Geoproximity

- Маршрутизирует трафик по географическому расположению **с поправкой bias**
- Положительный bias (+1 до +99): расширяет зону охвата региона
- Отрицательный bias (-1 до -99): сужает зону охвата региона
- **Требует Route 53 Traffic Flow** (визуальный редактор политик, дополнительная стоимость)

## Health Checks

### Типы

| Тип | Что мониторит |
|---|---|
| **Endpoint** | HTTP, HTTPS или TCP-проверка по IP или домену |
| **Calculated** | Объединяет несколько health checks (логика AND/OR) |
| **CloudWatch Alarm** | Мониторит состояние CloudWatch alarm |

### Детали проверки endpoint

- Проверки из нескольких глобальных точек AWS
- Порог: 3/5 проверяющих должны согласиться о состоянии (настраивается)
- HTTP/HTTPS: Route 53 ожидает 2xx или 3xx в течение 2 с; проверяет первые 5 120 байт ответа
- Поддерживает сопоставление строк в теле ответа

### Health Checks с приватными ресурсами

Проверяющие Route 53 находятся в публичном интернете — они **не могут** напрямую достичь приватных ресурсов VPC. Решение: создайте CloudWatch-метрику + alarm, мониторящие приватный ресурс, затем используйте **CloudWatch alarm health check** в Route 53.

## Приватные Hosted Zones

- Разрешает DNS-имена в пределах одного или нескольких VPC
- Требует `enableDnsSupport = true` на связанных VPC
- Та же структура, что и публичные hosted zones; записи не видны за пределами VPC
- Можно связать с VPC из других аккаунтов (через CLI/API)

```bash
# Связать hosted zone с VPC из другого аккаунта
aws route53 associate-vpc-with-hosted-zone \
  --hosted-zone-id Z1234567890ABC \
  --vpc VPCRegion=us-east-1,VPCId=vpc-abc12345
```

## Route 53 Resolver

Обеспечивает DNS-разрешение для гибридных облачных архитектур.

| Тип endpoint | Направление | Применение |
|---|---|---|
| **Входящий Resolver Endpoint** | Локальная сеть → AWS | Локальный DNS разрешает записи приватной hosted zone AWS |
| **Исходящий Resolver Endpoint** | AWS → Локальная сеть | EC2-инстансы разрешают доменные имена локальной сети |

**Правила Resolver** управляют пересылкой:
- **Правило пересылки:** пересылает запросы для конкретных доменов на локальный DNS
- **Системное правило:** Route 53 Resolver обрабатывает (по умолчанию для доменов AWS)
- **Автоопределённое правило:** AWS добавляет правила для приватных hosted zones и обратного DNS

```
EC2 в VPC запрашивает corp.internal.example.com
  → Исходящий Resolver Endpoint
    → Правило пересылки: corp.internal → 192.168.1.1 (локальный DNS)
      → Локальный DNS отвечает
```

## Traffic Flow

Визуальный редактор для создания сложных конфигураций маршрутизации с объединением нескольких политик.

- Поддерживает версионирование политик маршрутизации
- Можно комбинировать Geoproximity + Latency + Weighted в древовидной структуре
- Дополнительная стоимость за запись политики (~50 $/месяц)

## Сводка ловушек экзамена

| Ловушка | Правильный ответ |
|---|---|
| CNAME на корневом домене | **Запрещено** — использовать Alias |
| Alias-запись платная за запрос | **Alias бесплатен** — только CNAME имеет плату за запрос |
| Geolocation = Latency | Разные: Geolocation — по географии, Latency — по измерению сети |
| Multi-Value = Load Balancer | Multi-Value НЕ замена ELB — случайный выбор на стороне клиента из до 8 записей |
| Failover не требует health check | **Failover требует health check** на основной записи |
| Route 53 разрешает приватную зону без настроек VPC | Требует `enableDnsSupport = true` на VPC |
| Health checks достигают приватных IP | Проверяющие Route 53 публичные — используйте **CloudWatch alarm health check** для приватных ресурсов |
