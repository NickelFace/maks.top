## Direct Connect (DX)

Direct Connect обеспечивает **выделенное приватное сетевое подключение** между вашим on-premises ЦОД и AWS.

### Типы подключений

| Тип | Пропускная способность | Кто обеспечивает канал | Время ожидания |
|---|---|---|---|
| **Dedicated** | 1, 10 или 100 Гбит/с | Заказываете у AWS; физический кросс-коннект в DX-локации | Недели — месяцы |
| **Hosted** | 50 Мбит/с – 10 Гбит/с (менее 1G через партнёра) | Партнёр AWS обеспечивает подключение | Быстрее (дни — недели) |

> **📌 Совет:** DX **не шифруется** по умолчанию. Это приватное, но не зашифрованное соединение. Для шифрования поверх DX → запустите туннель **Site-to-Site VPN** поверх DX Private VIF.

### Виртуальные интерфейсы (VIF)

| Тип VIF | Подключается к | Требует BGP |
|---|---|---|
| **Private VIF** | Один VPC через VGW | Да |
| **Public VIF** | Публичные сервисы AWS (S3, DynamoDB, CloudFront endpoints) через публичные IP | Да |
| **Transit VIF** | Transit Gateway (несколько VPC, несколько регионов) | Да |

### DX Gateway

- Подключить **одно DX-соединение** к **нескольким VPC** в разных регионах (или одном регионе).
- DX Gateway НЕ позволяет обмен трафиком между VPC. Только для доступа on-premises → VPC.
- Один DX Gateway может быть связан с до 10 VGW или одним Transit Gateway.

> **📌 Совет:** DX Gateway НЕ маршрутизирует трафик между VPC. Для VPC-to-VPC через инфраструктуру DX → используйте **Transit VIF → Transit Gateway**.

## Direct Connect — Высокая доступность

### Паттерны HA (от слабейшего к сильнейшему)

| Паттерн | Устойчивость |
|---|---|
| Одно DX-подключение | Нет резервирования — единая точка отказа |
| Два DX-подключения к одной DX-локации | Защита от сбоя подключения, НО НЕ от сбоя локации |
| Два DX-подключения к **двум разным DX-локациям** | Рекомендуемый HA; выживает при сбое локации |
| DX как основной + **Site-to-Site VPN как резервный** | Экономичный HA; VPN — резервный путь через публичный интернет |

> **📌 Совет:** Два подключения к одной DX-локации — НЕ настоящий HA. Экзамен специально это проверяет — нужны **две разные DX-локации** для резервирования на уровне локации.

## DX + VPN Backup

Используйте метрики BGP для предпочтения DX над VPN:

- Путь DX: более низкий BGP local preference (предпочтительный путь)
- Путь VPN: более высокий BGP MED или более низкий local preference

При сбое DX BGP переконвергирует и трафик автоматически маршрутизируется через VPN.

## Site-to-Site VPN

Зашифрованный IPsec-туннель между on-premises и AWS VPC через публичный интернет.

### Компоненты

| Компонент | Расположение | Примечания |
|---|---|---|
| **VGW** (Virtual Private Gateway) | Сторона AWS | Прикреплён к VPC; поддерживает BGP и статическую маршрутизацию |
| **CGW** (Customer Gateway) | On-premises сторона | Представляет on-premises маршрутизатор/файрвол в AWS |
| **VPN connection** | Связывает VGW + CGW | Всегда создаёт **2 туннеля** для HA (разные AWS endpoints) |

### Лимиты

| Параметр | Лимит |
|---|---|
| Макс. пропускная способность на туннель | ~1.25 Гбит/с |
| Макс. туннелей на VGW | 10 |
| Маршрутизация | BGP (динамическая) или статическая |

> **📌 Совет:** Максимальная пропускная способность Site-to-Site VPN — **~1.25 Гбит/с на туннель**. Для большей пропускной способности создайте несколько VPN-подключений и распределите трафик (ECMP). С Transit Gateway ECMP поддерживается нативно.

## Client VPN

Управляемый OpenVPN-сервис для индивидуального пользовательского доступа к AWS-ресурсам.

| Функция | Детали |
|---|---|
| **Протокол** | OpenVPN (TCP/UDP) |
| **Аутентификация** | Active Directory, Cognito (SAML), на основе сертификатов или взаимная auth |
| **Subnet associations** | Прикрепление к одной или нескольким подсетям VPC; клиенты получают IP из CIDR-пула |
| **Split tunneling** | Только трафик, предназначенный для AWS, идёт через VPN (снижает стоимость полосы) |
| **Логирование** | CloudWatch Logs или S3 |

## Transit Gateway (TGW)

Transit Gateway — **региональный хаб-и-спица сетевой транзитный хаб**, соединяющий несколько VPC, VPN-подключений и Direct Connect.

### Типы подключений

| Тип подключения | Примечания |
|---|---|
| **VPC** | До 5000 VPC-подключений на TGW |
| **VPN** | Site-to-Site VPN-подключения |
| **DX** | Через **Transit VIF** (заменяет Private VIF для мульти-VPC DX-доступа) |
| **TGW peering** | Подключение TGW между регионами (межрегиональный) |
| **AWS RAM** | Шаринг TGW между аккаунтами в AWS Organizations |

### Таблицы маршрутизации

- Каждый TGW имеет одну или несколько таблиц маршрутизации.
- Каждое подключение ассоциируется с одной таблицей и может распространять маршруты в другие.
- По умолчанию: все подключения в одной таблице → полная связность.
- Сегментация: разные VPC в разных таблицах → изоляция.

### TGW — ключевые случаи использования

- **Централизованный egress**: один NAT Gateway для всех VPC (все VPC к TGW, маршрут по умолчанию → общий egress VPC с NAT GW).
- **Централизованная инспекция**: весь трафик через VPC-файрвол (inspection VPC с Network Firewall или сторонним устройством).
- **Multicast**: IP-мультикаст между VPC (уникально для TGW — VPC Peering не поддерживает мультикаст).

## TGW vs VPC Peering

| Параметр | TGW | VPC Peering |
|---|---|---|
| **Масштаб** | Хаб-и-спица; 5000 VPC-подключений | 1:1 пиринг; макс. 125 активных пирингов на VPC |
| **Стоимость** | Почасовая плата за TGW ($0.05/ч) + обработка данных ($0.02/ГБ) | Нет почасовой платы; только стоимость передачи данных |
| **Транзитная маршрутизация** | Да (A → TGW → B → TGW → C) | Нет (нетранзитная; A→B→C недопустимо) |
| **Кросс-аккаунт** | Да (через AWS RAM) | Да (принять запрос пиринга) |
| **Кросс-регион** | Да (TGW peering) | Да (межрегиональный пиринг) |
| **Лучше для** | Большой масштаб (много VPC), централизованные сервисы | Несколько VPC, экономически эффективно, простая топология |

> **📌 Совет:** VPC Peering **нетранзитный**. Если A пирингует с B, а B пирингует с C, A не может достичь C через B. TGW обеспечивает транзитную маршрутизацию.

## Ловушки экзамена

| Ловушка | Правильный ответ |
|---|---|
| "DX зашифрован" | **Ложь** — DX приватный, но НЕ зашифрованный; добавьте VPN для шифрования |
| "Два DX-подключения к одной локации = HA" | **Ложь** — нужны две **разные DX-локации** для HA на уровне локации |
| "DX Gateway маршрутизирует трафик между VPC" | **Ложь** — DX Gateway только для on-prem → VPC; VPC-to-VPC требует TGW |
| "Пропускная способность VPN неограниченна" | Ложь — макс. **~1.25 Гбит/с на туннель**; ECMP + TGW для большего |
| "TGW позволяет транзитную маршрутизацию; VPC Peering — нет" | **Правда** — классическое отличие на экзамене |
| "Private VIF для доступа к S3 через DX" | **Ложь** — нужен **Public VIF** для S3/DynamoDB (у них публичные endpoints); Private VIF только для ресурсов VPC |
| "TGW всегда дешевле VPC Peering" | **Ложь** — TGW имеет почасовую плату; peering может быть дешевле для 2–3 VPC |
| "Client VPN для site-to-site подключения" | **Ложь** — Client VPN для индивидуальных пользователей; Site-to-Site VPN для сети-к-сети |
