## Модель данных

DynamoDB — **полностью управляемая, бессерверная, key-value и документная NoSQL БД** с задержкой единиц миллисекунд при любой нагрузке.

| Понятие | Описание |
|---|---|
| **Table** | Верхнеуровневый контейнер; схема не навязывается (кроме первичного ключа) |
| **Item** | Одна строка; до **400 КБ** |
| **Attribute** | Поле внутри item; может быть любого типа (строка, число, бинарный, список, map, set, boolean, null) |
| **Simple PK** | Только partition key — должен быть уникален по всей таблице |
| **Composite PK** | Partition key + sort key — комбинация уникальна; items с одинаковым PK группируются вместе |

DynamoDB распределяет данные по хешу partition key. Высокая кардинальность partition key критична для равномерного распределения и предотвращения «горячих разделов».

## Режимы ёмкости

### Provisioned Mode
- Задаёте **RCU (Read Capacity Units)** и **WCU (Write Capacity Units)**
- Предсказуемые нагрузки; поддерживает **auto-scaling** по целевому проценту утилизации
- **Reserved Capacity**: коммит на 1 или 3 года → экономия до 77%
- Burst: DynamoDB допускает кратковременные всплески выше провижн, используя burst-кредиты (300 секунд накопленной ёмкости)

### On-Demand Mode
- Оплата за запрос; **планирование ёмкости не нужно**
- Мгновенно масштабируется до любого уровня трафика
- Примерно **в 2× дороже** Provisioned при высокой устойчивой нагрузке
- Лучше для: новых таблиц с неизвестным трафиком, пиковых нагрузок, dev/test

> **📌 Tip:** On-Demand может быть ~в 2 раза дороже Provisioned при стабильно высоком трафике. Используйте Provisioned + auto-scaling для предсказуемых нагрузок.

## Математика RCU / WCU

### Read Capacity Units (RCU)
| Тип чтения | Потребление |
|---|---|
| Strongly consistent read (≤ 4 КБ) | **1 RCU** |
| Eventually consistent read (≤ 4 КБ) | **0.5 RCU** |
| Transactional read (≤ 4 КБ) | **2 RCU** |

Элементы > 4 КБ: округлять вверх до кратного 4 КБ.

**Пример**: 10 strongly consistent чтений/сек по 6 КБ = 10 × ceil(6/4) = 10 × 2 = **20 RCU**

### Write Capacity Units (WCU)
| Тип записи | Потребление |
|---|---|
| Стандартная запись (≤ 1 КБ) | **1 WCU** |
| Transactional запись (≤ 1 КБ) | **2 WCU** |

Элементы > 1 КБ: округлять вверх до кратного 1 КБ.

**Пример**: 5 записей/сек по 4.5 КБ = 5 × ceil(4.5/1) = 5 × 5 = **25 WCU**

## Индексы

### Local Secondary Index (LSI)
- Тот же **partition key**, что у базовой таблицы; другой **sort key**
- Создаётся **только при создании таблицы** — добавить позже нельзя
- До 5 LSI на таблицу
- Разделяет ёмкость с базовой таблицей
- Поддерживает strongly consistent reads (в отличие от GSI)

### Global Secondary Index (GSI)
- **Любые атрибуты** в качестве partition key и sort key
- Можно создавать и удалять **в любое время**
- До 20 GSI на таблицу (лимит по умолчанию, можно увеличить)
- **Отдельная ёмкость RCU/WCU**
- Только **eventually consistent** чтения

| Свойство | LSI | GSI |
|---|---|---|
| Partition key | Как у таблицы | Любой атрибут |
| Sort key | Другой | Любой атрибут |
| Создаётся | Только при создании таблицы | В любое время |
| Ёмкость | Общая с таблицей | Отдельная |
| Consistent reads | Strongly + eventually | Только eventually |
| Максимум на таблицу | 5 | 20 |

> **📌 Tip:** LSI создаётся только при создании таблицы. GSI имеет отдельную ёмкость — провижните её отдельно, иначе throttling будет независим от основной таблицы.

## DynamoDB Streams

- Захватывает **изменения элементов** (INSERT, MODIFY, REMOVE) в почти реальном времени
- Retention: **24 часа** (не увеличивается)
- Каждая запись содержит: ключ, старый образ, новый образ или оба (настраивается)
- Интеграция с **Lambda** для событийной обработки (exactly-once, порядок внутри шарда)
- Применения: репликация, аудит, кросс-региональная агрегация, запуск рабочих процессов

Типы потока:
| Тип | Что фиксируется |
|---|---|
| `KEYS_ONLY` | Атрибуты PK изменённых элементов |
| `NEW_IMAGE` | Весь элемент после изменения |
| `OLD_IMAGE` | Весь элемент до изменения |
| `NEW_AND_OLD_IMAGES` | Оба образа |

## DAX (DynamoDB Accelerator)

DAX — **полностью управляемый in-memory кеш**, созданный специально для DynamoDB.

| Свойство | Детали |
|---|---|
| Задержка | Микросекунды (против единиц мс у DynamoDB) |
| Тип кеша | Write-through (записи идут в DAX и DynamoDB одновременно) |
| Strongly consistent reads | Не кешируются — идут напрямую в DynamoDB |
| Кластер | Несколько узлов: primary + read replicas |
| API | Drop-in совместимость — те же DynamoDB API-вызовы |
| Применение | Read-heavy нагрузки, повторяющиеся чтения одних данных |

**DAX vs ElastiCache для DynamoDB:**
- DAX: специфичен для DynamoDB, обрабатывает DynamoDB API, проще в настройке
- ElastiCache: универсальный; используйте при кастомной логике кеша, кешировании результатов запросов, или источниках данных не из DynamoDB

> **📌 Tip:** DAX НЕ кеширует strongly consistent reads — они всегда идут напрямую в DynamoDB. Если нужна строгая согласованность, DAX не даёт преимуществ.

## Global Tables

- **Мультирегиональная, active-active** репликация — все таблицы принимают запись
- Разрешение конфликтов: **last-write-wins** (по временной метке)
- Задержка репликации — меньше секунды
- Требует включения DynamoDB Streams
- Применение: глобальные приложения с низкой задержкой чтения и записи в нескольких регионах

---

## TTL (Time to Live)

- Автоматически **истекает и удаляет элементы** по атрибуту с временной меткой
- Без дополнительной оплаты; удалённые элементы убираются в течение 48 часов после истечения
- Истёкшие элементы не потребляют WCU
- Удаления TTL отображаются в DynamoDB Streams (для аудита/архивации)

```yaml
# Пример: элемент с атрибутом TTL
{
  "userId": "abc123",
  "sessionData": "...",
  "expiresAt": 1748736000   # Unix timestamp; DynamoDB удалит, когда текущее время > значения
}
```

## Транзакции

- **ACID-транзакции** через несколько элементов и несколько таблиц
- Две операции: `TransactGetItems` (чтения) и `TransactWriteItems` (записи)
- До **100 элементов** в одной транзакции
- Потребляет **2× RCU/WCU** (одна для подготовки, одна для фиксации)
- Применение: банковские переводы, размещение заказа (атомарно: списать из инвентаря + создать заказ)

## Резервные копии и восстановление

### On-Demand Backup
- Полный бэкап таблицы в определённый момент времени
- Хранится до ручного удаления; в DynamoDB (не напрямую в S3)
- Восстановление создаёт **новую таблицу**

### Continuous Backup (PITR)
- **Point-in-Time Recovery** до **35 дней** назад
- Включается на уровне таблицы; небольшая дополнительная стоимость
- Гранулярность: до секунды
- Восстанавливает в новую таблицу

## Ловушки на экзамене

| Ловушка | Правильный ответ |
|---|---|
| LSI можно добавить после создания таблицы | Ложь — только при создании |
| GSI разделяет ёмкость с таблицей | Ложь — у GSI своя RCU/WCU |
| DAX кеширует strongly consistent reads | Ложь — проходит насквозь в DynamoDB |
| On-Demand всегда дешевле | Ложь — в 2× дороже при высоком устойчивом трафике |
| Retention DynamoDB Streams > 24ч | Ложь — максимум 24ч |
| Разрешение конфликтов Global Tables | Last-write-wins |
| Стоимость транзакций | 2× RCU/WCU |
| Максимальный размер элемента | 400 КБ |
