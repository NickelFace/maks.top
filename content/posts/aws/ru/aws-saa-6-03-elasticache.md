## Redis vs Memcached — сравнение функций

ElastiCache предлагает два движка. Разница между ними — гарантированная тема на экзамене.

| Функция | Redis | Memcached |
|---|---|---|
| Структуры данных | String, Hash, List, Set, Sorted Set, Bitmap, HyperLogLog, Streams | Только строки / простой key-value |
| Персистентность | RDB-снимки + AOF (Append Only File) | Нет |
| Репликация | Да — primary + read replicas | Нет |
| Multi-AZ + auto-failover | Да | Нет |
| Pub/Sub | Да | Нет |
| Lua-скрипты | Да | Нет |
| Cluster mode (шардинг) | Да — до 500 узлов | Да — только горизонтальный шардинг |
| RBAC / AUTH | Да — Redis AUTH token, RBAC (6.x+) | SASL опционально |
| Backup/Restore | Да | Нет |
| Транзакции | Да (`MULTI/EXEC`) | Нет |
| Геопространственные команды | Да | Нет |

> **📌 Tip:** Если на экзамене упоминается: персистентность, HA, репликация, pub/sub, лидерборды, session store с HA — ответ **Redis**. Memcached — только когда нужно простое кеширование с многопоточным горизонтальным масштабированием и ничего более.

## Детали архитектуры Redis

### Cluster Mode Disabled (Группа репликации)
- Один primary-узел + до 5 read replicas
- Все данные на каждом узле (полная репликация датасета)
- Multi-AZ: реплики в разных AZ; auto-failover повышает реплику при сбое primary

### Cluster Mode Enabled (Шардинг)
- Данные разбиты по до **500 шардам**; каждый шард = 1 primary + до 5 реплик
- До **500 узлов всего**
- Масштабирование записи — запросы распределяются по шардам
- Нельзя выполнять команды, требующие нескольких ключей из разных шардов (используйте хеш-теги `{key}` для совмещения)

### Redis AUTH
- Пароль/токен обязателен для подключения (задаётся при создании кластера)
- Для AUTH обязательно включить шифрование в транзите (TLS)
- RBAC (Role-Based Access Control) доступен в Redis 6.x — детальные ACL на пользователя

---

## Детали архитектуры Memcached

- Только in-memory, без персистентности
- Многопоточная архитектура — лучшая утилизация CPU для простого кеширования
- Горизонтальное масштабирование: добавляйте узлы для увеличения общей ёмкости кеша
- Без репликации — при потере узла теряются закешированные данные
- Без HA — при сбое узла всплеск cache miss до перезаполнения

## Стратегии кеширования

### Lazy Loading (Cache-Aside)
```
1. Приложение запрашивает кеш
2. Cache HIT → вернуть данные
3. Cache MISS → запросить БД → записать результат в кеш → вернуть данные
```
- Кеширует только реально запрашиваемые данные
- Кеш может содержать устаревшие данные (до истечения TTL)
- Дополнительная задержка при первом запросе (3 обращения при cache miss)

### Write-Through
```
1. Приложение пишет в БД
2. Приложение (или библиотека кеша) одновременно пишет в кеш
```
- Кеш всегда актуален
- Накладные расходы на запись (каждая запись — в БД и кеш)
- Может кешировать данные, которые никогда не читаются (трата памяти)

### Write-Behind (Write-Back)
```
1. Приложение пишет только в кеш
2. Кеш асинхронно сбрасывает данные в БД
```
- Минимальная задержка записи
- Риск потери данных при сбое кеша до сброса
- Сложно реализовать корректно

### TTL (Time to Live)
- Применяйте TTL ко всем закешированным элементам во избежание исчерпания памяти
- Баланс между свежестью (короткий TTL) и эффективностью кеша (длинный TTL)
- Критично для Lazy Loading: сокращает окно устаревших данных

## Типовые сценарии применения

| Сценарий | Движок | Паттерн |
|---|---|---|
| Кеш запросов к БД | Redis или Memcached | Lazy Loading с TTL |
| Session store (общий для инстансов) | Redis | Write-Through или Lazy Loading |
| Лидерборды | Redis (Sorted Sets) | Нативная структура данных Redis |
| Счётчики real-time аналитики | Redis (INCR/INCRBY) | Нативная структура данных Redis |
| Pub/Sub-сообщения | Redis | Нативный Pub/Sub |
| Rate limiting | Redis | INCR + TTL на ключ |
| Замена sticky sessions | Redis | Централизованный session store |

> **📌 Tip:** Когда в задаче написано «пользователей разлогинивает при маршрутизации на другой EC2» (проблема session affinity) — ответ: ElastiCache для Redis как централизованный session store, а не sticky sessions (которые — единая точка отказа).

## Безопасность

| Уровень | Механизм |
|---|---|
| Сеть | Развёртывание в VPC; Security Group контролирует доступ |
| Шифрование в транзите | TLS (включается при создании кластера для Redis AUTH) |
| Шифрование в покое | Ключи под управлением KMS (только Redis) |
| Аутентификация | Redis AUTH token; RBAC для Redis 6.x+ |
| IAM | Используется для управления кластером (API); не для data-plane |

## ElastiCache vs DAX

| Фактор | ElastiCache | DAX |
|---|---|---|
| Целевая БД | Любая (приложение, БД) | Только DynamoDB |
| Модель доступа | Кеширование на уровне приложения (явные вызовы кеша) | Прозрачный прокси DynamoDB API |
| Strongly consistent reads | Возможны (не кешировать их) | Не кешируются — проходят насквозь |
| Применение | Многоисточниковое кеширование, session store | Прозрачное ускорение чтения DynamoDB |

## CloudFront как edge-кеш

- **CDN** с глобальными точками присутствия (PoP / edge locations)
- Кеширует статический и динамический контент на границе сети
- Снижает нагрузку на origin; уменьшает задержку для географически распределённых пользователей
- TTL настраивается per-behavior; заголовки `Cache-Control` учитываются
- Работает с: S3, ALB, EC2, Lambda@Edge, API Gateway в качестве origins

---

## Кеширование API Gateway

- Кеш **per-stage**; размер 0.5 ГБ – 237 ГБ; TTL 0–3600 секунд (по умолчанию 300с)
- Снижает вызовы к бэкенду; уменьшает стоимость при высоком трафике
- Ключ кеша: метод + путь + опциональные query-параметры/заголовки
- Сброс вручную или через заголовок `Cache-Control: max-age=0` (если разрешено)

> **📌 Tip:** Кеш API Gateway работает на уровне stage, а не метода. Включение на stage применяется ко всем методам, если явно не настроены per-method переопределения.

## Ловушки на экзамене

| Ловушка | Правильный ответ |
|---|---|
| Memcached для HA-кеширования | Неверно — нет репликации; используйте Redis |
| Memcached для session store с HA | Неверно — Redis; нет потери данных при failover |
| ElastiCache вместо sticky sessions | Верно — централизованный session store |
| Cluster mode нужен для масштабирования записи | Верно — шардинг распределяет записи |
| DAX vs ElastiCache для DynamoDB | DAX проще; ElastiCache для кастомной логики |
| Расположение кеша API Gateway | Per-stage; не глобально |
| Варианты персистентности Redis | RDB (снимок) + AOF (лог); у Memcached — нет |
