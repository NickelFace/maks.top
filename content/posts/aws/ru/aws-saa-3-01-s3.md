## Основы S3

Amazon S3 (Simple Storage Service) — объектное хранилище с надёжностью 99.999999999% (11 девяток). Объекты хранятся в **бакетах** и адресуются **ключами** (полный путь внутри бакета).

---

## Бакеты, объекты, ключи

| Концепция | Детали |
|---|---|
| Имя бакета | Глобально уникально для **всех** аккаунтов и регионов AWS |
| Регион бакета | Выбирается при создании — данные остаются в этом регионе (без явной репликации) |
| Ключ объекта | Полный путь: `folder/subfolder/file.txt` — настоящих папок нет, только префиксы |
| Размер объекта | 0 Б – 5 ТБ; один PUT ≤ 5 ГБ; для > 5 ГБ — Multipart Upload |
| Метаданные | Системные (content-type, ETag) + пользовательские пары ключ/значение |

S3 использует **глобальное пространство имён** — имена бакетов уникальны во всём мире — но **бакеты привязаны к региону**. Обращение к бакету всегда идёт в регион создания (если не настроены Transfer Acceleration или репликация).

> **📌 Tip:** На экзамене может встретиться задача с глобально уникальным идентификатором для S3 — это встроенное свойство. Но не путайте «глобальное пространство имён» с «глобальной репликацией» — репликацию нужно настраивать явно.

## Классы хранения

Выбор зависит от частоты доступа, допустимого времени извлечения и стоимости.

| Класс | Надёжность | Доступность | Мин. срок | Извлечение | Применение |
|---|---|---|---|---|---|
| **S3 Standard** | 11 девяток | 99.99% | Нет | Немедленно | Частый доступ |
| **Intelligent-Tiering** | 11 девяток | 99.9% | Нет | Немедленно (частый/редкий); минуты (архив) | Непредсказуемый доступ |
| **Standard-IA** | 11 девяток | 99.9% | 30 дней | Немедленно | Редкий, но быстрый |
| **One Zone-IA** | 11 девяток* | 99.5% | 30 дней | Немедленно | Редкий, некритичный, одна AZ |
| **Glacier Instant Retrieval** | 11 девяток | 99.9% | 90 дней | Миллисекунды | Архивы, доступ раз в квартал |
| **Glacier Flexible Retrieval** | 11 девяток | 99.99% | 90 дней | Минуты – 12 ч | Архивы, доступ 1–2 раза в год |
| **Glacier Deep Archive** | 11 девяток | 99.99% | 180 дней | 12–48 ч | Долгосрочное хранение (7–10 лет) |

*Данные One Zone-IA хранятся в одной AZ — при уничтожении AZ данные **теряются**.

**Уровни Intelligent-Tiering (автоматические):**
- Frequent Access (по умолчанию)
- Infrequent Access (после 30 дней без обращений)
- Archive Instant Access (после 90 дней)
- Archive Access (90–730 дней, opt-in)
- Deep Archive Access (180–730 дней, opt-in)

> **📌 Tip:** Ловушка на экзамене — «Glacier» без уточнения. Glacier **Instant** = миллисекунды. Glacier **Flexible** = минуты–часы. Glacier **Deep Archive** = 12–48 ч. Сопоставляйте требование к времени извлечения с правильным классом.

## Версионирование

Настраивается на уровне бакета. Бакет может находиться в одном из трёх состояний:

| Состояние | Поведение |
|---|---|
| **Unversioned** (по умолчанию) | Нет ID версий; объекты перезаписываются |
| **Enabled** | Каждый PUT создаёт новую версию; DELETE добавляет маркер удаления |
| **Suspended** | Новые версии не создаются; существующие версии сохраняются |

После включения версионирование **нельзя отключить** — только приостановить.

**MFA Delete** — дополнительная защита:
- Требует токен MFA для постоянного удаления версии или смены состояния версионирования
- Включается/выключается только **root-аккаунтом** через CLI
- Версионирование должно быть включено

> **📌 Tip:** Версионирование + MFA Delete — ответ на вопрос «как защитить от случайного удаления».

## Иерархия управления доступом

S3 использует многоуровневую модель. Действие разрешено, только если **нет явного DENY** и есть хотя бы **одно явное ALLOW**.

```
IAM Policy (identity-based)
  └─ Bucket Policy (resource-based, JSON)
       └─ ACL (устаревший, для объекта или бакета)
            └─ Block Public Access (на уровне аккаунта/бакета)
```

### Политики бакета

JSON-политики, прикреплённые к бакету. Поддерживают условия (IP-диапазоны, VPC endpoint, MFA, TLS). Пример — разрешить только HTTPS:

```json
{
  "Effect": "Deny",
  "Principal": "*",
  "Action": "s3:*",
  "Resource": ["arn:aws:s3:::my-bucket/*"],
  "Condition": { "Bool": { "aws:SecureTransport": "false" } }
}
```

### ACL (устаревший механизм)

XML-списки контроля доступа на уровне объекта и бакета. Предопределённые гранты: `READ`, `WRITE`, `FULL_CONTROL`. ACL **отключены по умолчанию** через S3 Object Ownership.

### Block Public Access

Четыре настройки (на уровне аккаунта или бакета):

| Настройка | Блокирует |
|---|---|
| BlockPublicAcls | Новые публичные ACL |
| IgnorePublicAcls | Все публичные ACL (даже существующие) |
| BlockPublicPolicy | Новые публичные политики бакета |
| RestrictPublicBuckets | Все публичные политики бакета |

**Настройки на уровне аккаунта переопределяют настройки бакета** — даже если политика бакета разрешает публичный доступ, аккаунтная блокировка приоритетнее.

### S3 Object Ownership

С апреля 2023 г. ACL **отключены по умолчанию** на новых бакетах. По умолчанию установлен `BucketOwnerEnforced` — владелец бакета автоматически владеет всеми объектами, независимо от того, кто их загрузил.

> **📌 Tip:** Сценарий с кросс-аккаунтной загрузкой: при отключённых ACL (по умолчанию) владелец бакета всегда получает объекты в собственность. Если загрузчику нужен контроль над объектами — придётся включить ACL, но это устаревший путь.

## Шифрование

S3 шифрует все новые объекты по умолчанию (с января 2023 г.). Варианты:

| Тип | Управление ключом | Вызовы KMS API | Сигнал на экзамене |
|---|---|---|---|
| **SSE-S3** | AWS (AES-256) | Нет | По умолчанию; «без накладных расходов» |
| **SSE-KMS** | AWS KMS (CMK) | Да — логируется в CloudTrail | Аудит, ротация ключей, compliance |
| **DSSE-KMS** | AWS KMS (двойной слой) | Да | Регуляторные требования к 2 слоям |
| **SSE-C** | Клиент (ключ передаётся в запросе) | Нет | Клиент сохраняет контроль над ключом; только HTTPS |
| **Client-side** | Клиент (шифрование до загрузки) | Нет | Шифрование до выхода из сети клиента |

**Ловушка SSE-KMS:** каждый PUT и GET вызывает KMS `GenerateDataKey` / `Decrypt`. При высокой нагрузке можно **достичь лимита KMS API** (по умолчанию 5 500–30 000 запросов/с в зависимости от региона). Используйте S3 Bucket Keys, чтобы сократить вызовы KMS примерно на 99%.

> **📌 Tip:** SSE-C требует HTTPS — AWS отклонит запросы SSE-C по HTTP. SSE-KMS даёт аудиторский след в CloudTrail — используйте, когда compliance требует доказательств доступа.

## Статический хостинг

Включается в свойствах бакета. S3 отдаёт HTML/CSS/JS напрямую по HTTP.

- Имя бакета **должно совпадать с доменом** при использовании кастомного домена через Route 53
- Формат endpoint: `<bucket>.s3-website-<region>.amazonaws.com`
- Объекты должны быть публично доступны для чтения (Block Public Access должен быть отключён)
- HTTPS на кастомном домене требует **CloudFront** (endpoint статического сайта S3 не поддерживает HTTPS)

### CORS

Настраивается на бакете, который содержит ресурсы, а не на стороне запросчика.

```json
[{
  "AllowedOrigins": ["https://example.com"],
  "AllowedMethods": ["GET"],
  "AllowedHeaders": ["*"],
  "MaxAgeSeconds": 3000
}]
```

> **📌 Tip:** Если на экзамене написано «JS в браузере не может загрузить ресурсы из другого бакета S3» — ответ: **настроить CORS на бакете с ресурсами**.

## Сводка ловушек экзамена

| Ловушка | Правильный ответ |
|---|---|
| One Zone-IA теряет данные | При уничтожении AZ данные пропадут — не использовать для критичных данных |
| «Использовать Glacier» для архива | Уточнять: Glacier Instant / Flexible / Deep Archive — «Glacier» это семейство |
| SSE-KMS при высокой нагрузке | Возможен throttling KMS — включите S3 Bucket Keys |
| HTTPS для статического сайта S3 | Нужен CloudFront — endpoint S3 только HTTP |
| Загрузка из другого аккаунта | При BucketOwnerEnforced владелец бакета получает объект автоматически |
| Включение MFA Delete | Только root-аккаунт через CLI — не IAM-пользователи, не консоль |
