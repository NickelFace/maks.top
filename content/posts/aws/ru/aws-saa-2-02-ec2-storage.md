## EBS — Elastic Block Store

EBS предоставляет **постоянное блочное хранилище** для EC2-инстансов. Тома привязаны к AZ — они существуют в одной AZ и должны находиться в той же AZ, что и прикрепляемый инстанс.

### Типы томов EBS

| Тип | Категория | Макс. IOPS | Макс. пропускная способность | Макс. размер | Сценарий |
|---|---|---|---|---|---|
| **gp3** | SSD General Purpose | 16 000 | 1 000 МБ/с | 16 ТиБ | Загрузка, общие нагрузки — **выбор по умолчанию** |
| **gp2** | SSD General Purpose | 16 000 | 250 МБ/с | 16 ТиБ | Устаревший; burst-based IOPS |
| **io2 Block Express** | SSD Provisioned IOPS | 256 000 | 4 000 МБ/с | 64 ТиБ | Mission-critical БД (Oracle, SAP HANA) |
| **io1** | SSD Provisioned IOPS | 64 000 | 1 000 МБ/с | 16 ТиБ | High-IOPS приложения со стабильной производительностью |
| **st1** | HDD Throughput | 500 | 500 МБ/с | 16 ТиБ | Big data, хранилища данных, обработка логов |
| **sc1** | HDD Cold | 250 | 250 МБ/с | 16 ТиБ | Редкие обращения; самый дешёвый HDD |

### gp3 vs gp2

| | **gp2** | **gp3** |
|---|---|---|
| Базовый IOPS | 3 IOPS/ГиБ (мин 100, макс 16 000) | **3 000 IOPS всегда (бесплатно)** |
| Burst | До 3 000 (используя кредитный бакет) | Нет концепции burst; всегда 3 000 |
| Пропускная способность | 128–250 МБ/с (зависит от размера) | **До 1 000 МБ/с (задаётся независимо)** |
| Стоимость | Выше | ~20% дешевле; IOPS и пропускная способность задаются отдельно |

> **📌 Tip:** gp3 почти всегда лучше gp2. Дешевле И даёт 3 000 IOPS независимо от размера тома. У gp2 IOPS зависят от размера — том 100 ГиБ даёт только 300 IOPS базово.

### Provisioned IOPS: io1 vs io2

| | **io1** | **io2** | **io2 Block Express** |
|---|---|---|---|
| Макс. IOPS | 64 000 | 64 000 | 256 000 |
| Долговечность | 99,8–99,9% | **99,999%** | **99,999%** |
| Соотношение IOPS:ГиБ | 50:1 | 500:1 | 1000:1 |
| Multi-Attach | Да | Да | Да |

> **📌 Tip:** Для новых деплойментов всегда предпочитайте **io2** вместо io1 — выше долговечность (99,999%), выше соотношение IOPS:ГиБ, та же цена.

## Возможности EBS

### Multi-Attach

Позволяет одному EBS-тому (только io1 или io2) быть одновременно подключённым к **до 16** EC2-инстансам в одной AZ.

**Требования:**
- Инстанс должен использовать Nitro System
- Требуется кластерно-осведомлённая файловая система (например, GFS2, OCFS2) — **НЕ ext4/XFS**
- Все инстансы должны быть в одной AZ

> **📌 Tip:** Multi-Attach предназначен для приложений, самостоятельно управляющих параллельными записями (например, Oracle RAC). Стандартные ФС ext4 повредят данные. Экзамен может предлагать Multi-Attach как ответ на «общее блочное хранилище» — следите за ограничением на AZ и требованием к ФС.

### Шифрование EBS

- **В покое**: AES-256, используя KMS-ключ (дефолтный или CMK)
- **В транзите**: шифруется между EC2 и EBS (настройка не требуется)
- Шифрование существующего незашифрованного тома: создать снапшот → скопировать снапшот с включённым шифрованием → создать новый том из зашифрованного снапшота
- Все снапшоты зашифрованного тома — зашифрованы
- Тома из зашифрованных снапшотов — зашифрованы

```bash
# Включить шифрование EBS по умолчанию для новых томов в регионе
aws ec2 enable-ebs-encryption-by-default --region us-east-1
```

### Снапшоты EBS

- Снапшоты **инкрементальны** и хранятся в S3 (управляется AWS)
- Снапшоты **региональны**, но можно скопировать в другой регион
- **Fast Snapshot Restore (FSR)**: предварительный прогрев снапшота для устранения задержки при первом использовании (доп. стоимость)
- **EBS Snapshot Archive**: дешёвый тир хранения снапшотов; восстановление занимает 24–72 ч

```bash
# Создать снапшот
aws ec2 create-snapshot \
  --volume-id vol-0abc123 \
  --description "Pre-deploy backup"

# Скопировать снапшот в другой регион
aws ec2 copy-snapshot \
  --source-region us-east-1 \
  --source-snapshot-id snap-0abc123 \
  --region eu-west-1 \
  --description "DR copy"
```

## Instance Store (Эфемерное хранилище)

Instance store предоставляет **временное блочное хранилище** на физических дисках, непосредственно подключённых к хосту.

| Свойство | Значение |
|---|---|
| **Постоянность** | **Не постоянный** — данные теряются при остановке, завершении или отказе оборудования |
| **Производительность** | Максимально возможные IOPS и пропускная способность (NVMe SSD на некоторых типах) |
| **Стоимость** | Включена в цену инстанса — доп. плата отсутствует |
| **Размер** | Фиксирован; определяется типом инстанса |
| **Шифрование** | Шифрование в покое на Nitro-инстансах (на уровне железа) |

**Типы инстансов с большим instance store:**
- `i3`, `i4i` — NVMe SSD, до 7,5 млн IOPS (i4i.32xlarge) — NoSQL, кэширование
- `d2`, `d3` — HDD, высокая последовательная пропускная способность — Hadoop, data lakes
- `h1` — HDD — MapReduce, HDFS

> **📌 Tip:** Instance store переживает **перезагрузку**, но НЕ остановку или завершение. Экзамен любит это различие. «Максимальный IOPS, эфемерный, теряется при остановке» → instance store. «Постоянный, можно сделать снапшот, переживает остановку» → EBS.

## EFS — Elastic File System

EFS предоставляет **управляемую NFS v4 файловую систему**, которую можно одновременно монтировать на нескольких EC2-инстансах в разных AZ.

### Ключевые свойства

| Свойство | Значение |
|---|---|
| **Протокол** | NFS v4.0 / v4.1 |
| **Поддерживаемые ОС** | **Только Linux** (POSIX-совместимая) |
| **Доступность** | Multi-AZ по умолчанию (Regional mode) |
| **Масштабирование** | Автоматическое; до петабайт |
| **Монтирование** | Mount target в каждой AZ; используются Security Groups |

### Тиры хранения

| Тир | Стоимость | Паттерн доступа |
|---|---|---|
| **Standard** | Выше | Часто используемые файлы |
| **Standard-IA** | ~92% дешевле | Редко используемые (перемещаются политикой жизненного цикла) |
| **One Zone** | Ниже Standard | Одна AZ; менее надёжный |
| **One Zone-IA** | Минимальная | Редкий доступ + одна AZ |

### Режимы производительности

| Режим | Лучший для | Компромисс |
|---|---|---|
| **General Purpose** (по умолчанию) | Чувствительные к задержке приложения (веб, CMS) | Низкая задержка; макс. 35 000 IOPS |
| **Max I/O** | Высокопараллельные нагрузки (big data, медиа) | Выше задержка; неограниченный IOPS |

### Режимы пропускной способности

| Режим | Как определяется пропускная способность |
|---|---|
| **Bursting** | Зависит от объёма хранимых данных (1 МБ/с на ГиБ + burst-кредиты) |
| **Provisioned** | Задаётся независимо от объёма |
| **Elastic** | Автоматически масштабируется; рекомендован для переменных нагрузок |

> **📌 Tip:** EFS — **только для Linux**. Если в сценарии экзамена Windows-инстансам нужно общее файловое хранилище — EFS не подходит. Используйте FSx for Windows File Server. EFS = Linux NFS; FSx for Windows = Windows SMB.

## Семейство FSx

Amazon FSx — полностью управляемые сторонние файловые системы.

| FSx | Протокол | ОС | Лучший сценарий |
|---|---|---|---|
| **FSx for Windows File Server** | SMB 3.0 | Windows (+ Linux через SMB) | AD-интегрированные Windows file shares, DFS |
| **FSx for Lustre** | Lustre (POSIX) | Linux | HPC, обучение ML, рендеринг, интеграция с S3 |
| **FSx for NetApp ONTAP** | NFS, SMB, iSCSI | Linux + Windows | Enterprise storage; мультипротокол; тиринг данных |
| **FSx for OpenZFS** | NFS | Linux | Низкая задержка; возможности ZFS (снапшоты, клоны) |

### FSx for Lustre — интеграция с S3

```
S3 Bucket ←→ FSx for Lustre (ленивая загрузка из S3 при первом обращении)
                       ↓ (после обработки)
             запись результатов обратно в S3
```

- Ленивая загрузка: файлы загружаются из S3 только при первом обращении
- Data repository tasks: массовый импорт/экспорт между Lustre и S3
- Сценарий: обучение ML, где данные хранятся в S3, но нужен POSIX-доступ

### FSx for Windows — интеграция с Active Directory

- Поддерживает Microsoft AD (самоуправляемый или AWS Managed AD)
- Поддерживает DFS (Distributed File System) namespace и репликацию
- Поддерживает Windows ACL, shadow copies (VSS), Microsoft DFS-R

> **📌 Tip:** «Windows-приложению нужно общее хранилище с интеграцией Active Directory» → **FSx for Windows File Server**, не EFS. EFS не поддерживает Windows NTFS/ACL или SMB.

## Дерево решений по хранилищам

```
Нужно блочное хранилище для ОДНОГО инстанса:
  ├── Загрузка / общие задачи  → gp3
  ├── High-IOPS БД            → io2 Block Express
  ├── Логи / последовательное чтение → st1 (HDD)
  ├── Архив / холодные данные  → sc1 (HDD)
  └── Максимальный IOPS, временное → Instance Store

Нужно общее файловое хранилище:
  ├── Linux, multi-AZ, автомасштаб → EFS
  ├── Windows + интеграция AD   → FSx for Windows
  ├── HPC / ML / интеграция S3  → FSx for Lustre
  ├── Мультипротокол enterprise  → FSx for NetApp ONTAP
  └── Возможности ZFS + NFS      → FSx for OpenZFS
```

## Типичные ловушки на экзамене

| Ловушка | Правильное понимание |
|---|---|
| «EFS работает с Windows» | Неверно — EFS только Linux (NFS) |
| «st1/sc1 можно использовать как boot-том» | Неверно — только SSD-типы (gp2, gp3, io1, io2) поддерживают загрузку |
| «Базовый IOPS у gp3 — 100» | Неверно — у gp3 базовый IOPS всегда **3 000**, независимо от размера |
| «Instance store переживает остановку» | Неверно — данные теряются при stop/terminate; выживает только при reboot |
| «Снапшоты EBS — полные копии» | Неверно — только первый снапшот полный; последующие инкрементальны |
| «Multi-Attach работает с любым типом тома» | Неверно — только io1 и io2 поддерживают Multi-Attach |
| «Multi-Attach работает с любой ФС» | Неверно — требуется кластерно-осведомлённая ФС (не ext4/XFS) |
| «EBS том пропадает при завершении инстанса» | По умолчанию root-том удаляется; дополнительные тома — сохраняются |
| «FSx for Lustre постоянно хранит данные» | По умолчанию нет — для надёжности объединяйте с S3; Lustre — быстрый scratch |
