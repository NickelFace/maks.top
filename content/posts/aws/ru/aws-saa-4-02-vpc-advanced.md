## VPC Продвинутый уровень — Подключение

В этой статье разобрано, как VPC подключаются друг к другу и к локальной инфраструктуре — ключевая тема экзамена SAA-C03.

---

## VPC Endpoints

Доступ к сервисам AWS в частной сети без выхода трафика за пределы AWS (без IGW, NAT и публичных IP).

### Gateway Endpoints

| Параметр | Значение |
|---|---|
| Поддерживаемые сервисы | **Только S3 и DynamoDB** |
| Стоимость | **Бесплатно** |
| Реализация | Запись в **таблице маршрутизации** |
| Область действия | Региональный — без привязки к AZ |
| DNS | Без изменений DNS; таблица маршрутизации направляет трафик |

```
Таблица маршрутизации:
  10.0.0.0/16  → local
  pl-68a54001   → vpce-xxxxx   ← префикс-лист S3 → Gateway Endpoint
  0.0.0.0/0     → igw-xxxxx
```

### Interface Endpoints (AWS PrivateLink)

| Параметр | Значение |
|---|---|
| Поддерживаемые сервисы | Большинство сервисов AWS (100+) + пользовательские сервисы |
| Стоимость | **Почасово + за ГБ** |
| Реализация | **ENI** в вашей подсети (приватный IP) |
| DNS | Приватный DNS переопределяет endpoint сервиса |
| Security Groups | Можно прикрепить SG к ENI |

```bash
# После создания interface endpoint для S3, приватный DNS разрешает:
s3.us-east-1.amazonaws.com → 10.0.1.25 (ENI в вашей подсети)
```

> **📌 Tip:** Gateway Endpoint = бесплатно, таблица маршрутизации, только S3/DynamoDB. Interface Endpoint = платный, ENI, DNS, поддерживает почти всё. «Приватное подключение к SQS из Lambda в VPC» → Interface Endpoint.

## VPC Peering

Соединяет два VPC через приватные IP-адреса (одинаковый или разные регионы, одинаковый или разные аккаунты).

| Параметр | Значение |
|---|---|
| Маршрутизация | Оба VPC должны добавить маршруты друг к другу |
| Перекрывающиеся CIDR | **Не разрешены** — peering не создаётся при перекрытии CIDR |
| Транзитивность | **НЕ транзитивен** — A↔B, B↔C не означает A↔C |
| Межрегиональный | Поддерживается |
| Межаккаунтный | Поддерживается |
| Пропускная способность | Без ограничений; тарифицируется как межAZ или межрегиональный трафик |

**Диаграмма нетранзитивности:**
```
VPC-A ←→ VPC-B ←→ VPC-C
           ↑
    A не может достичь C без прямого peering A↔C
```

> **📌 Tip:** При большом числе VPC peering превращается в задачу полной связности (N×(N-1)/2 соединений). Для ≥ 3 VPC, которым нужно взаимодействовать, лучший ответ — **Transit Gateway**.

## Transit Gateway (TGW)

Модель hub-and-spoke — региональный роутер, соединяющий VPC, VPN и Direct Connect.

| Возможность | Детали |
|---|---|
| Вложения | VPC, Site-to-Site VPN, Direct Connect Gateway, пиринг TGW |
| Маршрутизация | Каждое вложение связано с **таблицей маршрутизации TGW** |
| Транзитивность | **Поддерживает транзитивную маршрутизацию** — ключевое отличие от VPC Peering |
| Межрегиональный | Через пиринг TGW |
| Multicast | Поддерживается (уникально среди сетевых сервисов AWS) |
| Пропускная способность | До 50 Гбит/с на вложение |
| Стоимость | За вложение/час + за ГБ обработанного трафика |

**Пример изоляции через таблицы маршрутизации:**

```
Shared-RT:  знает VPC-A, VPC-B, VPC-C (полная связность)
Prod-RT:    знает только Prod-VPC и Corp-VPN (изолирован от Dev)
Dev-RT:     знает только Dev-VPC
```

> **📌 Tip:** TGW — ответ, когда нужно соединить множество VPC или нужна транзитивная маршрутизация. VPC Peering — для простого прямого соединения двух VPC. TGW — для масштабируемой hub-and-spoke архитектуры.

## Site-to-Site VPN

Зашифрованный IPSec-туннель между локальной сетью и VPC AWS.

| Компонент | Роль |
|---|---|
| **Virtual Private Gateway (VGW)** | VPN-endpoint на стороне AWS, прикреплён к VPC |
| **Customer Gateway (CGW)** | VPN-устройство на стороне клиента (физическое или программное) |
| **VPN Connection** | Два IPSec-туннеля (HA — оба должны быть настроены на CGW) |

**Варианты маршрутизации:**
- **Статическая:** маршруты задаются вручную с обеих сторон
- **BGP (динамическая):** предпочтительнее — поддерживает propagation маршрутов

| Параметр | Virtual Private Gateway | Transit Gateway |
|---|---|---|
| Поддерживаемые VPC | Один VPC на VGW | Много VPC через TGW |
| BGP | Поддерживается | Поддерживается |
| ECMP | Не поддерживается | **Поддерживается** — агрегация полосы по туннелям |

**Пропускная способность VPN:** один туннель = до 1,25 Гбит/с. С TGW + ECMP — агрегация нескольких туннелей.

> **📌 Tip:** VPN настраивается быстро (минуты–часы), шифруется по умолчанию, но использует публичный интернет — задержка и полоса нестабильны. Direct Connect — выделенный канал, но монтируется недели.

## Direct Connect (DX)

Выделенное приватное сетевое соединение от локальной инфраструктуры до AWS.

### Типы подключений

| Тип | Скорость | Провайдер |
|---|---|---|
| **Dedicated** | 1, 10, 100 Гбит/с | Партнёры AWS Direct Connect |
| **Hosted** | 50 Мбит/с – 10 Гбит/с (доступен sub-1G) | Партнёры AWS |

**Время подготовки:** недели–месяцы на физический монтаж.

### Виртуальные интерфейсы (VIF)

| Тип VIF | Подключается к | Применение |
|---|---|---|
| **Private VIF** | VPC (через VGW или TGW) | Доступ к приватным ресурсам |
| **Public VIF** | Публичные endpoint AWS | S3, DynamoDB, публичные API |
| **Transit VIF** | Transit Gateway | Множество VPC через один DX |

### Шифрование

**DX по умолчанию НЕ шифруется.** Варианты добавления шифрования:
- Site-to-Site VPN поверх DX (IPSec через private VIF)
- MACsec (шифрование на физическом уровне, поддерживается на dedicated 10/100G)

> **📌 Tip:** «Нужна приватная стабильная полоса до AWS» → Direct Connect. «Нужна зашифрованная приватная связь» → VPN поверх Direct Connect или MACsec.

### Direct Connect Gateway

Подключает **один DX к нескольким VPC** в разных регионах через единый шлюз.

```
Локальная сеть ──DX──→ Direct Connect Gateway ──→ VGW-us-east-1 (VPC-A)
                                                ──→ VGW-eu-west-1 (VPC-B)
                                                ──→ VGW-ap-southeast-1 (VPC-C)
```

- Нельзя использовать для маршрутизации трафика между VPC через DX Gateway
- До 10 VGW на один DX Gateway

## AWS PrivateLink

Публикует сервис в вашем VPC для других VPC или аккаунтов **без Peering, IGW и NAT**.

| Компонент | Роль |
|---|---|
| **Провайдер сервиса** | NLB перед сервисом; создаёт VPC Endpoint Service |
| **Потребитель сервиса** | Создаёт Interface Endpoint, указывающий на сервис |

```
Consumer VPC ──Interface Endpoint (ENI)──→ NLB в Provider VPC
```

**Сценарии использования:**
- Публикация SaaS-сервиса между аккаунтами/VPC
- Сервисы AWS через Interface Endpoints (SQS, SNS и др.) внутри используют PrivateLink
- Безопасные интеграции с маркетплейсами — без peering и изменений таблиц маршрутизации

> **📌 Tip:** PrivateLink — ответ, когда нужно открыть доступ к одному сервису множеству потребителей без доступа ко всему VPC. Peering открывает доступ ко всему CIDR; PrivateLink — только к одному endpoint.

## Руководство по выбору подключения

| Сценарий | Решение |
|---|---|
| Приватный доступ к S3 с EC2 (бесплатно) | **Gateway Endpoint** |
| Приватный доступ к SQS из Lambda в VPC | **Interface Endpoint** |
| Соединить 2 VPC | **VPC Peering** |
| Соединить 10 VPC + on-prem VPN | **Transit Gateway** |
| Локальная сеть → AWS, < 1,25 Гбит/с, быстрый запуск | **Site-to-Site VPN** |
| Локальная сеть → AWS, выделенная полоса, низкая задержка | **Direct Connect** |
| Один DX → несколько региональных VPC | **DX Gateway** |
| Публикация сервиса между аккаунтами без полного доступа к VPC | **PrivateLink** |

## Сводка ловушек экзамена

| Ловушка | Правильный ответ |
|---|---|
| VPC Peering транзитивен | Peering **НЕ транзитивен** — нужно прямое соединение для каждой пары |
| DX шифруется | **DX НЕ шифруется** по умолчанию — добавьте VPN или MACsec |
| TGW не поддерживает транзитивную маршрутизацию | TGW **поддерживает** транзитивную маршрутизацию — главное преимущество перед peering |
| Gateway Endpoints стоят денег | **Бесплатны** — только Interface Endpoints платные |
| VPN поверх DX то же самое, что VPN через интернет | VPN поверх DX = приватный + зашифрованный; VPN через интернет = зашифрованный, но нестабильный |
| PrivateLink то же самое, что Peering | PrivateLink открывает один сервис; Peering открывает весь CIDR VPC |
