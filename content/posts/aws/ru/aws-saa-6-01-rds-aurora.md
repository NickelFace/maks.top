## Движки RDS

Amazon RDS — **управляемый сервис реляционных БД**: AWS берёт на себя патчинг, резервные копии и железо. Вы выбираете только движок и размер.

| Движок | Особенности |
|---|---|
| MySQL | Самый распространённый; совместим с Aurora MySQL |
| PostgreSQL | Богатый набор функций; совместим с Aurora PostgreSQL |
| MariaDB | Форк MySQL; community edition |
| Oracle | BYOL или License-Included |
| SQL Server | Express/Web/Standard/Enterprise; поддержка Windows-аутентификации |
| IBM Db2 | Добавлен в 2023; только BYOL |

> **📌 Tip:** Oracle и SQL Server не могут использовать Aurora. Если на экзамене написано «перенести Oracle на Aurora» — это неправильный ответ.

---

## Multi-AZ (высокая доступность)

Multi-AZ создаёт **синхронную резервную реплику** в другой зоне доступности. Это функция HA, а не производительности.

| Свойство | Значение |
|---|---|
| Репликация | Синхронная (нулевая потеря данных) |
| Резервная реплика доступна для чтения? | Нет — только для переключения |
| Триггер failover | Сбой AZ, сбой инстанса, перезагрузка с failover |
| Время failover | ~60–120 секунд |
| Эндпоинт | DNS-адрес не меняется — приложения переподключаются автоматически |
| Масштабирование чтения | Не поддерживается через standby |

> **📌 Tip:** Multi-AZ ≠ масштабирование чтения. Для чтения используйте Read Replicas. Экзамен специально путает.

---

## Read Replicas (масштабирование чтения)

Read Replicas используют **асинхронную репликацию** — небольшая задержка возможна (eventual consistency).

| Свойство | Значение |
|---|---|
| Максимум реплик | 5 на источник (15 для Aurora) |
| Область | Та же регион, cross-region или cross-account |
| Чтение | Да — через отдельный эндпоинт реплики |
| Promote | Можно повысить до standalone БД (репликация прерывается) |
| Cross-region реплика | Создаёт собственные автоматические бэкапы; дополнительная стоимость |
| Multi-AZ источник | Можно создать Read Replica от Multi-AZ источника — репликация идёт со standby (без нагрузки на primary) |

**Read Replica сама может быть Multi-AZ** — частый ловушечный вопрос на экзамене.

## RDS Proxy

RDS Proxy расположен между приложением и БД и поддерживает **пул соединений**, сокращая нагрузку на БД.

| Функция | Детали |
|---|---|
| Протоколы | MySQL, PostgreSQL, MariaDB, SQL Server |
| Аутентификация | IAM + Secrets Manager (без паролей в коде) |
| Пул соединений | Снижает нагрузку при всплесках трафика |
| Failover | Сокращает время переключения ~на 66% |
| Lambda | Lambda создаёт много коротких соединений — Proxy предотвращает исчерпание лимита |
| VPC | RDS Proxy недоступен публично — только внутри VPC |

> **📌 Tip:** Lambda + RDS + исчерпание соединений → ответ всегда RDS Proxy.

## Резервные копии и Point-in-Time Recovery

### Автоматические резервные копии
- Retention: **0–35 дней** (0 отключает бэкапы)
- Транзакционные логи сохраняются каждые **5 минут** → PITR до любой точки в окне retention
- Хранятся в S3; удаляются автоматически после окончания retention
- Происходят в окне бэкапа (незначительное влияние на I/O на single-AZ)

### Ручные снимки (Manual Snapshots)
- Создаются в любое время, хранятся **до удаления вручную**
- Копируются cross-region (для DR) — зашифрованные снимки остаются зашифрованными
- Восстановление из снимка создаёт **новый инстанс БД**

> **📌 Tip:** Автоматические бэкапы удаляются вместе с БД (если не взять финальный снимок). Ручные снимки остаются навсегда.

## Хранилище

| Тип | Применение |
|---|---|
| gp2 | Общее назначение; всплесковые IOPS (3 IOPS/ГБ, burst до 3000) |
| gp3 | Новее; IOPS и хранилище оплачиваются раздельно; базовые 3000 IOPS |
| io1 / io2 | Provisioned IOPS; до 64 000 IOPS; критические нагрузки |

**Автомасштабирование хранилища**: включается на уровне инстанса. Расширяется автоматически при < 10% свободного места. Задаёте максимум.

---

## Шифрование

| Уровень | Механизм |
|---|---|
| В покое | AWS KMS (AES-256). Включается только при создании. |
| В транзите | SSL/TLS. Приложения должны использовать `require_ssl`. |
| Снимки | Шифруются если источник зашифрован |
| Копирование снимка в другой регион | Перешифровывается KMS-ключом целевого региона |

Зашифровать незашифрованную БД на месте нельзя. Путь: снимок → копирование с шифрованием → восстановление.

## Parameter Groups и Option Groups

- **Parameter Group**: параметры конфигурации движка (`max_connections`, `innodb_buffer_pool_size`). Изменение статического параметра требует перезагрузки.
- **Option Group**: дополнительные функции для конкретных движков (Oracle TDE, MSSQL TDE, MySQL memcached plugin).

## Архитектура Aurora

Aurora — переработанный AWS реляционный движок, совместимый с MySQL и PostgreSQL, но с принципиально другим слоем хранения.

| Свойство | Aurora | Стандартный RDS |
|---|---|---|
| Копии хранилища | 6 копий в 3 AZ (по 2 на AZ) | 2 копии (Multi-AZ) |
| Самовосстановление | Сканирует и восстанавливает блоки автоматически | Нет |
| Авторост | До **128 ТБ** с шагом 10 ГБ | Autoscaling до 64 ТБ |
| Хранилище | Общий кластерный том (не на инстанс) | EBS на инстанс |
| Failover | ~30 секунд | ~60–120 секунд |
| Read replicas | До 15 | До 5 |
| Стоимость | ~на 20% дороже RDS | Базовая |

> **📌 Tip:** Хранилище Aurora общее — все инстансы кластера читают из одного тома. Добавление read replica не копирует данные.

---

## Эндпоинты Aurora

| Эндпоинт | Маршрут | Применение |
|---|---|---|
| **Cluster (writer)** | Текущий writer-инстанс | Все операции записи |
| **Reader** | Одна из read replica (round-robin) | Масштабирование чтения |
| **Custom** | Подмножество инстансов | Аналитика на более мощных инстансах |
| **Instance** | Конкретный инстанс | Диагностика |

---

## Aurora Global Database

- Один primary-регион (чтение/запись) + до 5 вторичных регионов (только чтение)
- Задержка репликации: **< 1 секунды** (RPO)
- Failover (повышение вторичного): **< 1 минуты** (RTO)
- Применение: глобальные чтения с низкой задержкой, DR с быстрым переключением

---

## Aurora Serverless v2

- Нет фиксированного размера инстанса — масштабирование в единицах **ACU (Aurora Capacity Units)**
- Задаёте минимум и максимум ACU (например, 0.5–128 ACU)
- Масштабируется **за секунды** (v1 — за минуты с холодным стартом)
- Оплата за ACU-секунду — выгодно при непредсказуемой или пиковой нагрузке
- Полный набор функций Aurora: Multi-AZ, Global, read replicas

---

## Aurora Multi-Master

- Несколько writer-инстансов — все принимают записи одновременно
- Разрешение конфликтов: first-write-wins
- Редко встречается на экзамене; применяется для непрерывной доступности записи

> **📌 Tip:** Multi-Master ≠ Global Database. Multi-Master — несколько писателей в одном регионе. Global Database — репликация между регионами.

---

## Ловушки на экзамене

| Ловушка | Правильный ответ |
|---|---|
| Multi-AZ для масштабирования чтения | Неверно — используйте Read Replicas |
| Read Replica для HA failover | Неверно — используйте Multi-AZ |
| Lambda + RDS = исчерпание соединений | RDS Proxy |
| Хранилище Aurora на инстанс | Неверно — общий кластерный том |
| Зашифровать незашифрованный RDS | Снимок → копирование с шифрованием → восстановление |
| RPO < 1с cross-region | Aurora Global Database |
| Минимальное retention бэкапов | 1 день (0 полностью отключает бэкапы) |
| Read Replicas могут быть Multi-AZ | Верно — реплика сама может быть Multi-AZ |
