## Основы DR-стратегий

### RPO vs RTO

| Термин | Определение | Определяется |
|---|---|---|
| **RPO** (Recovery Point Objective) | Максимально допустимая **потеря данных** в единицах времени ("насколько старыми могут быть восстановленные данные?") | Частота резервного копирования, задержка репликации |
| **RTO** (Recovery Time Objective) | Максимально допустимое **время простоя** ("как долго до восстановления систем?") | Скорость процесса восстановления, уровень автоматизации |

Меньше RPO/RTO = более дорогая архитектура.

## Четыре DR-стратегии

От наименее затратных / высокий RPO+RTO до наиболее затратных / низкий RPO+RTO:

### 1. Backup & Restore

- Периодические снимки / резервные копии в S3 или другой регион.
- При катастрофе: развернуть новое окружение, восстановить из резервной копии.
- **RPO**: часы. **RTO**: часы.
- **Стоимость**: наименьшая — платите только за хранение резервных копий.

### 2. Pilot Light

- Минимальное "всегда включённое" ядро (например, репликация БД запущена, AMI подготовлены).
- Всё остальное выключено или не provisioned до катастрофы.
- При катастрофе: развернуть полный стек, перенаправить DNS в DR-регион.
- **RPO**: минуты — несколько часов. **RTO**: десятки минут.

### 3. Warm Standby

- Уменьшенная, но **полностью функциональная копия** production в DR-регионе.
- При катастрофе: масштабировать DR-окружение до полной production-мощности.
- **RPO**: секунды — минуты. **RTO**: минуты.

### 4. Multi-Site Active/Active

- Полная production-нагрузка одновременно в двух или более регионах.
- Трафик балансируется между регионами (Route 53 / Global Accelerator).
- При катастрофе: исключить отказавший регион из маршрутизации; остальные регионы принимают всю нагрузку.
- **RPO**: близко к нулю. **RTO**: близко к нулю.
- **Стоимость**: наибольшая — полная инфраструктура в каждом регионе.

### Сравнительная таблица DR

| Стратегия | RPO | RTO | Стоимость | Ключевая характеристика |
|---|---|---|---|---|
| Backup & Restore | Часы | Часы | $ | Только стоимость хранения |
| Pilot Light | Минуты–часы | Десятки минут | $$ | Основные сервисы всегда включены |
| Warm Standby | Секунды–минуты | Минуты | $$$ | Полный стек, уменьшенный |
| Multi-Site Active/Active | ~0 | ~0 | $$$$ | Полный стек в каждом регионе |

> **📌 Совет:** "Warm standby" ≠ "hot standby" (active/active). Warm standby всё ещё требует шага масштабирования. Active/active имеет нулевой RTO, потому что полная мощность уже существует.

## AWS Backup

Централизованный, полностью управляемый сервис резервного копирования для AWS-сервисов.

| Функция | Детали |
|---|---|
| **Поддерживаемые сервисы** | EC2, EBS, RDS, Aurora, DynamoDB, EFS, FSx, S3, Storage Gateway |
| **Backup plans** | Задают расписание (ежедневно/еженедельно), хранение, жизненный цикл (переход в холодное хранилище) |
| **Кросс-аккаунт / кросс-регион** | Копировать резервные копии в другой аккаунт или регион для DR |
| **Vault Lock (WORM)** | Запись-один-раз-чтение-много; предотвращает удаление даже администраторами; для соответствия требованиям |
| **On-demand backups** | Немедленное резервное копирование вне расписания плана |

> **📌 Совет:** AWS Backup Vault Lock — ответ, когда экзамен спрашивает про **неизменяемые резервные копии для compliance**, которые нельзя удалить.

## Elastic Disaster Recovery (DRS)

Заменяет более старый CloudEndure Disaster Recovery.

- **Непрерывная репликация на уровне блоков** с on-premises или AWS в staging-область в целевом регионе.
- Экономичный staging: реплицированные данные на дешёвых EBS-томах; запущенных инстансов нет.
- **Failover**: запуск recovery инстансов в течение минут; RTO/RPO в **минутах**.
- **Failback**: репликация изменений обратно на источник после катастрофы.

> **📌 Совет:** DRS = быстрый (в масштабе минут) RTO/RPO для серверов. Отличается от DMS (который мигрирует базы данных) и MGN (lift-and-shift миграция). Все три используют непрерывную репликацию на уровне блоков, но для разных целей.

## Семейство Snow — офлайн-передача данных

Используйте Snow когда: данных > 10 ТБ, пропускная способность ограничена/дорога, air-gapped среды.

| Устройство | Полезное хранилище | Вычисления на edge | Случай использования |
|---|---|---|---|
| **Snowcone** | 8 ТБ HDD / 14 ТБ SSD | Да (2 vCPU, 4 ГБ RAM) | Удалённые/суровые условия, малые передачи, IoT edge |
| **Snowball Edge Storage** | 80 ТБ | Ограниченные | Крупномасштабная миграция данных |
| **Snowball Edge Compute** | 42 ТБ | Да (52 vCPU, опциональный GPU) | ML на edge, видеоанализ |
| **Snowmobile** | До 100 ПБ | Нет | Миграция петабайтного масштаба (грузовик) |

### Порядок операций

1. Запросить устройство через консоль AWS.
2. AWS доставляет устройство.
3. Скопировать данные на устройство (шифруется с AWS KMS).
4. Отправить устройство обратно в AWS.
5. AWS загружает данные в S3.

> **📌 Совет:** Snowball для **офлайн** передачи. Для **онлайн** передачи используйте DataSync. Точка безубыточности — примерно 10 ТБ при медленном канале или когда передача займёт больше недели.

## AWS DataSync — онлайн-передача данных

Онлайн-сервис передачи данных на основе агентов.

| Функция | Детали |
|---|---|
| **Источники** | On-premises NFS, SMB, HDFS, Lustre, S3-совместимое хранилище |
| **Назначения** | Amazon S3, EFS, FSx for Windows, FSx for Lustre, FSx for OpenZFS |
| **Агент** | Лёгкая VM, развёртываемая on-premises; подключается к DataSync service endpoint |
| **Планирование** | Однократно или периодически (ежечасно/ежедневно/еженедельно) |
| **Шифрование** | TLS при передаче; AWS KMS в покое |
| **Ограничение пропускной способности** | Настройка максимальной полосы во избежание насыщения канала |
| **Целостность данных** | Контрольные суммы проверяются от начала до конца |

> **📌 Совет:** DataSync НЕ инструмент миграции для запущенных приложений — он передаёт файловые данные. Для миграции живых баз данных → DMS.

## AWS Storage Gateway

Гибридный сервис хранения, соединяющий on-premises приложения с AWS-хранилищем.

### Типы шлюзов

| Тип | Протокол | AWS бэкенд | Случай использования |
|---|---|---|---|
| **S3 File Gateway** | NFS, SMB | S3 (Standard, IA, Glacier) | Файловые шары → облако; локальный кэш с малой задержкой |
| **Volume Gateway (Cached)** | iSCSI | S3 + снимки EBS | On-prem тома в AWS; локальное кэширование |
| **Volume Gateway (Stored)** | iSCSI | Снимки EBS | Полные данные on-prem; асинхронный бэкап в S3 |
| **Tape Gateway** | VTL (iSCSI) | S3 + Glacier | Замена физической ленточной библиотеки; виртуальные ленты |

> **📌 Совет:** Storage Gateway расширяет on-premises хранилище до AWS — приложения продолжают использовать привычные протоколы (NFS/SMB/iSCSI). Им не нужно знать, что данные в S3/Glacier.

## DMS — Database Migration Service

DMS мигрирует базы данных в AWS с минимальным простоем.

### Типы миграции

| Тип | Пример | SCT нужен? |
|---|---|---|
| **Гомогенная** | MySQL → RDS MySQL, Oracle → Oracle | Нет |
| **Гетерогенная** | Oracle → Aurora PostgreSQL, SQL Server → MySQL | Да — сначала используйте **Schema Conversion Tool (SCT)** |

### Функции DMS

- **Непрерывная репликация данных (CDC)**: захват текущих изменений из источника во время миграции.
- **Full load + CDC**: перенос существующих данных с последующим применением изменений → минимальное окно перехода.
- **Комбинации источник/назначение**: on-prem, на базе EC2, RDS, Aurora, Redshift, DynamoDB, S3, MongoDB и другие.
- **Replication instance**: инстанс EC2, выполняющий задачи репликации (выбирайте размер по объёму данных).

> **📌 Совет:** Для гетерогенных миграций (разные типы движков) НЕОБХОДИМО запустить **Schema Conversion Tool (SCT)** перед DMS для конвертации схемы и хранимых процедур.

## Application Migration Service (MGN)

Lift-and-shift миграция серверов (не баз данных).

- Установить лёгкий **агент репликации** на исходные серверы.
- Непрерывная репликация на уровне блоков в экономичную staging-область в AWS.
- При переключении: запустить полностью реплицированные EC2 инстансы.
- Минимальный простой; окно переключения — минуты.
- Заменяет более старый Server Migration Service (SMS).

## 7 стратегий миграции (7 Rs)

| Стратегия | Описание | Пример |
|---|---|---|
| **Retire** (Вывод из эксплуатации) | Списание — больше не нужно | Устаревшее приложение, которым никто не пользуется |
| **Retain** (Сохранение) | Оставить on-premises пока | Требования соответствия |
| **Relocate** (Перемещение) | Перенос в AWS без изменений (уровень контейнеров) | Перенос VMware на VMware Cloud on AWS |
| **Rehost** (Повторный хостинг) | Lift-and-shift на EC2 | Перенос VM на EC2 с помощью MGN |
| **Replatform** (Смена платформы) | Lift-tinker-and-shift | MySQL на EC2 → RDS MySQL (преимущества managed сервиса) |
| **Refactor / Re-architect** (Рефакторинг) | Перепроектирование с cloud-native сервисами | Монолит → Lambda + SQS + DynamoDB |
| **Repurchase** (Перепокупка) | Переход на SaaS | CRM → Salesforce |

> **📌 Совет:** Rehost = без изменений кода; Replatform = незначительные оптимизации, базовая архитектура не меняется; Refactor = значительное перепроектирование для облачных преимуществ.

## Ловушки экзамена

| Ловушка | Правильный ответ |
|---|---|
| "Snowball для онлайн-передачи данных" | **Ложь** — Snowball офлайн; используйте **DataSync** для онлайн |
| "DMS для гетерогенной миграции без SCT" | **Ложь** — гетерогенная требует **Schema Conversion Tool** сначала |
| "Warm standby = нулевой RTO" | **Ложь** — warm standby ещё требует шага масштабирования; **active/active = нулевой RTO** |
| "Multi-site active/active — самый дешёвый DR" | **Ложь** — самый дорогой; дешевейший — backup & restore |
| "DRS заменяет DMS для миграции баз данных" | **Ложь** — DRS для DR серверов (блочная репликация); DMS для миграции баз данных |
| "Storage Gateway полностью заменяет on-prem хранилище" | **Ложь** — он **расширяет** его; on-prem приложения по-прежнему используют локальные протоколы |
| "AWS Backup Vault Lock может удалить администратор" | **Ложь** — Vault Lock WORM предотвращает удаление даже привилегированными пользователями |
| "DataSync для миграции живых баз данных приложений" | **Ложь** — DataSync перемещает файловые данные; используйте **DMS** для баз данных |
