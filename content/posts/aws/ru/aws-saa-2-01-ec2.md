## Семейства инстансов EC2

Инстансы группируются по **основному оптимизируемому ресурсу**. Формат имени: `<семейство><поколение>.<размер>`, например `m7g.xlarge`.

| Семейство | Буква | Оптимизировано для | Типичные сценарии |
|---|---|---|---|
| **General Purpose** | M, T | Баланс CPU/RAM/сеть | Веб-серверы, серверы приложений, dev/test |
| **Compute Optimized** | C | Высокое соотношение CPU/RAM | Пакетная обработка, HPC, игры, видеокодирование |
| **Memory Optimized** | R, X, z | Большой объём RAM | In-memory DB (Redis, SAP HANA), аналитика реального времени |
| **Storage Optimized** | I, D, H | Высокий IOPS или последовательная пропускная способность | NoSQL DB, хранилища данных, Hadoop |
| **Accelerated Computing** | P, G, Inf, Trn | GPU / ML-ускорители | Обучение ML, инференс, 3D-рендеринг, HPC |
| **HPC Optimized** | Hpc | Сверхнизкая задержка сети (EFA) | Тесносвязанные HPC, симуляции |

### T-серия: burstable-инстансы

T-инстансы накапливают **CPU-кредиты** в простое и тратят их при нагрузке.

| Режим | Поведение |
|---|---|
| `standard` | Нельзя превысить накопленные кредиты |
| `unlimited` | Можно превысить кредиты; избыточный CPU оплачивается |

> **📌 Tip:** T-инстансы в режиме `unlimited` могут приносить неожиданные расходы при длительной высокой нагрузке. Экзамен может спрашивать о неожиданных счетах — виновник режим unlimited.

## Варианты покупки

Правильный выбор варианта покупки — один из наиболее частых вопросов экзамена SAA-C03.

| Вариант | Оплата | Скидка vs OD | Лучший сценарий |
|---|---|---|---|
| **On-Demand** | По секунде/часу; без обязательств | 0% (база) | Непредсказуемые нагрузки; dev/test; краткосрочно |
| **Reserved (Standard)** | 1 или 3 года; all/partial/no upfront | До 72% | Стабильные, предсказуемые нагрузки |
| **Reserved (Convertible)** | 1 или 3 года | До 66% | Стабильные нагрузки; возможно изменение типа инстанса |
| **Savings Plans (Compute)** | Обязательство $/ч; 1 или 3 года | До 66% | Гибкий: любое семейство, регион, ОС, tenancy |
| **Savings Plans (EC2 Instance)** | Обязательство $/ч; конкретное семейство+регион | До 72% | Менее гибкий; привязан к семейству+региону |
| **Spot** | Ставка на свободные мощности | До 90% | Отказоустойчивые, stateless, гибкие нагрузки |
| **Dedicated Host** | За хост; on-demand или reserved | — | BYOL по сокетам/ядрам; комплайнс |
| **Dedicated Instance** | За инстанс | ~10% доплата | Изоляция workload; нет совместного использования |
| **Capacity Reservation** | Оплата по OD-тарифу; гарантированная ёмкость | 0% | Гарантия наличия в конкретной AZ |

### Сравнение стоимости: 1 год, m5.xlarge, us-east-1 (приблизительно)

| Вариант | В месяц |
|---|---|
| On-Demand | ~$140 |
| Reserved Standard (no upfront) | ~$72 |
| Compute Savings Plan | ~$79 |
| Spot (переменный) | ~$14–$42 |

### Spot: подробнее

- **Прерывание:** AWS может забрать Spot с предупреждением за 2 минуты
- **Spot Block:** (устарел) резервировал инстанс на 1–6 ч без прерывания
- **Spot Fleet:** mix из типов инстансов; поддерживает целевую ёмкость; может комбинировать Spot + On-Demand
- **EC2 Auto Scaling + Spot:** используйте несколько типов инстансов и AZ; стратегия `capacity-optimized`

```bash
# Запрос Spot-инстанса с обработкой прерываний
aws ec2 request-spot-instances \
  --spot-price "0.05" \
  --instance-count 1 \
  --type "one-time" \
  --launch-specification file://spec.json
```

> **📌 Tip:** Dedicated Host vs Dedicated Instance: **Dedicated Host** даёт видимость физических сокетов/ядер — нужно для **BYOL** (Windows Server, SQL Server, Oracle). Dedicated Instance только гарантирует отсутствие совместного железа, без контроля над размещением.

## Amazon Machine Images (AMI)

AMI — шаблон, содержащий ОС, сервер приложений и программы для запуска EC2-инстансов.

### Типы AMI

| Тип | Источник | Примечание |
|---|---|---|
| **От AWS** | Amazon (Amazon Linux 2023, Windows, Ubuntu) | Поддерживается AWS; бесплатно |
| **AWS Marketplace** | Сторонние вендоры | Могут включать дополнительную лицензионную плату |
| **Community** | Публичные вклады сообщества | Использовать осторожно; не проверены AWS |
| **Custom (private)** | Вы создаёте из EC2-инстанса или снапшота | Принадлежит вашему аккаунту; можно расшарить |

### Область применения и копирование AMI

- AMI привязаны к **региону** — для использования в другом регионе нужно **скопировать** AMI
- Копирование зашифрованной AMI в другой регион: копия шифруется KMS-ключом целевого региона
- AMI можно расшарить с конкретными AWS-аккаунтами или сделать публичными

```bash
# Скопировать AMI в другой регион
aws ec2 copy-image \
  --source-region us-east-1 \
  --source-image-id ami-0abcd1234 \
  --region eu-west-1 \
  --name "MyApp-v2-eu"
```

### AMI и EBS-снапшоты

- EBS-backed AMI состоит из одного или нескольких EBS-снапшотов
- Снапшоты хранятся в S3 (управляется AWS; напрямую как S3-объекты недоступны)
- Зашифрованный инстанс → зашифрованная AMI; новые инстансы наследуют шифрование

> **📌 Tip:** AMI — региональный объект. Если вопрос экзамена описывает DR-сценарий с копированием инстансов в другой регион — сначала скопируйте AMI, затем запускайте из копии. Запустить AMI в регионе, где её нет, невозможно.

## User Data и Instance Metadata

### User Data (cloud-init)

Скрипт, выполняемый **один раз при первом запуске** от имени root. Используется для бутстрапа инстанса.

```bash
#!/bin/bash
yum update -y
amazon-linux-extras install nginx1 -y
systemctl start nginx
systemctl enable nginx
echo "<h1>Hello from $(hostname)</h1>" > /usr/share/nginx/html/index.html
```

- Запускается только при первом старте (если не настроено иначе)
- Доступен через IMDS: `http://169.254.169.254/latest/user-data`
- Максимальный размер: 16 КБ (base64: ~21 КБ)

### Instance Metadata Service (IMDS)

Предоставляет информацию об инстансе коду, работающему на нём.

| Версия | Способ запроса | Безопасность |
|---|---|---|
| **IMDSv1** | Простой GET-запрос; без аутентификации | Уязвим к SSRF-атакам |
| **IMDSv2** | Требует session token (сначала PUT) | Защищён от SSRF; рекомендуется |

```bash
# IMDSv2 — шаг 1: получить токен
TOKEN=$(curl -sX PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

# IMDSv2 — шаг 2: использовать токен
curl -sH "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/instance-id

# Часто используемые пути метаданных
# /latest/meta-data/instance-id
# /latest/meta-data/instance-type
# /latest/meta-data/placement/availability-zone
# /latest/meta-data/iam/security-credentials/<role-name>
# /latest/dynamic/instance-identity/document
```

> **📌 Tip:** IMDSv2 — экзаменационный ответ на любой вопрос о защите IMDS. Типичный сценарий: веб-приложение на EC2 уязвимо к SSRF — злоумышленник крадёт IAM credentials через IMDSv1. Решение: принудительно включить IMDSv2 (требует заголовок с токеном).

## Группы размещения (Placement Groups)

Группы размещения управляют тем, как инстансы распределяются по физическому оборудованию.

| Тип | Расположение | Лимит на AZ | Сценарий | Компромисс |
|---|---|---|---|---|
| **Cluster** | Один стойка, одна AZ | Без жёсткого лимита | HPC, низкая задержка 10+ Gbps | Единая точка отказа (стойка) |
| **Spread** | Разные стойки, разные AZ | **7 на AZ** | Критические инстансы (ZooKeeper, Kafka) | Жёсткий лимит 7 на AZ |
| **Partition** | Группы инстансов на отдельных партициях (стойках) | 7 партиций на AZ | Hadoop, HDFS, HBase, Cassandra | Инстансы делят риск партиции |

```bash
# Создать cluster placement group
aws ec2 create-placement-group \
  --group-name my-hpc-cluster \
  --strategy cluster
```

> **📌 Tip:** Spread placement group → максимум 7 инстансов на AZ. Это жёсткий лимит. Если в сценарии экзамена 8+ инстансов должны быть на разном железе — Spread в одной AZ не справится; нужно использовать несколько AZ.

## Жизненный цикл инстанса: Stop, Hibernate, Terminate

| Действие | RAM | EBS root | Instance store | Биллинг |
|---|---|---|---|---|
| **Stop** | Очищается | Сохраняется | Потеря | Нет оплаты за compute; EBS оплачивается |
| **Hibernate** | Сохраняется на EBS | Сохраняется | Потеря | Нет оплаты за compute; EBS оплачивается |
| **Terminate** | Очищается | Удаляется (по умолчанию) | Потеря | Оплата прекращается |
| **Reboot** | Сохраняется | Сохраняется | Сохраняется | Непрерывная оплата |

**Требования для Hibernate:**
- Инстанс должен быть EBS-backed
- RAM ≤ 150 ГБ
- Root EBS-том должен быть зашифрован и достаточно большим для дампа RAM
- Не поддерживается на bare metal-инстансах
- Максимальная продолжительность hibernate: 60 дней

> **📌 Tip:** Hibernate часто встречается в вопросах «сохранить состояние приложения между Stop/Start без повторной инициализации». Ключевое требование: **зашифрованный root EBS-том**. Если экзамен спрашивает о сохранении данных в памяти после остановки — ответ hibernate.

## Типичные ловушки на экзамене

| Ловушка | Правильное понимание |
|---|---|
| «Spot-инстансы всегда доступны» | Неверно — могут быть прерваны с уведомлением за 2 минуты |
| «Dedicated Instance = BYOL» | Неверно — нужен Dedicated HOST для видимости сокетов/ядер |
| «Reserved Instance привязан к одной AZ» | Standard RI может быть AZ-specific (с резервированием ёмкости) или региональным (гибкий) |
| «AMI — глобальный объект» | Неверно — AMI привязаны к региону; нужно копировать |
| «User Data запускается при каждом старте» | Неверно — по умолчанию только при первом запуске |
| «IMDSv1 нормально использовать» | Неверно — IMDSv2 безопасный вариант; IMDSv1 уязвим к SSRF |
| «Cluster placement group охватывает несколько AZ» | Неверно — только одна AZ |
| «Spread group не имеет лимитов» | Неверно — максимум 7 инстансов на AZ |
| «T-инстансы никогда не дают дополнительных расходов» | Неверно — режим unlimited может начислять плату за burst CPU |
